💡
Bytecode is platform-independent; JVM is platform-dependent. The same .class file runs on any OS that has a JVM. Each JVM is compiled specifically for its target OS/CPU. This is what "Write Once, Run Anywhere" means.

Steps ②③④ together form the Linking phase.

🔷 Phase 1 — Loading

ClassLoader reads the binary data from a .class file and creates a java.lang.Class object in the Method Area (Metaspace).

  • Finds the .class file using fully qualified class name (e.g., com.example.MyClass)
  • Reads the binary bytecode from filesystem, JAR, network, or anywhere
  • Creates a Class<?> object in Method Area representing the loaded class
  • Only loaded once per ClassLoader — subsequent requests return the same Class object
JavaClass loading observation
// When does class loading happen?
// 1. new MyClass() — instantiation
// 2. MyClass.staticMethod() — static method call
// 3. MyClass.STATIC_FIELD  — static field access (except compile-time constants)
// 4. Class.forName("MyClass") — explicit loading
// 5. Loading a subclass triggers loading of parent class

// Observe class loading with JVM flag: -verbose:class
// java -verbose:class Main → prints every class as it's loaded

// Get ClassLoader of a class
System.out.println(String.class.getClassLoader());        // null (Bootstrap)
System.out.println(MyClass.class.getClassLoader());      // AppClassLoader
System.out.println(MyClass.class.getClassLoader()
                         .getParent());                   // ExtClassLoader / PlatformCL
System.out.println(MyClass.class.getClassLoader()
                         .getParent().getParent());        // null (Bootstrap)

// Class.forName — loads AND initializes (runs static blocks)
Class<?> c1 = Class.forName("com.example.MyClass");

// ClassLoader.loadClass — loads but does NOT initialize
ClassLoader cl = ClassLoader.getSystemClassLoader();
Class<?> c2 = cl.loadClass("com.example.MyClass"); // no <clinit> run

🔶 Phase 2 — Linking (3 sub-phases)

2a. Verification

JVM checks that the bytecode is structurally correct and safe. Ensures the JVM won't crash or violate security by executing it.

  • File format check: Magic number 0xCAFEBABE, valid version
  • Bytecode validity: Valid opcodes, correct operand types, stack depth consistency
  • Class/method references: Referenced classes and methods actually exist
  • Access rules: No accessing private members of other classes

2b. Preparation

JVM allocates memory for static variables and sets them to default values (0, false, null) — NOT the values from the source code.

JavaPreparation phase — default values
class MyClass {
    static int     count   = 100;   // Preparation: count = 0 (not 100!)
    static boolean active  = true;  // Preparation: active = false
    static String  name    = "Java"; // Preparation: name = null
    static final int MAX  = 50;    // compile-time constant: MAX = 50 directly!
}
// Actual values (100, true, "Java") are assigned during Initialization phase
// static final primitive constants assigned during Preparation (not Initialization)

2c. Resolution

Replaces symbolic references in the constant pool with direct references (memory addresses / method table offsets).

💡
In bytecode, method calls are stored as symbolic references like "java/io/PrintStream.println:(Ljava/lang/String;)V". Resolution converts this to an actual pointer to the method in memory. This can be done lazily (when first used) or eagerly depending on JVM implementation.

🔷 Phase 3 — Initialization

JVM executes the class initializer <clinit> method — assigns actual values to static variables and runs static initialization blocks, in top-to-bottom order.

JavaInitialization — static blocks and order
class InitDemo {
    static int x = 10;         // ← executed 1st (top-to-bottom)

    static {
        System.out.println("Static block 1, x = " + x); // x=10
        x = 20;
    }

    static int y = x * 2;      // y = 40 (x is 20 at this point)

    static {
        System.out.println("Static block 2, y = " + y); // y=40
    }

    public static void main(String[] args) {
        System.out.println("main, x=" + x + ", y=" + y);
    }
}
// Output:
// Static block 1, x = 10
// Static block 2, y = 40
// main, x=20, y=40

// Class initialization is thread-safe!
// JVM guarantees <clinit> is called ONCE and atomically
// If two threads try to load same class simultaneously, only one runs <clinit>

// Parent class is initialized BEFORE child class
class Parent {
    static { System.out.println("Parent init"); }
}
class Child extends Parent {
    static { System.out.println("Child init"); }
}
// new Child() → "Parent init" then "Child init"
⚠️
Tricky Interview Question: A class is initialized at most ONCE per ClassLoader. Accessing a static final compile-time constant (e.g., static final int X = 5) does NOT trigger class initialization — the value is inlined by the compiler. But static final String s = new String("hello") DOES trigger initialization (it's not a compile-time constant).

Section 4
Parent Delegation Model
How class loading requests propagate, why it exists, how to break it

🔄 How the Delegation Model Works

When any ClassLoader is asked to load a class, it follows a strict delegation chain before trying to load it itself.

Parent Delegation — "Load java.lang.String" Flow AppClassLoader.loadClass("java.lang.String") │ │ 1. Already loaded? Check cache → NO │ 2. Delegate to parent first! ↓ PlatformClassLoader.loadClass("java.lang.String") │ │ 1. Already loaded? Check cache → NO │ 2. Delegate to parent first! ↓ BootstrapClassLoader.loadClass("java.lang.String") │ │ 1. Already loaded? Check cache → YES (if loaded before) or │ look in java.base module → FOUND! ↓ returns Class<String> BootstrapCL returns Class → PlatformCL returns it → AppCL returns it ↓ java.lang.String loaded by Bootstrap (not App CL)

💻 loadClass() Algorithm — Source Code Logic

Javajava.lang.ClassLoader.loadClass() — simplified
// This is the ACTUAL logic inside ClassLoader.loadClass()
protected Class<?> loadClass(String name, boolean resolve)
        throws ClassNotFoundException {

    synchronized (getClassLoadingLock(name)) {

        // Step 1: Check if class already loaded (cache)
        Class<?> c = findLoadedClass(name);

        if (c == null) {
            try {
                // Step 2: Delegate to parent FIRST
                if (parent != null) {
                    c = parent.loadClass(name, false); // recurse up
                } else {
                    // Parent is Bootstrap (null) — ask Bootstrap
                    c = findBootstrapClassOrNull(name);
                }
            } catch (ClassNotFoundException e) {
                // Parent couldn't find it — that's OK
            }

            if (c == null) {
                // Step 3: Parent failed — try to find it OURSELVES
                c = findClass(name); // ← subclasses override THIS method
            }
        }

        if (resolve) {
            resolveClass(c); // link the class
        }
        return c;
    }
}
✅
Key insight: When creating a custom ClassLoader, you should override findClass(), not loadClass(). Overriding loadClass() breaks the delegation model. findClass() is called only after the parent chain has failed.

❓ Why Parent Delegation Model?

  • Security: Prevents malicious code from replacing core JDK classes. Even if you create a class called java.lang.String, the Bootstrap ClassLoader's version will always be loaded first.
  • Uniqueness: Ensures a class is loaded exactly once. Same class definition = same Class object across the application.
  • Consistency: java.lang.* classes behave the same everywhere — they're always loaded by Bootstrap.

💡 Breaking the Delegation Model (When & Why)

Some frameworks intentionally break parent delegation for valid reasons:

  • Tomcat / App Servers: Each web app has its own ClassLoader to isolate their classes. webapp-A's version of Hibernate ≠ webapp-B's version.
  • OSGi (Eclipse): Bundle ClassLoaders load their own dependencies, different bundles can use different versions of same library.
  • Hot Reload (dev tools): Throw away old ClassLoader, create new one to reload changed classes without restarting JVM.
  • Thread Context ClassLoader: SPI (Service Provider Interface) like JDBC uses Thread Context ClassLoader to break delegation — Bootstrap needs to load user's DB driver.
🎯 Interview Questions — ClassLoader
Q1What happens if you create a class named java.lang.String in your application?
Due to the Parent Delegation Model, when your code tries to load "java.lang.String", the request is delegated up to Bootstrap ClassLoader, which finds the real JDK String and returns it. Your custom String class will never be loaded. Additionally, the JVM has a security restriction: you cannot define classes in the java.* package at all — it throws SecurityException.
Q2Can the same class be loaded by two different ClassLoaders? Are they the same?
Yes! You can load the same .class file with two different ClassLoader instances. However, they are treated as completely different classes by the JVM. Instances of one cannot be cast to the other — you'll get ClassCastException. Class.equals() will return false. This is why ClassLoader isolation works in app servers.
Q3What is the difference between Class.forName() and ClassLoader.loadClass()?
Class.forName("X"): Loads AND initializes the class — runs static blocks, assigns static values.
ClassLoader.loadClass("X"): Loads the class but does NOT initialize it — static blocks don't run.
Example: JDBC driver registration uses Class.forName("com.mysql.Driver") specifically to trigger the static block that registers the driver with DriverManager.

🗂️ 5 Memory Areas — Overview

📦 Method Area
Class metadata, bytecode, static variables, constant pool, method/field info. Java 8+: Replaced PermGen → Metaspace (native memory).
Shared — all threads
🌿 Heap
All object instances and arrays. GC operates here. Divided: Young Gen (Eden+Survivor) + Old Gen + (String Pool Java 7+).
Shared — all threads
📚 JVM Stack
One per thread. Holds Stack Frames. Each frame = local variables + operand stack + frame data. StackOverflowError if too deep.
Per-thread
🔢 PC Register
Program Counter — holds address of currently executing bytecode instruction. Undefined (native) when running native methods.
Per-thread
⚡ Native Method Stack
Like JVM Stack but for native (non-Java) methods called via JNI. C stack for native code execution.
Per-thread

🔷 Method Area (Metaspace in Java 8+)

Stores all class-level data. In Java 7 and earlier, it was called PermGen (Permanent Generation) — a fixed-size heap area. In Java 8+, it became Metaspace — stored in native memory (off-heap), can grow dynamically.

🔄 Object Lifecycle in Heap — Step by Step

Object Lifecycle — From Birth to Collection 1. new MyObject() → allocated in EDEN space Eden: [obj1][obj2][obj3][obj4]... ↓ Eden gets full → Minor GC triggered 2. MINOR GC (Young Gen GC) - Mark live objects in Eden + S0 (from-space) - Copy live objects to S1 (to-space) - Increment age counter on each survived object - Clear Eden and S0 - Swap S0 ↔ S1 (to becomes from for next GC) ↓ Repeat: survivors move between S0 and S1 3. TENURING — after surviving N minor GCs (default N=15) Object promoted from Young Gen → Old Gen (Tenured) Also: large objects may go directly to Old Gen ↓ Old Gen gets full 4. MAJOR GC / FULL GC - Collects entire heap (Young + Old + Metaspace) - Stop-The-World pause (all application threads paused) - More expensive, less frequent than Minor GC
⚠️
Stop-The-World (STW): During GC, all application threads are paused — this causes latency spikes. Modern GC algorithms (G1, ZGC, Shenandoah) minimize STW pauses by doing most work concurrently with application threads.

JVM Stack & Stack Frame
Per-thread, stack frames, local variable array, operand stack, StackOverflowError

📋 JVM Stack — Structure

Each thread has its own JVM Stack created at thread creation time. The stack holds Stack Frames. One frame is pushed for each method invocation and popped when the method returns.

JVM Stack — Thread execution visualization Thread's JVM Stack ┌─────────────────────────────────────────────┐ │ ← TOP (current executing frame) │ │ Frame 3: calculateTax(double) │ │ Local Variables: [this, amount=1000.0] │ │ Operand Stack: [200.0] │ │ Return Address: processOrder (line 45) │ ├─────────────────────────────────────────────┤ │ Frame 2: processOrder(Order) │ │ Local Variables: [this, order, total=0] │ │ Operand Stack: [] │ │ Return Address: main (line 12) │ ├─────────────────────────────────────────────┤ │ Frame 1: main(String[]) │ │ Local Variables: [args, order] │ │ Operand Stack: [] │ │ Return Address: (none — bottom) │ └─────────────────────────────────────────────┘ ← BOTTOM

🔷 Stack Frame — 3 Components

ComponentWhat It ContainsSize
Local Variable Array Method parameters + local variables. Primitives stored directly (int, long, double...). Objects stored as references (pointer to heap object). Index 0 = this for instance methods. Fixed at compile time

Frame Data Reference to constant pool (for dynamic linking). Normal method completion info. Abnormal completion (exception) table. Variable

💥 Stack vs Heap — What Goes Where

JavaStack vs Heap allocation
void example() {
    // ── ON STACK ──────────────────────────────────────────────
    int x = 42;               // primitive → stored directly in stack frame
    double pi = 3.14;          // primitive → stored directly in stack frame
    boolean flag = true;       // primitive → stored directly in stack frame

    // Reference (pointer) → on stack; Object → on heap
    String name = new String("Alice");
    //  ^--- reference (4/8 bytes) on Stack
    //                              ^--- String object on HEAP

    // ── ON HEAP ───────────────────────────────────────────────
    MyObject obj = new MyObject();  // obj → Stack; new MyObject() → Heap
    int[] arr = new int[10];         // arr → Stack; array → Heap

    // Instance variables are on HEAP (inside the object)
    obj.name = "Bob";  // obj.name is inside obj on heap
    obj.age  = 30;    // obj.age  is inside obj on heap

    // Static variables → Method Area (Metaspace), not stack or instance heap
    MyObject.count = 5; // stored in Method Area
}
// When example() returns: stack frame is popped, x/pi/flag/references gone
// Heap objects (MyObject, String) are candidates for GC (if no other references)

Execution Engine
Interpreter, JIT Compiler, GC, JNI — how bytecode becomes machine code

🔷 Three Components of the Execution Engine

  • Interpreter: Reads and executes bytecode instructions ONE at a time. Fast startup, but slow for repeated code (re-interprets every time).
  • JIT Compiler: Identifies "hot" methods (frequently called), compiles them to native machine code. Subsequent calls run at native speed.
  • Garbage Collector: Automatically manages heap memory — reclaims memory of unreachable objects.

🔄 Interpreter vs JIT — The Hybrid Approach

Bytecode Execution — Interpreter → JIT Pipeline .class bytecode ↓ Interpreter (starts immediately) - Executes each bytecode instruction directly - No warmup time, immediate startup - Slow: re-interprets same code every invocation ↓ (method call counter exceeds threshold) JIT Compiler detects "hot method" Threshold: -XX:CompileThreshold=10000 (default) (method called >10000 times OR 10000 combined invocations+loops) ↓ C1 Compiler (Client JIT — Tier 1-3) - Quick compilation, basic optimizations - Profiling instrumentation inserted ↓ (more profiling data collected) C2 Compiler (Server JIT — Tier 4) - Aggressive optimizations - Inlining, loop unrolling, escape analysis - Compiles to highly optimized native code ↓ Native machine code in Code Cache (called directly, no interpretation overhead)

JNI allows Java code to call, and be called by, native C/C++ code. The Execution Engine uses it when Java code invokes native methods.

💻 Custom ClassLoader — File System Loader

JavaCustom ClassLoader — load from custom directory
import java.io.*;
import java.nio.file.*;

public class FileClassLoader extends ClassLoader {

    private final String classDir; // directory containing .class files

    public FileClassLoader(String classDir) {
        super(); // parent = AppClassLoader (maintains delegation)
        this.classDir = classDir;
    }

    public FileClassLoader(String classDir, ClassLoader parent) {
        super(parent); // custom parent
        this.classDir = classDir;
    }

    // ✅ Override findClass(), NOT loadClass()
    // findClass() is called only AFTER parent delegation has failed
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytecode = loadBytecode(name);
        if (bytecode == null) {
            throw new ClassNotFoundException("Cannot find class: " + name);
        }
        // defineClass — converts raw bytecode to Class object
        return defineClass(name, bytecode, 0, bytecode.length);
    }

    private byte[] loadBytecode(String className) {
        // Convert class name to file path: "com.example.Foo" → "com/example/Foo.class"
        String filePath = classDir + File.separator
            + className.replace(".", File.separator) + ".class";
        File classFile = new File(filePath);
        if (!classFile.exists()) return null;
        try {
            return Files.readAllBytes(classFile.toPath());
        } catch (IOException e) {
            return null;
        }
    }
}

// ─── Usage ───
FileClassLoader loader = new FileClassLoader("/plugins/classes");

// Load class dynamically
Class<?> clazz = loader.loadClass("com.example.Plugin");

// Instantiate (use interface for type safety)
Runnable plugin = (Runnable) clazz.getDeclaredConstructor().newInstance();
plugin.run();

// Get class metadata
System.out.println(clazz.getClassLoader()); // FileClassLoader
System.out.println(clazz.getName());        // com.example.Plugin

🔥 Hot Reload Pattern — Key Use Case

JavaHot Class Reload — recreate ClassLoader
// Hot reload: throw away old CL, create new one for updated .class
class HotReloader {
    private FileClassLoader currentLoader;
    private final String classDir;

    HotReloader(String classDir) {
        this.classDir = classDir;
        this.currentLoader = new FileClassLoader(classDir);
    }

    public Object loadFresh(String className) throws Exception {
        // DISCARD old ClassLoader → all its classes become GC-eligible
        this.currentLoader = new FileClassLoader(classDir);

        // Load fresh copy from updated .class file
        Class<?> clazz = currentLoader.loadClass(className);
        return clazz.getDeclaredConstructor().newInstance();
    }
}

// Note: Java ClassLoader cache is per-instance!
// The OLD ClassLoader still has the cached old class.
// NEW ClassLoader has no cache → loads fresh from disk.
// Old ClassLoader (and its Class objects) are GC'd when no references remain.

🔐 Encrypted ClassLoader — Security Use Case

JavaEncrypted ClassLoader — decrypt then load
public class EncryptedClassLoader extends ClassLoader {
    private final byte XOR_KEY = 0x42; // simple XOR "encryption"
    private final String encryptedDir;

    public EncryptedClassLoader(String dir) {
        this.encryptedDir = dir;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            // 1. Read encrypted .class file
            byte[] encrypted = readEncryptedFile(name);
            // 2. Decrypt
            byte[] bytecode = decrypt(encrypted);
            // 3. Define class from decrypted bytecode
            return defineClass(name, bytecode, 0, bytecode.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }

    private byte[] decrypt(byte[] data) {
        byte[] result = new byte[data.length];
        for (int i = 0; i < data.length; i++)
            result[i] = (byte) (data[i] ^ XOR_KEY);
        return result;
    }

    private byte[] readEncryptedFile(String name) throws IOException {
        String path = encryptedDir + "/"
            + name.replace(".", "/") + ".enc";
        return Files.readAllBytes(Paths.get(path));
    }
}

Section 12
Master Interview Q&A
20+ most asked JVM/ClassLoader interview questions with detailed answers

📦 JDK / JRE / JVM Questions

Q1What is the difference between JDK, JRE, and JVM?
JVM (Java Virtual Machine): Abstract machine that executes Java bytecode. Platform-specific (different JVM for Windows/Linux/Mac). Contains ClassLoader, Memory Areas, and Execution Engine.
JRE (Java Runtime Environment): JVM + Java Standard Class Libraries (java.lang, java.util, etc.). Used to run Java programs. No compiler.
JDK (Java Development Kit): JRE + development tools (javac, jar, javadoc, jdb, jshell). Used to develop and run Java programs.
Java 9+ note: JRE is no longer separately distributed. JDK includes everything.
Q2Is Java compiled or interpreted?
Both! Java uses a two-phase approach:
1. javac compiles .java source to platform-independent bytecode (.class).
2. JVM interprets bytecode at startup, then the JIT compiler compiles hot methods to native machine code at runtime.
This hybrid approach gives: fast startup (interpreter), fast steady-state (JIT native code), and platform independence (bytecode is portable).

🔄 Class Loading Questions

Q3What are the phases of class loading?
3 main phases (5 steps):
1. Loading: ClassLoader finds .class file, reads binary data, creates Class object in Method Area.
2. Linking: 3 sub-phases:
  • Verification: Verify bytecode correctness and security.
  • Preparation: Allocate memory for static variables, set to default values (0/false/null).
  • Resolution: Convert symbolic references to direct memory references.
3. Initialization: Execute <clinit> — run static blocks, assign actual static variable values in source order.
Q4Explain the Parent Delegation Model with an example.
When a ClassLoader is asked to load a class, it delegates to its parent FIRST before trying itself.
Example — Loading java.util.ArrayList:
1. AppClassLoader asks: "did parent load it?" → delegates to PlatformCL
2. PlatformCL asks: "did parent load it?" → delegates to Bootstrap
3. Bootstrap finds it in java.base module → returns Class object
4. Returns all the way back to AppClassLoader → uses Bootstrap's version.
Why? Security (prevent hijacking core classes), uniqueness (one definition per ClassLoader), consistency (same String class everywhere).
Q5What is a ClassNotFoundException vs NoClassDefFoundError?
ClassNotFoundException (checked): ClassLoader tried to load a class dynamically (via Class.forName(), loadClass()) and could not find it on the classpath. Handle explicitly.
NoClassDefFoundError (unchecked Error): Class was present at compile-time but missing at runtime. OR class was found but its static initializer threw an exception (subsequent attempts get ExceptionInInitializerError → NoClassDefFoundError). Much harder to debug.
Q6What triggers class initialization?
A class is initialized (static blocks run) when:
1. An instance is created: new MyClass()
2. A static method is called: MyClass.staticMethod()
3. A non-constant static field is read/written
4. Class.forName("MyClass") is called (with initialize=true)
5. A subclass is initialized (triggers parent class init first)
Does NOT trigger initialization: Accessing a compile-time static final constant, accessing class via array type (MyClass[] arr), classLoader.loadClass().

🧠 Memory & JVM Architecture Questions

Q7What are the runtime data areas in JVM?
Shared by all threads:
• Method Area (Metaspace): Class metadata, bytecode, static variables, constant pool.
• Heap: All object instances and arrays. GC's domain.
Per-thread (each thread gets its own):
• JVM Stack: Stack frames for method invocations (local vars + operand stack).
• PC Register: Address of currently executing bytecode instruction.
• Native Method Stack: Stack for native (JNI) method calls.
Q8What is PermGen and why was it replaced by Metaspace in Java 8?
PermGen was a fixed-size region of the Java Heap (default 64MB) storing class metadata. Problems: Fixed size caused OutOfMemoryError: PermGen space especially in app servers loading many classes. Hard to tune correctly.
Metaspace (Java 8+): Moved to native memory (off-heap). Grows dynamically as needed. Only limited by available system memory. Configure with -XX:MaxMetaspaceSize to prevent unbounded growth. Also: String Pool moved to Heap in Java 7 (before Metaspace change).
Q9Where are local variables, instance variables, and static variables stored?
Local variables (primitives): Directly in JVM Stack frame's local variable array.
Local variables (object references): Reference is in Stack; actual object is in Heap.
Instance variables: Inside the object on the Heap (both primitives and references).
Static variables: In the Method Area (Metaspace), in the class's static fields section.
String literals: In the String Pool (part of Heap since Java 7).
Q10What is StackOverflowError? How to prevent it?
Thrown when the JVM Stack for a thread runs out of space — typically due to infinitely (or too deeply) recursive method calls. Each method call creates a Stack Frame; too many frames overflow the stack.
Prevention: Fix infinite recursion, convert to iterative approach, increase stack size with -Xss4m (but this uses more memory per thread).

🚀 JIT & Execution Engine Questions

Q11What is the JIT compiler? How does it work?
JIT (Just-In-Time) compiler monitors method invocation counts. When a method becomes "hot" (called > ~10,000 times), JIT compiles it to native machine code stored in the Code Cache. Subsequent calls bypass interpretation and execute native code directly.
C1 (Client): Quick compilation, basic optimization, profiling instrumented.
C2 (Server): Aggressive optimization using profiling data (inlining, escape analysis, loop unrolling).
Tiered compilation (default): Starts with C1 (fast compile), promotes to C2 (optimal) for hottest methods.
Q12What is escape analysis?
JIT analysis that determines whether an object "escapes" the scope of the method where it was created. If it doesn't escape:
1. Stack Allocation: JIT can allocate the object on the stack instead of heap (instant deallocation when method returns, no GC needed).
2. Scalar Replacement: Decompose object into individual scalar fields (object is never actually created).
3. Lock Elision: Remove synchronization on objects that are thread-local.
Q13Why does Java warm up slowly? What is JVM warm-up?
At startup, JVM interprets all code — slow per-instruction execution. JIT hasn't had enough invocations to compile methods yet. Over time (~seconds to minutes depending on app), JIT compiles hot methods to native code — performance improves dramatically. This is "warm-up time."
Solutions for slow start: GraalVM Native Image (AOT compilation), AppCDS (class data sharing), Project Leyden (future Java).

🧩 Tricky / Advanced Questions

Q14Why does ClassLoader.getParent() return null for Bootstrap ClassLoader's child?
Bootstrap ClassLoader is written in native C++ and has NO Java object representation. When PlatformClassLoader's getParent() is called, it returns null because the Bootstrap loader cannot be represented as a Java ClassLoader object. This is why we check for null in the delegation chain to identify the Bootstrap ClassLoader.
Q15What is the Thread Context ClassLoader and why is it needed?
When code in Bootstrap ClassLoader needs to load classes provided by the application (e.g., JDBC driver via SPI), normal parent delegation fails — Bootstrap can't load application-level classes by going UP the chain. The Thread Context ClassLoader (Thread.currentThread().getContextClassLoader()) solves this by allowing Bootstrap-loaded code to access a ClassLoader set by the application thread. This is used by JDBC, JNDI, XML parsers, and other SPI-based frameworks.
Q16What is the difference between Minor GC, Major GC, and Full GC?
Minor GC: Collects Young Generation (Eden + Survivor spaces) only. Fast (Young Gen is small), frequent. Causes STW pause but very brief (milliseconds).
Major GC: Collects Old Generation. Slower than Minor GC (large area). Triggered when Old Gen is full.
Full GC: Collects ENTIRE heap — Young + Old + Metaspace. Slowest, longest STW pause. Should be minimized. Triggered by: explicit System.gc(), promotion failure, concurrent mode failure.
Q17What is the String Pool? How does String interning work?
String Pool is a special cache in the Heap (since Java 7) that stores unique String literals. When you write "hello", JVM checks the pool — if it exists, returns the cached reference; if not, creates it and adds to pool.
"abc" == "abc" → true (same pool reference)
new String("abc") == "abc" → false (new heap object)
new String("abc").intern() == "abc" → true (intern() returns pool reference)
Memory benefit: Thousands of "GET", "POST" strings reuse same pool entry.
Q18What is 0xCAFEBABE and why does it exist?
0xCAFEBABE is the "magic number" — the first 4 bytes of every valid Java .class file. The JVM verifies this magic number during the Loading → Verification phase. If the file doesn't start with CA FE BA BE, the JVM rejects it as not a valid Java class file. It was named by James Gosling (Java's creator) — "cafe babe" was the food/drink theme of Grateful Dead hang-out spots. The version bytes follow (minor/major version).
Q19What is the difference between -Xms and -Xmx?
-Xms512m: Initial heap size — JVM starts with this much heap. Setting it equal to Xmx avoids heap resizing overhead.
-Xmx2g: Maximum heap size — JVM cannot grow beyond this. Throws OutOfMemoryError: Java heap space when reached.
Best practice: Set -Xms = -Xmx in production to avoid GC overhead from heap resizing and ensure predictable memory usage. E.g., -Xms2g -Xmx2g
Q20How can you cause a memory leak in Java?
Java GC prevents traditional memory leaks but you can still prevent GC from collecting objects:
1. Static collections: static List<Object> cache — objects never removed.
2. ThreadLocal without remove(): Thread pool keeps reference via ThreadLocalMap.
3. Unclosed resources: DB connections, file streams — keep internal buffers alive.
4. Listeners not deregistered: Event listeners hold references to their publishers.
5. Inner classes: Non-static inner class holds implicit reference to outer class.
6. Custom ClassLoader: If CL is not GC'd, all its loaded classes stay in memory.

🎯 Rapid Fire — JVM Interview Questions
Q1What is the difference between JDK, JRE, and JVM?
JVM (Java Virtual Machine): Abstract machine that executes bytecode. Platform-specific implementation (Windows JVM, Linux JVM).
JRE (Java Runtime Environment): JVM + standard class libraries (rt.jar, java.util, java.lang etc.) — enough to RUN Java programs.
JDK (Java Development Kit): JRE + development tools (javac compiler, javadoc, jdb debugger, jconsole, jstack etc.) — needed to DEVELOP Java programs.
Relationship: JDK ⊃ JRE ⊃ JVM. In Java 9+, modular system (JPMS) replaced monolithic JRE — you can create custom runtime images with jlink.
Q2Explain the ClassLoader delegation model (Parent Delegation).
When a ClassLoader is asked to load a class it: (1) Checks its own cache first. (2) Delegates to parent ClassLoader. (3) Only if parent cannot find it, loads it itself.
Hierarchy: Bootstrap CL → Extension/Platform CL → Application CL → Custom CL
Why? Prevents malicious code from replacing core classes (e.g., a fake java.lang.String). Security and consistency.
Q3Can we load the same class twice in the same JVM?
Yes — by different ClassLoaders. Each ClassLoader maintains its own namespace. The same fully qualified class name loaded by two different ClassLoaders creates two distinct Class objects in the JVM. They are NOT type-compatible — casting between them throws ClassCastException. This is used in application servers (each webapp has its own ClassLoader) and hot-reloading frameworks.
Q4What are the phases of class loading?
1. Loading: Read .class file bytes, create Class object in Method Area.
2. Linking:
  • Verification — bytecode validity check (structure, types, security)
  • Preparation — allocate memory for static fields, set to default values (0, null, false)
  • Resolution — replace symbolic references with direct memory references
3. Initialization: Run static initializers and static blocks in top-down order. Triggered on first active use.
Q5What triggers class initialization?
Active use triggers initialization: (1) Creating instance with new. (2) Calling a static method. (3) Reading/writing a non-constant static field. (4) Class.forName() call. (5) A subclass being initialized triggers superclass initialization. (6) JVM main class at startup.
Does NOT trigger (passive use): Accessing a static final compile-time constant (inlined by compiler), creating array of a class type, ClassLoader.loadClass().
Q6What is the difference between Stack and Heap in JVM?
Stack: Per-thread. Stores stack frames (local variables, method arguments, partial results, return address). LIFO. Fixed size (-Xss). Fast allocation. StackOverflowError if exceeded. Automatically managed (frame pushed/popped).
Heap: Shared across all threads. Stores all object instances and arrays. Managed by GC. -Xms/-Xmx controls size. OutOfMemoryError if full. Slower than stack (GC overhead, synchronization).
Q7What is Metaspace and how is it different from PermGen?
PermGen (before Java 8): Fixed-size JVM heap region. Stored class metadata, interned strings, static variables. Common cause of OutOfMemoryError: PermGen space in apps that load many classes dynamically.
Metaspace (Java 8+): Uses native OS memory (outside JVM heap). No fixed upper limit by default (auto-grows). Can be limited with -XX:MaxMetaspaceSize. String pool moved to heap. Static variable references moved to heap. Significantly reduces OOM errors from class loading.
Q8How does JIT compilation work? What is tiered compilation?
JIT monitors method invocation counts. Initially code runs interpreted. After ~10,000 invocations (threshold), method is compiled to native machine code by JIT.
Tiered Compilation (Java 8+ default):
Level 0: Interpreter (no profiling)
Level 1–3: C1 compiler (fast compile, basic optimizations + profiling data collection)
Level 4: C2 compiler (aggressive optimizations using profiling data — inlining, loop unrolling, escape analysis)
Methods can be deoptimized if assumptions become invalid (e.g., new subclass loaded invalidating an inlined call).
Q9Explain Garbage Collection — how does generational GC work?
Based on the "generational hypothesis" — most objects die young.
Young Generation: Eden + 2 Survivor spaces (S0, S1). New objects go to Eden. Minor GC: copies live objects to Survivor, increments age. Objects aging past threshold promoted to Old Gen.
Old Generation: Long-lived objects. Major/Full GC collects here — more expensive, longer STW pauses.
GC algorithms: Serial (single thread), Parallel (multi-thread throughput), CMS (deprecated, low-pause), G1 (default Java 9+, region-based, predictable pauses), ZGC (Java 11+, <1ms pauses), Shenandoah (Red Hat, concurrent).
Q10What is the difference between Minor GC, Major GC, and Full GC?
Minor GC: Collects Young Generation only. Fast (young gen is small). STW pause is short. Triggered when Eden is full.
Major GC: Collects Old Generation. Slower, longer STW pause. Triggered when old gen is full.
Full GC: Collects ENTIRE heap (Young + Old + Metaspace). Most expensive. Triggered by: System.gc(), promotion failure, concurrent mode failure (CMS), explicit request. Should be minimized in production.
Q11What is a memory leak in Java? How to detect it?
Java memory leak = objects are not GC'd because references are unintentionally held. Common causes:
• Static collections growing without bound
• Unclosed resources (streams, connections)
• ThreadLocal without remove() in thread pools
• Listener/callback not unregistered
• Inner class holding outer class reference
Detection: Heap dump analysis (jmap -dump + Eclipse MAT / VisualVM), monitoring heap growth over time with JConsole, GC log analysis (-verbose:gc), profilers (YourKit, JProfiler, Async-profiler).
Q12What happens when you call System.gc()?
It is a hint to the JVM to run GC — the JVM may ignore it entirely. It is non-deterministic, non-blocking, and not guaranteed to do anything. In production code, calling System.gc() is considered bad practice because: (1) It may trigger an unnecessary Full GC causing long STW pause. (2) The JVM's GC algorithms are better at deciding when to collect. (3) It can be disabled with -XX:+DisableExplicitGC. Use case: before capturing a heap dump in diagnostics.
Q13What are Strong, Soft, Weak, and Phantom references?
Strong (default): Object obj = new Object() — never GC'd while reference exists.
Soft: SoftReference<T> — GC'd only when JVM is low on memory. Good for memory-sensitive caches.
Weak: WeakReference<T> — GC'd at next GC cycle regardless of memory. Used in WeakHashMap, ThreadLocal internals.
Phantom: PhantomReference<T> — get() always returns null. Used for pre-mortem cleanup (resource release) before GC. Used with ReferenceQueue. Cleaner API (Java 9+) is preferred modern alternative.
Q14How does the JVM execute bytecode — describe the execution engine?
Execution Engine components:
1. Interpreter: Reads and executes bytecode instructions one at a time. Fast startup, slower execution.
2. JIT Compiler: Compiles hot methods to native code. C1 (quick) → C2 (optimized). Results cached in Code Cache.
3. Garbage Collector: Automatic memory reclamation.
4. Native Method Interface (JNI): Bridge to native (C/C++) code.
5. Native Method Libraries: Platform-specific native implementations.
Q15What is the G1 GC and why is it the default since Java 9?
G1 (Garbage-First) GC divides heap into equal-sized regions (1–32MB). Regions are dynamically assigned as Eden, Survivor, Old, or Humongous (for large objects). G1 prioritizes collecting regions with most garbage first (hence "Garbage-First"). Benefits: predictable pause times (-XX:MaxGCPauseMillis=200 target), works well for heaps 6GB+, concurrent marking, no fragmentation. Good balance of throughput vs latency. For ultra-low latency use ZGC (Java 15+ production-ready, <1ms pauses).
