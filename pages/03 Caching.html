Caching

The Art of Remembering What You Already Know

Foundations — Why, Where & What Gets Evicted

Build the mental model before you memorize any terms.

⚡ §1 — Why Caching Exists

🧠
The Mental Model — Your Desk vs. Your Filing Cabinet

Your filing cabinet (the database) holds everything — but you have to walk over to it, flip through folders, and find what you need. That takes time. Your desk (the cache) holds the things you grabbed most recently. They're right there, instantly accessible. Caching is just: keep the things you use most in the place that's fastest to reach.

The Four Problems Caching Solves

Each of these is a reason an interviewer might ask you "why did you add a cache here?" — and you should be able to answer for all four.

🔴 Database Overload

A single MySQL instance handles roughly ~10K QPS (queries per second) for simple reads. Twitter at peak serves >300K tweets/second. Without caching, that database dies. The cache absorbs the reads so the DB only handles writes and cache misses.

⚡ Response Latency

A DB query across the network: ~1–10ms. A Redis lookup in RAM: ~0.1–0.5ms. That's 10–100× faster. When you're building a product page that fires 20 DB queries, those milliseconds add up to a noticeably slow product.

💰 Compute Savings

Imagine calculating a user's recommendation feed: 200ms of ML computation. Now imagine 50,000 users opening the app simultaneously. Cache the result for 5 minutes — you run that computation once, not 50,000 times. This is pure cost reduction.

☁️ Infrastructure Cost

Cloud databases bill per read/write. Fewer DB calls = lower cost. Companies like Pinterest reported 80–90% reduction in DB load after adding Redis. Redis RAM costs money too — but it's far cheaper than equivalent DB compute.

Numbers to Remember

L1 CPU cache: ~1ns · L2 CPU cache: ~4ns · RAM (Redis): ~100ns · SSD DB: ~100µs · HDD DB: ~10ms · Network DB: ~1–10ms. Caching moves you from milliseconds to microseconds — a 1,000× improvement that users actually feel.

🏗 §2 — Where to Cache: The 5 Layers

The Caching Hierarchy

Every request passes through multiple potential caching layers. The higher in the stack you cache something, the faster the response — but the fewer users share that cache. Understand each layer so you can place caches intelligently in a design.

1
Client-Side Cache
Browser HTTP cache (Cache-Control, ETag), mobile app local cache, service worker cache. Lives on the user's device. Zero network cost for a hit.
~0ms
2
CDN Cache
Cloudflare, Akamai, AWS CloudFront. Caches static assets (images, JS, CSS) and increasingly API responses at edge nodes close to users. Perfect for read-heavy global content.
~5–20ms
3
Load Balancer Cache
Some LBs (e.g., NGINX, HAProxy) support basic response caching. Uncommon in modern architectures — but exists. Handles identical repeated requests before they hit app servers.
~1–5ms
4
Application Cache (In-Process)
A dictionary/HashMap in your app server's memory. Guava Cache (Java), functools.lru_cache (Python). Zero network hop — but not shared across instances. Good for static config data, lookup tables.
~0.01ms
5
Distributed Cache
Redis, Memcached. Shared across all app server instances. Network hop required but still ~10–100× faster than a database. This is what people mean 99% of the time when they say "add a cache."
~0.5–2ms

When to use CDN cache

Static assets (always). API responses that are the same for all users (leaderboards, trending content, home feed skeleton). Not for personalized data.

Netflix: ~95% of traffic served from CDN edge. Zero origin server load for those requests.

When to use in-process cache

Configuration that changes rarely (feature flags, DB connection metadata, country codes). Fastest possible access. Problem: each server has its own copy — if one server updates, others don't know.

Watch out for stale state when deploying updates to config.

⚠️
Don't Cache Everything

Every cache adds complexity: stale data risk, invalidation logic, extra infrastructure. Only cache data that is: (1) read far more often than it's written, (2) expensive to compute or retrieve, and (3) acceptable to be slightly stale. Real-time stock prices? Don't cache. User profile data? Cache aggressively.

🗑 §3 — Cache Eviction Policies

The Fundamental Problem: Cache is Finite

RAM is expensive. Your cache can hold 10GB worth of data but your dataset is 10TB. So when the cache is full and a new item arrives, something has to go. Which item? That's the eviction policy.

The right policy depends on your access patterns. Getting this wrong means you're evicting the wrong things — and your cache hit rate tanks.

🕐 LRU — Least Recently Used

Analogy: Your browser tabs — you close the one you haven't clicked in 3 days. It might still be useful, but recency is your proxy for "needed."

Evicts: The key that was accessed least recently.

Use when: Access patterns are temporal — recently used data is likely to be used again. This is the correct default for most workloads.

Real-world: Redis default eviction. Used by virtually every production cache.

📊 LFU — Least Frequently Used

Analogy: You keep books you've read 10 times, but donate the one you picked up once years ago. Frequency beats recency.

Evicts: The key with the lowest access count.

Use when: Access patterns are stable — some items are always popular (product catalog's top 100 items) and you want to keep them regardless of when they were last accessed.

Problem: A brand new viral item starts with frequency=1 and gets immediately evicted. Needs aging mechanisms.

📬 FIFO — First In, First Out

Analogy: A grocery store shelf — oldest stock goes first, regardless of what's popular.

Evicts: The oldest entry by insertion time.

Use when: You have a natural time-based data lifecycle. Rare in application caching. More common in buffers and queues.

Problem: Ignores both recency and frequency. A key accessed every second could get evicted just because it was inserted first.

⏱ TTL — Time To Live

Analogy: Milk with an expiry date. After the date, throw it out regardless of whether you drink it daily or haven't touched it.

Evicts: Entries automatically after a set duration (e.g., 5 minutes).

Use when: Data has a known staleness window. Session tokens, OTP codes, weather data, pricing data.

Key insight: TTL is not a replacement for LRU/LFU — it works alongside them. Redis supports both simultaneously.

🎯
Interview Default: Always Start with LRU + TTL

If asked which eviction policy you'd use, say: "I'd use LRU as the primary policy combined with TTL per key type. LRU handles memory pressure, TTL handles data freshness. I'd only move to LFU if I had evidence that my top items were getting evicted due to a recency skew in access patterns."

The Hard Parts — Invalidation, Failures & Redis

This is where most engineers get it wrong. This is also what interviewers test hardest.

🔄 §4 — Cache Invalidation: The Hardest Problem

"There are only two hard things in computer science: cache invalidation and naming things." — Phil Karlton

This isn't a joke. Every production system eventually has a cache invalidation bug. Understanding the three strategies — and their trade-offs — is table stakes for any system design interview.

Why Is It Hard?

The fundamental tension: your cache has a copy of data that lives in the database. When the database changes, your cache has stale data. The question is: when do you update the cache, and who is responsible for doing it?

There's no perfect answer. Every strategy trades off something — write latency, consistency, complexity, or durability. Your job as a designer is to pick the trade-off that fits your use case.

Strategy A: Write-Through

Analogy: Carbon copy paper — every time you write something, a copy goes to the filing cabinet simultaneously. Your desk and the cabinet always agree.

1
App receives a write request
User updates their profile picture URL.
2
Write to cache AND database simultaneously
Both are updated as part of the same write operation. Some implementations write DB first, then cache. Some write cache first.
3
Acknowledge success to client
Only after both writes succeed. This means write latency = max(cache write time, DB write time).

✅ What you gain

  • Cache is always consistent with DB
  • Read-after-write always hits cache
  • No stale data window
  • Simple mental model

❌ What you give up

  • Higher write latency (two writes per operation)
  • Cache gets polluted with data that's never read again
  • What if cache write succeeds but DB write fails? Rollback complexity.
  • Two failure points per write

Best for: Systems where reads immediately follow writes (user profiles, settings, financial dashboards).

Strategy B: Write-Back (Write-Behind)

Analogy: A whiteboard in a meeting. You jot notes on the whiteboard immediately (fast), and someone transfers them to the official document later (async). The whiteboard is always current; the document lags.

1
App receives a write request
User updates their profile picture URL.
2
Write to cache ONLY — return success immediately
Ultra-fast write. Client gets acknowledgement in sub-millisecond time.
3
Background process writes to DB asynchronously
A worker drains the "dirty" cache keys and flushes them to DB. Could be 100ms later, could be 10 seconds later.
💀
The Scary Part: Data Loss Window

If your cache server crashes between step 2 and step 3, those writes are gone. Forever. That's why write-back is mostly used for non-critical data where eventual persistence is acceptable: analytics counters, view counts, "recently viewed" lists. Never for financial transactions or order data.

Best for: High-write workloads where throughput > durability (gaming leaderboards, real-time analytics, social media like/view counters).

Strategy C: Cache-Aside (Lazy Loading) ⭐ Most Common

Analogy: You only put things on your desk when you need them. You don't pre-populate your desk with every file in the cabinet — you grab things as you reach for them. Lazy, but space-efficient.

1
App receives a read request for user:42
Check the cache first.
✗
Cache MISS — key not found
Go to the database and fetch the data.
3
Store result in cache with TTL
e.g., SET user:42 {data} EX 300 (expires in 5 minutes). Now future reads are fast.
✓
Cache HIT on all subsequent reads (within TTL)
Sub-millisecond response. DB not touched.

✅ What you gain

  • Simple to implement — app logic controls everything
  • Cache only fills with data that's actually requested
  • Cache node failure is recoverable (warm up on restart)
  • Works with any database

❌ What you give up

  • First request always slow (cache miss)
  • Thundering herd when cache is cold (see §5)
  • Stale data until TTL expires after a write
  • App code must handle miss logic everywhere
🎯
Interview Answer: Why Cache-Aside is the Default

Cache-Aside is the most common pattern because it gives you flexibility. The cache is managed by the application layer — you can add it incrementally, it degrades gracefully if Redis is down (you just hit the DB), and it doesn't require special cache infrastructure. In most interviews, start here and only deviate if you have a specific reason (ultra-low write latency → Write-Back; strict consistency → Write-Through).

💥 §5 — The 3 Cache Failure Modes You Will Be Asked About

Why These Matter

Any senior engineer who's run a production cache has seen these. Interviewers at Google and Meta specifically probe for these because they reveal whether you've thought about failure modes — not just the happy path.

⚡ Failure Mode A: Cache Stampede / Thundering Herd

What happens: A heavily-accessed cache key expires. At that exact moment, 1,000 concurrent requests all check the cache, all get a miss, and all simultaneously fire a database query for the same data. Your DB — which was handling 100 QPS gracefully — suddenly gets 1,000 concurrent queries. It falls over.

Real world: This killed several production systems at Reddit during AMAs, when a cached post expired and thousands of users were on the page.

Solutions
1
Mutex / Lock (Thundering Herd Prevention)
When a cache miss occurs, acquire a distributed lock before querying the DB. Only one request goes to the DB; all others wait. When the DB returns, the result is cached, all waiters get the cached result. Redis SET NX (set if not exists) is perfect for this.
2
Probabilistic Early Expiration (PER)
Before a key actually expires, a small probability triggers a background refresh. If key TTL = 5min and we're at 4:50, there's a 10% chance any request will proactively refresh it. This smooths out the expiry cliff.
3
Background Refresh Thread
A dedicated worker monitors key TTLs and refreshes them before they expire. The user-facing code never sees a cache miss. Used heavily at Facebook and Twitter for high-traffic keys.
👻 Failure Mode B: Cache Penetration

What happens: Requests come in for data that doesn't exist in your database. Example: a malicious actor (or just a bug) sends requests for user IDs 99999999, 88888888, 77777777 — none of which exist. These always result in a cache miss → DB query → empty result. The cache never helps because there's nothing to cache. Your DB takes the full load.

Real world: Common in DDoS attacks and web scraping attacks. Also happens innocently with typos or invalid IDs from client bugs.

Solutions

Solution 1: Cache Null Results

When a DB query returns empty, cache the null/empty result with a short TTL (e.g., 30 seconds).

Trade-off: Cache gets filled with useless null entries. If the data is later created legitimately, it serves null until TTL expires.

Solution 2: Bloom Filter ⭐

A probabilistic data structure that sits in front of the cache. Before any DB query, check: "does this key possibly exist?" If the Bloom filter says NO — definitely no — skip DB entirely.

False positives possible (it says "maybe yes" for non-existent keys) but never false negatives. Perfect for this use case. ~10MB of RAM can index billions of keys.

❄️ Failure Mode C: Cache Avalanche

What happens: Many cache keys all expire at the same time — causing a tsunami of simultaneous DB requests. This is different from stampede: stampede is one popular key expiring; avalanche is many keys expiring simultaneously.

Common cause: System restart after a cache failure. All keys were set with TTL=300s during cache warm-up at t=0. At t=300s, they all expire at once. Boom.

Also caused by: A Redis node failure. All traffic that was hitting the dead node suddenly hits the DB.

Solutions
1
Randomize TTL Values
Instead of SET key TTL=300, use SET key TTL=(300 + random(0,60)). This spreads expirations across a 60-second window instead of a single second. Simple, effective, always do this.
2
Cache Warm-Up on Restart
After a Redis restart, proactively pre-populate the cache with the most critical keys before opening traffic. Use a startup job that loads top N users, top N products, etc.
3
Circuit Breaker on DB
If the DB is getting hammered, trip a circuit breaker that returns a degraded response instead of crashing. Better to show "temporarily unavailable" than to take down the entire service.
4
Redis Clustering / Replication
Prevent single-node failure from triggering avalanche. Redis Sentinel for HA, Redis Cluster for horizontal scaling. A single node failure only impacts a fraction of keyspace.
🎯
Interview Tip: Name All Three Unprompted

When an interviewer asks "what problems can arise with your caching strategy?" — name all three: "I'd watch out for cache penetration (fake IDs — mitigated with Bloom filters), cache stampede (popular key expiry — mitigated with mutex), and cache avalanche (mass expiration — mitigated with randomized TTL)." This signals you've run real production systems.

🔴 §6 — Redis: What It Actually Is

⚠️
Common Mistake: "Redis = Cache"

Redis is a multi-model in-memory data store. It can be a cache, yes — but also a message queue, a leaderboard engine, a session store, a pub/sub broker, a rate limiter, and a distributed lock manager. Calling Redis "just a cache" in an interview is a junior-level answer. Knowing its data structures and when to use each one is the senior answer.

📝
String
Simple key → value. Supports atomic increment (INCR). Values up to 512MB. The building block everything else extends.
Rate limiting (INCR + EXPIRE), simple cache, feature flags, OTP codes, session tokens
🗂
Hash
A map of field → value under one key. Like a mini-object. Efficient for partial updates — change one field without reserializing the whole object.
User profiles, session data, product metadata. user:42 → {name:"Alice", email:"...", plan:"pro"}
📋
List
Doubly-linked list. O(1) push/pop from both ends. Ordered by insertion. Supports blocking pop (BLPOP) — a key queue primitive.
Activity feed (recent 50 items), job queue, message queue, "recent searches" list
🎯
Set
Unordered collection of unique strings. O(1) add/remove/check. Supports union, intersection, difference operations across sets.
Unique visitors per day, user tags, friend lists, "users who liked post X", mutual connections
🏆
Sorted Set (ZSet)
Each member has a floating-point score. Members sorted by score. Range queries by score or rank. O(log N) operations.
Leaderboards, priority queues, time-series (score = timestamp), range queries on scores
📡
Pub/Sub
Publish messages to channels; subscribers receive in real-time. Fire-and-forget: if subscriber is offline, message is lost. Not a persistent queue.
Real-time notifications, live chat, cache invalidation broadcast, collaborative editing

🏆 Deep Dive: Sorted Set Leaderboard

This is the most commonly asked Redis data structure in system design interviews. Here's exactly how you'd build a real-time leaderboard:

Java Redis Sorted Set — Leaderboard Implementation 
// Add/update a player's score — O(log N)
// ZADD game:leaderboard 15000 "alice"
jedis.zadd("game:leaderboard", 15000, "alice");
jedis.zadd("game:leaderboard", 12500, "bob");
jedis.zadd("game:leaderboard", 18200, "charlie");

// Get top 10 players, highest score first — O(log N + K)
// ZREVRANGE game:leaderboard 0 9 WITHSCORES
Set<Tuple> top10 = jedis.zrevrangeWithScores("game:leaderboard", 0, 9);

// Get a player's rank (0-indexed from top) — O(log N)
// ZREVRANK game:leaderboard "alice" → returns 1 (2nd place)
Long rank = jedis.zrevrank("game:leaderboard", "alice");

// Increment score atomically — O(log N)
// ZINCRBY game:leaderboard 500 "alice"
jedis.zincrby("game:leaderboard", 500, "alice");

// Get players in a score range — range queries are O(log N + K)
// ZRANGEBYSCORE game:leaderboard 10000 20000
Set<String> midTier = jedis.zrangeByScore("game:leaderboard", 10000, 20000);
💡
Why Sorted Set is perfect for leaderboards

A SQL query for SELECT user, score FROM scores ORDER BY score DESC LIMIT 10 requires a full sort at query time — O(N log N). Redis Sorted Set maintains the sorted order on every write, making reads O(log N + K). For 10M users, that's the difference between 260ms and 0.2ms.

🔒 Redis as a Distributed Lock (Redlock)

This comes up in interviews when you need exactly-once processing or mutual exclusion. The simplest form:

Java Redis Distributed Lock — SET NX EX pattern 
// Acquire lock — SET key value NX EX 30
// NX = only set if key does not exist (atomic compare-and-set)
// EX 30 = auto-expire in 30 seconds (prevents deadlock if process crashes)
String lockKey   = "lock:payment:order_123";
String lockValue = UUID.randomUUID().toString(); // unique per holder

SetParams params = SetParams.setParams().nx().ex(30);
String result = jedis.set(lockKey, lockValue, params);

if ("OK".equals(result)) {
    try {
        processPayment(); // critical section — only one process at a time
    } finally {
        // Release lock ONLY if we still own it (check value matches)
        // Use Lua script for atomic check-and-delete
        String luaScript = "if redis.call('get',KEYS[1])==ARGV[1] then " +
                           "return redis.call('del',KEYS[1]) else return 0 end";
        jedis.eval(luaScript, 1, lockKey, lockValue);
    }
} else {
    // Lock held by another process — retry or fail fast
}
📊
Redis Numbers to Know for Interviews

Single Redis instance: ~100K–1M ops/second (depending on operation complexity) · Typical latency: 0.1–0.5ms · Max memory per instance: typically ~50–100GB RAM in production · Redis Cluster supports horizontal scaling with automatic sharding across nodes.

📐 §7 — Cache Sizing & TTL Strategy

The 80/20 Rule of Caching

In most production systems, 20% of the data is responsible for 80% of the reads. You don't need to cache everything — you need to cache the right things. That 20% is your target.

Start by identifying your hot data: what are your most-read DB tables? What queries run thousands of times per second? Those are your cache candidates.

Data TypeCache?TTL GuidanceInvalidationReal Example
User profile Yes 5–15 min On write Twitter user object
Product catalog Yes 1–24 hr On update Amazon product page
Home feed / timeline Yes 30s – 2 min On new post Instagram home tab
Leaderboard Yes 10s – 1 min On score update Gaming apps
Config / feature flags Yes 5 min + local On change LaunchDarkly flags
Real-time stock prices Short TTL only < 1s Continuous Trading platforms
Financial transactions Avoid N/A N/A Bank balance, orders
Personalized recs Yes (keyed) 1–6 hr On model refresh Netflix suggestions

How to Estimate Cache Size

Start with: What is my working set?

Working set = data that's accessed in a given time window. For most apps, the "hot" data in any 1-hour window fits in 10–50GB of RAM.

Formula: avg_object_size × hot_key_count × safety_factor(1.5)

Example: 1M active users × 1KB profile = 1GB. Add 1.5× safety = 1.5GB. A tiny Redis instance handles this.

How to Set TTL Values

Ask yourself: "How stale can this data be before a user notices or before it causes a bug?"

  • User profile: 5 minutes is fine (changes rarely)
  • Post like count: 30s is fine (approximate is ok)
  • Auth token: matches token expiry (e.g., 1 hour)
  • Static content: hours to days

Always add ±10–20% jitter to prevent avalanche.

💡
Key Insight: Cache Hit Rate is Your North Star Metric

Target cache hit rate: >95% for a well-tuned production cache. Below 90% means you're either caching the wrong things, your TTLs are too short, or your cache is too small. Monitor this metric. If hit rate drops, your DB will feel it immediately.

🎯 §8 — Interview Playbook

How to Talk About Caching in Any System Design Interview

You don't wait for the interviewer to ask about caching. You proactively bring it up when you've identified a read-heavy bottleneck. Here's the exact framework.

The Standard Places to Add Cache in Any Design

1. User/Session Data

Every request that requires authentication checks who the user is. Cache the session → user mapping. Without this, your auth service or DB takes every API request's auth check hit.

Key: session:{token} → user object. TTL = session lifetime.

2. Aggregated/Computed Results

Anything that requires a complex query or computation: leaderboards, feed rankings, recommendation lists, search facets. Compute once, cache the result.

Key: feed:{userId} → array of post IDs. TTL = 30s–2min.

3. Frequently Read, Rarely Written

Product catalog, user profiles, config data, geo/location data. High read-to-write ratio makes caching extremely effective with simple cache-aside + TTL.

Key: product:{id} → product object. TTL = 1–24 hr.

4. Rate Limiting Counters

Track API usage per user per minute using Redis String + INCR + EXPIRE. Atomic operations make this consistent. Can't do this reliably in a stateless DB.

Key: ratelimit:{userId}:{minute} → count. TTL = 60s.

Exact Phrases That Signal Senior-Level Thinking

1
When introducing cache
"I'd add a Redis cache in front of the user service here. User profile reads are happening on every API request — that's easily 100K QPS at scale — and profiles change rarely, so a 5-minute TTL gives us a high hit rate with acceptable staleness."
2
When explaining invalidation
"I'd use cache-aside with TTL-based expiration. When a user updates their profile, I'd proactively delete the cache key rather than wait for TTL to expire — this keeps it fresh while keeping the architecture simple. I'd avoid write-through here because the extra write latency isn't worth it for profile updates."
3
When discussing failure modes
"Two failure modes I'd design against: cache stampede on popular keys — I'd use a mutex lock or probabilistic early refresh — and cache avalanche on restart, which I'd handle by randomizing TTLs by ±20% and pre-warming critical keys before opening traffic."
4
When discussing Redis specifically
"For the leaderboard I'd use Redis Sorted Set — members are automatically kept sorted by score, top-N queries are O(log N), and score increments are atomic. That's a better fit than re-sorting the DB on every read. Single Redis instance handles ~100K ops/sec, so this scales well into the mid-tiers."
5
When discussing trade-offs (always score points here)
"The trade-off with caching is complexity: we now have two sources of truth. We accept that reads might be up to 5 minutes stale — that's fine for profiles but not for inventory counts. For anything that needs strong consistency like payment status or stock levels, I wouldn't cache or I'd use a very short TTL with explicit invalidation on write."

Questions the Interviewer Will Ask — and What They Want to Hear

How do you handle cache invalidation when data changes?

They want to see that you know it's hard and have a strategy. Say: "It depends on consistency requirements. For most data I use TTL + proactive delete on write (cache-aside pattern). For strictly consistent data like inventory, I use write-through or skip caching entirely. The key insight is that there's no perfect solution — it's always a trade-off between freshness, complexity, and performance."

What happens if your Redis cache goes down?

They want to see graceful degradation. Say: "All traffic falls through to the database — it's degraded performance, not an outage. The DB handles the load at increased latency. I'd handle this with: (1) Redis Sentinel for auto-failover to a replica, (2) a circuit breaker that detects Redis failure and temporarily bypasses cache instead of hammering a dead Redis, (3) rate limiting on the DB to prevent it from being overwhelmed during the cache-cold period."

How do you prevent one user's data from being served to another user?

Key namespacing. Say: "Always prefix cache keys with the entity type and ID: user:42:profile, not just profile. For sensitive data, never use shared keys. I'd also ensure the application layer validates that the cached data belongs to the requesting user before returning it — defense in depth."

Would you use Redis or Memcached? Why?

Say: "Redis in almost all cases. Memcached is slightly faster for simple string k-v operations because it's a simpler system, but Redis gives you persistence options, rich data structures (sorted sets, hashes, pub/sub), atomic operations, and Lua scripting. Unless you're doing pure string caching at extreme scale and every nanosecond matters, Redis's feature richness wins."

🔥
Senior Engineer Perspective: The Real Goal of Caching

After years of production systems, here's what I want you to internalize: caching is not about making your system faster. It's about making your database survivable at scale. Without caching, every user request reaches the DB, and at 100K users the DB becomes your single point of failure. Caching is the thing that makes your system resilient. Get this right and your system scales. Get it wrong and you're up at 3am with an exploding database.

📋 Quick Reference: Caching Decision Checklist

  • ✅ Identified high-read, low-write data
  • ✅ Chosen cache-aside as default strategy
  • ✅ Set TTL with ±20% jitter (avalanche prevention)
  • ✅ Namespaced all cache keys (user:42:profile)
  • ✅ Addressed stampede (mutex or background refresh)
  • ✅ Addressed penetration (Bloom filter or cache-null)
  • ✅ Defined graceful degradation if Redis goes down
  • ✅ Identified data that should NEVER be cached
  • ✅ Planned cache warm-up for cold start
  • ✅ Discussed hit rate monitoring (>95% target)
