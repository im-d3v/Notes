🏛️ DAO & DTO — What They Are and Why They Exist

DAO — Data Access Object

A design pattern that isolates all database operations behind an interface. Your business logic never speaks directly to a database — it speaks to a DAO. The DAO translates that into SQL/JPA queries.

Controller ──► Service ──► DAO Interface │ ┌───────────────┤ ▼ ▼ JdbcUserDAO JpaUserDAO (raw SQL) (EntityManager) │ │ └───────┬───────┘ ▼ Database
💡
Why DAO matters: If tomorrow you switch from MySQL to MongoDB, only the DAO implementation changes — zero changes to Service or Controller. The interface is the contract.

DAO vs Repository — What's the Difference?

DAORepository (Spring Data)
OriginClassic J2EE patternDomain-Driven Design (DDD)
FocusDatabase table / persistence technologyDomain aggregate (business concept)
MethodsfindById, insert, update, deletefindByEmailAndActive, findTopByOrderByCreatedDesc
Speaks inDatabase terms (rows, tables)Business terms (User, Order)
SpringManual implementationInterface only — Spring generates impl
JavaTraditional DAO vs Spring Repository
// ── Old-school DAO (manual implementation) ──────────────────
public interface UserDAO {
    User findById(Long id);
    List<User> findAll();
    void save(User user);
    void delete(Long id);
}

@Repository
public class JpaUserDAO implements UserDAO {
    @PersistenceContext
    private EntityManager em;

    public User findById(Long id) { return em.find(User.class, id); }
}

// ── Modern Spring Data Repository (no implementation needed) ─
public interface UserRepository extends JpaRepository<User, Long> {
    // Spring generates ALL implementation at runtime
    Optional<User> findByEmail(String email);
    List<User> findByActiveTrue();
}

DTO — Data Transfer Object

A plain object used to carry data between layers (Controller ↔ Service, or Service ↔ DB). It is not an entity — it has no JPA annotations, no database identity, no Hibernate proxy magic.

⚠️
Never expose your Entity directly from a REST API.
Entities may contain sensitive fields (passwords, internal IDs), lazy collections that trigger queries when serialized, bidirectional references that cause infinite JSON loops, and @Version fields that leak internal state.
JavaEntity vs DTO — practical example
// ── Entity (database representation) ────────────────────────
@Entity
public class User {
    @Id @GeneratedValue private Long id;
    private String email;
    private String passwordHash;    // NEVER expose this!
    private String internalCode;    // business-sensitive
    @Version private Long version;
    @OneToMany(mappedBy = "user", fetch = FetchType.LAZY)
    private List<Order> orders;     // lazy — triggers query if serialized!
}

// ── Response DTO (what the API sends back) ───────────────────
public record UserResponse(Long id, String email, String displayName) {}

// ── Request DTO (what the API receives) ─────────────────────
public record CreateUserRequest(
    @NotBlank String email,
    @Size(min = 8) String password,
    String displayName
) {}

// ── Mapper (Entity ↔ DTO conversion) ────────────────────────
@Component
public class UserMapper {
    public UserResponse toResponse(User user) {
        return new UserResponse(user.getId(), user.getEmail(), user.getDisplayName());
    }
    public User toEntity(CreateUserRequest req) {
        User u = new User();
        u.setEmail(req.email());
        u.setPasswordHash(passwordEncoder.encode(req.password()));
        return u;
    }
}

Flow: Request → Response (full picture)

HTTP POST /users JSON body │ ▼ @RestController │ @RequestBody → deserialize JSON → CreateUserRequest (DTO) │ validates DTO (@Valid) │ ▼ @Service │ mapper.toEntity(dto) → User (Entity) │ calls repository.save(entity) │ ▼ @Repository (JpaRepository) │ EntityManager.persist(entity) │ Hibernate generates INSERT SQL │ ▼ Database — row inserted ── Response path ────────────────────────────── Database returns generated ID ▼ Entity with id populated ▼ Service mapper.toResponse(entity) → UserResponse (DTO) ▼ Controller ResponseEntity.created(location).body(userResponse) ▼ HTTP 201 JSON body

🔌 The Journey: Raw SQL → JDBC → ORM

Level 0: Raw JDBC — The Pain

Before any abstraction, Java talks to databases via JDBC (Java Database Connectivity) — a low-level API from 1997. Here's what saving a simple User looked like:

JavaRaw JDBC — every line is boilerplate you write by hand
public User saveUser(User user) throws SQLException {
    Connection conn = null;
    PreparedStatement ps = null;
    ResultSet rs = null;
    try {
        // 1. Get connection from driver (no pooling!)
        conn = DriverManager.getConnection(
                "jdbc:mysql://localhost/mydb", "root", "pass");

        // 2. Write SQL by hand — typo? Boom at runtime
        ps = conn.prepareStatement(
                "INSERT INTO users (email, name) VALUES (?, ?)",
                Statement.RETURN_GENERATED_KEYS);

        // 3. Map object fields → SQL params manually
        ps.setString(1, user.getEmail());
        ps.setString(2, user.getName());

        ps.executeUpdate();

        // 4. Extract generated key manually
        rs = ps.getGeneratedKeys();
        if (rs.next()) user.setId(rs.getLong(1));
        return user;

    } catch (SQLException e) {
        // 5. Catch checked exception on every operation
        conn.rollback(); throw e;
    } finally {
        // 6. Close EVERYTHING manually or connection leak!
        if (rs   != null) rs.close();
        if (ps   != null) ps.close();
        if (conn != null) conn.close();
    }
}
// This is just ONE method. Multiply by every table × every operation.
🚫
Problems with raw JDBC:
1. Massive boilerplate — connection, statement, resultset, close, catch
2. No object mapping — you manually map every column to every field
3. SQL mixed into Java code — not type-safe, refactoring nightmares
4. No connection pooling — DriverManager.getConnection creates new connection every time
5. Vendor-specific SQL — switch DB = rewrite queries

How JDBC Works Internally

Your Java Code │ calls java.sql.* interfaces ▼ JDBC API (java.sql.Connection, Statement, ResultSet) │ abstract interfaces — no DB-specific code here ▼ JDBC Driver (e.g., mysql-connector-j-8.x.jar) │ vendor-specific implementation of JDBC interfaces │ translates Java calls → native DB protocol packets ▼ MySQL / PostgreSQL / Oracle (TCP socket connection)

JDBC Driver Types

Type 4 (Pure Java) — most common today. Communicates directly over TCP using DB's native wire protocol. No native code, no OS dependency. Example: mysql-connector-java.

DataSource vs DriverManager

DriverManager = creates new connection each call (bad).
DataSource = interface for connection pooling. Spring Boot uses HikariCP as the default DataSource implementation.

🧰 JdbcTemplate — Spring's JDBC Abstraction

What JdbcTemplate Does

JdbcTemplate is a Spring class that handles all the boilerplate JDBC code (getting connection, handling exceptions, closing resources). You only write the SQL and the row mapping. It sits below JPA/Hibernate but above raw JDBC.

JdbcTemplate │ handles: connection acquire/release, exception translation, │ PreparedStatement creation, ResultSet closing ▼ DataSource (HikariCP pool) ▼ Database
JavaJdbcTemplate — same operations, 80% less code
@Repository
public class UserJdbcRepository {

    private final JdbcTemplate jdbc;

    public UserJdbcRepository(JdbcTemplate jdbc) { this.jdbc = jdbc; }

    // ── Query single row → object ────────────────────────────
    public Optional<User> findById(Long id) {
        try {
            return Optional.ofNullable(
                jdbc.queryForObject(
                    "SELECT * FROM users WHERE id = ?",
                    this::mapRow,   // RowMapper — maps ResultSet → User
                    id
                )
            );
        } catch (EmptyResultDataAccessException e) {
            return Optional.empty();
        }
    }

    // ── Query multiple rows ──────────────────────────────────
    public List<User> findByActive(boolean active) {
        return jdbc.query(
            "SELECT * FROM users WHERE active = ?",
            this::mapRow,
            active
        );
    }

    // ── Insert with generated key ────────────────────────────
    public Long insert(User user) {
        KeyHolder keyHolder = new GeneratedKeyHolder();
        jdbc.update(conn -> {
            PreparedStatement ps = conn.prepareStatement(
                "INSERT INTO users (email, name) VALUES (?, ?)",
                Statement.RETURN_GENERATED_KEYS);
            ps.setString(1, user.getEmail());
            ps.setString(2, user.getName());
            return ps;
        }, keyHolder);
        return keyHolder.getKey().longValue();
    }

    // ── RowMapper — maps one DB row → one Java object ────────
    private User mapRow(ResultSet rs, int rowNum) throws SQLException {
        User u = new User();
        u.setId(rs.getLong("id"));
        u.setEmail(rs.getString("email"));
        u.setName(rs.getString("name"));
        return u;
    }
}

JPA Architecture

The specification, the implementation, the persistence unit, and the object graph that makes it all work.

🏗️ JPA Architecture — The Full Picture

JPA Is a Specification, Not an Implementation

JPA (Jakarta Persistence API) defines the interfaces and rules — @Entity, EntityManager, JPQL. It does NOT execute queries. That's done by an ORM provider that implements the spec.

Your Code uses only JPA annotations and interfaces │ @Entity EntityManager @Query @Id ▼ JPA API (jakarta.persistence.*) │ interfaces: EntityManagerFactory, EntityManager, │ EntityTransaction, Query, TypedQuery ▼ JPA Provider / ORM Implementation ├──► Hibernate (default in Spring Boot) ├──► EclipseLink (reference implementation) └──► OpenJPA │ ▼ JDBC Layer ▼ Database

All JPA Components and Relationships

ComponentWhat It IsScope
PersistenceBootstrap class — reads persistence.xml / Spring config, creates EntityManagerFactoryApplication startup, once
EntityManagerFactory (EMF)Factory for EntityManagers. Holds the DB connection pool and metadata about all entities.Application-scoped singleton
EntityManager (EM)The main API — manages entities, runs queries, controls transactionsShort-lived (per request / per transaction)
EntityTransactionControls begin/commit/rollback for resource-local (non-JTA) transactionsPer EntityManager
PersistenceContextFirst-level cache managed by EntityManager — all currently managed entities live hereTied to EntityManager lifecycle
Query / TypedQueryJPQL or native SQL queries built from EntityManagerPer query execution

📄 persistence.xml — The Original JPA Config

What Is persistence.xml and Why It Exists

Before Spring Boot, all JPA configuration was in META-INF/persistence.xml. It defines a Persistence Unit — a named configuration block specifying which entities belong together and how to connect to the database.

Spring Boot replaces persistence.xml entirely with auto-configuration + application.properties. But you may see it in legacy projects or non-Spring JPA apps.

XMLpersistence.xml — classic JPA config (legacy/standalone)
<persistence xmlns="https://jakarta.ee/xml/ns/persistence"
             version="3.0">

    <!-- A persistence unit = one database connection + its entities -->
    <persistence-unit name="myAppPU" transaction-type="RESOURCE_LOCAL">

        <!-- The JPA provider (implementation class) -->
        <provider>org.hibernate.jpa.HibernatePersistenceProvider</provider>

        <!-- List ALL entity classes (or use <exclude-unlisted-classes>false</exclude-unlisted-classes>) -->
        <class>com.app.entity.User</class>
        <class>com.app.entity.Order</class>

        <properties>
            <!-- JDBC connection -->
            <property name="jakarta.persistence.jdbc.driver"   value="org.postgresql.Driver"/>
            <property name="jakarta.persistence.jdbc.url"      value="jdbc:postgresql://localhost/mydb"/>
            <property name="jakarta.persistence.jdbc.user"     value="myuser"/>
            <property name="jakarta.persistence.jdbc.password"  value="secret"/>

            <!-- Hibernate-specific -->
            <property name="hibernate.dialect"    value="org.hibernate.dialect.PostgreSQLDialect"/>
            <property name="hibernate.hbm2ddl.auto" value="validate"/>
            <property name="hibernate.show_sql"   value="true"/>
        </properties>

    </persistence-unit>
</persistence>
Using It in Standalone Java (No Spring)
JavaManual bootstrap without Spring
// Persistence.createEntityManagerFactory reads META-INF/persistence.xml
EntityManagerFactory emf =
    Persistence.createEntityManagerFactory("myAppPU");

EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();

try {
    tx.begin();
    em.persist(new User("alice@example.com"));
    tx.commit();
} catch (Exception e) {
    tx.rollback(); throw e;
} finally {
    em.close();        // always close EM
}
emf.close();           // close when app shuts down
✅
In Spring Boot: All of this is automatic. Spring Boot reads application.properties, creates the EMF as a singleton bean, injects EntityManager via @PersistenceContext, and wraps transactions with @Transactional. You never write this code manually.

🔤 JPA Vendor & Dialect

JpaVendorAdapter — Bridges Spring to the ORM Provider

In Spring, JpaVendorAdapter is an interface that configures the JPA provider (Hibernate, EclipseLink) into a Spring-aware setup. It sets the dialect, show_sql, etc. Spring Boot auto-configures HibernateJpaVendorAdapter automatically.

JavaManual JpaVendorAdapter (Spring Boot does this for you)
@Bean
public LocalContainerEntityManagerFactoryBean entityManagerFactory(
        DataSource dataSource) {

    LocalContainerEntityManagerFactoryBean em =
        new LocalContainerEntityManagerFactoryBean();

    em.setDataSource(dataSource);
    em.setPackagesToScan("com.app.entity");  // scan for @Entity

    // Tell Spring to use Hibernate as the JPA provider
    HibernateJpaVendorAdapter vendorAdapter = new HibernateJpaVendorAdapter();
    vendorAdapter.setShowSql(true);
    vendorAdapter.setGenerateDdl(false);     // use Flyway instead
    em.setJpaVendorAdapter(vendorAdapter);

    Properties props = new Properties();
    props.setProperty("hibernate.dialect",
        "org.hibernate.dialect.PostgreSQLDialect");
    props.setProperty("hibernate.jdbc.batch_size", "50");
    em.setJpaProperties(props);

    return em;
}

Dialect — What It Is and Why It Matters

A Dialect is a Hibernate class that knows the SQL syntax quirks of a specific database. SQL is not truly standard — every DB has different syntax for things like sequences, data types, pagination, and locking.

JavaWhat dialect generates differently per DB
// LIMIT/OFFSET pagination — different SQL per DB:
// MySQL Dialect:
//   SELECT * FROM orders LIMIT 20 OFFSET 40

// Oracle Dialect:

//   WHERE rn BETWEEN 41 AND 60

// PostgreSQL Dialect:
//   SELECT * FROM orders LIMIT 20 OFFSET 40

// SQL Server Dialect:
//   SELECT * FROM orders ORDER BY id OFFSET 40 ROWS FETCH NEXT 20 ROWS ONLY

// Your JPQL (database-agnostic):
// "SELECT o FROM Order o" + Pageable.of(2, 20)

⛓️ Persistence → EMF → EntityManager — The Full Chain

Every Layer Explained

1
Persistence (Bootstrap)
javax.persistence.Persistence is the entry point. It reads persistence.xml (or Spring's config), loads the JPA provider, and calls createEntityManagerFactory(). This happens ONCE at application startup.
2
EntityManagerFactory — Heavy, Shared, Singleton
Created once. Holds: entity metadata (class mappings, column names, associations), the JDBC connection pool (DataSource), second-level cache (optional), SQL dialect knowledge. Very expensive to create — never create per-request.
3
EntityManager — Lightweight, Per-Transaction
Created from EMF via emf.createEntityManager(). Manages the Persistence Context (first-level cache). Must be closed after each unit of work. In Spring, this is handled automatically — @PersistenceContext injects a transaction-scoped proxy.
4
EntityTransaction — Manual or Managed
Controls begin(), commit(), rollback(). In plain JPA you manage it. In Spring with @Transactional, Spring's JpaTransactionManager manages it for you via AOP proxy.
JavaEntityManager API — core operations
@Service
public class UserService {

    @PersistenceContext          // Injects a thread-safe proxy (not raw EM)
    private EntityManager em;   // The proxy delegates to the current tx's EM

    public void demonstrate() {

        // ── persist() — INSERT new entity ──────────────────────
        User user = new User("alice@example.com");
        em.persist(user);    // state: TRANSIENT → MANAGED
                              // SQL INSERT happens at flush (not yet!)

        // ── find() — SELECT by primary key ─────────────────────
        User found = em.find(User.class, 1L);  // checks PC cache first!

        // ── merge() — UPDATE detached entity ───────────────────
        user.setName("Alice");
        User managed = em.merge(user);  // returns new managed instance

        // ── remove() — DELETE entity ───────────────────────────
        em.remove(em.find(User.class, 2L));  // must be managed to remove

        // ── createQuery() — JPQL query ─────────────────────────
        List<User> users = em
            .createQuery("SELECT u FROM User u WHERE u.active = true", User.class)
            .getResultList();

        // ── flush() — sync PC to DB (before commit) ────────────
        em.flush();  // executes pending INSERTs/UPDATEs/DELETEs NOW

        // ── detach() — remove from PC (stop tracking) ──────────
        em.detach(found);   // found is now DETACHED — changes not tracked

        // ── clear() — detach ALL entities from PC ──────────────
        em.clear();   // use in batch processing to free memory

        // ── refresh() — reload from DB (discard local changes) ─
        em.refresh(managed);  // re-read from DB — overwrites in-memory changes
    }
}

@PersistenceContext — Thread Safety Trick

A raw EntityManager is NOT thread-safe. Spring injects a proxy that, on every method call, looks up the correct EntityManager for the current transaction from a ThreadLocal. This is why you can safely inject it as a field into a singleton service bean.

Thread 1 (Request A) → calls em.find() Thread 2 (Request B) → calls em.find() │ Both call same proxy object ──► proxy checks ThreadLocal │ ┌───────────────────┤ ▼ ▼ EM for Tx A EM for Tx B (own PC cache) (own PC cache)

Spring Data JPA — Repository Hierarchy

From raw interface to generated implementation — the complete inheritance chain and what each level adds.

🏛️ Repository → CrudRepository → JpaRepository

The Full Hierarchy

Repository<T, ID> (marker interface — no methods) │ just tells Spring Data "this is a repository" ▼ CrudRepository<T, ID> │ save(S), saveAll(Iterable), findById(ID), existsById(ID), │ findAll(), findAllById(Iterable), count(), deleteById(ID), │ delete(T), deleteAllById(Iterable), deleteAll() ▼ PagingAndSortingRepository<T, ID> │ findAll(Sort) ← return all sorted │ findAll(Pageable) ← return one page ▼ JpaRepository<T, ID> (JPA-specific additions) │ getById(ID) ← returns proxy (no DB hit until accessed) │ findAll(Example<S>) ← query by example │ flush() ← sync persistence context to DB │ saveAndFlush(T) ← save + immediate flush │ deleteAllInBatch() ← single DELETE without loading entities │ getReferenceById(ID) ← like getById (JPA 3 name) ▼ SimpleJpaRepository<T, ID> │ THE ACTUAL IMPLEMENTATION — Spring generates a bean of this class │ for every interface extending JpaRepository

What Each Level Adds — With Code

JavaBuilding your own repository — what you get free
// Your interface — you write ZERO implementation
public interface ProductRepository extends JpaRepository<Product, Long> {

    // ── Inherited from CrudRepository ───────────────────────
    // save(product), findById(1L), deleteById(1L), count() etc.

    // ── Inherited from PagingAndSortingRepository ────────────
    // findAll(PageRequest.of(0, 20, Sort.by("name")))  → Page<Product>

    // ── Derived query methods (you name, Spring generates SQL) ─
    List<Product> findByCategory(String category);
    List<Product> findByCategoryAndActiveTrueOrderByNameAsc(String category);
    Optional<Product> findBySkuIgnoreCase(String sku);
    long countByCategoryAndActiveTrue(String category);
    boolean existsBySku(String sku);

    // ── Custom JPQL ─────────────────────────────────────────
    @Query("SELECT p FROM Product p WHERE p.price BETWEEN :min AND :max")
    List<Product> findByPriceRange(@Param("min") BigDecimal min,
                                   @Param("max") BigDecimal max);

    // ── Modifying query ─────────────────────────────────────
    @Modifying @Transactional
    @Query("UPDATE Product p SET p.active = false WHERE p.category = :cat")
    int deactivateByCategory(@Param("cat") String category);

    // ── Projection (fetch only needed fields → faster) ───────

}

// Projection interface

    Long getId();
    String getName();
    BigDecimal getPrice();

🔍 SimpleJpaRepository — What Spring Generates

How Spring Data Creates Implementations at Runtime

1
@EnableJpaRepositories scans for repository interfaces
Finds all interfaces extending Repository in the configured packages.
2
JpaRepositoryFactory creates proxy
For each interface, a JDK dynamic proxy is created backed by a SimpleJpaRepository instance. The proxy delegates all calls to it.
3
Method name parsing for derived queries
PartTreeJpaQuery parses findByEmailAndActiveTrue → builds JPQL → creates TypedQuery. All at startup time.
4
Bean registered in ApplicationContext
The proxy bean is registered and can be injected via @Autowired anywhere.
JavaSimpleJpaRepository — simplified source (what Spring generates for you)
// Spring Data's actual implementation class (simplified)
public class SimpleJpaRepository<T, ID>
        implements JpaRepositoryImplementation<T, ID> {

    private final EntityManager em;
    private final JpaEntityInformation<T, ?> entityInformation;

    // findById — checks persistence context cache first
    @Transactional(readOnly = true)
    public Optional<T> findById(ID id) {
        return Optional.ofNullable(em.find(entityInformation.getJavaType(), id));
    }

    // save — decides persist (new) vs merge (existing)
    @Transactional
    public <S extends T> S save(S entity) {
        if (entityInformation.isNew(entity)) {
            em.persist(entity);    // INSERT
            return entity;
        } else {
            return em.merge(entity); // SELECT + UPDATE
        }
    }

    // deleteById — loads entity first, then removes
    @Transactional
    public void deleteById(ID id) {
        findById(id).ifPresent(this::delete);
    }

    // findAll — full table scan (use with Pageable in prod!)
    @Transactional(readOnly = true)
    public List<T> findAll() {
        return em.createQuery("SELECT e FROM " + entityName + " e", domainClass)
                 .getResultList();
    }
}
⚠️
save() and isNew(): How does Spring know if an entity is new or existing? By checking if the @Id field is null/zero (for auto-generated IDs). If your entity uses a manually assigned ID (e.g., UUID set in constructor), implement Persistable<ID> and override isNew() to avoid a SELECT before every INSERT.

Entity Lifecycle & Persistence Context

The four states every JPA entity passes through and how the first-level cache works.

🔄 Entity States — Transient, Managed, Detached, Removed

The Four States and Transitions

new User() │ object exists in JVM heap │ no DB row, not tracked by JPA ▼ TRANSIENT │ │ │ em.persist() │ (object GC'd without persist → lost) ▼ │ MANAGED ◄────────────────────────────┐ │ tracked by Persistence Context │ │ changes auto-detected (dirty) │ em.merge(detached) │ SQL sent on flush/commit │ │ │ ├─────── em.detach() / em.clear() ──────────────► DETACHED │ tx ends (EXTENDED scope) │ │ session closes │ changes NOT tracked │ │ can re-attach via merge() │ ├─────── em.remove() ──────────────────────────► REMOVED │ │ DELETE on flush/commit ▼ (gone from DB)
JavaWalking through all four states
// ── 1. TRANSIENT — new object, unknown to JPA ─────────────────
User user = new User("alice@example.com");
// user.id = null, not tracked, no SQL sent

// ── 2. MANAGED — persisted into the current Persistence Context ─
em.persist(user);
// user is now managed. JPA tracks every field change.
// SQL INSERT is NOT sent yet — queued until flush

user.setName("Alice Wonderland");
// Dirty checking detects this change — UPDATE will be sent on flush

em.flush();
// NOW: INSERT INTO users... then UPDATE users SET name=... WHERE id=...

// ── 3. DETACHED — disconnected from Persistence Context ─────────
em.detach(user);      // or em.clear() / tx ends
user.setName("Different");
// Change is NOT tracked — no SQL, no dirty checking

// Re-attach via merge
User managedUser = em.merge(user);
// SELECT first (to load latest DB state)
// Then UPDATE with merged values → managedUser is a NEW managed instance

// ── 4. REMOVED ───────────────────────────────────────────────────
em.remove(managedUser);
// Scheduled for DELETE. SQL runs on flush/commit.
em.persist(managedUser); // can undo remove before flush!

📦 Persistence Context — First-Level Cache

What the Persistence Context Is

The Persistence Context (PC) is Hibernate's in-memory identity map. It's essentially a Map<EntityType, Map<PrimaryKey, Entity>>. Every entity you load, persist, or query goes into this map. It provides:

Identity Guarantee

Two calls to em.find(User.class, 1L) within the same transaction return the same Java object instance — second call hits the cache, not the DB.

Change Tracking

Hibernate takes a snapshot of every managed entity. On flush, it compares current state to snapshot — changed fields generate UPDATE statements automatically.

Write-Behind

Changes are not sent to DB immediately. Hibernate batches them and sends on flush (before query, before commit, or explicit em.flush()).

Transactional Scope

Default: PC lives for one transaction. EXTENDED scope: PC lives across multiple transactions (stateful session beans — rare in Spring).

JavaPersistence Context caching in action
@Transactional
public void demonstratePC() {

    // Call 1: hits database → loads User → stores in PC cache
    User u1 = em.find(User.class, 1L);   // SELECT executed

    // Call 2: SAME transaction → returns cached object, NO SQL
    User u2 = em.find(User.class, 1L);   // cache hit!

    System.out.println(u1 == u2);           // true — same object!

    // Change tracking — no explicit save() needed
    u1.setName("Updated");
    // At transaction commit, Hibernate compares to snapshot
    // → UPDATE users SET name='Updated' WHERE id=1
    // You never call save() for this!
}

// ── Second-Level Cache (optional, not PC) ────────────────────────
// PC is cleared after each transaction.
// 2nd level cache (e.g., Ehcache, Redis) persists across transactions.
// Enable with: @Cache(usage = CacheConcurrencyStrategy.READ_WRITE) on entity

🔎 Dirty Checking — How Hibernate Detects Changes

The Mechanism

When an entity enters the Persistence Context, Hibernate stores a deep copy (snapshot) of its state. On flush, it compares every managed entity's current state to its snapshot. Any difference = "dirty" = generates an UPDATE. This happens without you calling any save method.

1
em.find() or query loads entity
Hibernate stores entity in PC + takes a snapshot copy of all fields.
2
Your code modifies entity fields
Normal Java setter calls — no JPA API involved.
3
Flush triggered (before query / before commit)
Hibernate iterates all managed entities, compares to snapshots using equals() / deep comparison.
4
Dirty entities generate UPDATE SQL
Only changed columns are included (with @DynamicUpdate) or all columns (default behavior).
💡
Performance tip: @Transactional(readOnly = true) tells Hibernate to skip dirty checking entirely — no snapshots taken. Use this on all read-only service methods for better performance.

Entity Relationships & Mappings

One-to-One, One-to-Many, Many-to-Many, ownership, JSON serialization traps, and how to fix infinite recursion.

🗺️ Mapping Types — Overview

The Four Relationship Types

AnnotationMeaningDB StructureExample
@OneToOne One entity relates to exactly one other FK in either table User ↔ UserProfile
@OneToMany One entity relates to many others FK in "many" side table Order → OrderItems
@ManyToOne Many entities relate to one other FK in "many" side table (same FK as OneToMany) OrderItem → Order
@ManyToMany Many entities relate to many others Join table with two FKs Product ↔ Category

1️⃣ @OneToOne Mapping

User ↔ UserProfile

Java@OneToOne — shared primary key vs FK approach
// ── Approach 1: FK in UserProfile table (most common) ────────
@Entity
public class User {
    @Id @GeneratedValue private Long id;
    private String email;

    @OneToOne(mappedBy = "user",           // UserProfile owns the FK
              cascade = CascadeType.ALL,
              fetch = FetchType.LAZY,       // ALWAYS lazy for @OneToOne!
              orphanRemoval = true)
    private UserProfile profile;
}

@Entity
public class UserProfile {
    @Id @GeneratedValue private Long id;
    private String bio;
    private String avatarUrl;

    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id", unique = true) // FK col with UNIQUE constraint
    private User user;
}

// ── Approach 2: Shared Primary Key (profile reuses user PK) ──
@Entity
public class UserProfile {
    @Id
    private Long id;         // same as User.id — no @GeneratedValue!

    @OneToOne(fetch = FetchType.LAZY)
    @MapsId                  // maps this entity's PK to the FK
    @JoinColumn(name = "user_id")
    private User user;
}
⚠️
@OneToOne Lazy Loading Gotcha: Even with fetch = LAZY, Hibernate may still do an eager SELECT for @OneToOne on the non-owning (mappedBy) side. This is because Hibernate needs to know whether to set the field to null or a proxy. Use @MapsId shared key approach or Bytecode Enhancement to truly lazy-load OneToOne.

📋 @OneToMany / @ManyToOne — Parent-Child

Order → OrderItems (the most common mapping)

JavaCorrect bidirectional OneToMany setup
@Entity
public class Order {
    @Id @GeneratedValue private Long id;

    // ── @OneToMany (inverse / parent side) ───────────────────
    @OneToMany(
        mappedBy   = "order",           // "order" = field name in OrderItem
        cascade    = CascadeType.ALL,   // persist/remove items with order
        orphanRemoval = true,           // remove from list = DELETE from DB
        fetch      = FetchType.LAZY     // ALWAYS lazy for collections!
    )
    private List<OrderItem> items = new ArrayList<>();

    // ── Synchronization helpers (CRITICAL for bidirectional) ──
    public void addItem(OrderItem item) {
        items.add(item);
        item.setOrder(this);      // keep both sides in sync!
    }

    public void removeItem(OrderItem item) {
        items.remove(item);
        item.setOrder(null);     // orphanRemoval will delete it
    }
}

@Entity
public class OrderItem {
    @Id @GeneratedValue private Long id;

    // ── @ManyToOne (owning / FK owner side) ───────────────────
    @ManyToOne(fetch = FetchType.LAZY)   // lazy: don't load Order unless needed
    @JoinColumn(
        name       = "order_id",          // actual FK column name in DB
        nullable   = false
    )
    private Order order;               // THIS SIDE OWNS THE FOREIGN KEY

    private int quantity;
    private BigDecimal price;
}
💡
Rule of thumb for ownership: The entity with the @JoinColumn (the FK column in its table) is the owning side. The entity with mappedBy is the inverse side. Hibernate only looks at the owning side to write the FK value. Always keep both sides in sync manually.

🔀 @ManyToMany Mapping

Student ↔ Course (classic many-to-many)

Java@ManyToMany with join table
@Entity
public class Student {
    @Id @GeneratedValue private Long id;
    private String name;

    // ── Owning side — defines the join table ─────────────────
    @ManyToMany(cascade = { CascadeType.PERSIST, CascadeType.MERGE })
    @JoinTable(
        name               = "student_course",     // join table name
        joinColumns        = @JoinColumn(name = "student_id"),
        inverseJoinColumns = @JoinColumn(name = "course_id")
    )
    private Set<Course> courses = new HashSet<>(); // Set not List!
}

@Entity
public class Course {
    @Id @GeneratedValue private Long id;
    private String title;

    // ── Inverse side — no join table definition ───────────────
    @ManyToMany(mappedBy = "courses")         // "courses" = field in Student
    private Set<Student> students = new HashSet<>();
}
⚠️
Never use CascadeType.ALL or orphanRemoval on @ManyToMany. Deleting a Student could cascade to delete Courses — affecting all other students enrolled! Only cascade PERSIST and MERGE.
When You Need Extra Data on the Join — Use @Embeddable
JavaMany-to-many with enrollment date — explicit join entity
// When you need data ON the relationship (enrollment date, grade),
// model it as a proper entity instead of a @ManyToMany

@Entity
public class Enrollment {

    @EmbeddedId
    private EnrollmentId id;

    @ManyToOne @MapsId("studentId") private Student student;
    @ManyToOne @MapsId("courseId")  private Course course;

    private LocalDate enrolledAt;   // extra data on the join
    private String grade;
}

@Embeddable
public class EnrollmentId implements Serializable {
    private Long studentId;
    private Long courseId;
    // equals + hashCode required
}

↔️ Unidirectional vs Bidirectional

The Core Difference

➡️ Unidirectional

  • Navigation in ONE direction only
  • Only ONE class has the reference field
  • Simpler — no sync helper methods needed
  • Best for: when you never need to navigate back
Java
// OrderItem → Product (one way only)
@Entity
public class OrderItem {
    @ManyToOne
    @JoinColumn(name = "product_id")
    private Product product;  // ✅
}

// Product has NO reference to OrderItem
// Product doesn't know who references it

↔️ Bidirectional

  • Navigation in BOTH directions
  • Both classes have reference fields
  • One side owns the FK (@JoinColumn)
  • Other side uses mappedBy
  • Requires manual sync helpers
Java
@Entity
public class Order {
    @OneToMany(mappedBy = "order")
    private List<OrderItem> items; // ✅
}
@Entity
public class OrderItem {
    @ManyToOne
    @JoinColumn(name = "order_id")
    private Order order;        // ✅ FK owner
}

♾️ Infinite Recursion Problem & Solutions

Why It Happens

When you have a bidirectional relationship and Jackson serializes the entities to JSON, it follows every reference. Order → OrderItems → Order → OrderItems → … until StackOverflowError.

Jackson serializes Order │ → serializes items: [...] │ └─► serializes OrderItem │ └─► serializes order: ← CYCLES BACK! │ └─► serializes items: [...] │ └─► ... ▼ java.lang.StackOverflowError

Solution 1: @JsonIgnore (simplest)

Tells Jackson to completely ignore this field during serialization. The relationship still works in Java — just not serialized to JSON.

Java@JsonIgnore — breaks the cycle by ignoring one side
@Entity
public class OrderItem {
    @Id @GeneratedValue private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    @JsonIgnore                      // ← stop Jackson here
    private Order order;            // not included in JSON output

    private int quantity;
}

// JSON result:
// { "id": 1, "quantity": 5 }  ← order field absent
// Pro: simple. Con: you can never include "order" in this DTO.
// Best practice: use DTOs instead and don't annotate entities with Jackson.

Solution 2: @JsonManagedReference + @JsonBackReference

A pair of annotations. Managed = serialized. Back = not serialized but deserialized back. The "managed" side always wins for output; the "back" side is the back-reference.

Java@JsonManagedReference + @JsonBackReference
@Entity
public class Order {
    @Id @GeneratedValue private Long id;

    @OneToMany(mappedBy = "order")
    @JsonManagedReference            // ← "I am the parent — serialize items"
    private List<OrderItem> items;
}

@Entity
public class OrderItem {
    @Id @GeneratedValue private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    @JsonBackReference              // ← "I am the child — DON'T serialize order"
    private Order order;
}

// JSON: Order → { "id":1, "items": [{"id":1,"quantity":5}, ...] }
//                               └── items are serialized ✅
//                               └── items don't have "order" field ✅ (no loop)

Solution 3: @JsonIdentityInfo (object references by ID)

Jackson serializes the first occurrence fully, then replaces subsequent occurrences with just the ID. Breaks cycles without losing data.

Java@JsonIdentityInfo — reference by ID on repetition
@Entity
@JsonIdentityInfo(
    generator = ObjectIdGenerators.PropertyGenerator.class,
    property  = "id"                // use "id" field as identity
)
public class Order {
    @Id @GeneratedValue private Long id;

    @OneToMany(mappedBy = "order")
    private List<OrderItem> items;
}

// JSON output:
// { "id": 1, "items": [
//     { "id": 10, "order": 1, "quantity": 5 },  ← order is just the ID (1)
//     { "id": 11, "order": 1, "quantity": 2 }
// ]}
// "order": 1 → reference to previously seen object with id=1 (no recursion)

Solution 4: Use DTOs (Best Practice — No Annotations on Entity)

The cleanest solution. Entities should have NO Jackson annotations. Map to DTOs in the service layer — DTOs have exactly the fields the API needs, no circular references possible.

JavaDTO approach — cleanest solution
// No Jackson annotations on entities ← clean, portable
@Entity
public class Order { ... }  // pure JPA entity

// DTO — only what the API needs, no cycles possible
public record OrderDTO(
    Long id,
    String status,
    List<OrderItemDTO> items   // no back-reference to OrderDTO → no cycle
) {}

public record OrderItemDTO(
    Long id,
    String productName,        // just the name, not the whole Product
    int quantity,
    BigDecimal price
) {}

// Mapper converts entity → DTO (no Jackson annotations needed anywhere)
public OrderDTO toDTO(Order order) {
    return new OrderDTO(
        order.getId(),
        order.getStatus(),
        order.getItems().stream().map(this::toItemDTO).toList()
    );
}

@JsonIgnoreSimple, one annotationField NEVER serializedQuick fix / prototypes
@JsonManagedReferenceParent-child semantics clearStill annotating entity with JacksonSimple parent-child APIs
@JsonIdentityInfoNo data loss — back-refs by IDConfusing JSON formatGraph structures

🔗 JoinColumn, JoinTable, JoinColumns, mappedBy — Explained

@JoinColumn — Single Foreign Key

Specifies the foreign key column in the current entity's table. Used with @ManyToOne, @OneToOne, and unidirectional @OneToMany.

Java@JoinColumn attributes
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(
    name             = "customer_id",    // FK column name in THIS table
    referencedColumnName = "id",        // PK column in REFERENCED table (default: id)
    nullable         = false,           // NOT NULL constraint on FK
    updatable        = false,           // once set, FK cannot change
    foreignKey       = @ForeignKey(name = "fk_order_customer") // FK constraint name
)
private Customer customer;

@JoinColumns — Composite Foreign Key

When the referenced entity has a composite primary key, you need multiple FK columns.

Java@JoinColumns for composite FK
// Referenced entity has composite PK (country_code + zone_code)
@ManyToOne
@JoinColumns({
    @JoinColumn(name = "country_code",  referencedColumnName = "country_code"),
    @JoinColumn(name = "zone_code",     referencedColumnName = "zone_code")
})
private ShippingZone shippingZone;

@JoinTable — For @ManyToMany and Unidirectional @OneToMany

Defines the intermediate join table that holds the FKs from both sides.

Java@JoinTable full anatomy
@ManyToMany
@JoinTable(
    name               = "student_course",   // join table name in DB

    // FK column pointing to THIS entity (Student)
    joinColumns        = @JoinColumn(
        name = "student_id",
        referencedColumnName = "id"
    ),

    // FK column pointing to THE OTHER entity (Course)
    inverseJoinColumns = @JoinColumn(
        name = "course_id",
        referencedColumnName = "id"
    ),

    // Optional: add unique constraint to prevent duplicate enrollments
    uniqueConstraints  = @UniqueConstraint(columnNames = {"student_id", "course_id"})
)
private Set<Course> courses;

// Creates table: student_course(student_id FK, course_id FK)

mappedBy — The Non-Owning Side

mappedBy tells JPA: "the FK is defined on the OTHER entity — look at its field named X for the join column definition." It marks the inverse side of a bidirectional relationship. Hibernate ignores the inverse side for DB writes — it only reads it for object graph navigation.

JavamappedBy — what it references
@Entity
public class Order {

    // mappedBy = "order" means:
    //   "The FK is defined in OrderItem's field named 'order'
    //   Don't look here for join column — look in OrderItem.order"
    @OneToMany(mappedBy = "order")     // ← "order" = field name in OrderItem
    private List<OrderItem> items;
}

@Entity
public class OrderItem {

    @ManyToOne
    @JoinColumn(name = "order_id")   // ← this defines the actual FK column
    private Order order;             // ← field name referenced by mappedBy = "order"
}

// ⚠️ Critical: If you add an item only to Order.items but don't set
//    item.setOrder(this), Hibernate won't write the FK! It reads from the
//    OWNING SIDE (OrderItem.order) to write the column value.

Fetch Strategies & Performance

Eager vs Lazy loading, what really happens under the hood, and fixing the notorious N+1 problem.

⚡ Eager vs Lazy Loading

What FetchType Controls

FetchType decides WHEN Hibernate loads associated entities from the database — immediately with the parent (EAGER) or only when you first access the field (LAZY).

⚡ EAGER — Load Now

  • Associated entity loaded in SAME SQL (JOIN) or immediately after
  • Always fetched — even if you never use it
  • Predictable — no LazyInitializationException
  • Can cause performance problems for large collections

Default for: @ManyToOne, @OneToOne

💤 LAZY — Load on Demand

  • Hibernate creates a proxy object with just the ID
  • Real SQL SELECT fired only when you access the field
  • Efficient — skip loading unused data
  • Requires open Session/EntityManager when accessed

Default for: @OneToMany, @ManyToMany

How Lazy Loading Works Internally — The Proxy

1
Entity loaded — LAZY association gets a proxy
When em.find(Order.class, 1L) runs, order.getCustomer() is NOT fetched. Hibernate sets it to a CGLIB proxy of Customer that knows only the customer ID.
2
Proxy intercepts first access
When your code calls order.getCustomer().getName(), the proxy intercepts .getName(). It checks: "have I been initialized?" — No.
3
SQL fired, proxy initialized
SELECT * FROM customers WHERE id = ? fires. The proxy populates itself with real data. All subsequent calls use the cached data — no more SQL.
🚫
LazyInitializationException — the most common JPA error:
could not initialize proxy — no Session

Happens when you access a LAZY field after the EntityManager / transaction has closed. For example, loading an entity in a service method, closing the transaction, then trying to access entity.getOrders() in the controller. Solutions: use JOIN FETCH, @EntityGraph, or load data within the transaction.
Default Fetch Types — Important to Memorise
AnnotationDefault FetchTypeRecommended FetchType
@ManyToOneEAGERLAZY — change this!
@OneToOneEAGERLAZY — but tricky (see note)
@OneToManyLAZYLAZY — keep default
@ManyToManyLAZYLAZY — keep default
⚠️
Always use LAZY fetch by default for ALL associations. Load eagerly only when needed using JOIN FETCH or @EntityGraph on a per-query basis. This gives you maximum control and avoids accidental N+1 problems from EAGER defaults.

🔥 N+1 Problem — The Silent Performance Killer

What Is the N+1 Problem?

When loading a list of entities and then accessing a LAZY association on each one, Hibernate fires a separate SELECT for every entity. 1 query for the list + N queries for the associations = N+1 queries total.

JavaN+1 in action — 101 SQL queries for 100 orders
// ⚠️ This looks innocent but fires 101 queries!
List<Order> orders = orderRepo.findAll();       // Query 1: SELECT * FROM orders

for (Order order : orders) {
    // Each iteration fires: SELECT * FROM customers WHERE id = ?
    System.out.println(order.getCustomer().getName()); // Queries 2-101!
}

// Total: 1 + 100 = 101 queries for 100 orders
// With 10,000 orders? 10,001 queries. App crawls or times out.

Solution 1: JOIN FETCH in JPQL — Best for Simple Cases

JavaJOIN FETCH — one SQL with a JOIN
// ✅ Single SQL with LEFT JOIN — fetches orders + customers together
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.customer")
List<Order> findAllWithCustomer();

// Generated SQL:
// SELECT DISTINCT o.*, c.* FROM orders o
// INNER JOIN customers c ON c.id = o.customer_id
// ← ONE query, all data loaded ✅

// Multiple associations — can cause cartesian product!
@Query("SELECT DISTINCT o FROM Order o "
     + "JOIN FETCH o.customer "
     + "LEFT JOIN FETCH o.items i "   // LEFT for optional items
     + "LEFT JOIN FETCH i.product")
List<Order> findAllWithDetails();
// DISTINCT prevents duplicate Order objects from join multiplication
⚠️
Cartesian Product Warning: Fetching two collections with JOIN FETCH (e.g., order.items AND order.tags) creates a Cartesian product — if an order has 5 items and 3 tags, you get 15 rows. Use @EntityGraph or separate queries to avoid this.

Solution 2: @EntityGraph — Declarative Fetch Control

Java@EntityGraph — specify what to load
// ── Inline EntityGraph ─────────────────────────────────────
@EntityGraph(attributePaths = { "customer", "items", "items.product" })
List<Order> findByStatus(String status);
// Spring generates LEFT JOIN for customer, items, and items.product
// All in ONE or TWO SQL statements (Hibernate may split to avoid cartesian)

// ── Named EntityGraph (reusable) ──────────────────────────
@Entity
@NamedEntityGraph(
    name = "Order.withCustomerAndItems",
    attributeNodes = {
        @NamedAttributeNode("customer"),
        @NamedAttributeNode(value = "items",
            subgraph = "items.product")         // nested association
    },
    subgraphs = @NamedSubgraph(
        name  = "items.product",
        attributeNodes = @NamedAttributeNode("product")
    )
)
public class Order { ... }

// Use in Repository:
@EntityGraph("Order.withCustomerAndItems")
Optional<Order> findById(Long id);

Solution 3: Hibernate Batch Fetching — Best for Complex Graphs

Instead of 1 SELECT per entity, Hibernate groups them into batches: WHERE id IN (1, 2, 3, ..., 20). With N=100 and batch=20, you get 1 + 5 = 6 queries instead of 101.

PropertiesEnable batch fetching globally
# application.properties
spring.jpa.properties.hibernate.default_batch_fetch_size=20

# Or per-association in entity:
JavaPer-association batch fetch
@OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
@BatchSize(size = 20)              // fetch in groups of 20
private List<OrderItem> items;

// What happens:
// 1. Load 100 orders (1 query)
// 2. Access first order's items → Hibernate fetches items for orders 1-20 (1 query)
// 3. Next 20 orders accessed → fetch items for 21-40 (1 query)
// Total: 1 + ceil(100/20) = 6 queries instead of 101 ✅

Solution 4: DTO Projection — Only Fetch What You Need

JavaConstructor expression — fetch directly into DTO
// ── JPQL Constructor Expression ──────────────────────────────
@Query("SELECT new com.app.dto.OrderSummary(o.id, o.status, c.name) "
     + "FROM Order o JOIN o.customer c WHERE o.status = :status")
List<OrderSummary> findSummaries(@Param("status") String status);
// Hibernate SELECTs only o.id, o.status, c.name — not the full entity graph
// No N+1, no extra data loaded, no LazyInitializationException ✅

// ── Interface Projection ─────────────────────────────────────
public interface OrderSummaryView {
    Long getId();
    String getStatus();
    String getCustomerName();     // open projection — SpEL supported
}

List<OrderSummaryView> findByStatus(String status);

The Golden Rules of JPA Performance

1. Always use LAZY fetch on all associations — load eagerly per-query via JOIN FETCH or @EntityGraph
2. Always use DTOs for read-only API responses — avoids entities, avoids lazy issues, avoids N+1
3. Set batch_fetch_size=20 globally — cheap and effective for graph traversal
4. Use @Transactional(readOnly=true) on all read methods — skips dirty checking
5. Never use findAll() without pagination on large tables
6. Monitor SQL with show-sql=true in development — count your queries
