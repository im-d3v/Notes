1. Exception Hierarchy

How Java organizes all throwable things
💡
Golden Rule In Java, anything that can be "thrown" MUST extend java.lang.Throwable.
There are two direct children: Error and Exception.
📦 java.lang.Throwable
├── 🔴 Error (JVM problems — DO NOT catch)
│ ├── OutOfMemoryError
│ ├── StackOverflowError
│ ├── VirtualMachineError
│ └── AssertionError
└── 🟡 Exception (Your code problems — handle these)
├── 🟠 RuntimeException (Unchecked)
│ ├── NullPointerException
│ ├── ArrayIndexOutOfBoundsException
│ ├── ClassCastException
│ ├── ArithmeticException
│ ├── IllegalArgumentException
│ ├── NumberFormatException
│ ├── IllegalStateException
│ └── UnsupportedOperationException
└── 🟢 Checked Exceptions
├── IOException
│ ├── FileNotFoundException
│ └── EOFException
├── SQLException
├── ClassNotFoundException
├── CloneNotSupportedException
└── InterruptedException
⚡
Key Memory Trick Error = JVM Level (unrecoverable, never catch)
RuntimeException = Programmer's fault (unchecked, optional to handle)
Exception (non-Runtime) = External factors (checked, must handle)

🔍 What is Throwable?

Throwable is the root class of all errors and exceptions in Java. It has two important constructors: Throwable(String message) and Throwable(String message, Throwable cause).

It is NOT a marker interface — it's a real class with methods like getMessage(), getCause(), printStackTrace(), getStackTrace().

🟠

2. Checked Exceptions

Compile-time exceptions — must handle or declare

📌 Definition

Exceptions that extend Exception but NOT RuntimeException are called Checked Exceptions. The compiler forces you to either handle (try-catch) or declare (throws) them. If you don't, you get a compile error.

📏
Rule of Thumb Checked exceptions represent situations outside your program's control — file not found, database down, network timeout. The caller needs to know this can happen.
Checked Exception — Two Ways to Handle Java 
// ─────────────────────────────────────────
// WAY 1: Handle it with try-catch
// ─────────────────────────────────────────
public void readFile(String path) {
    try {
        FileReader fr = new FileReader(path); // throws FileNotFoundException
        BufferedReader br = new BufferedReader(fr);
        String line = br.readLine();            // throws IOException
        System.out.println(line);
        br.close();
    } catch (FileNotFoundException e) {
        System.out.println("File not found: " + e.getMessage());
    } catch (IOException e) {
        System.out.println("IO Error: " + e.getMessage());
    }
}

// ─────────────────────────────────────────
// WAY 2: Declare with throws (pass to caller)
// ─────────────────────────────────────────
public void readFile(String path) throws IOException {
    FileReader fr = new FileReader(path);
    // ↑ Not handled here, caller must handle it
}
📋 Common Checked Exceptions
  • IOException — file / stream / network operations
  • FileNotFoundException — file path doesn't exist
  • SQLException — database access error
  • ClassNotFoundException — class not found at runtime via reflection
  • InterruptedException — thread interrupted during wait/sleep
  • CloneNotSupportedException — clone() called without Cloneable
  • ParseException — date/number parsing failure
🔴

3. Unchecked Exceptions

Runtime exceptions — optional to handle

📌 Definition

Exceptions that extend RuntimeException are called Unchecked Exceptions. The compiler does NOT force you to handle them. They typically indicate programming bugs — null pointer access, out-of-bounds index, wrong type cast.

Most Common Unchecked Exceptions Java 
// 1. NullPointerException (NPE)
String s = null;
s.length();  // ❌ NullPointerException

// 2. ArrayIndexOutOfBoundsException
int[] arr = {1, 2, 3};
arr[5];  // ❌ ArrayIndexOutOfBoundsException

// 3. ClassCastException
Object obj = "hello";
Integer i = (Integer) obj;  // ❌ ClassCastException

// 4. ArithmeticException
int result = 10 / 0;  // ❌ ArithmeticException: / by zero

// 5. NumberFormatException
int n = Integer.parseInt("abc");  // ❌ NumberFormatException

// 6. StackOverflowError (not RuntimeException but common)
void infinite() { infinite(); }  // ❌ StackOverflowError

// 7. IllegalArgumentException
if (age < 0) throw new IllegalArgumentException("Age can't be negative");

// 8. ConcurrentModificationException
List<String> list = new ArrayList<>(Arrays.asList("a", "b"));
for (String item : list) {
    list.remove(item);  // ❌ ConcurrentModificationException
}

5. try — catch — finally Block

Flow, rules, and tricky edge cases
Basic Structure + Flow Java 
try {
    // Code that might throw exception
    int result = 10 / 0;       // Throws ArithmeticException
    System.out.println(result); // ← SKIPPED if exception thrown

} catch (ArithmeticException e) {
    // Handles specific exception
    System.out.println("Caught: " + e.getMessage()); // "/ by zero"

} catch (Exception e) {
    // Generic catch — always last (more specific → less specific)
    System.out.println("General error");

} finally {
    // ALWAYS runs — exception or not, return or not
    // Used for cleanup: close files, DB connections, etc.
    System.out.println("Finally always runs!");
}
⚠️ Critical Rules — finally Block
  • Always executes — whether exception occurs or not
  • Even after return — finally runs BEFORE the method returns
  • Exception in catch — finally STILL runs before propagating
  • Only exception — System.exit() or JVM crash skips finally
  • Exception in finally — replaces/suppresses the original exception!
⚠️ TRICKY — finally overrides return value! Java · Tricky 
public int trickMethod() {
    try {
        System.out.println("try");
        return 1;  // ← this is staged, but finally runs FIRST
    } finally {
        System.out.println("finally");
        return 2;  // ← THIS return wins! Overrides try's return 1
    }
}
// Output:
// try
// finally
// Returns: 2  ← NOT 1!
🚨
Interview Trap "What does this return?" — If finally has a return statement, it ALWAYS overrides the try/catch return. This is considered bad practice — never use return in finally!
⚠️ What if both catch AND finally throw? Java · Tricky 
public void bothThrow() {
    try {
        throw new RuntimeException("from try");
    } catch (RuntimeException e) {
        System.out.println("Caught: " + e.getMessage());
        throw new RuntimeException("from catch");  // ← staging
    } finally {
        throw new RuntimeException("from finally"); // ← THIS wins!
    }
}

// Result: Exception "from finally" propagates
// Exception "from catch" is LOST (suppressed silently)
// This is why: NEVER throw from finally!

6. try-with-resources

AutoCloseable, suppressed exceptions, Java 7+

📌 What is try-with-resources?

Introduced in Java 7. Any object that implements AutoCloseable (or Closeable) can be opened in the try header. Java automatically calls close() on it at the end — even if an exception occurs. No more forgetting to close in finally!

Old Way (error-prone) vs New Way (try-with-resources) Java 
// ───────────────────────────────
// ❌ OLD WAY — manual close in finally (risky)
// ───────────────────────────────
BufferedReader br = null;
try {
    br = new BufferedReader(new FileReader("file.txt"));
    String line = br.readLine();
} catch (IOException e) {
    e.printStackTrace();
} finally {
    try {
        if (br != null) br.close(); // ← What if THIS throws? Ugly!
    } catch (IOException e2) {
        e2.printStackTrace();
    }
}

// ───────────────────────────────
// ✅ NEW WAY — try-with-resources (Java 7+)
// ───────────────────────────────
try (BufferedReader br = new BufferedReader(new FileReader("file.txt"))) {
    String line = br.readLine();
    System.out.println(line);
    // br.close() called AUTOMATICALLY — even if exception occurs!
} catch (IOException e) {
    e.printStackTrace();
}

// ───────────────────────────────
// Multiple resources — closed in REVERSE order
// ───────────────────────────────
try (
    Connection        conn = DriverManager.getConnection(url);
    PreparedStatement ps   = conn.prepareStatement("SELECT * FROM users");
    ResultSet         rs   = ps.executeQuery()
) {
    // All three closed automatically: rs → ps → conn
    while (rs.next()) {
        System.out.println(rs.getString("name"));
    }
} catch (SQLException e) {
    e.printStackTrace();
}
Creating your own AutoCloseable Java 
// Any class implementing AutoCloseable works with try-with-resources
class MyConnection implements AutoCloseable {

    public MyConnection() {
        System.out.println("Connection opened");
    }

    public void query() {
        System.out.println("Querying...");
    }

    @Override
    public void close() {
        System.out.println("Connection closed automatically!");
    }
}

// Usage:
try (MyConnection conn = new MyConnection()) {
    conn.query();
}
// Output:
// Connection opened
// Querying...
// Connection closed automatically!

⚠️ Suppressed Exceptions in try-with-resources

If an exception is thrown in the try body AND another exception is thrown in close(), Java suppresses the close exception and adds it to the primary exception via addSuppressed(). You can retrieve them with e.getSuppressed(). This is BETTER than the old finally approach where the close exception would silently swallow the original!

Suppressed Exceptions Java 
class BuggyResource implements AutoCloseable {
    public void doWork() throws Exception {
        throw new Exception("Exception from doWork");
    }
    @Override
    public void close() throws Exception {
        throw new Exception("Exception from close");
    }
}

try (BuggyResource r = new BuggyResource()) {
    r.doWork();
} catch (Exception e) {
    System.out.println("Primary: " + e.getMessage());
    for (Throwable suppressed : e.getSuppressed()) {
        System.out.println("Suppressed: " + suppressed.getMessage());
    }
}
// Output:
// Primary:   Exception from doWork   ← main exception
// Suppressed: Exception from close  ← close exception saved here

7. Custom Exceptions

Design, best practices, when to use checked vs unchecked

📌 Why Custom Exceptions?

Custom exceptions make your code more readable, provide meaningful error messages, allow domain-specific errors (like InsufficientBalanceException), and let callers catch exactly what they need without catching generic exceptions.

Custom CHECKED Exception (extends Exception) Java 
// ─── CHECKED: Use when caller MUST handle it ───
public class InvalidAgeException extends Exception {

    // Best Practice: Always provide multiple constructors

    public InvalidAgeException(String message) {
        super(message);
    }

    public InvalidAgeException(String message, Throwable cause) {
        super(message, cause); // preserves original exception (exception chaining)
    }
}

// ─── Usage ───
public void registerUser(int age) throws InvalidAgeException {
    if (age < 18) {
        throw new InvalidAgeException("Age must be 18+. Provided: " + age);
    }
    System.out.println("User registered!");
}

// ─── Caller must handle ───
try {
    registerUser(15);
} catch (InvalidAgeException e) {
    System.out.println("Error: " + e.getMessage()); // Error: Age must be 18+. Provided: 15
}
Custom UNCHECKED Exception (extends RuntimeException) ← Spring Boot Way Java 
// ─── UNCHECKED: Preferred in Spring Boot ───
public class UserNotFoundException extends RuntimeException {

    private final long userId;

    public UserNotFoundException(long userId) {
        super("User not found with ID: " + userId);
        this.userId = userId;
    }

    public UserNotFoundException(String message) {
        super(message);
        this.userId = -1;
    }

    public long getUserId() { return userId; }
}

// ─── Service Layer — NO throws declaration needed ───
@Service
public class UserService {

    @Autowired
    private UserRepository userRepository;

    public User findById(long id) {
        return userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id)); // Clean!
    }
}

// ─── Global Handler in Spring Boot ───
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleUserNotFound(UserNotFoundException ex) {
        return new ErrorResponse(404, ex.getMessage());
    }

    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleGeneral(Exception ex) {
        return new ErrorResponse(500, "Internal Server Error");
    }
}
Aspect extends Exception (Checked) extends RuntimeException (Unchecked)
Caller forced to handle? ✅ Yes — compile error if not ❌ No — optional
Method signature polluted? Yes — "throws MyException" needed No — clean signatures
Use case API contract: caller MUST know and handle Programming error / domain error in Spring
Spring Boot preference ❌ Rarely used ✅ Preferred — use @ControllerAdvice

8. Multi-catch, Re-throw & Exception Chaining

Java 7 multi-catch, chaining, and wrapping exceptions
Multi-catch (Java 7+) + Exception Chaining Java 
// ─── Multi-catch: Handle multiple exceptions the same way ───
try {
    someRiskyOperation();
} catch (IOException | SQLException e) {
    // Both handled the same way — no code duplication
    System.out.println("Data access error: " + e.getMessage());
    // NOTE: 'e' is effectively final here — cannot reassign
}

// ─── Re-throw: Catch, log, then throw again ───
try {
    riskyCode();
} catch (IOException e) {
    logger.error("Failed to read file", e);
    throw e;  // re-throw same exception
}

// ─── Exception Chaining: Wrap in custom exception ───
// BEST PRACTICE: Always pass cause — preserves full stack trace!
try {
    connection.executeQuery(sql);
} catch (SQLException e) {
    throw new DataAccessException("Query failed for SQL: " + sql, e);
    //                                                         ↑ cause!
}

// ─── BAD PRACTICE: Losing the cause ───
catch (SQLException e) {
    throw new DataAccessException("Query failed"); // ❌ cause LOST!
}
💡
Exception Chaining Rule ALWAYS pass the original exception as the cause when wrapping. Without it, you lose the original stack trace and debugging becomes very hard. Use: new MyException("message", originalException)

9. Serialization & Exception Handling

serialVersionUID, NotSerializableException, transient

📌 What is Serialization?

Converting an object's state to a byte stream (Serialization) and converting it back (Deserialization). Used in file I/O, network transmission, caching, session persistence.

A class must implement java.io.Serializable (marker interface) to be serialized.

Serialization + All Exception Scenarios Java 
import java.io.*;

// ─── BASIC SERIALIZABLE CLASS ───
public class Employee implements Serializable {

    // ALWAYS define serialVersionUID to avoid InvalidClassException
    private static final long serialVersionUID = 1L;

    private String  name;
    private int     age;
    private transient String password;   // transient = NOT serialized
    private transient Logger logger;     // Logger is not Serializable, mark transient!

    public Employee(String name, int age, String password) {
        this.name     = name;
        this.age      = age;
        this.password = password;
    }
    // getters...
}

// ─── SERIALIZE ───
try (ObjectOutputStream oos =
        new ObjectOutputStream(new FileOutputStream("employee.ser"))) {
    oos.writeObject(new Employee("Rahul", 25, "secret123"));
    System.out.println("Serialized!");
} catch (IOException e) {
    e.printStackTrace();
}

// ─── DESERIALIZE ───
try (ObjectInputStream ois =
        new ObjectInputStream(new FileInputStream("employee.ser"))) {
    Employee emp = (Employee) ois.readObject();
    System.out.println(emp.getName());    // Rahul
    System.out.println(emp.getPassword()); // null ← transient field!
} catch (IOException | ClassNotFoundException e) {
    e.printStackTrace();
}
💥 Serialization Exception Scenarios — Asked in Interviews
  • NotSerializableException — class doesn't implement Serializable.
    Fix: implement Serializable or mark non-serializable fields as transient
  • InvalidClassException — class was modified after serialization, serialVersionUID mismatch.
    Fix: always define private static final long serialVersionUID = 1L;
  • ClassNotFoundException (during deserialization) — class not in classpath of reading JVM.
    Occurs in distributed systems when versions differ
  • StreamCorruptedException — byte stream is corrupted or not a valid serialized stream.
  • Transient field — field marked transient is set to null after deserialization (primitives → default value).
serialVersionUID — Why it Matters Java 
// ─── Scenario: What happens if you don't define serialVersionUID ───

// STEP 1: Serialize with version 1 of class (2 fields)
class Person implements Serializable {
    String name;
    int    age;
}
// → Serialize to file

// STEP 2: Add a new field to Person class
class Person implements Serializable {
    String name;
    int    age;
    String email; // ← NEW FIELD ADDED
}
// → Try to deserialize old file
// ❌ InvalidClassException: local class incompatible:
//    stream classdesc serialVersionUID = 5463..., local class serialVersionUID = 8821...

// ─── FIX: Always explicitly define it ───
class Person implements Serializable {
    private static final long serialVersionUID = 1L; // ← Always include this!
    String name;
    int    age;
    String email; // New field — old files still work, email will be null
}

10. Exception in Generic List

Type erasure, heap pollution, ClassCastException traps
Generic List Exception Scenarios Java 
// ─── 1. ClassCastException via Raw Type ───
List rawList = new ArrayList();  // Raw type — no generics
rawList.add("hello");
rawList.add(42);

List<String> stringList = rawList;  // Compiler warning, no error
String s = stringList.get(1);      // ❌ ClassCastException at runtime! (42 is Integer)

// ─── 2. Heap Pollution (@SafeVarargs) ───
@SafeVarargs
public static <T> List<T> asList(T... elements) {
    return new ArrayList<>(Arrays.asList(elements));
}

// ─── 3. UnsupportedOperationException ───
List<String> immutable = List.of("a", "b", "c");
immutable.add("d"); // ❌ UnsupportedOperationException (List.of returns immutable)

List<String> fromArrays = Arrays.asList("a", "b");
fromArrays.add("c");    // ❌ UnsupportedOperationException
fromArrays.set(0, "x");  // ✅ OK — can update, but fixed size

// ─── FIX: Wrap in new ArrayList to make mutable ───
List<String> mutable = new ArrayList<>(List.of("a", "b"));
mutable.add("c"); // ✅ Works!

// ─── 4. ConcurrentModificationException ───
List<String> names = new ArrayList<>(Arrays.asList("a", "b", "c"));

// ❌ BAD: Modifying while iterating with for-each
for (String name : names) {
    if (name.equals("b")) names.remove(name); // ❌ ConcurrentModificationException
}

// ✅ FIX 1: Use Iterator.remove()
Iterator<String> it = names.iterator();
while (it.hasNext()) {
    if (it.next().equals("b")) it.remove(); // ✅ Safe
}

// ✅ FIX 2: removeIf (Java 8)
names.removeIf(name -> name.equals("b")); // ✅ Cleanest

// ✅ FIX 3: Use CopyOnWriteArrayList (thread-safe)
List<String> safe = new CopyOnWriteArrayList<>(names);
for (String n : safe) {
    if (n.equals("b")) safe.remove(n); // ✅ No exception (iterates snapshot)
}
🧠
Type Erasure — Why Generic Exceptions Are Tricky Java generics use type erasure — at runtime, List<String> and List<Integer> are both just List. You CANNOT do catch (MyException<String> e) — generics in catch blocks are not allowed. You also cannot do new T() or throw new T() with generic types.

11. ClassNotFoundException vs NoClassDefFoundError

Most confused pair in Java interviews

🟠 ClassNotFoundException

Checked Exception

Thrown when JVM tries to load a class dynamically at runtime using Class.forName(), ClassLoader.loadClass(), or ClassLoader.findSystemClass() — and the class is NOT found in the classpath.

🔴 NoClassDefFoundError

Error (not Exception)

Thrown when a class was available at compile time but is missing at runtime. JVM found it during compilation but can't find it when the code actually runs.

Aspect ClassNotFoundException NoClassDefFoundError
Type Checked Exception Error (unchecked)
Parent Exception → ReflectiveOperationException Error → LinkageError
When Dynamic class loading (reflection) Class present at compile time, missing at runtime
Caused by Class.forName("com.example.Foo") — Foo not in classpath JAR removed after compilation, or static initializer failed
Must handle? Yes — checked exception No — it's an Error
Common Scenario JDBC driver loading, plugin systems Missing JAR in production, class initialization failure
Both exceptions — Code Examples Java 
// ─── ClassNotFoundException ───
try {
    Class<?> clazz = Class.forName("com.mysql.jdbc.Driver"); // must handle
} catch (ClassNotFoundException e) {
    System.out.println("Driver not in classpath: " + e.getMessage());
}

// ─── NoClassDefFoundError ─── (cannot be handled gracefully)
// Scenario: Foo.java was compiled but Foo.class deleted from classpath
// At runtime: Error: Could not find or load main class Foo
// This is an Error, JVM terminates

// ─── Also: Static Initializer Failure causes NoClassDefFoundError ───
class ConfigLoader {
    static Properties props;
    static {
        // If this block throws, ConfigLoader is marked as failed
        // Every subsequent attempt to use ConfigLoader throws NoClassDefFoundError
        props = loadFromFile(); // If this fails...
    }
}
// Any code using ConfigLoader will get:
// java.lang.NoClassDefFoundError: Could not initialize class ConfigLoader

12. Edge Cases & Tricky Questions

Output prediction questions — asked in every interview

🎯 Tricky 1: Can you catch Error?

Catching Error — Technically Yes, Practically Never Java 
try {
    int[] huge = new int[Integer.MAX_VALUE]; // OutOfMemoryError
} catch (OutOfMemoryError e) {
    // Technically compiles and catches it
    // BUT: JVM is in bad state — don't do this in production
    // Only valid use: logging before graceful shutdown
    System.err.println("Out of memory! Shutting down.");
}
// Rule: You CAN catch Error, but you almost NEVER should

🎯 Tricky 2: finally with System.exit()

System.exit() — The only way to skip finally Java 
public static void main(String[] args) {
    try {
        System.out.println("try");
        System.exit(0);     // ← JVM terminates HERE
    } finally {
        System.out.println("finally"); // ← NEVER prints!
    }
}
// Output: try
// "finally" does NOT print — System.exit() kills JVM

🎯 Tricky 3: Exception in static initializer

Static Block Exception → ExceptionInInitializerError Java 
class BadClass {
    static int value;
    static {
        value = 10 / 0; // ArithmeticException in static block
    }
}

try {
    BadClass obj = new BadClass();
} catch (ExceptionInInitializerError e) {
    // Wraps the original exception
    System.out.println("Cause: " + e.getCause()); // ArithmeticException
}
// Second access → NoClassDefFoundError (class initialization already failed)

🎯 Tricky 4: Can main() throw an exception?

main() can declare throws Java 
// YES! main() can throw exceptions
public static void main(String[] args) throws Exception {
    throw new Exception("Unhandled exception in main");
    // JVM catches it, prints stack trace, and terminates
    // Exception in thread "main" java.lang.Exception: Unhandled exception in main
}

🎯 Tricky 5: NPE in Unboxing

NullPointerException during auto-unboxing Java 
Integer boxed = null;
int     value = boxed; // ❌ NullPointerException! Auto-unboxing null Integer

// Why? Java tries: value = boxed.intValue() — NPE because boxed is null

// Another trap: conditional expressions
Integer a = null;
Integer b = true ? a : 0;  // ← int 0 forces unboxing of a → NPE!

// Map get() returning null then auto-unboxing
Map<String, Integer> map = new HashMap<>();
int count = map.get("missing"); // ❌ NPE! get() returns null, unboxing fails

// ✅ Safe way:
int count = map.getOrDefault("missing", 0); // Returns 0, no NPE

13. Interview Q&A — Most Asked

Direct answers you can say in interviews
Q What is the difference between throw and throws?
throw — keyword used to actually throw an exception object inside a method. throw new IllegalArgumentException("invalid")

throws — keyword in method signature declaring that the method might throw a checked exception. Caller must handle it. void readFile() throws IOException
Q Can we have try without catch?
Yes! You can have try-finally without catch (to ensure cleanup). Also try-with-resources without catch (Java 7+). But you cannot have try alone without either catch or finally.
Q Why should we extend RuntimeException for custom exceptions in Spring Boot?
In Spring, if you extend Exception (checked), every service method must declare throws YourException, polluting all signatures. With RuntimeException, exceptions propagate freely up the call stack and are caught centrally by @ControllerAdvice, keeping service layer clean and readable.
Q What is exception chaining and why is it important?
Exception chaining is wrapping an original exception inside a new one using new MyException("msg", originalException). It preserves the full stack trace and root cause. Without it, you lose the original exception details making debugging very hard. Always pass the cause when wrapping!
Q What is the difference between final, finally, and finalize?
final — keyword: variable cannot be reassigned, method cannot be overridden, class cannot be extended.
finally — block in try-catch: always executes for cleanup (close resources).
finalize() — deprecated method in Object class: called by GC before collecting object. Avoid using it — GC doesn't guarantee when it runs. Use try-with-resources or explicit close instead.
Q Can you override a method and add new checked exceptions?
No! An overriding method cannot add new checked exceptions or broader checked exceptions than the parent. It can:
✅ Remove the declared exception
✅ Throw a subclass of the declared exception
✅ Add unchecked (RuntimeException) exceptions
❌ Cannot add new/broader checked exceptions — would break Liskov Substitution Principle
Q What is a suppressed exception?
In try-with-resources, if the try body throws exception A and the close() method also throws exception B, Java keeps A as the primary exception and adds B as a suppressed exception via e.addSuppressed(B). You can retrieve them with e.getSuppressed(). This prevents losing exceptions (unlike old finally approach).
Q What exceptions can occur during serialization?
NotSerializableException — class doesn't implement Serializable
InvalidClassException — serialVersionUID mismatch after class change
ClassNotFoundException — class not in classpath during deserialization
StreamCorruptedException — corrupted byte stream
Transient fields — become null/default after deserialization (not an exception but common surprise)
Q What is the difference between ClassNotFoundException and NoClassDefFoundError?
ClassNotFoundException — Checked exception. Thrown during explicit dynamic class loading (Class.forName()) when class is not in classpath.

NoClassDefFoundError — Error (not exception). Class was available at compile time but missing at runtime (e.g., JAR deleted from classpath), or static initializer failed. Cannot recover — JVM is in bad state.
Q What is ConcurrentModificationException and how to fix it?
Thrown when you modify a collection while iterating over it using for-each (which uses Iterator internally). Collection tracks a modCount; if it changes during iteration, CME is thrown.

Fix 1: Use Iterator.remove() instead
Fix 2: Use list.removeIf(predicate) (Java 8)
Fix 3: Collect to remove list, then remove after loop
Fix 4: Use CopyOnWriteArrayList (thread-safe, iterates snapshot)
⚡ Quick Reference — Exception Handling Best Practices
  • Never swallow exceptions — catch (Exception e) {} is evil — always log or rethrow
  • Catch specific, not general — catch IOException, not Exception
  • Always pass cause when wrapping exceptions — preserve stack trace
  • Use try-with-resources for any AutoCloseable — never manual finally close
  • Never return from finally — it overrides try/catch return silently
  • Never throw from finally — suppresses original exception
  • Define serialVersionUID for every Serializable class
  • Mark non-serializable fields transient — Logger, Connection, etc.
  • Prefer RuntimeException for custom exceptions in Spring Boot
  • Use @ControllerAdvice for centralized exception handling in REST APIs
