Foundations

Why database selection is an architectural decision, and the deep dive into SQL.

🏗️ Why Database Choice Matters More Than Most Think

Here's the most common mistake junior engineers make in system design interviews: the interviewer describes a system and the candidate immediately says "I'll use PostgreSQL" or "I'll use MongoDB" — and then moves on. The interviewer silently marks them down.

Why? Because database choice is not a detail. It determines your consistency model, your query patterns, your scaling strategy, your operational complexity, and whether your system can survive 10× growth. Choosing wrong means a rewrite later — and rewrites at scale are terrifying.

⚠️
Common Mistake

"I'll use Postgres for everything" is not a thoughtful answer. It signals that you see databases as interchangeable utilities instead of design decisions. The interviewer wants to see you reason, not recite.

The Mental Model: Databases Are Tools, Not Religions

Think of it like choosing a vehicle. A motorcycle, a truck, and a bus all "get you somewhere" — but you wouldn't haul 20 tonnes of gravel on a motorcycle, and you wouldn't commute across town on an 18-wheeler. The right tool depends entirely on the job.

Every database was designed to solve a specific class of problem really well — and to accept trade-offs in other areas. Your job is to match the tool to the problem.

🎯
Interview Tip

When the interviewer describes a system, write down: What does the data look like? How is it accessed? How often is it written vs. read? What breaks if data is slightly stale? Answer those four questions before naming any database.

The Question That Unlocks Everything

"What are the access patterns?" — ask this in every system design interview. The answer tells you almost everything about which database to use.

🐘 SQL Databases — When and Why

Relational databases have been around since the 1970s. They're not "legacy" — they're battle-tested. Companies like Amazon, Airbnb, and GitHub ran on MySQL/PostgreSQL at massive scale. The reason they've survived 50 years is simple: ACID guarantees.

⚡ ACID Properties — What They Actually Mean

Every textbook defines ACID. Let's talk about what breaks in the real world without each property.

A — Atomicity

Real-world analogy: You're transferring $500 from your savings to checking. The database must deduct $500 and add $500 as a single all-or-nothing unit.

Without atomicity: the server crashes after the deduction but before the addition. Your money vanishes. Your bank customers riot.

✅
Either everything commits or nothing does.

C — Consistency

Real-world analogy: Your database has a rule — every order must link to a valid user. Consistency means that rule can never be violated, even mid-transaction.

Without consistency: you delete a user and leave orphaned orders floating with no owner. Your reports break. Your queries return NULL everywhere.

✅
Constraints and foreign keys are enforced at all times.

I — Isolation

Real-world analogy: Two cashiers both check the inventory at the same time — both see 1 widget remaining, both sell it. You've now oversold.

Isolation ensures concurrent transactions don't see each other's in-progress state. This is the hardest property to get right — it's why transaction isolation levels exist.

✅
Concurrent transactions behave as if they ran serially.

D — Durability

Real-world analogy: The database confirms your payment. One second later, the server loses power. When it restarts, your payment is still there — because it was written to disk before the confirmation.

Durability is achieved through the Write-Ahead Log (WAL) — changes are written to a durable log before the actual data pages are modified.

✅
Committed data survives crashes. Always.
💡
Key Insight

ACID is expensive. Enforcing all four properties requires locking, logging, and coordination overhead. This is why SQL databases are slower at raw writes than NoSQL — they're doing more work to give you stronger guarantees.

🎯 When SQL Is the Right Choice

ScenarioWhy SQL WinsReal Example
Financial transactions Atomicity is non-negotiable. Money cannot be half-moved. Banking systems, Stripe, PayPal
E-commerce orders Orders, items, users, payments all relate to each other. JOINs are needed. Amazon order DB, Shopify
User accounts / auth Unique constraints, foreign keys keep data clean. Rarely need to scale to extreme write volumes. GitHub users, LinkedIn profiles
Inventory systems Isolation prevents overselling. Constraints prevent negative stock. Ticketmaster, airline seats

📈 SQL Scaling Challenges — The Real Problems

1
Read Replicas: The First Scaling Move
Your primary (write) node gets overwhelmed. The solution: spin up replicas that are copies of the primary. All writes go to primary. Read traffic is spread across replicas. A single PostgreSQL primary handles roughly ~5,000–10,000 read QPS. With 5 replicas, you've got ~50,000 read QPS. What it doesn't solve: write bottleneck. All writes still hit one machine.
2
Connection Pooling: Why Raw DB Connections Are Expensive
Opening a database connection is slow — it involves TCP handshakes, authentication, and memory allocation on the DB side. A PostgreSQL instance can safely handle ~100–300 concurrent connections before performance degrades. With 50 app servers each opening 20 connections, you've already hit 1,000. Solution: PgBouncer or HikariCP sits in front of the DB, maintains a small pool of real connections, and queues requests.
3
JOINs at Scale: The Hidden Time Bomb
JOINs are SQL's superpower — and its Achilles heel. Joining two tables of 1M rows each is fine. Joining five tables of 50M rows each with filters? You'll bring down production. At scale, companies denormalize: they duplicate data to avoid JOINs. This trades storage for query speed. Instagram and Twitter eventually moved away from JOIN-heavy schemas for exactly this reason.
4
Read/Write Splitting: Route Traffic Intentionally
Send writes → primary. Send reads → replicas. Sounds simple, but requires your application to be aware of the split. Frameworks like Spring's @Transactional(readOnly = true) can help route automatically. The gotcha: replication lag (covered in Section 7) means reads from replicas might be slightly stale.
📊
Numbers to Remember
  • Single MySQL/PostgreSQL instance: ~5–15K QPS (reads), ~2–5K QPS (writes)
  • Max safe concurrent connections (PostgreSQL): ~200–400
  • Replication lag: typically <1 second, but can be seconds/minutes under high load
  • A single table with >100M rows is where you start thinking seriously about sharding

The NoSQL World

"NoSQL" is four completely different tools wearing the same name badge. Let's fix that.

🗃️ NoSQL Databases — The 4 Types

⚠️
Stop treating NoSQL as one category.

"Should I use SQL or NoSQL?" is the wrong question. Redis and Cassandra are both NoSQL. They have almost nothing in common. The right question is: what access pattern do I need to optimize for?

Type A

🔑 Key-Value Stores — Redis, DynamoDB, Memcached

Mental model: Imagine a giant hash map. You hand it a key, it gives you a value in microseconds. That's it. No schema. No relationships. No joins. Just get and set.

Redis can serve ~100,000–1,000,000 operations per second on a single instance because everything lives in RAM. The trade-off: data can be lost on restart (unless you configure persistence), and you can only look up by exact key — no "find all users where age > 25."

Use CaseWhy Key-Value Fits
Session storagesession_id → user data. Exact-key lookup. TTL built-in.
CachingCache expensive DB query results. Evict when stale.
Rate limitingIncrement a counter per user per minute. Atomic and fast.
LeaderboardsRedis Sorted Sets: O(log N) insert + rank query.
Distributed locksRedis SETNX (set if not exists) implements distributed mutex.

✅ You gain

  • Extreme speed (in-memory)
  • Simple mental model
  • Atomic operations
  • Built-in TTL (expiry)

❌ You give up

  • Complex queries
  • Relationships between data
  • Durability (unless configured)
  • Data size limited by RAM
Type B

📄 Document Stores — MongoDB, Firestore, CouchDB

Mental model: Imagine a filing cabinet full of folders. Each folder holds a JSON document. Documents in the same collection can have completely different fields. You query by any field — not just a primary key.

The killer feature: flexible schema. When your product is evolving fast and the data structure changes every sprint, rigid SQL schemas become a bottleneck. Document stores let you add new fields without migrations.

Use CaseWhy Document Fits
Content management (CMS)Blog posts, products — every item has different fields.
User profilesNested preferences, settings, history in one document.
Product catalogsA TV and a T-shirt have completely different attributes.
Real-time appsFirestore syncs changes to connected clients instantly.
⚠️
Watch out: the JOIN problem.

MongoDB doesn't do real JOINs. If your data has deep relationships (orders → users → addresses → payment methods), you'll either denormalize (duplicate data) or do multiple round trips. If relationships matter, SQL is probably better.

Type C

📊 Column-Family Stores — Cassandra, HBase, ScyllaDB

Mental model: Think of a spreadsheet where rows are identified by a partition key, and columns within a row are stored sorted together. Cassandra is designed for one thing above all else: extreme write throughput at global scale.

Cassandra has no single point of failure, no primary node for writes. Every node accepts writes. Data is replicated to N nodes automatically. A single Cassandra cluster can handle millions of writes per second.

💡
Why Discord Uses Cassandra for Messages

Discord stores billions of messages. Messages are written once, read occasionally, and almost never updated. The access pattern is: "give me all messages in channel X, sorted by time, paginated." This is exactly what Cassandra excels at — partition by channel_id, cluster by timestamp. No JOINs needed. Write volume is enormous. Cassandra was the obvious choice.

Use CaseWhy Column-Family Fits
Time-series data (IoT, metrics)Partition by device, sort by timestamp. Efficient range scans.
Chat / messaging historyWrite-heavy, partition by conversation, sort by time.
Activity feeds / logsAppend-only, massive write volume, time-ordered reads.
Recommendation scoresStore pre-computed scores per user, fast reads by user_id.

✅ You gain

  • Massive write throughput
  • Linear horizontal scalability
  • No single point of failure
  • Tunable consistency

❌ You give up

  • No ad-hoc queries (must know access pattern upfront)
  • No multi-row transactions
  • Updates are expensive (tombstones)
  • Steep learning curve for data modeling
Type D

🕸️ Graph Databases — Neo4j, Amazon Neptune, JanusGraph

Mental model: Think of a social network diagram. Circles are people (nodes). Lines connecting them are friendships (edges). A graph database stores this structure natively and can traverse millions of edges in milliseconds — something that would require exponentially expensive JOINs in SQL.

Graph databases shine when the relationships ARE the data. "Who are my second-degree connections?" or "Which users are connected to this fraudulent account within 3 hops?" are queries that would destroy a relational database but are trivial in a graph DB.

Use CaseReal CompanyWhy Graph?
Social graphFacebook, LinkedInFriends-of-friends, degree of connection queries.
Fraud detectionPayPal, Uber"Is this new account connected to known fraud accounts?"
Recommendation engineNetflix, Amazon"Users who like X also connected to Y via 2 hops."
Knowledge graphsGoogle Knowledge GraphEntity relationships and inference.
⚠️
Graph DBs are specialized tools.

Don't use a graph database unless your core queries require graph traversal. They're operationally complex and don't scale horizontally as easily as Cassandra. Most systems don't need them. When they do — nothing else compares.

Design Decisions

Frameworks for choosing and optimizing — the skills interviewers actually test.

🧭 The Database Selection Framework

The best system designers don't have a favourite database — they have a decision process. Here are the five questions you should ask out loud in every interview before naming a database. The interviewer will love this.

Q1
Is the schema fixed or flexible?
Fixed, well-defined schema with known relationships → lean SQL. Highly variable data where different records have different fields (products, user preferences) → lean Document Store. No schema at all (cache, session) → Key-Value.
Q2
Are there complex relationships between entities?
If you need JOINs between multiple tables regularly → SQL. If relationships form a graph (friends-of-friends, fraud networks) → Graph DB. If data is self-contained per entity (a message, a product) → NoSQL.
Q3
Read-heavy or write-heavy?
Massively write-heavy (millions of events/second) → Cassandra or DynamoDB. Read-heavy with complex queries → SQL with read replicas. Read-heavy with simple key lookups → Redis cache in front of your SQL DB.
Q4
What are the consistency requirements?
Money, inventory, anything that cannot afford to be wrong even for a millisecond → Strong consistency → SQL or DynamoDB with strong reads. Social feeds, view counts, recommendations → Eventual consistency is fine → Cassandra, DynamoDB, or Redis.
Q5
What is the scale? (QPS and data volume)
<10M rows, <10K QPS → a well-tuned SQL database is almost always fine. 100M+ rows, 100K+ writes/sec → you need to think about sharding or switching to a database built for horizontal scale (Cassandra, DynamoDB).

Interview Tip: Polyglot Persistence

Real-world systems use multiple databases. Uber uses MySQL for trips, Redis for rate limiting, Cassandra for location history. 🔍 Indexing — The Most Important Optimization

If you design a great database schema but forget indexes, your queries will crawl. If you add too many indexes, your writes will crawl. Understanding indexes is the difference between a database that handles millions of queries and one that melts under load.

What an Index Actually Is

Analogy: Imagine a 1,000-page textbook with no index. To find "B-tree", you'd read every single page. That's a full table scan. With a book index, you flip to "B" alphabetically and jump directly to the right page. That's an index scan.

Most SQL databases use a B-tree (balanced tree) for indexes. The index is a separate data structure, stored on disk, sorted by the indexed column. Finding a value in a B-tree of 1 million entries takes just ~20 comparisons (log₂ of 1,000,000 ≈ 20).

B-Tree Index (simplified) [ 50 ] / \ [ 25 ] [ 75 ] / \ / \ [ 12 ] [ 37 ] [ 62 ] [ 88 ] Query: WHERE age = 62 → Compare 62 to 50: go right → Compare 62 to 75: go left → Found: 62 → points to row in table ✓ 3 comparisons instead of scanning all rows

Composite Indexes — Column Order Matters

A composite index on (last_name, first_name) can answer:

  • WHERE last_name = 'Smith' → ✅ uses index
  • WHERE last_name = 'Smith' AND first_name = 'John' → ✅ uses index
  • WHERE first_name = 'John' → ❌ cannot use this index (leftmost prefix rule)

The leftmost prefix rule: An index on (A, B, C) can be used for queries filtering on A, A+B, or A+B+C — but not B alone or C alone. This catches junior engineers constantly. Design composite indexes based on your most common query patterns.

🚫
When NOT to add indexes
  • High write volume tables: Every INSERT/UPDATE/DELETE must also update all indexes. More indexes = slower writes. A table with 10 indexes on a write-heavy workload can be 5× slower than one with 2 indexes.
  • Small tables (<10K rows): A full scan of 10K rows is trivially fast. Index overhead isn't worth it.
  • Low-cardinality columns: An index on a boolean column (true/false) is almost useless — half the table matches. The DB optimizer may ignore it.
🎯
The EXPLAIN Mindset

In production, the first thing any experienced engineer does with a slow query is run EXPLAIN ANALYZE (PostgreSQL) or EXPLAIN (MySQL). It shows whether the DB is doing a full scan or using an index. In an interview, mentioning "I would run EXPLAIN to verify the query plan" signals real-world experience. Interviewers notice this.

Scaling Your Database

When one machine isn't enough — sharding, replication, and the hard problems they create.

✂️ Database Sharding

Problem: Your users table has 2 billion rows. No single machine can store that much data with acceptable query performance. Indexes that once fit in RAM now don't. Writes back up. Everything slows down.

Solution: Sharding — split the data horizontally across multiple database machines (shards). Each shard holds a subset of the rows. A shard is just a regular database instance — what makes it special is that together they hold all the data.

Before Sharding: ┌─────────────────────────────────┐ │ users table: 2 billion rows │ ← One machine is overwhelmed │ All on ONE database server │ └─────────────────────────────────┘ After Sharding (4 shards): ┌────────────┐ ┌────────────┐ ┌────────────┐ ┌────────────┐ │ Shard 0 │ │ Shard 1 │ │ Shard 2 │ │ Shard 3 │ │ 500M rows │ │ 500M rows │ │ 500M rows │ │ 500M rows │ │ user 0,4,8 │ │ user 1,5,9 │ │ user 2,6.. │ │ user 3,7.. │ └────────────┘ └────────────┘ └────────────┘ └────────────┘ Shard key: user_id % 4

Three Sharding Strategies

A — Hash-Based Sharding

How: shard = hash(user_id) % num_shards

✅ Pro: Uniform data distribution. No hotspots.

❌ Con: Range queries are impossible (WHERE user_id BETWEEN 100 AND 200 would hit all shards). Adding shards requires rehashing everything — painful.

Used when: You mostly do single-key lookups.

B — Range-Based Sharding

How: Users A–M on Shard 1. Users N–Z on Shard 2.

✅ Pro: Range queries stay on one shard. Easy to reason about.

❌ Con: Uneven distribution. If most users have names starting with A–M, Shard 1 is overloaded. This is called a hotspot.

Used when: Data has natural ranges (time-series by date).

C — Directory-Based Sharding

How: A separate lookup service (directory) maps each key to a shard. No algorithm — explicit mapping.

✅ Pro: Maximum flexibility. You can move data between shards without changing the algorithm.

❌ Con: The directory service is a single point of failure. Adds a network hop to every request. More complex to operate.

Used when: You need fine-grained control over data placement.

Consistent Hashing

The solution to hash-based sharding's "rehash everything" problem.

Nodes and keys are placed on a virtual ring. Adding a new shard only moves a fraction of keys (≈ 1/N). Cassandra and DynamoDB use this natively.

In interviews: Mention consistent hashing when discussing how to add shards without downtime.

Problems Sharding Creates

🚫
Cross-Shard Queries: The Nightmare

Once you shard, queries that span multiple shards are expensive or outright impossible. SELECT COUNT(*) FROM users now requires querying all shards and summing the results. JOINs across shards require fetching data into your application layer and joining in memory. This is called a scatter-gather query — and it kills performance.

⚠️
The Celebrity / Hotspot Problem

Imagine Instagram shards by user_id. Cristiano Ronaldo has 600 million followers. Every time he posts, millions of read requests hit the same shard. That shard melts. The solution: special handling for "celebrity" accounts — replicate their data across many shards, or cache at a higher layer (CDN/Redis). This is a genuinely hard problem. Twitter and Instagram have war stories about it.

Interview Tip — When to Mention Sharding

Don't jump to sharding immediately. It's a last resort. The correct escalation in an interview is: single DB → add indexes → add read replicas → introduce caching → then shard if you still need more write throughput. Saying this shows maturity.

🔁 Replication — One Database, Multiple Copies

Replication solves two problems: availability (if the primary dies, a replica takes over) and read scalability (spread reads across replicas). It's usually the first scaling technique you reach for — before sharding and before a major architectural overhaul.

Primary–Replica (Master–Slave) Replication

All writes All reads │ │ │ │ ▼ ▼ ▼ ▼ ┌─────────┐ replication ┌────┐ ┌────┐ ┌────┐ │ PRIMARY │ ──────────────► │ R1 │ │ R2 │ │ R3 │ │ (1 node)│ (async WAL) └────┘ └────┘ └────┘ └─────────┘ Replicas (can be stale!)

The primary writes to its Write-Ahead Log (WAL). Replicas stream that log and replay the changes. This is almost always asynchronous — meaning replicas apply changes after the primary commits. The gap between primary and replica is called replication lag.

The Read-Your-Own-Writes Problem

This trips up junior engineers every time. Here's the scenario:

  1. User updates their profile picture → write goes to primary. ✅
  2. Page refreshes → read goes to replica. ❌
  3. Replica is 500ms behind. User sees their old profile picture.
  4. User thinks the update failed and hits save again. Now you have duplicate writes.

Solutions:

  • Route reads-after-writes to primary for the same user for a short window.
  • Track a replication timestamp: client sends the timestamp of their last write; if the replica hasn't caught up to that timestamp, route the read to primary.
  • Sticky sessions: the same user always reads from the same replica.

Multi-Primary Replication — When and Why (and Why It's Hard)

Sometimes you need writes in multiple data centres simultaneously — for geographic redundancy or to reduce write latency for global users. Multi-primary replication (also called multi-master) lets multiple nodes accept writes.

🚫
Write conflicts are real and hard

Two users update the same row on two different primaries at the same millisecond. Which write wins? There's no universally correct answer. Databases use "last write wins" (based on timestamps — but clocks drift!), application-level conflict resolution, or CRDTs (conflict-free replicated data types). This is genuinely hard. Most teams avoid multi-primary unless they truly need global writes. Prefer routing writes to a single region's primary where possible.

💡
Failover: What Happens When Primary Dies

When the primary fails, one replica must be promoted to primary. This involves: detecting the failure (usually a few seconds of timeout), electing the new primary (Raft or Paxos consensus, or external tools like Patroni for PostgreSQL), and redirecting all write traffic. During this window (typically 15–60 seconds), writes are unavailable. Design systems that can tolerate brief write outages, or use multi-primary for zero-downtime failover.

Interview Playbook

Exact phrases, frameworks, and red flags — walk in ready.

🎯 Interview Playbook — Databases

The "I Would Choose X Because..." Template

Never just name a database. Always follow this structure. Interviewers at Google and Meta are specifically trained to probe whether you understand trade-offs — or are just name-dropping.

The Template

"For [this component], I would use [Database] because the data is [schema characteristics], the access pattern is [read/write description], and the consistency requirement is [strong/eventual]. The trade-off is [what we give up], which is acceptable here because [reason]."

Example Answers — Copy These Patterns

Design Twitter's tweet storage. What database?

"Tweets are written once and read millions of times. The schema is simple (tweet_id, user_id, text, timestamp). There are no complex JOINs needed for serving tweets. The write volume is high but predictable. The read volume is enormous — a viral tweet can get millions of reads in minutes.

I'd use a combination: Cassandra for tweet storage (partition by user_id, cluster by timestamp — efficient for timeline reads), and Redis as a cache for the home timeline of active users. The trade-off: eventual consistency on reads, but for a social feed that's totally acceptable — nobody cares if they see a tweet 500ms late."

Design a payment processing system. What database?

"For payment transactions, I'd use PostgreSQL. Money movement requires ACID guarantees — I need atomicity so funds can't be half-transferred, isolation so concurrent payments on the same account don't cause race conditions, and durability so committed payments survive crashes.

The trade-off: SQL doesn't scale writes as horizontally as Cassandra. But payment write volume, even at Stripe scale, is manageable with a well-tuned primary + replicas + connection pooling. I'd rather have correctness than raw throughput for money. I'd never sacrifice ACID for payments — that's how you lose money and customers."

Your users table has grown to 500M rows. Queries are slow. What do you do?

Escalate in this order — don't jump straight to sharding:

  1. Check indexes first. Run EXPLAIN ANALYZE on slow queries. 90% of performance problems at this stage are missing indexes or bad query plans. Fix those first.
  2. Add read replicas if read load is the problem. Route reads to replicas.
  3. Add a caching layer (Redis) for frequently accessed user data (profile, session, preferences). Cache-hit rate should be 90%+ before touching the DB architecture.
  4. Vertical scale the DB instance (more RAM, faster SSDs). Cheaper and faster than sharding.
  5. Shard by user_id only if writes are the bottleneck and nothing else works. Use consistent hashing to allow adding shards later without full rehashing.
What's the difference between sharding and replication?

"Replication creates copies of the same data on multiple machines. It solves availability (if one dies, others take over) and read scalability (spread read load across copies). Every replica holds all the data.

Sharding splits data across machines. Each shard holds a subset of the data. It solves write scalability and storage limits — no single machine needs to hold everything.

In production you typically use both: shard your data for write scalability, and replicate each shard for availability. DynamoDB does this under the hood — every partition is replicated to 3 availability zones."

🚩 Red Flags — What NOT to Say

🚫
  • "I'll use MongoDB because it's flexible" — Flexibility is a cost, not a benefit. NoSQL flexibility means you're giving up schema enforcement and JOINs. Always justify the trade-off.
  • "I'll add more indexes to fix the slow query" — Without running EXPLAIN first, this is a guess. Show that you'd diagnose before prescribing.
  • "Let's just shard it" — Sharding is a last resort. Proposing it before exhausting simpler options signals inexperience.
  • Picking a database and never mentioning trade-offs — Every choice has costs. If you don't mention them, the interviewer will think you don't know them.
  • "We can handle consistency issues later" — Red flag for data integrity questions. Say instead: "The consistency requirement here is X, which this database handles by Y."
