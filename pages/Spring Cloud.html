Why Microservices Exist — The Problem Story

The Monolith Problem

Before microservices, every enterprise application was a monolith — a single, large, deployable unit containing ALL business logic. Imagine an e-commerce app where payment, inventory, orders, notifications, and user management all live in one giant codebase and get deployed together.

😤
Deployment Hell

Changing one line in payments requires deploying the ENTIRE app. High risk. Entire system down during deploy.

🐌
Scaling Inefficiency

Can't scale just the checkout service. Must scale the ENTIRE monolith, wasting resources and money.

💥
Single Point of Failure

One memory leak in recommendations brings down the entire platform. No isolation.

🔗
Technology Lock-in

Entire app must use same language, same DB, same framework. Impossible to modernize selectively.

👥
Team Bottlenecks

100 developers all working in same codebase — merge conflicts, coordination overhead, slow releases.

🧪
Testing Nightmares

Full regression test suite takes hours. Any change could break anything. Fearful releases.

The Microservices Solution

Microservices decompose the monolith into small, independently deployable services — each owning its own data, logic, and deployment lifecycle. Each service does one thing well (Single Responsibility Principle at the service level).

📱 Mobile App 🌐 Web Browser 🤖 3rd Party API

📘 Key Principle: Bounded Context

Each microservice owns a Bounded Context — a well-defined boundary in which a particular domain model applies. This comes from Domain-Driven Design (DDD). The Order Service doesn't know or care about the internals of the Inventory Service.

⚠ Martin Fowler's Warning

"Don't start with microservices. Start with a monolith, understand your domain boundaries, THEN extract microservices." — This is the Strangler Fig Pattern: gradually replace monolith pieces with microservices over time.

The "8 Fallacies of Distributed Computing" — Every Dev Must Know

These are false assumptions developers make when building distributed systems. They will burn you in production:

1

The network is reliable

Networks drop packets, connections timeout. Always handle network failures. Use retries, circuit breakers.

2

Latency is zero

In-process calls take nanoseconds. HTTP calls take milliseconds. 100 microservice hops = huge latency. Design for it.

3

Bandwidth is infinite

Don't send huge payloads over the wire. Use pagination, compression, and selective field responses.

4

The network is secure

Always use HTTPS, mTLS between services, authentication at every hop. Never trust the internal network blindly.

5

Topology doesn't change

Service instances go up and down. Use service discovery (Eureka) instead of hardcoded IPs.

6

There is one administrator

Distributed systems are operated by multiple teams. Observability, tracing, and centralized logging are essential.

7

Transport cost is zero

Serialization/deserialization of JSON/XML has CPU cost. Consider binary protocols (Protobuf, Avro) for high throughput.

8

The network is homogeneous

Different services may use different protocols. Don't assume all services speak the same language/protocol.

REST API Fundamentals — Deep Internals

What is REST?

REST (Representational State Transfer) is an architectural style defined by Roy Fielding in his 2000 PhD dissertation. It's NOT a protocol or standard — it's a set of constraints that, when followed, yield a predictable, scalable API.

💡 REST vs RESTful

REST is the set of constraints. An API that follows these constraints is called RESTful. Most APIs people call "REST APIs" are actually just HTTP APIs that partially follow REST principles.

REST's 6 Architectural Constraints

1. Client-Server Separation

Client and server evolve independently. The server doesn't know about the UI. Separation of concerns enables portability of the client and scalability of the server.

2. Statelessness

Each request contains ALL information needed. Server stores ZERO session state between requests. This is why JWT tokens or session tokens must be sent with every request.

3. Cacheability

Responses must declare themselves cacheable or non-cacheable (via Cache-Control headers). Caching eliminates some client-server interactions, improving scalability.

4. Uniform Interface

Standardized way to interact — resource identification in URIs, manipulation through representations, self-descriptive messages, HATEOAS.

5. Layered System

Client can't tell if it's talking directly to the server or through intermediaries (load balancers, API gateways, caches). Each layer only knows adjacent layers.

6. Code on Demand (Optional)

Servers can transfer executable code to clients (like JavaScript). The only optional constraint. Rarely used in microservice contexts.

REST Resource Design — The Right Way

Resources are nouns, not verbs. The HTTP method expresses the action.

❌ Bad (verb-based)✅ Good (noun-based)Why
/getUser/123GET /users/123GET expresses the action
/createOrderPOST /ordersPOST creates resources
/deleteProduct/5DELETE /products/5DELETE expresses intent

HATEOAS — The Most Misunderstood REST Constraint

HATEOAS (Hypermedia As The Engine Of Application State) means your API responses should include links telling the client what it can do next. The client should not need to hardcode URLs.

// HATEOAS Response — Client gets order, also learns what to do next
{
  "orderId": 12345,
  "status": "PENDING",
  "total": 99.99,
  "_links": {
    "self": { "href": "/orders/12345", "method": "GET" },
    "cancel": { "href": "/orders/12345/cancel", "method": "POST" },
    "payment": { "href": "/orders/12345/payment", "method": "POST" },
    "customer": { "href": "/customers/67890", "method": "GET" }
  }
}
ℹ Spring HATEOAS

Spring provides spring-boot-starter-hateoas. Use EntityModel<T> and WebMvcLinkBuilder to add links. Most production APIs skip HATEOAS due to complexity vs benefit tradeoff, but it's Level 3 of Richardson Maturity Model (discussed in Section 6).

🔑 HTTP/2 Multiplexing — Why It Matters for Microservices

In HTTP/1.1, each request needs its own TCP connection or must wait in line (pipelining has head-of-line blocking). In HTTP/2, multiple requests fly over ONE TCP connection simultaneously as separate "streams". This is crucial for high-throughput microservices. Spring WebFlux + Netty supports HTTP/2 natively.

TCP Connection Internals — What Happens Under the Hood

When your microservice makes an HTTP call, here's exactly what happens at the network level:

1

DNS Resolution

Convert hostname (order-service) to IP address. Can take 10-100ms if not cached. Service discovery (Eureka) bypasses public DNS for internal services.

2

TCP 3-Way Handshake

SYN → SYN-ACK → ACK. Takes 1 Round Trip Time (RTT). With connection pooling, this cost is amortized across many requests.

3

TLS Handshake (if HTTPS)

Certificate exchange, cipher negotiation, session key generation. Adds 1-2 RTT. TLS 1.3 reduces this to 1 RTT. Use TLS session resumption for repeated connections.

4

HTTP Request Sent

Request bytes travel over established TCP connection. Connection pooling means steps 2-3 happen only once per pool entry.

5

Server Processing

Application logic executes. DB queries, business rules, downstream calls. This is where 99% of your latency budget is spent.

6

Response + Connection Handling

With Keep-Alive, connection returned to pool. Without it, TCP 4-way handshake (FIN/FIN-ACK) closes connection. Always use connection pooling!

⚡ Connection Pooling is Critical

Creating new TCP+TLS connections for every HTTP request is catastrophically slow. RestTemplate, FeignClient, WebClient all use connection pools under the hood. Tune pool sizes based on your concurrency needs: max-connections, max-connections-per-route, connection-timeout, socket-timeout.

Idempotency — Critical for Distributed Systems

🚨 Why Idempotency Matters in Microservices

When a network call fails, you don't know if the server received it. Was the payment processed before the connection dropped? If you retry a non-idempotent operation, you might charge the customer twice. Always design your APIs to handle duplicate requests gracefully.

Idempotent = Calling the operation multiple times produces the same result as calling it once. The state of the system is identical after 1 call or 100 calls.

// Making POST idempotent with Idempotency-Key header
POST /api/v1/payments HTTP/1.1
Idempotency-Key: uuid-a1b2c3d4-e5f6  ← Client generates unique key per operation
Content-Type: application/json

{
  "orderId": "order_123",
  "amount": 99.99
}

// Server logic:
@PostMapping("/payments")
public ResponseEntity<Payment> processPayment(
    @RequestHeader("Idempotency-Key") String idempotencyKey,
    @RequestBody PaymentRequest request) {

  // Check if this key was already processed
  Optional<Payment> existing = paymentRepo.findByIdempotencyKey(idempotencyKey);
  if (existing.isPresent()) {
    return ResponseEntity.ok(existing.get()); // Return cached result
  }

  // Process payment and store with idempotency key
  Payment payment = paymentService.process(request, idempotencyKey);
  return ResponseEntity.status(201).body(payment);
}

PUT vs PATCH — The Practical Difference

// Current User: { "id": 1, "name": "Alice", "email": "alice@example.com", "age": 30 }

// PUT — Replace ENTIRE resource. Missing fields become null!
PUT /users/1
{ "name": "Alice Updated" }
// Result: { "id": 1, "name": "Alice Updated", "email": null, "age": null } ← DANGER!

// PATCH — Partial update. Only specified fields change.
PATCH /users/1
{ "name": "Alice Updated" }
// Result: { "id": 1, "name": "Alice Updated", "email": "alice@example.com", "age": 30 } ← SAFE

// Spring implementation
@PatchMapping("/{id}")
public User patchUser(@PathVariable Long id, @RequestBody Map<String, Object> updates) {
    User user = userRepo.findById(id).orElseThrow();
    // Only apply fields that are present in the request
    updates.forEach((key, value) -> {
        Field field = ReflectionUtils.findField(User.class, key);
        field.setAccessible(true);
        ReflectionUtils.setField(field, user, value);
    });
    return userRepo.save(user);
}

// Spring Boot Global Exception Handler
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.UNPROCESSABLE_ENTITY)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatus(422);
        pd.setTitle("Validation Failed");
        pd.setDetail("Request body validation failed");

        List<Map<String, String>> errors = ex.getBindingResult()
            .getFieldErrors().stream()
            .map(e -> Map.of("field", e.getField(), "message", e.getDefaultMessage()))
            .collect(Collectors.toList());

        pd.setProperty("errors", errors);
        pd.setProperty("traceId", MDC.get("traceId"));
        return pd;
    }

    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ProblemDetail handleNotFound(ResourceNotFoundException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setTitle("Resource Not Found");
        return pd;
    }

    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ProblemDetail handleGeneric(Exception ex) {
        ProblemDetail pd = ProblemDetail.forStatus(500);
        pd.setTitle("Internal Server Error");
        pd.setDetail("An unexpected error occurred"); // Never expose ex.getMessage() in prod!
        return pd;
    }
}

REST Design Principles — Richardson Maturity Model

Richardson Maturity Model (RMM)

Leonard Richardson defined a 4-level model to measure how "RESTful" an API is. Most APIs are at Level 2.

0

Level 0 — The Swamp of POX

Single endpoint, single method. Like SOAP or XML-RPC. Everything is a POST to one URL. No HTTP semantics used at all.

POST /api — action: "getUser", id: 123
1

Level 1 — Resources

Multiple URIs representing different resources. But still using only POST. Better than Level 0 — at least resources are distinct.

POST /users — body: action: "get", id: 123
2

Level 2 — HTTP Verbs (Where most APIs live)

Using proper HTTP methods (GET, POST, PUT, DELETE) and proper status codes. This is what most people mean when they say "REST API".

GET /users/123 → 200 OK | DELETE /users/123 → 204 No Content
3

Level 3 — Hypermedia Controls (HATEOAS)

Responses include links to related actions. Client drives state through hypermedia. True REST. Client doesn't need out-of-band documentation to navigate the API.

GET /users/123 → body includes _links: {activate, delete, orders}

API Versioning Strategies

API versioning is how you evolve your API without breaking existing clients. This is one of the most debated topics in REST API design.

StrategyExampleProsConsUse When
URL Path /api/v1/users Most visible, easy to test in browser, works everywhere Violates REST (URI should represent resource, not version), pollutes URL structure Public APIs, external consumers, most common choice
Query Param /api/users?version=1 URL stays clean, easy to test Optional = easy to forget, can be stripped by proxies Internal APIs only
Header API-Version: 1 Clean URLs, REST-compliant, allows granular versioning Not visible in URL, harder to test without tools, browser caching issues Enterprise APIs, internal services
Media Type (Content Negotiation) Accept: application/vnd.myapp.v1+json Most REST-compliant, leverages HTTP protocol Complex, verbose headers, hard to explain to developers Highly standards-compliant APIs

✅ Versioning Best Practices
  • Never break existing versions. Add new versions instead.
  • Support at minimum 2 versions simultaneously (current + previous).
  • Set a sunset date for deprecated versions and communicate it via Sunset and Deprecation headers.
  • Design for backward compatibility first — additive changes (new fields) don't need a new version.

Microservice Architecture Deep Dive

Microservice Design Principles

These are the guiding principles that separate well-designed microservices from distributed monoliths:

🎯 Single Responsibility

Each service owns ONE business capability. Payment Service handles payments. Not payments AND invoicing AND reporting.

📦 Own Your Data

Each service has its OWN database. No service directly reads another service's database. Communicate only via APIs or events.

🚀 Independently Deployable

Deploy Order Service without touching Payment Service. This requires backward-compatible API contracts.

🔄 Decentralized Governance

Each team chooses the best technology for their service. Java, Python, Go — whatever fits the problem.

💥 Design for Failure

Downstream services WILL fail. Use circuit breakers, timeouts, fallbacks, retries with backoff.

📊 Observability First

Logs, metrics, and traces from day one. You can't debug what you can't see. Centralized logging + distributed tracing is non-negotiable.

Service Decomposition Strategies

How do you decide what should be a separate service? Two main approaches:

🔵 Decompose by Business Capability (Preferred)

Align services with business functions as understood by domain experts. Examples: OrderManagement, CustomerRelationships, InventoryManagement, PaymentProcessing. Each maps to a business department or function that could exist independently.

🟣 Decompose by Subdomain (DDD)

Apply Domain-Driven Design to find bounded contexts. A "Customer" in the Order domain means something different than a "Customer" in the Loyalty Program domain. Each subdomain becomes a service with its own model.

Anti-Patterns to Avoid

Anti-PatternDescriptionFix
🚫 Distributed Monolith Services are physically separate but logically coupled. Every deploy requires updating multiple services simultaneously. Enforce contract-first API design, independent deployment capability
🚫 Shared Database Multiple services read/write the same database tables. One service's schema change breaks others. Database per service. Share data via APIs or events only.
🚫 Chatty Services Service A makes 10 calls to Service B for a single user request. Amplifies latency and failures. Aggregate APIs, batch requests, or switch to async events for non-critical data.
🚫 Too Fine Grained Nano-services that do almost nothing. Overhead of managing and networking outweighs benefits. Start coarser, extract only when pain is felt.
🚫 Synchronous All The Way Every operation is synchronous HTTP. Creates brittle chains of failure. Use async messaging (Kafka, RabbitMQ) for non-real-time operations.

Synchronous vs Asynchronous Communication

The Fundamental Difference

🔵 Synchronous (Blocking)

Caller waits for response. If called service is slow, caller is slow. If called service is down, caller fails. Direct HTTP/gRPC calls. Request-Reply pattern. Tight temporal coupling.


Examples: REST, gRPC, GraphQL

🟢 Asynchronous (Non-Blocking)

Caller fires message and continues. Consumer processes when ready. Caller and consumer are temporally decoupled. Consumer failure doesn't affect caller. Scales independently.


Examples: Kafka, RabbitMQ, SQS

↓ ↓→─→ Inventory Service
↓ ↓ waiting...
↓ ↓←─← Response
Client ←──────────────← Response
⚠ If Inventory is down, Order fails

↓ (immediately returns 202 Accepted)

⚡ The Key Decision Rule

Use synchronous when: You need the result NOW to continue processing (user is waiting for a response).
Use asynchronous when: You're triggering a side effect that can happen later (notification, audit log, index update, downstream data sync).

When one microservice needs to call another, you need an HTTP client. Spring provides several options, each with different trade-offs.

WebClient (Reactive — For High Concurrency)

WebClient is part of Spring WebFlux. It's non-blocking — it doesn't hold threads while waiting for responses. Under load, this is dramatically more efficient.

@Configuration
public class WebClientConfig {
    @Bean
    public WebClient userWebClient(WebClient.Builder builder) {
        return builder
            .baseUrl("http://user-service")
            .defaultHeader(HttpHeaders.CONTENT_TYPE, MediaType.APPLICATION_JSON_VALUE)
            .filter(ExchangeFilterFunction.ofRequestProcessor(req -> {
                System.out.println("Request: " + req.method() + " " + req.url());
                return Mono.just(req);
            }))
            .codecs(cfg -> cfg.defaultCodecs().maxInMemorySize(16 * 1024 * 1024)) // 16MB buffer
            .build();
    }
}

// Reactive (Non-Blocking) Usage
public Mono<User> getUserAsync(Long userId) {
    return webClient.get()
        .uri("/api/v1/users/{id}", userId)
        .retrieve()
        .onStatus(HttpStatus.NOT_FOUND::equals, resp ->
            resp.bodyToMono(String.class)
                .flatMap(body -> Mono.error(new UserNotFoundException(userId))))
        .bodyToMono(User.class)
        .timeout(Duration.ofSeconds(5))
        .onErrorReturn(User.defaultUser()); // Fallback
}

// Block (convert to sync) — only when you MUST (e.g., in non-reactive code)
User user = getUserAsync(userId).block(); // Avoid in reactive pipelines!

public Mono<OrderDetails> getOrderDetails(Long orderId) {

    Mono<User> userMono = getUser(orderId);
    Mono<List<Product>> productsMono = getProducts(orderId);

    return Mono.zip(orderMono, userMono, productsMono)
        .map(tuple -> new OrderDetails(
            tuple.getT1(), tuple.getT2(), tuple.getT3()
        ));
}

WebFlux and Reactive Programming Internals

The Problem with Traditional Blocking I/O

In a traditional Spring MVC app, each HTTP request consumes ONE thread for its entire duration. If the request makes a database query (50ms) and then an HTTP call (100ms), that thread is blocked and idle for 150ms. With 200 threads in the pool and 1000 concurrent users, you hit a wall.

🧵 Blocking vs Non-Blocking Thread Model
BLOCKING (Spring MVC)
Thread-1: [req1]─DB WAIT──HTTP WAIT─[done]
Thread-2: [req2]─DB WAIT──────────────...
Thread-3: [req3]─DB WAIT───────────────
Thread-4: [req4]───────────────────────
Thread-N: WAITING IN QUEUE... (501 users get rejected)
Default tomcat pool: ~200 threads. 200 concurrent = limit.
NON-BLOCKING (WebFlux + Netty)
EventLoop-1: req1..req2..req3..req4..req5..
→ fires DB call, returns to loop
→ DB responds, resumes req1
→ fires HTTP call, returns to loop
→ handles req2,3 while waiting...
Few threads, thousands of concurrent connections.

Mono and Flux — The Two Core Types

🔵 Mono<T>

A publisher that emits 0 or 1 item and then completes (or errors). Use for single-item responses: getting a user by ID, creating an order, etc.

🟣 Flux<T>

A publisher that emits 0 to N items and then completes (or errors). Use for collections, streams, pagination: getting all users, streaming events, etc.

@RestController
@RequestMapping("/api/v1/users")
public class UserController {

    // Mono — returns 0 or 1 user
    @GetMapping("/{id}")
    public Mono<User> getUser(@PathVariable Long id) {
        return userRepository.findById(id)  // Returns Mono
            .switchIfEmpty(Mono.error(new UserNotFoundException(id)));
    }

    // Flux — returns stream of users
    @GetMapping
    public Flux<User> getAllUsers() {
        return userRepository.findAll();  // Returns Flux
    }

    // Server-Sent Events streaming
    @GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<User> streamUsers() {
        return userRepository.findAll()
            .delayElements(Duration.ofMillis(100)); // Push one every 100ms
    }

    // Complex reactive pipeline
    @PostMapping
    public Mono<ResponseEntity<User>> createUser(@RequestBody Mono<CreateUserRequest> request) {
        return request
            .flatMap(req -> userRepository.findByEmail(req.getEmail())  // Check duplicate
                .flatMap(existing -> Mono.<User>error(new DuplicateEmailException()))
                .switchIfEmpty(userRepository.save(userMapper.toEntity(req)))
            )
            .map(user -> ResponseEntity.status(201).body(user));
    }
}

// R2DBC — Reactive database access (no blocking JDBC!)
public interface UserRepository extends ReactiveCrudRepository<User, Long> {
    Mono<User> findByEmail(String email);
    Flux<User> findByStatus(String status);
}
⚠ The Golden Rule of Reactive Programming

NEVER block inside a reactive pipeline. No Thread.sleep(), no blocking DB calls (JdbcTemplate), no .block() calls inside reactive chains. This defeats the purpose entirely and can cause deadlocks. Use R2DBC for reactive database access, not JPA.

Backpressure — Reactive's Superpower

Backpressure is the ability of a consumer to signal to the producer how fast it can consume. Without it, a slow consumer gets overwhelmed by a fast producer (OutOfMemoryError).

// Backpressure strategies
Flux.range(1, 1000000)  // Producer generates 1M items fast
    .onBackpressureBuffer(100)  // Buffer up to 100, drop rest
    .publishOn(Schedulers.boundedElastic())
    .subscribe(
        item -> processSlowly(item),  // Slow consumer
        error -> log.error("Error", error),
        () -> log.info("Completed")
    );

Eureka's Self-Preservation Mode — Understanding and Dangers

⚠ Self-Preservation Mode Explained

When Eureka server stops receiving heartbeats from more than 85% of instances, it assumes a network partition (not that services are down) and stops evicting instances. This prevents mass eviction during a brief network hiccup. Problem: in dev, when you stop a service, Eureka keeps it in registry for minutes. Always disable in dev, leave enabled in prod.

Custom Gateway Filter

@Component
public class AuthenticationFilter implements GlobalFilter, Ordered {

    @Autowired
    private JwtTokenProvider jwtProvider;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();

        // Whitelist public endpoints
        if (isPublicEndpoint(request.getPath().toString())) {
            return chain.filter(exchange);
        }

        String authHeader = request.getHeaders().getFirst("Authorization");

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            return onError(exchange, "Missing or invalid Authorization header", HttpStatus.UNAUTHORIZED);
        }

        String token = authHeader.substring(7);
        try {
            Claims claims = jwtProvider.validateAndGetClaims(token);
            // Add user info to downstream request headers
            ServerHttpRequest mutatedRequest = request.mutate()
                .header("X-User-Id", claims.getSubject())
                .header("X-User-Role", claims.get("role", String.class))
                .build();
            return chain.filter(exchange.mutate().request(mutatedRequest).build());
        } catch (JwtException e) {
            return onError(exchange, "Invalid JWT token", HttpStatus.UNAUTHORIZED);
        }
    }

    private Mono<Void> onError(ServerWebExchange exchange, String message, HttpStatus status) {
        ServerHttpResponse response = exchange.getResponse();
        response.setStatusCode(status);
        response.getHeaders().add("Content-Type", "application/json");
        DataBuffer buffer = response.bufferFactory().wrap(
            ("{\"error\": \"" + message + "\", \"status\": " + status.value() + "}").getBytes()
        );
        return response.writeWith(Mono.just(buffer));
    }

    @Override
    public int getOrder() { return -1; } // Run before other filters
}

Circuit Breaker — State Machine Deep Dive

The Problem: Cascading Failures

Imagine Order Service depends on Inventory Service. Inventory Service becomes slow (100ms → 5000ms). Now Order Service threads pile up waiting for responses. Order Service's thread pool exhausts. Users see Order Service as down. They retry. More load. Everything cascades. The Circuit Breaker prevents this.

The State Machine

✅ CLOSED
Normal operation. Requests flow through. Failures counted. If failure rate exceeds threshold → OPEN
🚫 OPEN
Calls blocked immediately. No requests reach the service. Fallback called. After wait duration → HALF-OPEN
⚡ HALF-OPEN
Probe state. Limited requests allowed through. If enough succeed → CLOSED. If fail again → OPEN
// Transitions between states

CLOSED (normal):
  Track: success/failure ratio in sliding window
  Threshold: failureRateThreshold (e.g., 50%)
  Slow calls: slowCallRateThreshold also counted as failures
  If failure rate > threshold → OPEN
  
OPEN (blocking):
  Duration: waitDurationInOpenState (e.g., 60 seconds)
  All calls immediately fail with CallNotPermittedException
  Fallback method is invoked
  After duration → HALF_OPEN
  
HALF_OPEN (testing):
  Allow: permittedNumberOfCallsInHalfOpenState (e.g., 10 calls)
  If success rate > successThreshold → CLOSED
  If failures occur → back to OPEN

Sliding Window Types

📊 COUNT_BASED Window

Track last N calls. If last 100 calls have more than 50% failures → open. Best when traffic is consistent. Configuration: slidingWindowType: COUNT_BASED, slidingWindowSize: 100

⏱ TIME_BASED Window

Track calls in last N seconds. If in last 60 seconds more than 50% failed → open. Best when traffic varies. Configuration: slidingWindowType: TIME_BASED, slidingWindowSize: 60 (seconds)

Resilience4j Complete Internals

Resilience4j Modules Overview

⚡ Circuit Breaker

Stops calls to a failing service. The core fault-tolerance pattern.

🔄 Retry

Automatically retry failed operations with configurable backoff. Handles transient failures.

⏱ Rate Limiter

Limit the number of calls a service makes to a dependency in a time period.

🏊 Bulkhead

Isolate different parts of the system. Prevent one failing operation from consuming all threads.

⏰ Time Limiter

Cancel operations that take too long. Prevents thread exhaustion from slow calls.

💾 Cache

Cache results of operations to reduce load on dependencies.

Applying Resilience4j with Annotations

@Service
public class OrderService {

    @CircuitBreaker(name = "userServiceCB", fallbackMethod = "getUserFallback")
    @Retry(name = "userServiceRetry", fallbackMethod = "getUserFallback")
    @TimeLimiter(name = "userServiceCB")
    @Bulkhead(name = "userServiceBH")
    public CompletableFuture<User> getUserById(Long userId) {
        return CompletableFuture.supplyAsync(() ->
            userClient.getUserById(userId)
        );
    }

    // Fallback methods — must have same signature + extra Throwable param
    private CompletableFuture<User> getUserFallback(Long userId, CallNotPermittedException ex) {
        log.warn("Circuit breaker OPEN for user {}: {}", userId, ex.getMessage());
        return CompletableFuture.completedFuture(
            User.builder().id(userId).name("Service Unavailable").build()
        );
    }

    private CompletableFuture<User> getUserFallback(Long userId, Throwable ex) {
        log.error("User service failed for id {}: {}", userId, ex.getMessage());
        return CompletableFuture.completedFuture(
            User.defaultUser(userId)
        );
    }
}

// Important: Annotation order matters! 
// Processing order (outer to inner): TimeLimiter → CircuitBreaker → Retry → Bulkhead
⚡ Retry + Circuit Breaker: Be Careful!

If you stack Retry (3 attempts) with CircuitBreaker (50% failure threshold over 10 calls), and 5 calls each retry 3 times, that's 15 calls to the circuit breaker. The circuit may open faster than expected. Design the interaction carefully. Usually: CircuitBreaker wraps Retry (outer layer) so that if the circuit opens during retries, remaining attempts are short-circuited immediately.

Distributed Tracing Internals

The Problem: Debugging Across Services

A user reports their order is slow. The request touches: API Gateway → Order Service → User Service → Inventory Service → Payment Service. Which one is slow? Regular logs per service don't help because you can't correlate them.

🔍 Distributed Tracing — Trace and Span Model
TraceID: abc-123 (spans entire request)
API Gateway [SpanID: span-1 ←─────────────────────────────────────── 450ms ──────────────────────────────]
Order Service [SpanID: span-2 ←────────────────────────────── 420ms ─────────────────────────]
User Service [SpanID: span-3 ←── 30ms ──]
Inventory Svc [SpanID: span-4 ←──────────────── 200ms ──────────] ← THIS IS SLOW!
Payment Svc [SpanID: span-5 ←── 80ms ──]
↑ You can now pinpoint: Inventory Service taking 200ms is the bottleneck

Key Concepts

ConceptDescriptionExample
Trace The entire journey of a single request across all services. Has one unique Trace ID. traceId: abc-123-def-456
Span A single unit of work within a trace. Has a Span ID, start time, end time, tags. Can have a parent span. spanId: span-001, parentSpanId: null
Trace Context The TraceID + SpanID propagated in HTTP headers between services. X-B3-TraceId, X-B3-SpanId headers (Brave/Zipkin format)
Sampling What % of requests to trace. 100% in dev, 1-10% in prod to manage overhead. management.tracing.sampling.probability: 0.1

Distributed Transactions and Saga Pattern

Why ACID Doesn't Work Across Services

Traditional ACID transactions rely on a single database and a transaction coordinator. In microservices, each service has its OWN database. You cannot do a BEGIN TRANSACTION across Service A's PostgreSQL AND Service B's MongoDB simultaneously.

🚨 The Distributed Transaction Problem

Scenario: Place order → Deduct inventory → Charge payment. What if payment fails after inventory is deducted? You need to roll back inventory. But inventory service has already committed. This is the distributed transaction problem — and it's why Saga exists.

Two-Phase Commit (2PC) — Why We Don't Use It

2PC uses a coordinator to orchestrate a commit across multiple services. Phase 1: coordinator asks all participants "can you commit?". Phase 2: if all say yes → commit; if any says no → abort. Problems: coordinator is single point of failure, blocking protocol (all services block during Phase 1), not supported by most NoSQL databases, poor performance.

The Saga Pattern — The Solution

A Saga is a sequence of local transactions. Each local transaction updates a single service's database and publishes an event or message. If a step fails, compensating transactions are executed to undo the effects of preceding steps.

🎭 Choreography-Based Saga

Each service listens for events and reacts. No central coordinator. Services publish events when they complete their local transaction. Decentralized. Risk of circular dependencies.

Good for: simple flows, few services, event-driven systems
🎪 Orchestration-Based Saga

A central Saga Orchestrator tells each service what to do and handles failures. Easier to visualize, monitor, and debug. Introduces a central coordinator (can use Axon, Conductor, Temporal).

Good for: complex flows, many services, clear ownership needed
📊 Order Saga — Choreography Pattern
Order Service → ORDER_CREATED event
↓
Inventory Service consumes → reserves stock → publishes INVENTORY_RESERVED event
↓

Outbox Pattern — Reliable Event Publishing

⚠ The Dual Write Problem

What if Order Service saves the order successfully to DB, but then the Kafka publish FAILS? The order is saved but the saga never starts. This is the dual write problem. Solution: Transactional Outbox Pattern.

// Outbox Pattern — atomically save order AND event in SAME transaction
@Transactional
public Order createOrder(CreateOrderRequest req) {
    // 1. Save order
    Order order = orderRepo.save(Order.pending(req));

    // 2. Save event to outbox table IN SAME TRANSACTION
    OutboxEvent outboxEvent = OutboxEvent.builder()
        .aggregateId(order.getId())
        .eventType("ORDER_CREATED")
        .payload(objectMapper.writeValueAsString(new OrderCreatedEvent(order)))
        .status(EventStatus.PENDING)
        .build();
    outboxRepo.save(outboxEvent);
    // DB transaction commits BOTH atomically. No dual write problem!
    return order;
}

// Separate Outbox Publisher (polls and publishes)
@Scheduled(fixedDelay = 1000)  // Every 1 second
public void publishPendingEvents() {
    outboxRepo.findByStatus(EventStatus.PENDING)
        .forEach(event -> {
            try {
                kafkaTemplate.send(event.getTopic(), event.getPayload()).get();
                outboxRepo.updateStatus(event.getId(), EventStatus.PUBLISHED);
            } catch (Exception e) {
                outboxRepo.updateStatus(event.getId(), EventStatus.FAILED);
                log.error("Failed to publish event: {}", event.getId(), e);
            }
        });
}

@Service

    public void handleOrderCreated(
            @Payload OrderCreatedEvent event,

        log.info("Processing order {} from partition {} at offset {}",
            event.getOrderId(), partition, offset);

        try {
            inventoryService.reserveStock(event.getOrderId(), event.getItems());

            log.error("Unexpected error processing order {}", event.getOrderId(), e);
            // Don't acknowledge — message will be redelivered
            // After max retries → Dead Letter Topic
        }
    }

Cascading Failures and Retry Storm

How Cascading Failures Happen

T+0s: Database gets slow
Inventory DB has lock contention. Queries take 10s instead of 100ms.
T+5s: Inventory Service threads exhaust
200 threads all blocked waiting for DB. Thread pool full. New requests queue up.
T+10s: Order Service starts failing
Order Service calls Inventory Service. Gets timeout. Retries (no circuit breaker). More connections pile up.
T+15s: Order Service threads exhaust
Order Service now also has full thread pool. Can't accept new requests.
T+20s: API Gateway overloaded
Gateway queues requests for Order Service. Memory fills. Gateway starts failing.
T+25s: Complete outage
Everything is down. Root cause was one slow DB. Could have been isolated with circuit breaker.

Retry Storm (Thundering Herd Problem)

When a service comes back online after being down, ALL waiting clients retry simultaneously. This creates a massive spike that can immediately bring the service down again.

🚨 Retry Storm Prevention
  • Exponential Backoff: Each retry waits longer: 1s → 2s → 4s → 8s
  • Jitter: Add random delay to backoff: 1s±0.5s → 2s±1s → 4s±2s. Spreads load.
  • Max Retries: Don't retry forever. Give up after 3-5 attempts.
  • Circuit Breaker: If service is down, stop retrying immediately. Wait for probe period.

resilience4j:
  retry:

      paymentRetry:

        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2  # 1s, 2s, 4s, 8s
        enableRandomizedWait: true
        randomizedWaitFactor: 0.5     # ±50% jitter: 0.5s-1.5s, 1s-3s, 2s-6s

Eventual Consistency Deep Dive

CAP Theorem — The Foundation

Eric Brewer's CAP Theorem states that a distributed system can guarantee only two of three properties simultaneously:

🔵 C — Consistency

Every read receives the most recent write or an error. All nodes see the same data at the same time.

🟢 A — Availability

Every request receives a (non-error) response, even if it's not the most recent data. System is always up.

🟠 P — Partition Tolerance

System continues operating even if network partitions (messages lost between nodes) occur. In distributed systems, P is mandatory — network WILL partition.

📘 CAP in Practice: You must choose CP or AP

Since network partitions (P) WILL happen, you must choose between Consistency (CP — like Zookeeper, HBase) or Availability (AP — like Eureka, Cassandra, DynamoDB). Most microservice databases are AP systems, accepting eventual consistency for higher availability.

Eventual Consistency in Practice

When Order Service creates an order and publishes an event to Kafka, the Inventory Service doesn't see the inventory update instantaneously. There's a small window where the inventory count in Inventory Service is "stale". This is eventual consistency — eventually, all nodes will be consistent.

What is the difference between Spring @Transactional and manual transaction management?

The N+1 problem occurs when fetching a list of N entities triggers N additional queries to load their associations — one query for the list plus one per entity.

How does Spring Security authentication work?
Spring Security authentication follows this flow:

  1. Request hits the Security Filter Chain.
  2. UsernamePasswordAuthenticationFilter extracts credentials.
  3. An Authentication object is passed to AuthenticationManager.
  4. AuthenticationManager delegates to an AuthenticationProvider.
  5. Provider calls UserDetailsService.loadUserByUsername().
  6. On success, the Authentication is stored in SecurityContextHolder.
@Service
public class CustomUserDetailsService implements UserDetailsService {

            .withUsername(user.getUsername())
            .password(user.getPassword())        // BCrypt hashed
            .roles(user.getRole())
            .build();
    }
}

What is JWT and how is it used for stateless authentication?
JWT (JSON Web Token) is a compact, self-contained token composed of three Base64-encoded parts: Header.Payload.Signature.

Flow:
  1. User logs in → server validates credentials → issues a signed JWT.
  2. Client stores token (localStorage / cookie).
  3. Client sends Authorization: Bearer <token> on every request.
  4. Server verifies signature → extracts claims → authorises.
  5. No session stored on server → stateless.
// Generate a JWT
public String generateToken(UserDetails userDetails) {
    return Jwts.builder()
        .setSubject(userDetails.getUsername())
        .setIssuedAt(new Date())
        .setExpiration(new Date(System.currentTimeMillis() + 86_400_000)) // 24h
        .signWith(secretKey, SignatureAlgorithm.HS256)
        .compact();
}

// Validate and extract username
public String extractUsername(String token) {
    return Jwts.parserBuilder()
        .setSigningKey(secretKey)
        .build()
        .parseClaimsJws(token)
        .getBody()
        .getSubject();
}

What is database indexing and when should you use it?
An index is a data structure (usually a B-Tree) that allows the database to locate rows without scanning the entire table.

Use an index when:
  • Columns are frequently used in WHERE, JOIN ON, ORDER BY, or GROUP BY.
  • Column has high cardinality (many distinct values).
  • Table has many rows and full scans are slow.
Avoid over-indexing — indexes slow down INSERT / UPDATE / DELETE because the index must be maintained.

-- Single-column index
CREATE INDEX idx_orders_customer_id ON orders(customer_id);

-- Composite index (order matters – most selective first)

-- Unique index (also enforces constraint)
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- Partial index (PostgreSQL) – only index active users
CREATE INDEX idx_active_users ON users(email) WHERE active = true;

    public ProductDTO getProduct(Long id) {
        log.info("Loading from DB...");  // only prints on cache MISS
        return productRepo.findById(id)
            .map(productMapper::toDTO)
            .orElseThrow(() -> new ProductNotFoundException(id));
    }

    // Update cache when product changes
    @CachePut(value = "products", key = "#result.id")
    public ProductDTO updateProduct(Long id, UpdateProductRequest req) {
        Product product = productRepo.findById(id)
            .orElseThrow(() -> new ProductNotFoundException(id));

    }

    @CacheEvict(value = "products", key = "#id")

        productRepo.deleteById(id);
    }

    // Clear entire cache
    @CacheEvict(value = "products", allEntries = true)

Approach 1 – Using HashSet (O(n)) 
public List<Integer> findDuplicates(List<Integer> numbers) {
    Set<Integer> seen      = new HashSet<>();
    Set<Integer> duplicates = new HashSet<>();

    for (int num : numbers) {
        if (!seen.add(num)) {      // add() returns false if already present
            duplicates.add(num);
        }
    }
    return new ArrayList<>(duplicates);
}

Approach 2 – Using Streams (functional) 
public List<Integer> findDuplicatesStream(List<Integer> numbers) {
    return numbers.stream()
        .collect(Collectors.groupingBy(
            Function.identity(),
            Collectors.counting()
        ))
        .entrySet().stream()
        .filter(e -> e.getValue() > 1)
        .map(Map.Entry::getKey)
        .collect(Collectors.toList());
}

// With frequency count
public Map<Integer, Long> findDuplicatesWithCount(List<Integer> numbers) {
    return numbers.stream()
        .collect(Collectors.groupingBy(
            Function.identity(),
            Collectors.counting()
        ))
        .entrySet().stream()
        .filter(e -> e.getValue() > 1)
        .collect(Collectors.toMap(Map.Entry::getKey,
                                   Map.Entry::getValue));
}

// Test

Approach 3 – Sorted array (O(n log n), O(1) space) 
public List<Integer> findDuplicatesSorted(List<Integer> numbers) {
    List<Integer> sorted = new ArrayList<>(numbers);
    Collections.sort(sorted);
    List<Integer> result = new ArrayList<>();

    for (int i = 1; i < sorted.size(); i++) {
        if (sorted.get(i).equals(sorted.get(i - 1))
                && (result.isEmpty() || !result.get(result.size()-1)
                                                .equals(sorted.get(i)))) {
            result.add(sorted.get(i));
        }
    }
    return result;
}

What happens if you call a @Transactional method from within the same class?
The transaction is NOT applied. Spring's @Transactional works via an AOP proxy. When you call a method from within the same bean, you bypass the proxy and call the method directly — the transaction annotation is ignored.

@Service
public class OrderService {

    // PROBLEM – internal call bypasses proxy
    public void placeOrder(CreateOrderRequest req) {
        validateOrder(req);
        createOrder(req);   // ← direct internal call – @Transactional ignored!
    }

    @Transactional
    public void createOrder(CreateOrderRequest req) {
        // This transaction NEVER starts when called from placeOrder()
        orderRepo.save(new Order(req));
    }
}

// SOLUTION 1 – Move createOrder to a separate bean
@Service
public class OrderCreationService {
    @Transactional
    public void createOrder(CreateOrderRequest req) {
        orderRepo.save(new Order(req));
    }
}

@Service
public class OrderService {
    private final OrderCreationService orderCreationService;

    public void placeOrder(CreateOrderRequest req) {
        validateOrder(req);
        orderCreationService.createOrder(req); // goes through proxy ✔
    }
}

// SOLUTION 2 – Self-inject the proxy (less preferred)
@Service
public class OrderService {
    @Autowired
    private OrderService self; // injects the proxy

    public void placeOrder(CreateOrderRequest req) {
        validateOrder(req);
        self.createOrder(req); // through proxy ✔
    }

    @Transactional
    public void createOrder(CreateOrderRequest req) {
        orderRepo.save(new Order(req));
    }
}
What is the difference between @Transactional(readOnly = true) and without it?
Setting readOnly = true gives Spring and the database a hint that no data will be modified in this transaction. Benefits:
  • Hibernate skips dirty-checking (no need to track changes).
  • Database may optimise (e.g. read from replica).
  • Prevents accidental writes.

What happens when two @Transactional methods call each other? (Transaction Propagation)
Spring's transaction propagation defines what happens to the transaction when a transactional method is called from another transactional method.

PropagationBehaviour
REQUIRED (default)Join existing transaction or create new one
REQUIRES_NEWAlways create new transaction, suspend existing
NESTEDCreate savepoint within existing transaction
SUPPORTSJoin if exists, run without if not
NOT_SUPPORTEDSuspend existing, run without transaction
NEVERThrow exception if transaction exists
MANDATORYThrow exception if NO transaction exists
 
@Service
public class OrderService {

    @Autowired
    private AuditService auditService;

    // REQUIRED (default) – auditService joins this transaction
    // if OrderService rolls back, audit log also rolls back
    @Transactional
    public void placeOrder(CreateOrderRequest req) {
        orderRepo.save(new Order(req));
        auditService.log("ORDER_PLACED");  // same transaction
    }
}

@Service
public class AuditService {

    // REQUIRES_NEW – audit log saved even if order fails
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String action) {
        auditRepo.save(new AuditLog(action, LocalDateTime.now()));
        // runs in its own transaction – independent of caller
    }

    // NESTED – savepoint created; inner rollback doesn't affect outer
    @Transactional(propagation = Propagation.NESTED)
    public void logNested(String action) {
        auditRepo.save(new AuditLog(action, LocalDateTime.now()));
    }
}

Common Mistakes

❌ Mistake 1 – Using Field Injection instead of Constructor Injection
Why it's a mistake: Field injection hides dependencies, makes testing harder (can't inject mocks without reflection), and the class can be instantiated without its dependencies being set.

// ❌ WRONG – field injection
@Service
public class OrderService {
    @Autowired
    private OrderRepository orderRepo;    // hidden dependency

    @Autowired
    private PaymentService paymentService; // hidden dependency
}

// Unit test is forced to use reflection or Spring context
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    @InjectMocks OrderService orderService;  // fragile
}

// ✅ CORRECT – constructor injection
@Service
public class OrderService {
    private final OrderRepository   orderRepo;
    private final PaymentService    paymentService;

    // Dependencies are explicit and mandatory
    public OrderService(OrderRepository orderRepo,
                        PaymentService  paymentService) {
        this.orderRepo       = orderRepo;
        this.paymentService  = paymentService;
    }
}

// Unit test is clean and explicit
class OrderServiceTest {
    private OrderService orderService;

    @BeforeEach
    void setUp() {
        orderService = new OrderService(
            mock(OrderRepository.class),
            mock(PaymentService.class)
        );
    }
}
❌ Mistake 2 – Catching Exception or Throwable and swallowing it
Why it's a mistake: Swallowing exceptions hides bugs, makes debugging impossible, and can leave the system in an invalid state silently.

// ❌ WRONG – swallowing exception
public void processPayment(Payment payment) {
    try {
        paymentGateway.charge(payment);
    } catch (Exception e) {
        // Silent catch – error disappears completely
        // System thinks payment succeeded when it didn't!
    }
}

// ❌ WRONG – catching Throwable (catches OutOfMemoryError too!)
try {
    riskyOperation();
} catch (Throwable t) {
    // Never do this
}

// ✅ CORRECT – catch specific exception, log, and handle or rethrow
public void processPayment(Payment payment) {
    try {
        paymentGateway.charge(payment);
    } catch (PaymentGatewayException e) {
        log.error("Payment failed for orderId={}, reason={}",
                   payment.getOrderId(), e.getMessage(), e);
        throw new PaymentProcessingException(
            "Payment processing failed: " + e.getMessage(), e);
    } catch (NetworkTimeoutException e) {
        log.warn("Payment gateway timeout for orderId={}",
                  payment.getOrderId());
        throw new ServiceUnavailableException("Payment service timeout", e);
    }
}
❌ Mistake 3 – N+1 queries due to LAZY loading without JOIN FETCH
Why it's a mistake: Loading 100 orders and then accessing their customers fires 101 SQL queries instead of 1, causing severe performance degradation.

// ❌ WRONG – triggers N+1 queries

            String name = o.getCustomer().getName(); // N queries!
            return new OrderDTO(o.getId(), name);
        })
        .collect(Collectors.toList());
    // Total: 1 + N SQL queries
}

❌ Mistake 4 – Storing plain text passwords
Why it's a mistake: If the database is compromised, all user passwords are exposed immediately. Passwords must always be hashed with a slow, salted algorithm.

// ❌ WRONG – plain text password storage
public User register(RegisterRequest req) {
    User user = new User();
    user.setEmail(req.getEmail());
    user.setPassword(req.getPassword()); // NEVER do this!
    return userRepo.save(user);
}

// ❌ WRONG – MD5 or SHA-1 (fast = crackable)
String hash = DigestUtils.md5Hex(password); // Not safe!

// ✅ CORRECT – BCrypt (slow, salted, industry standard)
@Service
public class UserService {
    private final PasswordEncoder passwordEncoder;

    public UserService(PasswordEncoder passwordEncoder) {
        this.passwordEncoder = passwordEncoder;
    }

    public User register(RegisterRequest req) {
        User user = new User();
        user.setEmail(req.getEmail());
        user.setPassword(passwordEncoder.encode(req.getPassword())); // BCrypt
        return userRepo.save(user);
    }

    public boolean verifyPassword(String raw, String hashed) {
        return passwordEncoder.matches(raw, hashed); // constant-time comparison
    }
}

// Configuration
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // strength 12 = ~250ms per hash
}

❌ Mistake 7 – Using == to compare Strings and Objects
Why it's a mistake: == compares references not content. Two strings with identical characters may be different objects in memory, making == return false unexpectedly.

// ❌ WRONG

    processOrder(order);
}

// ❌ WRONG – comparing wrapper types with ==
Integer a = 200;
Integer b = 200;
if (a == b) { ... }  // false! (outside -128 to 127 cache range)

Integer x = 100;
Integer y = 100;
if (x == y) { ... }  // true only by coincidence (cached range)

// ✅ CORRECT – use .equals() for objects
if ("PENDING".equals(status)) {   // null-safe when literal is on left
    processOrder(order);
}

// ✅ CORRECT – use Objects.equals() for null safety
if (Objects.equals(status, "PENDING")) {
    processOrder(order);
}

// ✅ CORRECT – use enum instead of string for status
public enum OrderStatus { PENDING, CONFIRMED, SHIPPED, DELIVERED }

if (order.getStatus() == OrderStatus.PENDING) { // enums use == correctly
    processOrder(order);
}

// ✅ CORRECT – Integer comparison
Integer a = 200, b = 200;
if (a.equals(b)) { ... }  // always correct

Requirements: Users submit a long URL, receive a short code (e.g. app.io/xK9mP). Visiting the short URL redirects to the original.

Step 2 – Short Code Generation 
@Service
public class UrlShortenerService {

    private static final String ALPHABET =
        "abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ0123456789";
    private static final int CODE_LENGTH = 6;

    private final ShortUrlRepository repo;
    private final Random random = new SecureRandom();

    // Strategy 1 – Random code with uniqueness check
    public String generateCode() {
        String code;
        do {
            StringBuilder sb = new StringBuilder();
            for (int i = 0; i < CODE_LENGTH; i++) {
                sb.append(ALPHABET.charAt(random.nextInt(ALPHABET.length())));
            }
            code = sb.toString();
        } while (repo.existsByShortCode(code));  // retry if collision
        return code;
    }

    // Strategy 2 – Base62 encoding of the auto-incremented ID
    public String encodeId(long id) {
        StringBuilder sb = new StringBuilder();
        while (id > 0) {
            sb.append(ALPHABET.charAt((int)(id % 62)));
            id /= 62;
        }
        return sb.reverse().toString();
    }
}

🏗️ Design Problem 2 – Design a Rate Limiter
Requirements: Limit each user/IP to N requests per time window. Exceed the limit → return 429 Too Many Requests.

Approach 1 – In-memory with Bucket4j 
// pom.xml
<dependency>
    <groupId>com.github.vladimir-bukhtoyarov</groupId>
    <artifactId>bucket4j-core</artifactId>
    <version>8.3.0</version>
</dependency>

@Component
public class RateLimiterFilter extends OncePerRequestFilter {

    // One bucket per client IP
    private final Map<String, Bucket> buckets = new ConcurrentHashMap<>();

    private Bucket createBucket() {
        return Bucket.builder()
            .addLimit(Bandwidth.classic(
                100,                               // 100 requests
                Refill.greedy(100, Duration.ofMinutes(1)) // per minute
            ))
            .build();
    }

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                    HttpServletResponse res,
                                    FilterChain chain)
            throws ServletException, IOException {

        String clientIp = req.getRemoteAddr();
        Bucket bucket   = buckets.computeIfAbsent(clientIp, k -> createBucket());

        if (bucket.tryConsume(1)) {
            chain.doFilter(req, res);
        } else {
            res.setStatus(HttpStatus.TOO_MANY_REQUESTS.value());
            res.setHeader("X-RateLimit-Retry-After", "60");
            res.getWriter().write("Rate limit exceeded. Try again in 60 seconds.");
        }
    }
}
Approach 2 – Redis sliding window (distributed) 
@Service
public class RedisRateLimiter {

    private final StringRedisTemplate redis;
    private static final int  LIMIT  = 100;
    private static final long WINDOW = 60; // seconds

    public boolean isAllowed(String clientId) {
        String key    = "rate:" + clientId;
        long   now    = System.currentTimeMillis();
        long   window = now - (WINDOW * 1000);

        // Sliding window log algorithm
        redis.opsForZSet().removeRangeByScore(key, 0, window);
        Long count = redis.opsForZSet().zCard(key);

        if (count == null || count < LIMIT) {
            redis.opsForZSet().add(key, String.valueOf(now), now);
            redis.expire(key, Duration.ofSeconds(WINDOW));
            return true;
        }
        return false;
    }
}

// Strategy pattern – one interface, multiple implementations
public interface NotificationChannel {
    void send(NotificationMessage message);
    ChannelType getType();
}

@Component
public class EmailChannel implements NotificationChannel {
    private final JavaMailSender mailSender;

    @Override
    public void send(NotificationMessage msg) {
        SimpleMailMessage email = new SimpleMailMessage();
        email.setTo(msg.getRecipient());
        email.setSubject(msg.getSubject());
        email.setText(msg.getBody());
        mailSender.send(email);
        log.info("Email sent to {}", msg.getRecipient());
    }

    @Override public ChannelType getType() { return ChannelType.EMAIL; }
}

@Component
public class SmsChannel implements NotificationChannel {
    @Override
    public void send(NotificationMessage msg) {
        smsGateway.sendSms(msg.getRecipient(), msg.getBody());
    }
    @Override public ChannelType getType() { return ChannelType.SMS; }
}

// Factory – select channel dynamically
@Service
public class NotificationService {

    private final Map<ChannelType, NotificationChannel> channels;

    // Spring injects all NotificationChannel implementations
    public NotificationService(List<NotificationChannel> channelList) {
        this.channels = channelList.stream()
            .collect(Collectors.toMap(
                NotificationChannel::getType,
                Function.identity()
            ));
    }

    // Send via all user-preferred channels
    @Async
    public void notify(User user, NotificationMessage message) {
        user.getPreferredChannels().forEach(channelType -> {
            NotificationChannel channel = channels.get(channelType);
            if (channel != null) {
                try {
                    channel.send(message);
                } catch (Exception e) {
                    log.error("Failed to send via {}: {}",
                               channelType, e.getMessage());
                    // Save failed notification for retry
                    failedNotificationRepo.save(
                        new FailedNotification(user.getId(), channelType, message));
                }
            }
        });
    }
}
