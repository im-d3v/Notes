Section 0
Thread Creation — 4 Ways
extends Thread, implements Runnable, Callable+Future, Virtual Threads

🔷 Way 1 — Extend Thread class

Override run() and call start(). Limitation: cannot extend any other class.

Javaextends Thread
class MyThread extends Thread {
    private String taskName;

    MyThread(String name) {
        super(name);           // sets thread name
        this.taskName = name;
    }

    @Override
    public void run() {
        System.out.println(
            Thread.currentThread().getName()
            + " is running task: " + taskName
        );
    }
}

MyThread t = new MyThread("Task-A");
t.start(); // ✅ Starts a new thread and calls run() on it
t.run();   // ❌ Runs on SAME (main) thread — no new thread created!
⚠️
Key Interview Trap: Always call start(), NEVER run() directly. run() is just a normal method call on the current thread. start() registers the thread with JVM and schedules it.

🔶 Way 2 — Implement Runnable (Preferred)

Better than extending Thread — class can still extend another class. Also can be used with lambdas (functional interface).

Javaimplements Runnable
// Traditional approach
class MyRunnable implements Runnable {
    @Override
    public void run() {
        System.out.println("Runnable running on: "
            + Thread.currentThread().getName());
    }
}

Thread t1 = new Thread(new MyRunnable(), "Worker-1");
t1.start();

// Lambda (Runnable is @FunctionalInterface)
Thread t2 = new Thread(() ->
    System.out.println("Lambda thread: " + Thread.currentThread().getName()),
    "Worker-2"
);
t2.start();

// Thread utility methods
Thread.sleep(1000);         // pause current thread for 1s (throws InterruptedException)
Thread.currentThread();    // reference to current executing thread
t1.join();                 // wait for t1 to finish before continuing
t1.join(2000);            // wait max 2 seconds
t1.interrupt();           // request thread to stop (sets interrupted flag)
Thread.interrupted();     // checks and CLEARS interrupted flag
t1.isInterrupted();      // checks without clearing flag
t1.setDaemon(true);      // must be set BEFORE start() — daemon = background thread
t1.setPriority(Thread.MAX_PRIORITY); // 1 (MIN) to 10 (MAX), default 5

🔷 Way 3 — Callable + Future (return value from thread)

Runnable cannot return a value or throw checked exceptions. Callable<V> solves both — returns a value and can throw exceptions. Result is accessed via Future<V>.

JavaCallable + Future
import java.util.concurrent.*;

// Callable — returns Integer, can throw exception
Callable<Integer> task = () -> {
    Thread.sleep(1000);
    return 42;
};

ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Integer> future = executor.submit(task);

// Future methods
future.isDone();           // check if completed
future.isCancelled();      // check if cancelled
future.cancel(true);       // attempt to cancel

Integer result = future.get();         // blocks until done
Integer r2     = future.get(2, TimeUnit.SECONDS); // timeout version

System.out.println("Result: " + result); // 42
executor.shutdown();

🔶 Way 4 — Virtual Threads (Java 21 — Project Loom)

Virtual threads are lightweight threads managed by the JVM (not OS). You can create millions of them without exhausting resources. Perfect for I/O-bound tasks.

Java 21Virtual Threads — Project Loom
// Create virtual thread (Java 21+)
Thread vt = Thread.ofVirtual().name("vt-1").start(() ->
    System.out.println("Virtual thread: " + Thread.currentThread().isVirtual())
);

// Virtual thread with factory
Thread.Builder builder = Thread.ofVirtual().name("worker-", 0);
Thread t1 = builder.start(() -> System.out.println("Task 1"));
Thread t2 = builder.start(() -> System.out.println("Task 2"));

// Virtual thread ExecutorService — creates a new virtual thread per task
try (ExecutorService exec = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10_000; i++) {
        final int id = i;
        exec.submit(() -> /* I/O or DB call */ System.out.println("Task: " + id));
    }
} // auto-shutdown via try-with-resources

// Check if virtual
System.out.println(Thread.currentThread().isVirtual()); // false for platform threads
💡
Virtual vs Platform Threads: Platform threads = OS threads (expensive, limited ~1000s). Virtual threads = JVM-managed (cheap, can create millions). Virtual threads block cheaply — JVM automatically unmounts from carrier thread during I/O. Don't pool virtual threads — create a new one per task.

🎯 Interview Questions — Thread Lifecycle
Q1What is the difference between BLOCKED and WAITING state?
BLOCKED: Thread is waiting to acquire a synchronized lock held by another thread. It will automatically become RUNNABLE when the lock is released — no explicit notification needed.
WAITING: Thread called wait(), join(), or LockSupport.park() and is indefinitely waiting for an explicit signal (notify(), notifyAll(), or thread completion).
Q2What is a daemon thread? Give examples.
Daemon threads are background threads that JVM automatically kills when all user (non-daemon) threads finish. You cannot set a thread as daemon after it starts.
Examples of JVM daemon threads: Garbage Collector, JIT compiler, Finalizer thread.
Use case: Background monitoring, heartbeats, cleanup tasks that should not prevent JVM from shutting down.

Synchronization & Monitors

⚙️ The Race Condition Problem

JavaRace Condition Demo — Problem
class Counter {
    private int count = 0;

    // ❌ NOT thread-safe — count++ is NOT atomic!
    // count++ = read count + increment + write back (3 operations)
    public void increment() { count++; }

    public int getCount() { return count; }
}

// Two threads each incrementing 1000 times
Counter counter = new Counter();
Runnable task = () -> {
    for (int i = 0; i < 1000; i++) counter.increment();
};
Thread t1 = new Thread(task), t2 = new Thread(task);
t1.start(); t2.start();
t1.join(); t2.join();
// Expected: 2000  Actual: ANYTHING from 1000-2000 (non-deterministic)
System.out.println(counter.getCount());

🔐 synchronized — 4 Variants

Javasynchronized — All 4 Forms
class BankAccount {
    private double balance;
    private static int totalAccounts = 0;
    private final Object balanceLock = new Object(); // private lock object

    // ── FORM 1: synchronized instance method ──
    // Locks on "this" (the BankAccount instance)
    public synchronized void deposit(double amount) {
        balance += amount;
    }

    // ── FORM 2: synchronized block — on "this" ──
    // Better: only critical section is locked, more granular
    public void withdraw(double amount) {
        synchronized (this) {
            if (balance >= amount) balance -= amount;
        }
    }

    // ── FORM 3: synchronized block — private lock (BEST practice) ──
    // Prevents external code from locking on "this" — more secure
    public void transfer(double amount, BankAccount to) {
        synchronized (balanceLock) {
            this.balance -= amount;
            to.balance   += amount;
        }
    }

    // ── FORM 4: synchronized static method ──
    // Locks on Class object (BankAccount.class) — shared across ALL instances
    public static synchronized void registerAccount() {
        totalAccounts++;
    }
}

// Reentrancy demo — synchronized is REENTRANT
// A thread that already holds a lock CAN re-acquire it
class ReentrantDemo {
    synchronized void outer() {
        System.out.println("outer");
        inner(); // ✅ allowed — same thread, reentrant lock
    }
    synchronized void inner() {
        System.out.println("inner");
    }
}
💡
How it works internally: Every Java object has an intrinsic lock (monitor). synchronized acquires the monitor before entering the block and releases it on exit (even if exception is thrown). Only one thread can hold a monitor at a time — others go to BLOCKED state.

📊 synchronized Granularity — Performance Impact

JavaFine-grained vs Coarse-grained locking
// ❌ Coarse-grained: entire method locked even for unrelated ops
synchronized void processOrder(Order order) {
    validateOrder(order);     // doesn't need lock
    updateInventory(order);   // needs lock
    sendNotification(order);  // doesn't need lock
}

// ✅ Fine-grained: only lock the critical section
void processOrder(Order order) {
    validateOrder(order);          // no lock needed
    synchronized (inventoryLock) {
        updateInventory(order);     // only this needs lock
    }
    sendNotification(order);       // no lock needed
}
🎯 Interview Questions — Synchronization
Q1Can two synchronized methods of the same object run concurrently?
No. Both lock on this — only one thread can hold the this monitor at a time. So if Thread-1 is in deposit(), Thread-2 cannot enter withdraw() either. However, if one is an instance method and another is static synchronized, they lock on different objects (instance vs Class) so they CAN run concurrently.
Q2What is object-level vs class-level locking?
Object-level: synchronized instance methods / synchronized(this) → locks on the specific object instance. Different instances can be accessed concurrently.
Class-level: static synchronized methods / synchronized(ClassName.class) → locks on the Class object. Only one thread can access across ALL instances simultaneously.

volatile Keyword
Visibility guarantee, happens-before, when to use it

🧠 Java Memory Model — Why volatile is needed

Each thread has its own CPU cache / working memory. Without volatile, a thread might read a stale cached value instead of the latest value from main memory.

Without volatile — CPU Cache Problem
Thread-1 (CPU Core 1) Thread-2 (CPU Core 2) ┌─────────────────────┐ ┌─────────────────────┐ │ L1 Cache: flag=false│ │ L1 Cache: flag=false│ ← stale! └─────────┬───────────┘ └─────────┬───────────┘ │ writes reads │ ▼ ▼ ┌─────────────────────────────────────┐ │ Main Memory: flag=true │ └─────────────────────────────────────┘ Thread-2 never sees the update! → infinite loop

💻 volatile — Correct Usage

Javavolatile — Stop flag pattern
class Worker implements Runnable {
    // ✅ volatile ensures Thread-2 always sees Thread-1's update
    private volatile boolean running = true;

    @Override
    public void run() {
        while (running) {
            // do work...
        }
        System.out.println("Worker stopped");
    }

    public void stop() {
        running = false; // ← write to volatile: immediately visible to all threads
    }
}

// Without volatile: Thread might loop forever reading cached flag=true
// With volatile: Write in stop() is immediately flushed to main memory
//               Read in run() always fetches from main memory

// ─── Double-Checked Locking + volatile (Singleton) ───
class Singleton {
    // volatile prevents instruction reordering during object creation
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {               // 1st check (no lock)
            synchronized (Singleton.class) {
                if (instance == null) {       // 2nd check (with lock)
                    instance = new Singleton(); // volatile prevents partial init
                }
            }
        }
        return instance;
    }
}

Section 4
wait() / notify() / notifyAll()
Inter-thread communication — Producer-Consumer pattern

📖 Rules for wait() / notify()

  • Must be called from a synchronized context — otherwise IllegalMonitorStateException
  • wait() releases the lock and puts thread in WAITING state
  • notify() wakes up ONE random waiting thread on the same object
  • notifyAll() wakes up ALL waiting threads — they compete for the lock
  • Always use wait() in a loop (while, not if) — guards against spurious wakeups

🏭 Classic Producer-Consumer Pattern

JavaProducer-Consumer using wait/notify
class SharedBuffer {
    private final Queue<Integer> buffer = new LinkedList<>();
    private final int CAPACITY = 5;

    // Producer: add item — waits if buffer is full
    public synchronized void produce(int item) throws InterruptedException {
        // ✅ while loop — not if — protects against spurious wakeups
        while (buffer.size() == CAPACITY) {
            System.out.println("Buffer full, producer waiting...");
            wait(); // ← releases lock, enters WAITING
        }
        buffer.add(item);
        System.out.println("Produced: " + item + " | Size: " + buffer.size());
        notifyAll(); // ← wake up waiting consumers
    }

    // Consumer: remove item — waits if buffer is empty
    public synchronized int consume() throws InterruptedException {
        while (buffer.isEmpty()) {
            System.out.println("Buffer empty, consumer waiting...");
            wait(); // ← releases lock, enters WAITING
        }
        int item = buffer.poll();
        System.out.println("Consumed: " + item + " | Size: " + buffer.size());
        notifyAll(); // ← wake up waiting producers
        return item;
    }
}

// Usage
SharedBuffer buf = new SharedBuffer();

Thread producer = new Thread(() -> {
    for (int i = 1; i <= 10; i++) {
        try { buf.produce(i); }
        catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}, "Producer");

Thread consumer = new Thread(() -> {
    for (int i = 1; i <= 10; i++) {
        try { buf.consume(); }
        catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
}, "Consumer");

producer.start(); consumer.start();
✅
Modern Alternative: Use BlockingQueue (ArrayBlockingQueue, LinkedBlockingQueue) instead of manual wait/notify. It handles all synchronization internally with put() (blocks when full) and take() (blocks when empty).
JavaProducer-Consumer using BlockingQueue (preferred)
import java.util.concurrent.*;

BlockingQueue<Integer> queue = new ArrayBlockingQueue<>(5);

// Producer
Thread producer = new Thread(() -> {
    for (int i = 1; i <= 10; i++) {
        try {
            queue.put(i);  // blocks if queue is full
            System.out.println("Produced: " + i);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
});

// Consumer
Thread consumer = new Thread(() -> {
    for (int i = 1; i <= 10; i++) {
        try {
            int item = queue.take(); // blocks if queue is empty
            System.out.println("Consumed: " + item);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
});

producer.start(); consumer.start();

Locks — ReentrantLock, ReadWriteLock, StampedLock
java.util.concurrent.locks — more flexible than synchronized

🔷 ReentrantLock — Explicit Lock

More flexible than synchronized: supports tryLock, timed lock, interruptible locking, and fairness.

JavaReentrantLock — Complete Usage
import java.util.concurrent.locks.*;

class ThreadSafeCounter {
    private int count = 0;
    private final ReentrantLock lock = new ReentrantLock(true); // true=fair

    public void increment() {
        lock.lock(); // acquire lock (blocks if unavailable)
        try {
            count++;
        } finally {
            lock.unlock(); // ALWAYS unlock in finally!
        }
    }

    // tryLock — non-blocking, returns false immediately if locked
    public boolean tryIncrement() {
        if (lock.tryLock()) { // returns true if lock acquired
            try { count++; return true; }
            finally { lock.unlock(); }
        }
        return false; // couldn't acquire lock
    }

    // tryLock with timeout
    public boolean timedIncrement() throws InterruptedException {
        if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
            try { count++; return true; }
            finally { lock.unlock(); }
        }
        return false;
    }

    // lockInterruptibly — can be interrupted while waiting
    public void interruptibleIncrement() throws InterruptedException {
        lock.lockInterruptibly(); // throws InterruptedException if interrupted
        try { count++; }
        finally { lock.unlock(); }
    }

    public int getCount()     { return count; }
    public int getHoldCount() { return lock.getHoldCount(); } // reentrancy count
    public boolean isLocked() { return lock.isLocked(); }
}

🔶 ReadWriteLock — Concurrent Reads, Exclusive Writes

Allows multiple threads to read simultaneously, but only one thread to write (no concurrent reads during write).

JavaReadWriteLock — Read-heavy workload optimization
class CachedData {
    private Map<String, String> cache = new HashMap<>();
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock  = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();

    // Multiple threads can read simultaneously
    public String get(String key) {
        readLock.lock();
        try {
            return cache.get(key);
        } finally {
            readLock.unlock();
        }
    }

    // Only one thread can write — all readers/writers blocked
    public void put(String key, String value) {
        writeLock.lock();
        try {
            cache.put(key, value);
        } finally {
            writeLock.unlock();
        }
    }
}

// ─── StampedLock (Java 8+) — even more efficient ───
StampedLock sl = new StampedLock();

// Optimistic read — no lock! Just check for concurrent write
long stamp = sl.tryOptimisticRead();
double x = /* read field */ 0;
if (!sl.validate(stamp)) { // check if write happened during read
    stamp = sl.readLock(); // fall back to regular read lock
    try { x = /* re-read */ 0; }
    finally { sl.unlockRead(stamp); }
}

📬 Condition — Advanced wait/notify with ReentrantLock

JavaCondition — separate wait sets
class BoundedBuffer<T> {
    private final Object[] items;
    private int head, tail, count;
    private final ReentrantLock lock = new ReentrantLock();

    // Two separate condition queues — more efficient than notifyAll
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    BoundedBuffer(int capacity) { items = new Object[capacity]; }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) notFull.await(); // wait until space
            items[tail] = item;
            tail = (tail + 1) % items.length;
            count++;
            notEmpty.signal(); // signal ONLY consumer — efficient!
        } finally { lock.unlock(); }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) notEmpty.await(); // wait until data
            T item = (T) items[head];
            head = (head + 1) % items.length;
            count--;
            notFull.signal(); // signal ONLY producer — efficient!
            return item;
        } finally { lock.unlock(); }
    }
}

Section 6
Thread Pools & ExecutorService
Executors factory, ThreadPoolExecutor internals, shutdown

🏗️ Why Thread Pools?

  • Thread creation is expensive — OS-level resource allocation
  • Reuse threads across multiple tasks — eliminates creation/destruction overhead
  • Bounded resources — prevents OutOfMemoryError from unlimited thread creation
  • Better throughput — queue tasks, execute with fixed thread count

🏭 5 Types of Executors — Executors factory

JavaExecutors Factory Methods
import java.util.concurrent.*;

// ── 1. Fixed Thread Pool ──
// Fixed N threads. Queue tasks when all busy.
// Use: CPU-intensive tasks, N = number of CPU cores
ExecutorService fixed = Executors.newFixedThreadPool(4);

// ── 2. Cached Thread Pool ──
// Creates new threads as needed, reuses idle ones (60s TTL)
// Use: large number of short-lived async tasks
// WARNING: can create unlimited threads if tasks arrive fast!
ExecutorService cached = Executors.newCachedThreadPool();

// ── 3. Single Thread Executor ──
// Exactly 1 thread. Tasks executed sequentially.
// Use: sequential task processing, event loops
ExecutorService single = Executors.newSingleThreadExecutor();

// ── 4. Scheduled Thread Pool ──
// For delayed or periodic tasks
ScheduledExecutorService sched = Executors.newScheduledThreadPool(2);
sched.schedule(() -> System.out.println("Delayed 2s"), 2, TimeUnit.SECONDS);
sched.scheduleAtFixedRate(() -> System.out.println("Every 1s"),
    0, 1, TimeUnit.SECONDS);   // initial delay=0, period=1s
sched.scheduleWithFixedDelay(() -> System.out.println("With 1s delay after each"),
    0, 1, TimeUnit.SECONDS);   // delay between END of last and START of next

// ── 5. Work Stealing Pool (Java 8+) ──
// Uses all available CPUs, tasks can steal from other queues (ForkJoin)
ExecutorService workSteal = Executors.newWorkStealingPool();

// ─── Submitting tasks ───
fixed.execute(runnable);                   // fire and forget
Future<?> f1    = fixed.submit(runnable);  // returns Future
Future<Integer> f2 = fixed.submit(callable); // returns Future with result
List<Future<String>> futures =
    fixed.invokeAll(callables);             // submit all, wait for all
String first = fixed.invokeAny(callables);  // return first successful result

// ─── Shutdown ───
fixed.shutdown();           // stop accepting new tasks, finish submitted ones
fixed.shutdownNow();        // interrupt all running tasks, return pending
fixed.isShutdown();         // true after shutdown() called
fixed.isTerminated();       // true when all tasks finished
fixed.awaitTermination(10, TimeUnit.SECONDS); // wait for termination

⚙️ ThreadPoolExecutor — Custom Pool (Under the Hood)

JavaThreadPoolExecutor — Full Configuration
// ThreadPoolExecutor is what Executors.newFixedThreadPool() creates internally
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    4,                              // corePoolSize: min threads always alive
    10,                             // maximumPoolSize: max threads
    60L, TimeUnit.SECONDS,          // keepAliveTime: idle thread TTL
    new ArrayBlockingQueue<>(100), // workQueue: task queue
    Executors.defaultThreadFactory(), // threadFactory
    new ThreadPoolExecutor.CallerRunsPolicy() // rejection policy
);

// Rejection Policies (when queue is full AND maxPoolSize reached):
// AbortPolicy (default) → throws RejectedExecutionException
// CallerRunsPolicy     → caller thread runs the task (back pressure)
// DiscardPolicy        → silently discards task
// DiscardOldestPolicy  → discards oldest queued task, retries

// Monitor the pool
pool.getPoolSize();          // current thread count
pool.getActiveCount();       // executing threads
pool.getCompletedTaskCount();// total completed tasks
pool.getQueue().size();      // pending tasks

// Hook methods (override for monitoring)
ThreadPoolExecutor poolWithHooks = new ThreadPoolExecutor(2,4,60L,
    TimeUnit.SECONDS, new LinkedBlockingQueue<>()) {
    @Override
    protected void beforeExecute(Thread t, Runnable r) {
        System.out.println("Before: " + t.getName());
    }
    @Override
    protected void afterExecute(Runnable r, Throwable t) {
        System.out.println("After task");
    }
};

🎯 Interview Questions — Thread Pools
Q1What is the difference between shutdown() and shutdownNow()?
shutdown(): Initiates orderly shutdown. No new tasks accepted. Previously submitted tasks continue to execute. Non-blocking — returns immediately.
shutdownNow(): Attempts to stop all actively executing tasks by calling interrupt() on all threads. Returns list of tasks that were waiting in queue. Threads must respond to interruption to actually stop.
Q2How many threads should be in a thread pool?
Depends on task type:
CPU-bound tasks: N = number of CPU cores (or N+1). Adding more won't help — CPU is the bottleneck.
I/O-bound tasks: N = CPU_cores × (1 + wait_time/compute_time). More threads can be beneficial since threads spend time waiting for I/O.
Formula: Runtime.getRuntime().availableProcessors() for CPU count.

🔴 Deadlock — 4 Necessary Conditions (Coffman)

  • Mutual Exclusion: At least one resource is non-shareable
  • Hold and Wait: Thread holds a resource while waiting for another
  • No Preemption: Resources can't be forcibly taken — only voluntarily released
  • Circular Wait: Thread-1 waits for Thread-2, Thread-2 waits for Thread-1
JavaDeadlock — Classic Demo
class DeadlockDemo {
    private static final Object LOCK_A = new Object();
    private static final Object LOCK_B = new Object();

    // ❌ DEADLOCK: Thread-1 holds A, wants B
    //             Thread-2 holds B, wants A
    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (LOCK_A) {
                System.out.println("T1: Holding A...");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (LOCK_B) {   // waits for B — DEADLOCK!
                    System.out.println("T1: Holding A and B");
                }
            }
        }, "Thread-1");

        Thread t2 = new Thread(() -> {
            synchronized (LOCK_B) {
                System.out.println("T2: Holding B...");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (LOCK_A) {   // waits for A — DEADLOCK!
                    System.out.println("T2: Holding B and A");
                }
            }
        }, "Thread-2");

        t1.start(); t2.start(); // Both threads hang forever
    }
}

    void transfer(Account from, Account to, double amount) {
        // Always lock account with smaller ID first
        Account first  = from.id < to.id ? from : to;
        Account second = from.id < to.id ? to : from;

class DeadlockFix2 {
    ReentrantLock lockA = new ReentrantLock();

                        try {
                            // Do work safely
                            return;
                        } finally { lockB.unlock(); }
                    }
                } finally { lockA.unlock(); }
            }

⏳ CountDownLatch — "Wait until N tasks done"

One-shot counter. Threads call countDown(). Main thread calls await() and blocks until count reaches zero. Cannot be reset.

JavaCountDownLatch — Parallel initialization
import java.util.concurrent.*;

CountDownLatch latch = new CountDownLatch(3); // count = 3

// 3 worker threads — each decrements latch
for (int i = 1; i <= 3; i++) {
    final int id = i;
    new Thread(() -> {
        try {
            System.out.println("Service " + id + " starting...");
            Thread.sleep(1000 * id); // simulate init time
            System.out.println("Service " + id + " ready!");
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            latch.countDown(); // decrement count (3→2→1→0)
        }
    }).start();
}

// Main thread waits until all 3 services are ready
latch.await();                       // blocks until count == 0
latch.await(10, TimeUnit.SECONDS);   // timeout version
System.out.println("All services ready! Starting application...");

🔄 CyclicBarrier — "All threads wait for each other"

All N threads must arrive at the barrier before any can proceed. Can be reset and reused — that's the difference from CountDownLatch.

JavaCyclicBarrier — Parallel computation phases
int numThreads = 4;
// Barrier action runs when all threads reach barrier
CyclicBarrier barrier = new CyclicBarrier(numThreads,
    () -> System.out.println("All threads reached phase boundary!")
);

for (int i = 0; i < numThreads; i++) {
    final int id = i;
    new Thread(() -> {
        try {
            // Phase 1
            System.out.println("Thread " + id + ": Phase 1 work");
            barrier.await(); // wait for ALL threads to finish phase 1

            // Phase 2 — all threads start together
            System.out.println("Thread " + id + ": Phase 2 work");
            barrier.await(); // cyclic! reused for phase 2 boundary

        } catch (InterruptedException | BrokenBarrierException e) {
            Thread.currentThread().interrupt();
        }
    }).start();
}

💡 ThreadLocal — Each Thread Gets Its Own Copy

ThreadLocal provides thread-isolated storage. Each thread accessing the variable gets its own independent copy. No synchronization needed — threads cannot interfere with each other's data.

JavaThreadLocal — Usage and Pitfalls
// ─── Basic ThreadLocal ───
class RequestContext {
    // Each thread has its own userId
    private static final ThreadLocal<String> userIdHolder =
        new ThreadLocal<>();

    public static void setUserId(String userId) {
        userIdHolder.set(userId);
    }
    public static String getUserId() {
        return userIdHolder.get();
    }
    // ⚠️ MUST call remove() — prevents memory leak in thread pools!
    public static void clear() {
        userIdHolder.remove();
    }
}

// Each request thread sets its own userId
new Thread(() -> {
    RequestContext.setUserId("user-123");
    System.out.println(RequestContext.getUserId()); // "user-123"
    // ... process request ...
    RequestContext.clear(); // ✅ clean up!
}).start();

// ─── ThreadLocal with initialValue ───
static final ThreadLocal<SimpleDateFormat> DATE_FORMAT =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));
// SimpleDateFormat is NOT thread-safe — ThreadLocal gives each thread its own instance

// ─── InheritableThreadLocal ───
// Child threads inherit value from parent thread
static final InheritableThreadLocal<String> CONTEXT =
    new InheritableThreadLocal<>();
CONTEXT.set("parent-context");
new Thread(() ->
    System.out.println(CONTEXT.get()) // "parent-context" — inherited!
).start();
🚨
Memory Leak Warning with Thread Pools! In thread pools, threads are reused. If you call set() but forget remove(), the value persists for the lifetime of the thread, even across different requests. This causes stale data bugs AND memory leaks. Always call ThreadLocal.remove() in a finally block.

📖 JMM — Key Rules (Happens-Before Relationships)

The JMM defines when one action is guaranteed to be visible to another. If action A happens-before action B, then A's effects are visible to B.

Program OrderEach action in a thread happens-before every subsequent action in the same thread
Monitor LockUnlock of a monitor happens-before every subsequent lock of the same monitor
volatile WriteWrite to a volatile field happens-before every subsequent read of that field
Thread StartThread.start() happens-before any action in the started thread

class EvenOddPrinter {
    private int number = 1;
    private final int max = 20;
    private final Object lock = new Object();

    public void printOdd() {
        while (number <= max) {
            synchronized (lock) {
                while (number <= max && number % 2 == 0) {
                    try { lock.wait(); } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                }

                    lock.notifyAll();
                }
            }
        }
    }

C5Implement a custom BlockingQueue from scratch
JavaCustom BlockingQueue
import java.util.concurrent.locks.*;

public class MyBlockingQueue<T> {
    private final Object[] buffer;
    private int head = 0, tail = 0, count = 0;
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    public MyBlockingQueue(int capacity) {
        buffer = new Object[capacity];
    }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == buffer.length)
                notFull.await(); // wait until space available
            buffer[tail] = item;
            tail = (tail + 1) % buffer.length;
            count++;
            notEmpty.signal(); // wake up consumer
        } finally { lock.unlock(); }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0)
                notEmpty.await(); // wait until item available
            T item = (T) buffer[head];
            buffer[head] = null; // help GC
            head = (head + 1) % buffer.length;
            count--;
            notFull.signal();  // wake up producer
            return item;
        } finally { lock.unlock(); }
    }

    public int     size()    { return count; }
    public boolean isEmpty() { return count == 0; }
    public boolean isFull()  { return count == buffer.length; }
}

🔥 Core Multithreading Q&A

Q1What is the difference between process and thread?
Process: Independent program in execution. Has its own memory space (heap, stack, code, data). Heavy-weight. IPC needed for communication.
Thread: Lightweight unit of execution within a process. Shares process memory (heap, code). Has own stack and registers. Communication via shared memory. Faster to create/destroy than processes.
Q2What is the difference between Runnable and Callable?
Runnable: void run() — no return value, cannot throw checked exceptions.
Callable: V call() throws Exception — returns a value, can throw checked exceptions. Used with ExecutorService.submit() which returns a Future.
Q3What is the purpose of the synchronized keyword?
Provides two guarantees: Mutual Exclusion (only one thread can execute the synchronized block at a time) and Visibility (changes made inside synchronized block are visible to other threads after the lock is released). Internally uses the object's monitor (intrinsic lock).
Q4Can we synchronize a constructor in Java?
No. Constructors cannot be synchronized because only one thread can access a new object during construction — other threads don't have a reference to it yet. Using synchronized on a constructor causes a compile error. However, you can use synchronized blocks inside a constructor.
Q5What is the difference between sleep() and wait()?
Thread.sleep(n): Static method. Pauses current thread for n milliseconds. Does NOT release lock. Thread goes to TIMED_WAITING. Doesn't need synchronized context.
object.wait(): Instance method on Object. Releases the lock. Thread goes to WAITING. Must be called from synchronized block. Thread resumes when notify()/notifyAll() is called.
Q6What is thread starvation and how to prevent it?
Starvation occurs when a thread is perpetually denied CPU time because other threads with higher priority keep running. Fixes: (1) Use fair locks — new ReentrantLock(true). (2) Avoid indefinite priority escalation. (3) Use ThreadPoolExecutor with bounded queue so all tasks eventually execute.
Q7What is a spurious wakeup and how to handle it?
A thread can wake up from wait() without being notified — this is a spurious wakeup (hardware/OS level, not a bug). Fix: Always call wait() in a loop (while, not if), re-checking the condition after wakeup: while (!condition) { lock.wait(); } — not if (!condition) { lock.wait(); }
Q8What is the difference between notify() and notifyAll()?
notify(): Wakes up ONE arbitrarily chosen thread waiting on the same object. Others remain WAITING. Risk: if wrong thread is woken up, it may wait again and never make progress (use notifyAll for safety).
notifyAll(): Wakes up ALL waiting threads. They all compete for the lock — only one gets it, others go back to BLOCKED/WAITING after re-checking condition. Safer but less efficient.
Q9What is the Java Memory Model (JMM)?
JMM defines the rules for how threads interact through memory. Key problems it addresses: (1) Visibility: Changes by one thread may not be visible to others (CPU caching). (2) Atomicity: Multi-step operations can be interleaved. (3) Reordering: Compiler/CPU may reorder instructions. Solutions: volatile, synchronized, Atomic classes, happens-before relationships.
Q10What is ThreadLocal memory leak and how to prevent it?
In thread pools, threads are reused. If ThreadLocal.remove() is not called, the value remains in the ThreadLocalMap for the life of the thread. This: (1) Leaks memory. (2) Causes stale data bugs where old request data bleeds into new request. Fix: Always call remove() in a finally block after use.
Q11What is ForkJoinPool and when to use it?
ForkJoinPool is designed for divide-and-conquer tasks (recursively split work). It uses work-stealing — idle threads steal tasks from busy threads' queues. Used by: parallel streams, CompletableFuture (default), Executors.newWorkStealingPool(). Best for CPU-intensive recursive tasks (merge sort, tree traversal, parallel reduction).
Q12Difference between ConcurrentHashMap and synchronizedMap?
Collections.synchronizedMap: Wraps entire map with single lock. Only one thread can access ANY operation at a time. Iteration requires external sync too.
ConcurrentHashMap: Fine-grained locking (Java 7: 16 segments, Java 8: bucket-level CAS + sync). Multiple threads can read/write different buckets simultaneously. No lock for reads. Iteration is weakly consistent (no CME). Also no null keys/values.
Q13What happens if an exception occurs in a thread?
The thread terminates abnormally. The exception does NOT propagate to the parent thread. To handle it: (1) Use Thread.setUncaughtExceptionHandler(). (2) For ExecutorService: exceptions from Runnable are swallowed; from Callable they're wrapped in ExecutionException and thrown when future.get() is called. (3) Use try-catch inside the thread's run().

🧠 Tricky Output & Snippet Questions

S1What is the output? How many times does "done" print?
JavaTricky visibility bug
class VisibilityTest {
    static boolean stop = false; // no volatile!
    public static void main(String[] args) throws Exception {
        new Thread(() -> {
            while (!stop) {} // spin
            System.out.println("done");
        }).start();
        Thread.sleep(1000);
        stop = true;
        System.out.println("main done");
    }
}
✅ Answer
Possible outputs: (a) "main done" then "done" OR (b) just "main done" — the loop may never exit!
Without volatile, the worker thread may cache stop=false in its CPU register/L1 cache and never see the updated value from main thread. Add volatile to stop to guarantee visibility.
S2Is this code thread-safe? What's the output range?
JavaRace condition with volatile
class RaceTest {
    volatile int x = 0;
    void increment() { x++; }
    public static void main(String[] args) throws Exception {
        RaceTest rt = new RaceTest();
        Thread t1 = new Thread(() -> {
            for(int i=0;i<1000;i++) rt.increment();
        });
        Thread t2 = new Thread(() -> {
            for(int i=0;i<1000;i++) rt.increment();
        });
        t1.start(); t2.start(); t1.join(); t2.join();
        System.out.println(rt.x);
    }
}
✅ Answer
NOT thread-safe. Output: 1000-2000 (non-deterministic).
volatile only ensures visibility — it does NOT make x++ atomic. x++ = read x + add 1 + write x (3 ops). Both threads can read x=5 simultaneously, both compute 6, both write 6 → increment lost. Fix: Use AtomicInteger or synchronized.
S3What is the output of this program?
JavaThread join() behavior
public class JoinTest {
    public static void main(String[] args) throws Exception {
        Thread t = new Thread(() -> {
            try { Thread.sleep(500); } catch(Exception e) {}
            System.out.println("Child");
        });
        t.start();
        t.join(); // main waits for t to finish
        System.out.println("Main");
    }
}
✅ Answer
Output: Child → Main (always in this order)
join() makes the calling thread (main) wait until thread t completes. Without join(), output could be "Main → Child" or "Child → Main" — non-deterministic. join() establishes a happens-before relationship.
