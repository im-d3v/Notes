Pattern 01 Arrays & Strings O(n) Time

Two Pointers

The Art of Coordinated Movement — eliminating nested loops by keeping two indices that move with purpose, not brute force.

2Varieties
O(n)Time Complexity
O(1)Space Complexity
14Practice Problems

🧠 The Mental Model

The Tug-of-War Analogy

Imagine two people pulling a rope from opposite ends. The rope is your sorted array. Person Left starts at the lightest end, person Right at the heaviest. They call out their combined pull (sum). If it's too strong → Right steps left to ease up. If too weak → Left steps right to pull harder. They close in until they meet.

This is the core insight: movement is not random — each step is a logical elimination. When Right steps left, you are not just trying the next value; you are discarding every row in a hypothetical O(n²) matrix in a single decision.

Why One Index Cannot Do This

A single pointer can answer "what is at position i?" but it cannot simultaneously reason about relationships between two positions without fixing one and scanning the rest — which is the O(n²) you are trying to escape.

Two pointers maintain a window of interest defined by both ends at once. Every move of either pointer is guided by a shared invariant (e.g., sum too large, duplicate found, distance exceeded). The invariant is what replaces the inner loop.

💡
The key abstraction: One index describes a position. Two indices describe a relationship — and relationships are what most array problems are actually asking about.

💡 The Core Idea

What is it actually doing?

Two Pointers is a technique that replaces a nested loop with guided, purposeful movement of two indices across a linear data structure. Instead of checking every pair (O(n²)), you use a property of the input — usually sortedness or a monotonic relationship — to guarantee that when you move a pointer, you can safely ignore everything behind it.

When does it apply?

  • The data structure is linear (array, string, linked list).
  • You need to find a pair, triplet, or subarray satisfying some condition.
  • There exists a monotonic property: moving one pointer in a direction always makes the situation "more" or "less" of something measurable.
  • The brute-force solution uses two nested loops over the same array.

The Invariant

Every two-pointer solution maintains an invariant: a truth that holds before and after every pointer move. For opposite-direction: "all elements outside [L, R] have been proven to not contain the answer." For same-direction: "slow always points to the last valid element seen." Identify the invariant first — the code writes itself after that.

📐 Code Templates

Python Template A — Opposite Direction 
def two_pointers_opposite(arr: list, target: int) -> list:
    # PRE-CONDITION: arr must be sorted (or sort it here at O(n log n) cost)
    left, right = 0, len(arr) - 1

    while left < right:            # loop until pointers cross
        current = arr[left] + arr[right]

        if current == target:
            return [left, right]   # FOUND — adapt to your return type

        elif current < target:
            left += 1              # INVARIANT: everything to the left of L
                                   # combined with R is provably too small
        else:
            right -= 1             # INVARIANT: everything to the right of R
                                   # combined with L is provably too large

    return []                      # no pair found
Python Template B — Same Direction (Fast / Slow) 
def two_pointers_same_dir(arr: list) -> int:
    # slow = boundary of "processed / valid" region
    # fast = exploration pointer
    slow = 0

    for fast in range(len(arr)):     # fast always advances every iteration

        if is_valid(arr, slow, fast):  # define your own validity condition
            arr[slow] = arr[fast]   # write valid element into slow position
            slow += 1              # advance slow boundary

        # if not valid: fast advances next iteration, slow stays
        # effectively "skipping" invalid elements

    return slow                    # new length of valid prefix


# Example: remove all instances of `val`
def is_valid(arr, slow, fast):
    return arr[fast] != val        # keep everything that's not val

🚶 Step-by-Step Walkthrough — LC #15: 3Sum

The Problem

Given an integer array nums, return all triplets [nums[i], nums[j], nums[k]] such that i ≠ j ≠ k and nums[i] + nums[j] + nums[k] == 0. No duplicate triplets in the answer.

Medium Sorted Two Pointers Skip Duplicates

Step 1 — Recognition

"Find all triplets summing to 0." → This is a k-Sum variant. Brute force is O(n³). Two pointers on a sorted array can reduce the inner two loops to O(n), making the total O(n²). ✅

Step 2 — Approach

1
Sort the array
Enables two-pointer technique and makes duplicate skipping trivial. O(n log n) one-time cost.
2
Fix one element with outer loop
Iterate i from 0 to n-3. The fixed element is nums[i]. Skip if nums[i] > 0 (sorted, so nothing can sum to 0).
3
Two-pointer on remaining subarray
L = i+1, R = n-1. Find pairs in nums[i+1..n-1] that sum to -nums[i].
4
Skip duplicates at every level
After fixing i: skip if nums[i] == nums[i-1]. After finding a triplet: advance L past all duplicate values, same for R.

Step 3 — Trace: nums = [-4, -1, -1, 0, 1, 2]

1
Sort → [-4, -1, -1, 0, 1, 2]. Already sorted in example.
2
i=0, nums[i]=-4. L=1(-1), R=5(2). sum=-4+(-1)+2=-3 < 0 → L++
3
L=2(-1), R=5(2). sum=-4+(-1)+2=-3 < 0 → L++
4
L=3(0), R=5(2). sum=-4+0+2=-2 < 0 → L++
5
L=4(1), R=5(2). sum=-4+1+2=-1 < 0 → L++. L≥R, done with i=0.
6
i=1, nums[i]=-1. L=2(-1), R=5(2). sum=-1+(-1)+2=0 → FOUND [-1,-1,2]. Skip dups on L and R.
7
L=3(0), R=4(1). sum=-1+0+1=0 → FOUND [-1,0,1]. Skip dups. L≥R, done with i=1.
8
i=2, nums[i]=-1 == nums[1]. Skip duplicate outer. Continue.
9
i=3, nums[i]=0 > 0? No. L=4(1), R=5(2). sum=0+1+2=3 > 0 → R--. L≥R, done.
✅
Result: [[-1,-1,2], [-1,0,1]]
Python LC 15 — 3Sum Solution 
def threeSum(self, nums: List[int]) -> List[List[int]]:
    nums.sort()
    result = []

    for i in range(len(nums) - 2):
        if i > 0 and nums[i] == nums[i - 1]:
            continue                   # skip duplicate outer element
        if nums[i] > 0:
            break                      # sorted: no triplet can sum to 0

        left, right = i + 1, len(nums) - 1

        while left < right:
            total = nums[i] + nums[left] + nums[right]

            if total == 0:
                result.append([nums[i], nums[left], nums[right]])
                while left < right and nums[left] == nums[left + 1]:
                    left += 1           # skip duplicate left
                while left < right and nums[right] == nums[right - 1]:
                    right -= 1          # skip duplicate right
                left += 1
                right -= 1

            elif total < 0:
                left += 1
            else:
                right -= 1

    return result

⚖️ The Decision Moment

Two Pointers vs. Sliding Window

🔵 Two Pointers

  • Looking for a specific pair/triplet satisfying exact condition
  • Pointers move independently toward each other
  • Works on sorted data exploiting ordering
  • Window shrinks from both ends simultaneously
  • Example: pair summing to target

🟣 Sliding Window

  • Looking for a subarray/substring of optimal length/sum
  • Window expands right, contracts left — both move right
  • Works on unsorted data; cares about contiguous elements
  • Maintains a running aggregate (sum, frequency map)
  • Example: longest substring without repeat
🔑
Quick test: Does order of elements within the window matter? Does "contiguous" matter? → Sliding Window. Is the data sorted and are you eliminating halves? → Two Pointers.

⚠️ Common Traps

Trap 1 — Forgetting to Sort
💀
What goes wrong: Running opposite-direction two pointers on an unsorted array. The logic "sum too big → move right" only holds if data is ordered. You'll miss valid pairs or hit infinite loops.
Fix: Sort first. If sorting isn't allowed (values must be original indices), use a hash map instead.
Trap 2 — Off-by-One in Loop Condition
💀
What goes wrong: Using while left <= right instead of while left < right. When L == R you're looking at the same element twice, producing self-pairs (e.g., 5+5=10 using index 2 twice).
Fix: Always use strict inequality: left < right.
Trap 3 — Not Skipping Duplicates (k-Sum)
💀
What goes wrong: In 3Sum / 4Sum, after finding a valid triplet you don't advance past duplicate values. You record the same triplet multiple times.
Fix: After recording a result, use inner while loops to skip past all identical values before doing the final L++/R--.
Trap 4 — Moving Both Pointers Simultaneously on No-Match
💀
What goes wrong: When the sum doesn't match, doing left++ ; right-- at the same time. You skip valid pairs because both sides changed when only one condition fired.
Fix: Move exactly one pointer per comparison — the one that is "contributing too much" or "contributing too little."
Trap 5 — Same-Direction: Forgetting Slow's Meaning
💀
What goes wrong: In remove-duplicates style problems, writing to arr[slow] before checking the condition, or returning slow instead of slow + 1 (or vice versa).
Fix: Be explicit about whether slow is the next write position (0-indexed) or the count of valid elements. Draw it out before coding.

📊 Complexity Analysis

Time Complexity
O(n)
Each pointer moves at most n steps total
Space Complexity
O(1)
Only two index variables, no auxiliary structure

Intuition — Why O(n) and not O(n²)?

Brute force checks every pair: n × (n-1) / 2 — that's O(n²). Two pointers avoid this by eliminating entire rows of a hypothetical matrix with each pointer move.

Think of it this way: place all values in an n×n grid where grid[i][j] = arr[i] + arr[j]. Brute force visits every cell. Two pointers start at the top-right corner. If the sum is too big, move left (eliminate the entire current column above — all those sums would be even bigger). If too small, move down (eliminate the entire current row to the left). Each move eliminates a full row or column → O(n) moves total.

📌
3Sum is O(n²): We run O(n) two-pointer passes inside an O(n) outer loop. The pattern itself is still O(n) per call — the extra factor comes from fixing each element.

The Fundamental Proof

At each step, either L or R moves closer to the center. The total number of steps before L ≥ R is at most n – 1. Therefore: O(n). No element is visited twice by the same pointer. QED.
