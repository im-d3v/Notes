🌳 Git Commit DAG (Directed Acyclic Graph)

Commit Graph — branches are just pointers to commits
┌─────────────────────────┐ ▼ │ C1 ──── C2 ──── C3 ──── C4 main │ feature branch │ │ └──── C5 ──── C6 feature ◀───┘ main → points to C4 (a SHA-1 hash) feature→ points to C6 (a SHA-1 hash) HEAD → points to current branch (or commit in detached state) origin/main → remote-tracking branch (read-only local copy)
💡
Key insight: Branches in Git are just 41-byte files containing a SHA-1 hash. They are extremely cheap to create, switch between, and delete. This is fundamentally different from older VCS like SVN where branches were expensive full copies.

git fetch origin                   # fetch from origin
git fetch --all                    # fetch from ALL remotes
git fetch --prune                  # fetch + delete stale remote-tracking branches
git fetch origin main:main         # fetch origin/main into local main
git fetch --dry-run                # preview what would be fetched

# ── git pull ── (fetch + merge/rebase)
git pull                           # pull from tracked upstream
git pull origin main               # pull origin/main into current branch
git pull --rebase                  # pull + rebase (cleaner history)
git pull --rebase=interactive      # pull + interactive rebase
git pull --no-rebase               # force merge even if config says rebase
git pull --ff-only                 # only if fast-forward (fail if diverged)
git pull --no-ff                   # always create merge commit

# ── git push ──
git push                           # push to tracked upstream
git push origin main               # push local main to origin/main
git push origin feature/xyz        # push feature branch
git push -u origin feature/xyz     # push + set upstream tracking (-u = --set-upstream)
git push --all                     # push all local branches
git push --tags                    # push all tags
git push --follow-tags             # push + annotated tags reachable from pushed commits
git push --force-with-lease        # ✅ SAFE force push (fails if remote changed since last fetch)
git push --force                   # ❌ DANGEROUS: overwrites remote (use --force-with-lease instead)
git push origin --delete feature/xyz  # delete remote branch
git push origin :feature/xyz       # delete remote branch (older syntax)

# ── Tracking upstream ──
git branch -u origin/main         # set upstream for current branch
git branch --unset-upstream        # remove upstream tracking
⚠️
Always use --force-with-lease instead of --force! --force blindly overwrites the remote branch, potentially discarding commits pushed by teammates since your last fetch. --force-with-lease checks that your remote-tracking branch is up-to-date before forcing — if someone else pushed, it FAILS safely.

Section 5
Merge — 3 Strategies
Fast-forward, 3-way merge, squash merge — when to use each

⚡ Strategy 1 — Fast-Forward Merge

When the target branch has no new commits since the feature branch diverged, Git simply moves the pointer forward. No merge commit created.

BEFORE: main: A──B──C └── D──E feature AFTER (fast-forward): main: A──B──C──D──E ↑ main & feature (same commit)
ShellFast-forward merge
git switch main
git merge feature/login        # fast-forward if possible
git merge --ff-only feature   # FAIL if not fast-forwardable
git merge --no-ff feature     # FORCE merge commit even if fast-forward possible

🔄 Strategy 2 — Three-Way Merge (Merge Commit)

When both branches have diverged. Git finds the common ancestor and creates a new merge commit with two parents.

BEFORE: main: A──B──C──F └── D──E feature AFTER: main: A──B──C──F──M (M = merge commit with 2 parents) ╱ D──E
ShellThree-way merge
git switch main
git merge feature/login                     # auto 3-way if diverged
git merge --no-ff feature/login             # always create merge commit
git merge --no-ff -m "Merge feature/login" feature/login
git merge --abort                            # abort mid-conflict merge
git merge --continue                         # continue after resolving conflicts
git merge --strategy=recursive -X ours      # prefer our side on conflicts
git merge -X theirs feature                  # prefer their side on conflicts

🗜️ Strategy 3 — Squash Merge

Combines all commits from the feature branch into a single new commit on the target branch. Clean history but loses individual commit context.

feature branch has: D──E──F──G (4 commits) AFTER squash: main: A──B──C──DEFG (one squashed commit — not a merge commit) ↑ feature history is GONE (only in new single commit)
ShellSquash merge
git switch main
git merge --squash feature/login   # squash all feature commits into staged changes
git commit -m "feat: add login feature"  # create the squashed commit
# Note: feature branch still exists — delete it manually
git branch -D feature/login         # force delete (not "merged" in git's view)

Section 6
Rebase — Regular & Interactive
Rewrite history, clean commits, the golden rule

🔁 Regular Rebase

Rebase moves (replays) commits on top of another branch, creating new commits with different SHA hashes but same changes. Result: linear history.

BEFORE: main: A──B──C──F └── D──E feature AFTER (git rebase main on feature branch): main: A──B──C──F └── D'──E' feature (D' and E' are NEW commits) ↑ replayed on top of F (the tip of main)
Shellgit rebase
# ── Basic rebase ──
git switch feature/login
git rebase main                    # replay feature commits on top of main
# Then to integrate back:
git switch main
git merge --ff-only feature/login  # clean fast-forward (linear history!)

# ── Rebase control ──
git rebase --abort                 # cancel rebase, restore original state
git rebase --continue              # after fixing conflicts, continue
git rebase --skip                  # skip current conflicting commit
git rebase --quit                  # stop rebase but keep changes so far (rare)

# ── Rebase with upstream ──
git rebase origin/main             # rebase on remote main (common for PRs)

# ── Rebase onto ── (transplant commits)
git rebase --onto main abc123 feature
# Take commits from feature that are NOT in abc123, replay onto main
# Use: extract part of a branch to put elsewhere

✏️ Interactive Rebase — Rewrite History

The most powerful Git command. Edit, squash, reorder, drop commits before they're shared.

Shellgit rebase -i — Interactive rebase
# ── Start interactive rebase ──
git rebase -i HEAD~4              # rewrite last 4 commits
git rebase -i abc123             # rewrite commits after abc123
git rebase -i HEAD~4 --autosquash # auto-apply fixup! and squash! commits
git rebase -i --root              # rewrite entire history from first commit

# ── Editor opens with list of commits (top=oldest, bottom=newest):

pick abc1234 feat: add login form
pick def5678 fix: correct email validation
pick ghi9012 wip: temp test code
pick jkl3456 fix: remove temp test code

# ── Available commands (replace "pick"): ──
# pick    (p) = use commit as-is
# reword  (r) = use commit, but edit message
# edit    (e) = use commit, but stop to amend files/message
# squash  (s) = combine with previous commit, edit message
# fixup   (f) = combine with previous, DISCARD this commit's message
# drop    (d) = remove this commit entirely
# exec    (x) = run shell command after this commit
# break   (b) = stop for manual changes
# label (l), reset (t), merge (m) — for complex rebase scripts

# ── Rewrite to be: ──
pick abc1234 feat: add login form
reword def5678 fix: correct email validation   # change message
fixup ghi9012 wip: temp test code              # merge into above, discard msg
drop  jkl3456 fix: remove temp test code       # delete commit
# Save and close → interactive rebase executes steps
🚨
The Golden Rule of Rebasing: Never rebase commits that have been pushed to a shared remote branch. Rebase rewrites SHA hashes — if teammates have your old commits, their history diverges and causes "force push needed" chaos. Only rebase local unpublished commits, or on your own private branches.

💥 Understanding Conflict Markers

TextConflict markers in a file
// file: UserService.java
public String getUserName() {

<<<<<<< HEAD               // ← YOUR changes (current branch)
    return user.getFullName();
=======                     // ← DIVIDER between the two versions
    return user.getDisplayName();
>>>>>>> feature/display-name  // ← THEIR changes (branch being merged)
}

// ── With diff3 style (shows common ancestor too) ──
// Enable: git config merge.conflictstyle diff3
<<<<<<< HEAD
    return user.getFullName();
||||||| merged common ancestor
    return user.getName();
=======
    return user.getDisplayName();
>>>>>>> feature/display-name

✅ Conflict Resolution — Complete Workflow

1
Conflict occurs during merge/rebase
Git stops and marks conflicted files. Merge commit is incomplete.
2
Find all conflicted files
git status → shows "both modified" | git diff --name-only --diff-filter=U
3
Edit each conflicted file
Choose "ours", "theirs", or manually write the combined version. Remove ALL conflict markers (<<<, ===, >>>).
4
Stage the resolved files
git add UserService.java — marks file as resolved
5
Complete the merge/rebase
git merge --continue or git rebase --continue — creates merge commit
ShellConflict resolution commands
# ── During merge conflict ──
git status                      # shows "both modified" files
git diff                        # shows conflict markers in working tree

# ── Accept one side entirely (without editing) ──
git checkout --ours   file.txt  # take OUR version (current branch)
git checkout --theirs file.txt  # take THEIR version (merging branch)
# Modern equivalent:
git restore --ours   file.txt
git restore --theirs file.txt

# ── After resolving each file ──
git add file.txt                # mark as resolved
git merge --continue            # complete the merge

# ── Or ABORT entire merge ──
git merge --abort               # cancel, restore pre-merge state
git rebase --abort              # cancel rebase

# ── Merge tools ──
git mergetool                   # open configured merge tool
git mergetool --tool=vimdiff    # specify tool
# Configure: git config --global merge.tool vscode
# git config --global mergetool.vscode.cmd "code --wait $MERGED"

# ── Check for unresolved markers ──
grep -r "<<<<<<<" src/          # find any remaining conflict markers

Shellgit stash — complete guide
# ── Save stash ──

git stash push                    # same as above (explicit)

git stash -p                      # interactive: choose which hunks to stash

git stash push -k                # stash but keep staged files staged (-k=--keep-index)

git stash list                    # stash@{0}: WIP on main: abc1234 message

git stash show -p                 # show full diff of latest stash

git stash pop                     # apply latest stash + REMOVE from stash list
git stash apply                   # apply latest stash + KEEP in stash list
git stash apply stash@{2}         # apply specific stash
git stash pop stash@{1}           # pop specific stash
git stash branch feature/wip     # create new branch from stash + apply (no conflicts!)

# ── Remove stash ──
git stash drop                    # delete latest stash
git stash drop stash@{2}          # delete specific stash
git stash clear                   # delete ALL stashes ⚠️

# ── Common stash workflow ──
# You're working on feature, urgent bug comes in:
git stash push -m "WIP login form"  # save current work
git switch main
git switch -c hotfix/null-pointer  # fix the bug
# ... fix and commit ...
git switch feature/login
git stash pop                       # restore your WIP

    echo "   Required: type(scope): description"
    echo "   Example:  feat(auth): add login with OAuth2"
    exit 1
fi

📖 CI/CD Definitions

  • Continuous Integration (CI): Automatically build + test every code change merged to shared branch. Goal: catch bugs early, validate code quality constantly.
  • Continuous Delivery (CD): CI + automatically prepare artifact for deployment to any environment. Deployment to production requires manual approval button.
  • Continuous Deployment (CD): CI + automatically deploy to production on every passing build. No human gate — requires very mature testing.
  • CI Pipeline: Automated sequence: Source → Build → Unit Test → Integration Test → Static Analysis → Package → Artifact Store
  • CD Pipeline: Artifact → Deploy to Dev → Integration Tests → Deploy to Staging → UAT → Deploy to Production

📖 GitHub Actions Core Concepts

  • Workflow: Automated process defined in YAML (.github/workflows/). Triggered by events.
  • Event: Trigger — push, pull_request, schedule (cron), workflow_dispatch (manual), release

GroovyJenkinsfile — Spring Boot declarative pipeline
pipeline {
    // Run on any available agent (or specify label: agent { label 'linux' })
    agent any

    // Tool versions configured in Jenkins Global Tool Configuration
    tools {
        jdk 'JDK-21'
        maven 'Maven-3.9'
    }

    options {
        buildDiscarder(logRotator(numToKeepStr: '10'))  // keep last 10 builds
        timeout(time: 30, unit: 'MINUTES')              // timeout
        skipStagesAfterUnstable()
        disableConcurrentBuilds()                          // no parallel same-branch builds
        timestamps()                                       // add timestamps to console
    }

    parameters {
        choice(name: 'ENVIRONMENT', choices: ['dev', 'staging', 'prod'],
            description: 'Target environment')
        booleanParam(name: 'RUN_INTEGRATION_TESTS', defaultValue: true)
        string(name: 'DOCKER_TAG', defaultValue: 'latest')
    }

    environment {
        // credentials() fetches from Jenkins credentials store
        DOCKER_CREDENTIALS = credentials('dockerhub-credentials')
        SONAR_TOKEN        = credentials('sonar-token')
        APP_VERSION        = ""
        IMAGE_NAME         = "mycompany/spring-app"
    }

    stages {

        stage('Checkout') {
            steps {
                git branch: 'main', url: 'https://github.com/org/repo.git'
                script {
                    APP_VERSION = sh(script: 'mvn help:evaluate -Dexpression=project.version -q -DforceStdout', returnStdout: true).trim()
                    echo "Building version: ${APP_VERSION}"
                }
            }
        }

        stage('Build') {
            steps {
                sh 'mvn clean package -DskipTests --batch-mode'
            }
            post {
                success {
                    archiveArtifacts artifacts: 'target/*.jar', fingerprint: true
                }
            }
        }

        stage('Test') {
            parallel {   // run unit and integration tests in parallel
                stage('Unit Tests') {
                    steps {
                        sh 'mvn test --batch-mode'
                    }
                    post {
                        always {
                            junit 'target/surefire-reports/*.xml'
                            jacoco(execPattern: 'target/jacoco.exec',
                                   classPattern: 'target/classes',
                                   minimumInstructionCoverage: '80')
                        }
                    }
                }
                stage('Integration Tests') {
                    when { expression { params.RUN_INTEGRATION_TESTS } }
                    steps {
                        sh 'mvn verify -Pintegration-tests --batch-mode'
                    }
                    post {
                        always {
                            junit 'target/failsafe-reports/*.xml'
                        }
                    }
                }
            }
        }

        stage('Code Quality') {
            steps {
                withSonarQubeEnv('SonarQube') {
                    sh """
                        mvn sonar:sonar \
                          -Dsonar.projectKey=my-project \
                          -Dsonar.token=${SONAR_TOKEN}
                    """
                }
                timeout(time: 2, unit: 'MINUTES') {
                    waitForQualityGate abortPipeline: true  // fail if quality gate fails
                }
            }
        }

        stage('Docker Build & Push') {
            when { branch 'main' }  // only on main branch
            steps {
                script {
                    def dockerImage = docker.build("${IMAGE_NAME}:${APP_VERSION}")
                    docker.withRegistry('https://registry.hub.docker.com', 'dockerhub-credentials') {
                        dockerImage.push(APP_VERSION)
                        dockerImage.push('latest')
                    }
                }
            }
        }

        stage('Deploy') {
            when { branch 'main' }
            steps {
                input message: "Deploy version ${APP_VERSION} to ${params.ENVIRONMENT}?",
                      ok: 'Deploy'  // manual approval gate
                script {
                    sh """
                        kubectl set image deployment/spring-app \
                          spring-app=${IMAGE_NAME}:${APP_VERSION} \
                          --namespace=${params.ENVIRONMENT}
                        kubectl rollout status deployment/spring-app \
                          --namespace=${params.ENVIRONMENT}
                    """
                }
            }
        }
    }

    post {   // runs after all stages
        always {
            cleanWs()   // clean workspace
        }
        success {
            slackSend channel: '#deployments',
                      color: 'good',
                      message: "✅ ${JOB_NAME} #${BUILD_NUMBER} - SUCCESS. Version: ${APP_VERSION}"
        }
        failure {
            slackSend channel: '#deployments',
                      color: 'danger',
                      message: "❌ ${JOB_NAME} #${BUILD_NUMBER} - FAILED. Check: ${BUILD_URL}"
            emailext subject: "Build Failed: ${JOB_NAME}",
                     body: "Build ${BUILD_NUMBER} failed. See ${BUILD_URL}",
                     to: 'team@company.com'
        }
        unstable {
            slackSend channel: '#deployments', color: 'warning',
                      message: "⚠️ ${JOB_NAME} #${BUILD_NUMBER} - UNSTABLE (test failures)"
        }
    }
}

Interview Q&A
25+ most-asked Git & CI/CD interview questions

🔥 Git Core Questions

Q1What is the difference between git merge and git rebase?
git merge: Creates a new merge commit with two parents. Preserves the complete history showing exactly when branches diverged and merged. Non-destructive — never changes existing commits.
git rebase: Replays commits from one branch on top of another, creating NEW commits with different SHAs. Results in a linear history. Rewrites history — never do on public/shared branches.
When to use: Merge for feature completion to main (preserves context). Rebase for syncing local feature with updated main (keeps linear history, cleaner PRs).
Q2What is git rebase --interactive used for?
Interactive rebase lets you rewrite history of unpushed commits by: squash (combine multiple commits into one), reword (edit commit message), drop (delete a commit), edit (amend files in old commit), reorder (change commit order). Common before opening a PR to clean up "WIP" commits into meaningful units.
Q3What is the difference between git reset and git revert?
git reset: Moves the branch pointer backward to a previous commit. Rewrites history. Three modes: --soft (keep staged), --mixed (unstage, keep files), --hard (discard everything). UNSAFE on shared branches.
git revert: Creates a new commit that undoes the changes of a specific commit. Does NOT rewrite history. Safe to use on shared/public branches. Always prefer revert for pushed commits.
Q4What is HEAD in Git?
HEAD is a pointer to the currently checked-out branch or commit. Normally it points to a branch (e.g., refs/heads/main), which in turn points to the latest commit. When you checkout a specific commit directly (git checkout abc123), HEAD points to that commit directly — this is called "detached HEAD" state. HEAD~1 = one commit back. HEAD^ = first parent. HEAD~2 = two commits back.
Q5What is a fast-forward merge? When does it NOT happen?
Fast-forward merge occurs when the target branch has NO new commits since the feature branch was created — Git simply moves the branch pointer forward to the feature branch tip. No merge commit is created.
It does NOT happen when: (1) The target branch has diverged (new commits on both branches), (2) You use --no-ff flag to force a merge commit, (3) Merging across distant histories.
Q6How do you undo the last commit without losing changes?
git reset --soft HEAD~1 — moves HEAD back one commit, all changes from that commit remain STAGED.
git reset HEAD~1 (--mixed, default) — moves HEAD back, changes are UNSTAGED but preserved in working tree.
git reset --hard HEAD~1 — DANGEROUS: moves HEAD back AND discards all changes (unrecoverable, use with care).
If already pushed: git revert HEAD creates a new undo commit (safe for shared branches).
Q7What is the difference between git fetch and git pull?
git fetch: Downloads objects and refs from remote but does NOT integrate changes into your working branch. Updates remote-tracking branches (origin/main etc.) only. Safe — never modifies your local work.
git pull: = git fetch + git merge (or rebase if configured). Fetches AND integrates remote changes into current branch. Can cause merge conflicts.
Best practice: git fetch first, inspect git log origin/main, then git merge manually.
Q8What is cherry-pick and when would you use it?
git cherry-pick <SHA> copies a specific commit from one branch and applies it to the current branch as a new commit (same changes, different SHA).
Use cases: (1) A hotfix was committed to the wrong branch — cherry-pick it to the correct one. (2) A feature branch has one useful commit but you don't want to merge the whole branch. (3) Backport a fix from main to an older release branch. Overuse is a code smell — prefer proper branching.
Q9What is git stash and when would you use it?
git stash temporarily saves your uncommitted changes (working tree + staged) and reverts the working directory to the last commit. Changes are stored in a stack. Retrieve with git stash pop or git stash apply.
Use cases: (1) Need to switch branches urgently but don't want to commit unfinished work. (2) Pull latest changes but working tree is dirty. (3) Try an experiment and save current WIP.
Q10How do you find the commit that introduced a bug?
Manual approach: git log --oneline to find suspect commits, git diff abc..def to compare.
git bisect (best): Binary search. Mark current commit as bad, an old working commit as good. Git checks out the midpoint. Test, mark good/bad. Repeat ~log2(n) times to find exact bad commit.
Automated bisect: git bisect run mvn test -Dtest=BugTest — runs your test automatically to find the commit.
Q11What is the difference between an annotated and lightweight tag?
Lightweight tag: git tag v1.0 — just a pointer to a commit (like a branch that doesn't move). Stores no extra metadata.
Annotated tag: git tag -a v1.0 -m "message" — full Git object with tagger name, email, date, message, and optional GPG signature. Has its own SHA. Checkable with git show v1.0.
Use annotated for releases — they record who tagged and when, can be GPG-signed, and git describe uses them by default.
Q12Explain GitFlow vs Trunk-Based Development.
GitFlow: Long-lived branches (main, develop, feature/*, release/*, hotfix/*). Structured, predictable releases. Good for versioned software. Con: complex, merge conflicts accumulate.
Trunk-Based Development (TBD): Everyone commits to main (trunk) daily. Feature flags hide incomplete features. Very short-lived branches (<2 days). Requires mature CI/CD and testing. Used by Google, Facebook. Encourages continuous integration. Con: requires discipline, feature flags complexity.
GitHub Flow: Middle ground — simple (main + feature branches), continuous deployment, widely adopted for web apps.
Q13What is CI/CD and what are its benefits?
CI (Continuous Integration): Automatically build + test every code change merged to main branch. Benefits: catch bugs early, validate integration, enforce code quality, reduce "integration hell".
CD (Continuous Delivery/Deployment): Automatically deploy to environments after CI passes. Benefits: faster time-to-market, smaller (less risky) releases, faster feedback, reduced manual errors, more frequent deployments.
Combined: enables teams to release multiple times per day confidently.
Q14How do you handle secrets in CI/CD pipelines?
Never hardcode secrets in code or YAML! Solutions:
1. GitHub Secrets / Jenkins Credentials — store in platform, inject as env vars
2. HashiCorp Vault — dynamic secret management, rotate secrets
3. AWS Secrets Manager / Azure Key Vault — cloud-native secret storage
4. Environment-specific .env files — NOT committed to git, provided at runtime
5. OIDC federation — GitHub Actions can assume AWS/GCP roles without long-lived credentials
Best practice: rotate secrets regularly, use least-privilege, audit access logs.
Q15What is the difference between Continuous Delivery and Continuous Deployment?
Continuous Delivery: Every change that passes all tests is packaged and ready to deploy to production. But deployment requires a manual approval / button click. Business can decide when to release. Suitable when releases need business sign-off.
Continuous Deployment: Every passing change is AUTOMATICALLY deployed to production without human intervention. Requires very high test confidence (>95% coverage, E2E, canary releases). Used by Netflix, Amazon. Faster feedback, no deployment backlog.
