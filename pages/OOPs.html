Section 0
OOPs Overview
What is OOP and why it exists

📖 What is Object-Oriented Programming?

OOP is a programming paradigm that organizes software around objects — instances of classes that bundle data (fields) and behavior (methods) together. Java is primarily an OOP language.

OOP was introduced to solve problems of procedural/structured programming like code reuse, maintainability, and modeling real-world entities.

💡
Is Java fully OOP? — No. Java is NOT fully object-oriented because it supports primitive types (int, char, float, boolean etc.) which are not objects. A fully OOP language (like Smalltalk) treats everything as an object.

Pillar 1
Abstraction
Hiding implementation details, exposing only what's necessary

📖 What is Abstraction?

Abstraction means hiding the internal implementation and showing only the relevant features to the user. Think of a car — you press the accelerator without knowing how the engine works internally.

In Java, abstraction is achieved via:

  • Abstract classes — partial abstraction (0% to 100%)
  • Interfaces — 100% abstraction (before Java 8)

🔷 Abstract Class

Declared with abstract keyword. Cannot be instantiated. May contain abstract + concrete methods.

Java Abstract Class Example 
// Abstract class — cannot instantiate directly
abstract class Animal {

    private String name;

    // Constructor in abstract class — YES it can have one!
    Animal(String name) {
        this.name = name;
        System.out.println("Animal constructor called: " + name);
    }

    // Abstract method — NO body, subclass MUST override
    abstract void makeSound();

    // Concrete method — has body, subclass can use or override
    void breathe() {
        System.out.println(name + " is breathing");
    }

    public String getName() { return name; }
}

class Dog extends Animal {

    Dog(String name) {
        super(name); // Must call parent constructor
    }

    @Override
    void makeSound() {
        System.out.println(getName() + " says: Woof!");
    }
}

public class Main {
    public static void main(String[] args) {
        // Animal a = new Animal();  ← ERROR! Cannot instantiate
        Animal dog = new Dog("Tommy");
        dog.makeSound();  // Tommy says: Woof!
        dog.breathe();    // Tommy is breathing
    }
}
Output
Animal constructor called: Tommy
Tommy says: Woof!
Tommy is breathing

🔶 Interface

A contract that a class must follow. Before Java 8: all methods were abstract. After Java 8: can have default and static methods. Java 9+: can have private methods.

Java Interface with default & static methods (Java 8+) 
interface Flyable {
    // implicitly public abstract
    void fly();

    // Java 8 — default method (has body, optional to override)
    default void land() {
        System.out.println("Landing safely");
    }

    // Java 8 — static method (called on interface, not object)
    static void info() {
        System.out.println("Flyable interface");
    }

    // Variables are implicitly public static final
    int MAX_ALTITUDE = 30000; // constant
}

class Bird implements Flyable {
    @Override
    public void fly() {
        System.out.println("Bird is flying at max " + MAX_ALTITUDE + " ft");
    }
    // land() is optional to override — default is available
}

class Plane implements Flyable {
    @Override
    public void fly() {
        System.out.println("Plane is flying!");
    }

    @Override
    public void land() {
        System.out.println("Plane landing on runway");
    }
}

Rule of thumb: Use abstract class when classes share common state/behavior (IS-A). Use interface when you want to define a capability contract that unrelated classes can implement (CAN-DO). Example: A Dog IS-A Animal but CAN-DO Swimmable.
🎯 Interview Questions — Abstraction
Q1Can an abstract class have a constructor?
Yes! Abstract classes CAN have constructors. They are called when a subclass is instantiated via super(). Purpose: to initialize fields defined in the abstract class.
Q2Can an interface have instance variables?
No. Interface variables are implicitly public static final (constants). They are shared across all implementors and cannot be changed.
Q3Can we instantiate an abstract class?
No directly. But we can create an anonymous subclass: Animal a = new Animal() { void makeSound(){} };

Pillar 2
Encapsulation
Data hiding + controlled access via access modifiers

📖 What is Encapsulation?

Encapsulation is the process of wrapping data (fields) and methods into a single unit (class) and restricting direct access to the internal state using access modifiers. Access is provided through getters and setters.

  • Protects data from unintended modification
  • Enables validation in setters before assigning values
  • Makes code maintainable — internal changes don't break external code
  • Supports immutability — no setter + final field = immutable

💻 Encapsulation — Full Example

Java Bank Account with Validation 
public class BankAccount {

    // private — only accessible within this class
    private String accountNumber;
    private double balance;
    private String ownerName;

    // Constructor
    public BankAccount(String accNum, String owner, double initialBalance) {
        this.accountNumber = accNum;
        this.ownerName = owner;
        this.balance = initialBalance;
    }

    // Getter — read-only access to accountNumber
    public String getAccountNumber() { return accountNumber; }

    // Getter for balance
    public double getBalance() { return balance; }

    // NO setter for accountNumber — it's immutable after creation

    // Controlled deposit with validation
    public void deposit(double amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("Deposit must be positive");
        }
        balance += amount;
    }

    // Controlled withdrawal with validation
    public void withdraw(double amount) {
        if (amount > balance) {
            throw new IllegalStateException("Insufficient funds");
        }
        balance -= amount;
    }
}

// Usage
BankAccount acc = new BankAccount("ACC001", "Alice", 1000.0);
acc.deposit(500);
acc.withdraw(200);
// acc.balance = 99999;  ← COMPILE ERROR — private!
System.out.println(acc.getBalance()); // 1300.0

🧊 Creating an Immutable Class

An immutable class cannot be changed after creation. To create one:

  • Make class final (can't be subclassed)
  • Make all fields private final
  • No setters
  • Initialize all fields via constructor
  • Return copies of mutable fields in getters (defensive copy)
Java Immutable Class Pattern 
public final class ImmutableStudent {

    private final int id;
    private final String name;
    private final List<String> courses; // mutable field!

    public ImmutableStudent(int id, String name, List<String> courses) {
        this.id = id;
        this.name = name;
        // Defensive copy — don't store reference to outside list
        this.courses = new ArrayList<>(courses);
    }

    public int getId()    { return id; }
    public String getName() { return name; }

    public List<String> getCourses() {
        // Return defensive copy — don't expose internal list
        return Collections.unmodifiableList(courses);
    }
}
🎯 Interview Questions — Encapsulation
Q1What is the difference between encapsulation and abstraction?
Abstraction = WHAT (hide complexity, show only what's needed) | Encapsulation = HOW (bundle data+methods, protect via access modifiers). Abstraction is design-level; encapsulation is implementation-level.
Q2Why is protected used?
When you want a field/method to be accessible in subclasses (even in different packages) but NOT public. Common in framework code and template method pattern.
Q3How does encapsulation support immutability?
By making fields private final with no setters and providing only getters. String in Java is a classic immutable example — once created, the value never changes.

Pillar 3
Inheritance
IS-A relationship, super(), constructor chaining, types

📖 What is Inheritance?

Inheritance allows a child class to acquire properties and methods of a parent class. It promotes code reusability and establishes an IS-A relationship.

  • IS-A: Dog IS-A Animal ✅ (use inheritance)
  • HAS-A: Car HAS-A Engine (use composition)
  • Java supports single inheritance for classes
  • Multiple inheritance via interfaces only

🔗 Constructor Chaining — super() and this()

Java Constructor Chain — Output Prediction (asked in interview!) 
class A {
    A() {
        System.out.print("A ");
    }
}

class B extends A {
    B() {
        // Compiler auto-inserts: super();  ← calls A()
        System.out.print("B ");
    }
}

class C extends B {
    C() {
        // Compiler auto-inserts: super();  ← calls B()
        System.out.print("C ");
    }
}

public class Main {
    public static void main(String[] args) {
        new C();
    }
}
Output
A B C
💡
Rule: When new C() is called → C() calls B() via super() → B() calls A() via super() → A() prints first. Then B, then C. Parent constructors always execute before child.

🔗 this() — Constructor Overloading Chain

Java this() for constructor chaining within same class 
class Employee {
    private String name;
    private int age;
    private String dept;

    // Main constructor
    Employee(String name, int age, String dept) {
        this.name = name;
        this.age = age;
        this.dept = dept;
    }

    // Delegating to main constructor using this()
    Employee(String name, int age) {
        this(name, age, "General"); // must be FIRST statement
    }

    Employee(String name) {
        this(name, 0); // chains to above
    }
}
⚠️
Rules: super() and this() must be the first statement in a constructor. You CANNOT use both in same constructor.

⬆️ super Keyword — 3 Uses

Java All 3 uses of super 
class Vehicle {
    String brand = "Generic";

    Vehicle(String brand) {
        this.brand = brand;
    }

    void display() {
        System.out.println("Vehicle: " + brand);
    }
}

class Car extends Vehicle {
    String brand = "Toyota"; // shadows parent brand

    Car() {
        super("Ford"); // USE 1: Call parent constructor
    }

    void show() {
        System.out.println(super.brand); // USE 2: Access parent field → "Ford"
        System.out.println(this.brand);  // → "Toyota"
        super.display();                   // USE 3: Call parent method
    }
}

🔄 IS-A vs HAS-A Relationship

Java IS-A (Inheritance) vs HAS-A (Composition) 
// IS-A — Dog IS-A Animal (Inheritance)
class Animal { void eat() {} }
class Dog extends Animal { void bark() {} }

// HAS-A — Car HAS-A Engine (Composition — preferred!)
class Engine {
    void start() { System.out.println("Engine started"); }
}

class Car {
    // Car HAS-A Engine — composition
    private Engine engine = new Engine();

    void drive() {
        engine.start();
        System.out.println("Car is moving");
    }
}

// Ask: Is Car a type of Engine? NO → HAS-A (composition)
// Ask: Is Dog a type of Animal? YES → IS-A (inheritance)
✅
Favour composition over inheritance (Effective Java rule). Composition is more flexible — you can change the behavior at runtime by swapping the composed object.
🎯 Interview Questions — Inheritance
Q1What happens if child class doesn't call super()?
Java compiler automatically inserts super() (no-arg) as the first line in child constructor. If parent has NO no-arg constructor (only parameterized), compile error occurs — must explicitly call super(args).
Q2Why doesn't Java support multiple class inheritance?
To avoid the Diamond Problem: if class C extends A and B, and both A and B have same method, Java won't know which one to use. Interfaces with default methods handle this via explicit override rule.
Q3Can a subclass reduce the visibility of an overridden method?
No! You can only increase (widen) access. If parent method is public, child cannot override it as protected or private — compile error.

Pillar 4
Polymorphism
Overriding (runtime) vs Overloading (compile-time)

📖 What is Polymorphism?

Poly = many, morph = forms. Polymorphism allows one interface to be used for different data types or behaviors. In Java there are two types:

  • Compile-time Polymorphism — Method Overloading (resolved at compile time)
  • Runtime Polymorphism — Method Overriding (resolved at runtime via dynamic dispatch)

🔷 Method Overloading — Compile-time Polymorphism

Same method name, different parameters (type, number, or order) in the SAME class.

Java Method Overloading Examples 
class Calculator {

    // Different number of params
    int add(int a, int b)              { return a + b; }
    int add(int a, int b, int c)       { return a + b + c; }

    // Different type of params
    double add(double a, double b)    { return a + b; }

    // NOTE: Return type ALONE cannot differentiate overloading!
    // int add(int a, int b)    ← ERROR if double add already exists with same params
}

class PrintTest {

    void print(int x)      { System.out.println("int: " + x); }
    void print(Object x)   { System.out.println("Object: " + x); }
    void print(String x)   { System.out.println("String: " + x); }

    public static void main(String[] args) {
        PrintTest p = new PrintTest();
        p.print(5);       // → int: 5
        p.print("hi");   // → String: hi
        p.print(null);   // → String: null  (most specific type chosen)
    }
}
⚠️
Tricky: print(null) calls print(String) because String is more specific than Object. If both String and Integer overloads exist without Object, print(null) would be ambiguous → compile error!

🔶 Method Overriding — Runtime Polymorphism

Child class provides its own implementation of a method defined in parent class.

Java Method Overriding — Dynamic Dispatch 
class Shape {
    void draw() {
        System.out.println("Drawing a shape");
    }
}

class Circle extends Shape {
    @Override
    void draw() {
        System.out.println("Drawing a CIRCLE");
    }
}

class Rectangle extends Shape {
    @Override
    void draw() {
        System.out.println("Drawing a RECTANGLE");
    }
}

public class Main {
    public static void main(String[] args) {

        // Runtime Polymorphism — reference is Shape, object is Circle/Rectangle
        Shape s1 = new Circle();
        Shape s2 = new Rectangle();

        s1.draw();  // Drawing a CIRCLE (decided at RUNTIME)
        s2.draw();  // Drawing a RECTANGLE (decided at RUNTIME)

        // Polymorphic array
        Shape[] shapes = { new Circle(), new Rectangle(), new Circle() };
        for (Shape shape : shapes) {
            shape.draw(); // Each calls its own version
        }
    }
}
Output
Drawing a CIRCLE
Drawing a RECTANGLE
Drawing a CIRCLE
Drawing a RECTANGLE
Drawing a CIRCLE

⚡ Can We Override a Static Method?

Java Method Hiding vs Overriding 
class Parent {
    static void greet() { System.out.println("Hello from Parent"); }
    void show()       { System.out.println("Parent show"); }
}

class Child extends Parent {
    // This is METHOD HIDING (not overriding)
    static void greet() { System.out.println("Hello from Child"); }

    @Override
    void show()       { System.out.println("Child show"); }
}

public class Test {
    public static void main(String[] args) {
        Parent obj = new Child();

        obj.greet(); // → Hello from Parent (static — bound at COMPILE time)
        obj.show();  // → Child show (instance — bound at RUNTIME)
    }
}
Output
Hello from Parent
Child show
🚨
Key point: Static methods are HIDDEN not overridden. The method called depends on the reference type (Parent), not the object type (Child). This is why static methods do NOT participate in runtime polymorphism.
🎯 Interview Questions — Polymorphism
Q1What is covariant return type?
In overriding, the child method's return type can be a subtype of the parent's return type. E.g., parent returns Animal, child can return Dog (which extends Animal). This is covariant return type — added in Java 5.
Q2Why does reducing scope from parent to child give error while overriding?
Liskov Substitution Principle (LSP). If parent says a method is public, any code using the parent reference expects public access. If child makes it protected, it breaks the contract — code expecting public behavior would fail. Java enforces widening only.
Q3What is static binding vs dynamic binding?
Static/Early binding: Method resolved at compile time — static, private, final methods, overloaded methods. Dynamic/Late binding: Method resolved at runtime — overridden methods. JVM uses vtable for dynamic dispatch.

Section 5
Static Keyword — In Depth
static variable, method, block, class, import

📖 What is static?

The static keyword means the member belongs to the class (not to any instance). It is loaded into memory when the class is loaded (even before any object is created).

🔷 All 5 Uses of static

Java Static — All Contexts 
public class StaticDemo {

    // 1. STATIC VARIABLE — shared by all instances
    static int instanceCount = 0;
    private int id;

    // 2. STATIC BLOCK — runs once when class is loaded
    static {
        System.out.println("Class loaded — static block runs ONCE");
        instanceCount = 100; // initialize
    }

    // Instance block — runs on every new object
    {
        instanceCount++;
        this.id = instanceCount;
        System.out.println("Instance block: id = " + id);
    }

    // 3. STATIC METHOD — no access to instance variables
    static int getCount() {
        return instanceCount;
        // return id;  ← COMPILE ERROR — cannot access instance var
    }

    // 4. STATIC INNER CLASS
    static class Helper {
        void help() { System.out.println("Helping!"); }
    }
}

// 5. STATIC IMPORT
// import static java.lang.Math.PI;
// System.out.println(PI);  ← No need to write Math.PI

class TestStatic {
    public static void main(String[] args) {
        // static block runs here first
        StaticDemo s1 = new StaticDemo(); // id = 101
        StaticDemo s2 = new StaticDemo(); // id = 102

        System.out.println(StaticDemo.getCount()); // 102

        // Static inner class — no outer object needed
        StaticDemo.Helper h = new StaticDemo.Helper();
        h.help();
    }
}
Output
Class loaded — static block runs ONCE
Instance block: id = 101
Instance block: id = 102
102
Helping!

JavaConstructor Order
class Base {
    int x = initX();
    int initX() {
        System.out.println("Base initX");
        return 10;
    }
    Base() { System.out.println("Base()"); }

class Child extends Base {

✅ Output & Explanation
Output: Base initX → Base() → Child initY → Child()

Why? Order when new Child() is called:
1. Parent fields initialized (x = initX() → prints "Base initX")
2. Parent constructor body (prints "Base()")
3. Child fields initialized (y = initY() → prints "Child initY")
4. Child constructor body (prints "Child()")

JavaStatic vs Instance
class A {
    static int x = 10;
    int y = 20;

        System.out.println("x = " + x);

JavaRuntime Polymorphism with Fields
class Parent {

S4What is the output?
Javathis() and super() trap
class X {
    X()    { System.out.print("X0 "); }
    X(int n) { System.out.print("X1 "); }
}

class Y extends X {
    Y()    { this(10); System.out.print("Y0 "); }

Why? new Y() → Y() calls this(10) → Y(10) called → Y(10) calls super(10) → X(10) prints "X1" → back in Y(10) prints "Y1" → back in Y() prints "Y0"

class Animal {

class Dog extends Animal {

Complete Interview Q&A
All OOPs questions asked in real interviews

🏛️ Abstraction — Interview Q&A

Q1Can abstract class have constructor? Why?
Yes. To initialize its own fields. Called by child via super(). Though abstract class can't be instantiated directly, the constructor runs when child class object is created.
Q2Difference: abstract class vs interface — when to use?
Use abstract class when: classes share state/code + IS-A relationship. Use interface when: defining capability contract, unrelated classes need same behavior, or for multiple inheritance of type.
Q3Can interface extend another interface?
Yes! And it can extend multiple interfaces: interface C extends A, B. A class implementing C must implement all methods from A, B, and C.
Q4What are Marker Interfaces? Why needed?
Interfaces with NO methods (empty). Used to tag a class to provide metadata to JVM or frameworks. Examples: Serializable, Cloneable, RandomAccess. JVM uses these tags to provide special behavior (e.g., allow serialization only if Serializable is implemented).

🔒 Encapsulation — Interview Q&A

Q5What are the rules for an immutable class?
1. Class must be final. 2. All fields private final. 3. No setters. 4. Initialize via constructor. 5. Return defensive copies of mutable fields (like List, Date) in getters. Example: String class is immutable.
Q6Why do we need getters/setters instead of public fields?
Setters allow validation (e.g., reject negative age), lazy initialization, logging, and maintain invariants. Public fields give unrestricted access — no way to add logic later without breaking API.

🧬 Inheritance — Interview Q&A

Q7What is the Diamond Problem?
If class D extends B and C, and both B and C inherit method from A and override it differently, D doesn't know which version to use. Java solves this by not allowing multiple class inheritance. With interfaces (Java 8 default methods), if conflict occurs, class MUST explicitly override the method.
Q8Composition vs Inheritance — which to prefer?
Prefer Composition (Effective Java, Item 18). Composition = HAS-A. More flexible — can change behavior at runtime, no tight coupling, no fragile base class problem. Use inheritance only for true IS-A relationships where subtype can substitute supertype.

🔄 Polymorphism — Interview Q&A

Q9What is method hiding? How is it different from overriding?
Method Hiding: When a child class defines a static method with same signature as parent's static method. The method called depends on reference type (compile-time). Overriding: Instance methods — method called depends on object type (runtime). @Override does NOT apply to static methods.
Q10Can we overload main method?
Yes! Java allows overloading main. But JVM only calls public static void main(String[] args) as the entry point. Other overloaded versions can be called manually.
Q11Can constructors be overloaded? Can they be overridden?
Overloaded: ✅ Yes — multiple constructors with different parameters in same class. Overridden: ❌ No — constructors are not inherited, so can't be overridden. Each class has its own constructors.

⚡ Static — Interview Q&A

Q12Can static method access instance variables?
No. Static methods don't have a this reference. They belong to the class, not any object. To access instance variables in a static method, you need to pass an object reference as parameter.
Q13Why are utility classes (Math, Collections) final with private constructor?
final prevents subclassing (no one can extend and override static methods). Private constructor prevents instantiation. This enforces the utility class pattern — all methods are static, no need to create objects.
Q14Difference between static and instance initializer blocks?
Static block static { }: runs once when class is loaded, before any object creation. Used for one-time setup. Instance block { }: runs every time a new object is created, before constructor body. Runs AFTER parent constructor.
