The JUnit Architecture

Why JUnit 5 is three separate things — and why that's brilliant.

🏗️ Why JUnit 5 Is Different From JUnit 4

JUnit 4 was a single monolithic JAR. Everything — the API you wrote tests with, the engine that ran them, and the extension model — was baked into one artifact. This worked fine until people wanted to run other kinds of tests on the same platform: Cucumber scenarios, Spock specs, Kotlin tests.

There was no hook for that. JUnit 4 either ran your tests, or it didn't.

JUnit 5 blew up this assumption. The team separated three distinct concerns into three separate modules. Understanding this split is understanding why JUnit 5 is so much more powerful.

JUnit 4 Reality

  • One monolithic JAR
  • API + engine tightly coupled
  • Extensions via @RunWith — clunky
  • Only one runner per test class
  • No nested test support

JUnit 5 Reality

  • Three separate modules
  • Clean separation: API vs engine vs platform
  • Extensions via @ExtendWith — composable
  • Multiple engines running simultaneously
  • First-class nested tests, parameterized, dynamic
💡
The Core Insight

JUnit 4's architecture forced IDE vendors, build tools, and test framework authors to hard-code JUnit 4 knowledge everywhere. JUnit 5 introduces a universal test platform that any test engine can plug into — decoupling the test execution infrastructure from the test programming model.

JUnit Platform — The Foundation

The Platform is the infrastructure layer. It doesn't know about @Test. It doesn't know what a "test class" is. Its only job is: find engines on the classpath, ask each engine to discover its tests, then run them.

Think of it as an airport. It doesn't care whether you're flying United or Delta. It provides gates, runways, and the control tower. Each airline (engine) operates its own flights (tests).

  • TestEngine API — the contract any engine must implement
  • Launcher API — how build tools (Maven/Gradle) trigger test runs
  • Console Launcher — run tests from the terminal
  • Build tool integrations — Maven Surefire, Gradle Test task

JUnit Jupiter — What You Write Every Day

Jupiter is the new programming model for JUnit 5. This is where all the APIs you use daily live: @Test, @BeforeEach, Assertions, @ParameterizedTest, etc.

Jupiter is itself split into two artifacts:

Artifact What it contains Who uses it
junit-jupiter-api All annotations, assertions, extension SPI — what your test code imports Your test source code
junit-jupiter-engine The engine that discovers and runs Jupiter tests on the platform Runtime only — never imported directly
junit-jupiter-params @ParameterizedTest, @ValueSource, @CsvSource, etc. Your test source code
🎯
Interview Tip

If asked "what is JUnit Jupiter?", say: "Jupiter is the new programming model and extension model introduced in JUnit 5. It consists of the API jar (what developers write against) and the engine jar (what runs those tests on the JUnit Platform)."

JUnit Vintage — The Bridge to the Past

If you have a project with 2,000 JUnit 4 tests, you can't migrate them overnight. Vintage lets you run those old tests on the new platform unchanged — side by side with new Jupiter tests in the same build.

It's purely a migration tool. If your project is greenfield, you never need it.

⚠️
Don't confuse Vintage with Legacy

Vintage doesn't mean "JUnit 5 supports running on old JUnit 4 runtime." It means "JUnit 4 tests can run on JUnit 5 Platform." The direction matters in interviews.

⚙️ The Test Engine API

A TestEngine is just an interface with two methods. That's it. Any framework — Spock, Cucumber, your own custom framework — can become a first-class citizen on the JUnit Platform by implementing this interface.

Java TestEngine interface (simplified) 
public interface TestEngine {

    // Step 1: Platform asks "what tests do you have?"
    TestDescriptor discover(EngineDiscoveryRequest discoveryRequest,
                                UniqueId uniqueId);

    // Step 2: Platform says "run these tests"
    void execute(ExecutionRequest request);

    // Optional: unique ID for this engine
    default String getId() { return "my-custom-engine"; }
}

discover() receives filtering criteria (run only these packages, these tags, etc.) and returns a tree of TestDescriptor nodes — essentially a map of everything the engine found to run.

execute() receives that tree plus a listener to report events: test started, passed, failed, skipped. The Platform aggregates these events from all engines and sends them to your IDE or build report.

🔥
Why This Design Is Brilliant

Before JUnit 5, IDEs had to special-case every test framework. Now any framework that implements TestEngine gets IDE support, build tool support, and coverage tool support for free. The Platform handles all of that.

🔄 How It All Connects in Practice

The Runtime Flow

Build Tool Triggers the Platform
Maven Surefire / Gradle Test task calls LauncherFactory.create() and invokes Launcher.discover() then Launcher.execute().
Platform Scans Classpath for Engines
Uses Java ServiceLoader to find all TestEngine implementations. Finds JupiterTestEngine, possibly VintageTestEngine.
Each Engine Discovers Its Tests
JupiterTestEngine.discover() scans for classes with @Test methods and builds a descriptor tree. Vintage does the same for @RunWith classes.
Platform Executes All Discovered Tests
Each engine's execute() is called with its descriptor tree. Engines run tests and fire events (started/passed/failed) back to the platform.
Results Aggregated and Reported
Platform aggregates all events into a unified report. Your IDE shows green/red. Maven prints the summary. Coverage tools record what ran.

JUnit Jupiter in Practice

The annotations, assertions, and patterns you use every day.

🔁 Test Lifecycle Annotations

The Lifecycle Order — Visualized

@BeforeAll Runs once before all tests in the class. Must be static (by default). Use for expensive setup: start a DB, load files.
↓
@BeforeEach Runs before every test method. Fresh setup per test. Use for creating objects, resetting state.
↓
@Test The actual test. A new class instance is created for each test method by default.
↓
@AfterEach Runs after every test. Cleanup: close connections, delete temp files.
↓
@AfterAll Runs once after all tests. Must be static (by default). Tear down shared resources.

Why New Instance Per Test?

JUnit creates a fresh instance of your test class before each test method. This is intentional — it prevents one test's side effects from leaking into the next.

The consequence: @BeforeAll and @AfterAll must be static because they run outside the context of any single instance.

Java @TestInstance workaround — when you need non-static @BeforeAll 
@TestInstance(TestInstance.Lifecycle.PER_CLASS)
class UserServiceTest {

    // With PER_CLASS: one instance for all tests
    // @BeforeAll no longer needs to be static!
    @BeforeAll
    void startDatabase() {
        // Called once, but on 'this' instance
        db.start();
    }

    @BeforeEach
    void resetState() {
        // Still runs before each test
        db.clearTables();
    }
}
⚠️
PER_CLASS is a double-edged sword

Sharing instance state across tests makes tests order-dependent and fragile. Use PER_CLASS only when you have a genuine reason (e.g., expensive setup that truly cannot be repeated). Always reset shared state in @BeforeEach.

🏷️ Core Test Annotations

Java All core annotations in one class 
@Tag("unit")                  // Label tests — run subsets with -Dgroups=unit
@DisplayName("Order Domain Tests")
class OrderTest {

    @Test
    @DisplayName("should calculate total correctly")
    void calculateTotal() { /* ... */ }

    @Test
    @Disabled("JIRA-1234: payment gateway down in CI")
    void processPayment() { /* skipped */ }

    @Test
    @Timeout(2)  // Fails if takes longer than 2 seconds
    void pricingCalculationIsFast() { /* ... */ }

    @RepeatedTest(5)
    void shouldBeIdempotent(RepetitionInfo info) {
        // Runs 5 times — test for flakiness or idempotency
    }

    // Nested tests — group related scenarios
    @Nested
    @DisplayName("when placing an order")
    class PlacingOrder {
        @Test void withValidItems() { /* ... */ }
        @Test void withEmptyCart_shouldFail() { /* ... */ }
    }

    @Nested
    @DisplayName("when cancelling an order")
    class CancellingOrder {
        @Test void beforeShipping() { /* ... */ }
        @Test void afterShipping_shouldFail() { /* ... */ }
    }
}
💡
@Nested is underused

@Nested classes have their own @BeforeEach/@AfterEach lifecycle that stacks on top of the outer class. They make test output read like a spec document, and your IDE groups them beautifully. Use them whenever you have multiple scenarios for the same class.

✅ Assertions

The Essential Assertions

Java Assertions cheatsheet 
import static org.junit.jupiter.api.Assertions.*;

// Basic equality — note: EXPECTED first, ACTUAL second
assertEquals(100, order.getTotal());
assertNotEquals(0, order.getTotal());

// Boolean
assertTrue(order.isActive());
assertFalse(order.isCancelled());

// Null checks
assertNull(order.getShippedAt());
assertNotNull(order.getCreatedAt());

// Reference equality (same object) vs value equality
assertSame(cachedUser, service.getUser(id));   // same reference
assertEquals(expectedUser, service.getUser(id)); // equals() match

// Exception: most important pattern — capture and inspect
IllegalArgumentException ex = assertThrows(
    IllegalArgumentException.class,
    () -> orderService.place(null)
);
assertEquals("order cannot be null", ex.getMessage());

// No exception expected
assertDoesNotThrow(() -> orderService.place(validOrder));

// Timeout
assertTimeout(Duration.ofSeconds(2), () -> service.heavyOp());

assertAll — The Hidden Gem

Normal assertions fail fast — the test stops at the first failure. If you have five fields to check, you might have to run the test five times to see all the problems.

assertAll() runs ALL assertions and reports ALL failures at once.

Java assertAll — all failures reported together 
User user = userService.createUser("Alice", "alice@test.com");

// BAD: stops at first failure — you won't see problems 2, 3, 4
assertEquals("Alice",          user.getName());
assertEquals("alice@test.com", user.getEmail());
assertNotNull(user.getId());
assertTrue(user.isActive());

// GOOD: all four checked — report shows all failures
assertAll("user fields",
    () -> assertEquals("Alice",          user.getName()),
    () -> assertEquals("alice@test.com", user.getEmail()),
    () -> assertNotNull(user.getId()),
    () -> assertTrue(user.isActive())
);
🔥
Use assertAll when testing an object's state

Any time you are verifying multiple fields on a single returned object, wrap them in assertAll(). It turns "run test 4 times to find 4 bugs" into "run test once, see all 4 bugs." This is a senior-level habit that interviewers notice.

🔢 Parameterized Tests

Parameterized tests exist for one reason: stop copy-pasting the same test with different input values. If you have three tests that are structurally identical but use different data, collapse them into one parameterized test.

Java @ValueSource — simplest: single values 
@ParameterizedTest
@ValueSource(strings = {"", "   ", "\t"})
@DisplayName("should reject blank usernames")
void shouldRejectBlankUsername(String username) {
    assertThrows(IllegalArgumentException.class,
        () -> userService.createUser(username));
}
Java @CsvSource — multiple parameters per test case 
@ParameterizedTest
@CsvSource({
    "1, ONE",
    "2, TWO",
    "3, THREE"
})
void shouldMapNumberToWord(int number, String expected) {
    assertEquals(expected, converter.toWord(number));
}
Java @MethodSource — most powerful: complex objects 
@ParameterizedTest
@MethodSource("provideOrderScenarios")
void shouldCalculateTax(BigDecimal price, String country, BigDecimal expectedTax) {
    assertEquals(expectedTax, taxService.calculate(price, country));
}

// Must be static — returns a Stream of Arguments
static Stream<Arguments> provideOrderScenarios() {
    return Stream.of(
        Arguments.of(new BigDecimal("100.00"), "US", new BigDecimal("8.50")),
        Arguments.of(new BigDecimal("200.00"), "UK", new BigDecimal("40.00")),
        Arguments.of(new BigDecimal("50.00"),  "DE", new BigDecimal("9.50"))
    );
}
💡
Which source type to use?

@ValueSource = single primitive/string values. @CsvSource = multiple simple values, readable inline. @MethodSource = complex objects, collections, or when setup requires logic. In real projects, @MethodSource is most common for service-layer tests.

🔌 Extensions — @ExtendWith

Extensions let you plug behavior into the test lifecycle without modifying test classes. They are composable — a test class can have multiple extensions, and they stack.

Java Using built-in extensions 
// MockitoExtension: initializes @Mock, @Spy, @Captor, @InjectMocks
@ExtendWith(MockitoExtension.class)
class UserServiceTest { }

// SpringExtension: loads Spring context, enables @Autowired in tests
@ExtendWith(SpringExtension.class)
class IntegrationTest { }

// Multiple extensions: compose them
@ExtendWith({MockitoExtension.class, TimingExtension.class})
class ComposedTest { }

// @TempDir: inject a temporary directory per test
@Test
void shouldWriteToFile(@TempDir Path tmpDir) {
    Path file = tmpDir.resolve("output.txt");
    // JUnit cleans up tmpDir after the test — no manual cleanup
}
Java Writing a simple custom extension 
public class TimingExtension implements BeforeEachCallback, AfterEachCallback {

    private long start;

    @Override
    public void beforeEach(ExtensionContext ctx) {
        start = System.currentTimeMillis();
    }

    @Override
    public void afterEach(ExtensionContext ctx) {
        long duration = System.currentTimeMillis() - start;
        System.out.printf("[%s] ran in %dms%n", ctx.getDisplayName(), duration);
    }
}
03

Mockito

Control your dependencies. Test your logic. No more guessing what happened.

🎭 Why Mockito Exists

Unit tests should test one thing in isolation: the class under test. But classes have dependencies — databases, HTTP clients, email services. If you let those real dependencies run during a unit test, you're not testing your logic anymore. You're testing the whole system.

Mockito creates fake ("mock") objects that stand in for real dependencies. You control exactly what they return, what exceptions they throw, and you can verify exactly how your code interacted with them.

💡

Mental model: A mock is a stunt double. Before filming, you hand the double a script: "When the director says jump, do this." Then you film. Mockito lets you write that script before your class runs.

The Three Things Mockito Lets You Do

1. Create Fakes

Replace real dependencies with fake objects that have no real implementation.

2. Define Behavior

"When findById(5) is called, return this User object." — Stubbing.

3. Verify Interactions

"Assert that save() was called exactly once with this argument."

The Result

Tests that are fast, deterministic, and isolate exactly the behavior you care about.

🆚 Mock vs Stub vs Spy — Clearly

These three terms are constantly confused. Let's fix that permanently.

🔵 Mock
A completely fake object. Not backed by any real implementation.
Default: all methods return null, 0, false, or empty collections.
You define only the behavior you need for the test.
Use for: isolating external dependencies (repos, HTTP clients, queues).
@Mock or Mockito.mock(UserRepo.class)
🟢 Spy
Wraps a real object. Real methods run by default.
You can override specific methods while the rest behave normally.
Use for: when you need mostly real behavior but want to stub one tricky method.
@Spy or Mockito.spy(new UserService())
⚠️ Use doReturn(), not when().thenReturn() with spies.
💡
Stub is not an object type — it's a verb

"Stubbing" means setting up the return value of a mock method. When you write when(mock.find()).thenReturn(user), you are stubbing the mock. Developers often confuse "stub" (the test double pattern) with "mock" (Mockito's object). In Mockito, you only create Mocks and Spies — stubbing is what you do to them.

⚠️
The Spy Gotcha — This Will Bite You
JavaWhy doReturn() is required for spies
UserService spy = Mockito.spy(new UserService());

// ❌ WRONG: when() evaluates spy.heavyOp() FIRST
//    The real method runs during setup → side effects, NPEs
when(spy.heavyOp()).thenReturn("result");

// ✅ CORRECT: doReturn() sets up the stub WITHOUT calling the real method
doReturn("result").when(spy).heavyOp();

Side-by-Side Comparison

Mock Spy
Backed by real object? No Yes
Default method behavior Returns null/0/false Calls real method
Use for Pure isolation Partial mocking
Stubbing style when().thenReturn() doReturn().when()
Annotation @Mock @Spy
Risk Missing stubs return null Real method runs if not stubbed

⚙️ Setting Up Mockito

Approach A — @ExtendWith (Preferred)

Java Complete test class structure — the template you copy every time 
@ExtendWith(MockitoExtension.class)  // ← Step 1: wire Mockito into JUnit 5
class UserServiceTest {

    @Mock
    private UserRepository userRepository;  // ← fully faked

    @Mock
    private EmailService emailService;       // ← fully faked

    @Spy
    private AuditLogger auditLogger = new AuditLogger();  // ← real object, stubbable

    @InjectMocks
    private UserService userService;         // ← class under test; mocks injected

    @Captor
    private ArgumentCaptor<User> userCaptor; // ← capture what was saved

    @Test
    void shouldCreateUser() {
        // Extension initializes all annotations before this runs
    }
}

Approach B — Manual Initialization (Legacy/Fallback)

Java Manual setup — when you can't use JUnit 5 extension 
class UserServiceTest {

    private AutoCloseable closeable;  // must close to release resources

    @Mock UserRepository userRepository;
    @InjectMocks UserService userService;

    @BeforeEach
    void setUp() {
        closeable = MockitoAnnotations.openMocks(this);
    }

    @AfterEach
    void tearDown() throws Exception {
        closeable.close();  // ← don't forget this
    }
}
⚠️
@InjectMocks Silent Failure

Mockito tries constructor injection, then setter injection, then field injection. If it can't inject — wrong types, no matching constructor, etc. — it silently leaves the field as is. You get a NullPointerException in the test, and Mockito says nothing. If you see an NPE on the class under test, check your @InjectMocks setup first.

🎬 Stubbing Behavior

Java All stubbing patterns 
// ── Return a value ────────────────────────────────────────────────
when(userRepository.findById(1L)).thenReturn(Optional.of(alice));

// ── Throw an exception ────────────────────────────────────────────
when(userRepository.findById(99L)).thenThrow(new UserNotFoundException("not found"));

// ── Dynamic response based on input ───────────────────────────────
when(userRepository.findById(anyLong())).thenAnswer(invocation -> {
    long id = invocation.getArgument(0);
    return id > 0 ? Optional.of(new User(id)) : Optional.empty();
});

// ── Call the real implementation (useful on Spy) ───────────────────
when(spy.format(anyString())).thenCallRealMethod();

// ── Different values on successive calls ──────────────────────────
when(idGenerator.next())
    .thenReturn(1L)   // 1st call
    .thenReturn(2L)   // 2nd call
    .thenReturn(3L);  // 3rd+ calls all return 3L

// ── Void methods ──────────────────────────────────────────────────
doNothing().when(emailService).sendWelcome(anyString());
doThrow(new MailException("SMTP down")).when(emailService).sendWelcome("bad@test.com");

// ── Chaining: different behavior per call ─────────────────────────
when(clock.now())
    .thenReturn(Instant.parse("2024-01-01T00:00:00Z"))
    .thenThrow(new RuntimeException("clock failed"));

🎯 Argument Matchers

Sometimes you don't care about the exact argument — you just want to stub "any call" to a method. Matchers let you express that.

Java Argument matchers reference 
import static org.mockito.ArgumentMatchers.*;

// ── Generic ───────────────────────────────────────────────────────
when(repo.save(any())).thenReturn(saved);          // any non-null arg
when(repo.findById(any(Long.class))).thenReturn(opt); // any Long

// ── Type-specific ─────────────────────────────────────────────────
when(svc.find(anyString())).thenReturn(result);
when(svc.find(anyInt())).thenReturn(result);
when(svc.find(anyLong())).thenReturn(result);
when(svc.findAll(anyList())).thenReturn(results);

// ── Exact value (use when mixing with other matchers) ─────────────
when(svc.query(anyString(), eq(10))).thenReturn(page);

// ── Null checks ───────────────────────────────────────────────────
when(svc.process(isNull())).thenThrow(new NullPointerException());
when(svc.process(isNotNull())).thenReturn(result);

// ── String matchers ───────────────────────────────────────────────
when(svc.find(startsWith("admin_"))).thenReturn(adminUser);
when(svc.find(contains("@"))).thenReturn(emailUser);
when(svc.find(matches("\\d{10}"))).thenReturn(phoneUser);

// ── Custom predicate ──────────────────────────────────────────────
when(repo.save(argThat(u -> u.getAge() >= 18))).thenReturn(saved);
⚠️
The All-or-Nothing Rule — Breaks Silently

The moment you use ANY argument matcher in a method call, every argument must use a matcher. Mixing matchers with raw values causes a Mockito exception.

✅ Correct

verify(svc.query(anyString(), eq(10)))

❌ Breaks at Runtime

verify(svc.query(anyString(), 10))

🔍 Verification

Java All verification patterns 
import static org.mockito.Mockito.*;

// ── Basic: was this method called? ────────────────────────────────
verify(userRepository).save(any(User.class));

// ── How many times? ───────────────────────────────────────────────
verify(emailService, times(1)).sendWelcome(anyString());
verify(auditLog,    times(3)).record(anyString());
verify(cache,       never()).evict(anyString());
verify(retryService, atLeast(2)).attempt(any());
verify(rateLimiter,  atMost(5)).check(anyString());

// ── only(): this method AND NOTHING ELSE was called on this mock ──
verify(emailService, only()).sendWelcome("alice@test.com");

// ── Nothing was called on these mocks at all ──────────────────────
verifyNoInteractions(emailService, auditLog);

// ── Only the verified calls happened (nothing extra) ──────────────
verify(userRepository).save(any());
verifyNoMoreInteractions(userRepository);  // no other methods called
🎯
Interview Tip: When to Verify vs When to Assert

Use verify() for side effects — things that don't come back as return values. "Did we save the user? Did we send the email? Did we log the event?"
Use assertEquals() for return values. Don't verify something you already asserted on — that's redundant and brittle.

📸 Argument Captor

Sometimes verify(mock.save(any())) is not enough. You need to inspect the actual object that was passed. That's what ArgumentCaptor is for.

Classic scenario: your method constructs an object internally and passes it to a dependency. You can't get that object back from the return value — you need to "catch" it mid-flight.

service.createUser("Alice")
constructs User internally
→
userRepo.save(user)
passes constructed object
→
captor.capture()
intercepts and holds it
→
captor.getValue()
you inspect it here
Java ArgumentCaptor — full example 
@Captor
private ArgumentCaptor<User> userCaptor;

@Test
void shouldSaveUserWithHashedPassword() {
    // given
    when(userRepository.save(any())).thenAnswer(i -> i.getArgument(0));

    // when
    userService.register("alice@test.com", "plaintext123");

    // then — capture what was actually saved
    verify(userRepository).save(userCaptor.capture());
    User savedUser = userCaptor.getValue();

    assertAll(
        () -> assertEquals("alice@test.com", savedUser.getEmail()),
        () -> assertNotEquals("plaintext123", savedUser.getPassword()), // must be hashed
        () -> assertTrue(savedUser.isActive()),
        () -> assertNotNull(savedUser.getCreatedAt())
    );
}

// Multiple calls: getAllValues()
@Test
void shouldSendEmailToAllAdmins() {
    userService.notifyAdmins("System restarted");

    ArgumentCaptor<String> emailCaptor =
        ArgumentCaptor.forClass(String.class);
    verify(emailService, times(3)).send(emailCaptor.capture());

    List<String> sentTo = emailCaptor.getAllValues();
    assertTrue(sentTo.containsAll(List.of("admin1@co", "admin2@co", "admin3@co")));
}

📖 BDD Mockito Style

BDD (Behavior Driven Development) promotes test structure as Given / When / Then. Mockito provides a BDD-flavored API that reads like a specification rather than test code.

Java Same test — standard Mockito vs BDD Mockito 
// ── Standard Mockito ──────────────────────────────────────────────
when(userRepository.findById(1L)).thenReturn(Optional.of(alice));

userService.deactivate(1L);

verify(userRepository, times(1)).save(any());
verify(emailService).sendDeactivation("alice@test.com");


// ── BDD Mockito ───────────────────────────────────────────────────
import static org.mockito.BDDMockito.*;

// given
given(userRepository.findById(1L)).willReturn(Optional.of(alice));

// when
userService.deactivate(1L);

// then
then(userRepository).should(times(1)).save(any());
then(emailService).should().sendDeactivation("alice@test.com");
💡
Which style to use?

Both are functionally identical. BDD style shines when your team structures tests as Given/When/Then — the code structure mirrors the intent. Standard style is more common in the wild. Pick one style per codebase and be consistent.

💥 Common Mistakes & Anti-Patterns

1. Mocking Everything — Testing Nothing

If you mock every collaborator of every class, your tests pass but prove nothing. You're testing that Mockito wires return values together, not that your logic is correct.

⚠️
Rule

Mock at the boundary: databases, HTTP clients, file systems, message queues. Don't mock value objects, DTOs, domain classes, or utility methods. If you're mocking a String or a simple POJO, stop and rethink.

2. Unused Stubs — Mockito Is Telling You Something

Java Strict stubbing will fail your test if a stub is never used 
// MockitoExtension uses STRICT_STUBS by default
// This FAILS the test because the stub was never called:
when(emailService.sendWelcome(anyString())).thenReturn(true);
// ... but sendWelcome is never called in the production code path tested

// ✅ Mockito is warning you: your test setup doesn't match reality.
//    Either the code path changed, or the test is wrong.
//    Listen to this warning — don't suppress it.

3. Verifying Implementation Detail, Not Behavior

✅ Verify the contract

Did we save the user to the repository? Did we send the welcome email? These are observable side effects that matter.

❌ Verify implementation

Did we call formatName()? Did we call buildQuery()? These are internal details — verifying them makes tests brittle.

4. Not Resetting Mocks Between Tests

If you're using @ExtendWith(MockitoExtension.class), Mockito creates fresh mocks for each test. You're safe.

If you're managing mocks manually (static fields, shared instances), call Mockito.reset(mock) in @BeforeEach or recreate them. Shared mock state between tests is a source of mysterious, flaky failures.

5. @InjectMocks on an Interface

Java Classic mistake 
// ❌ @InjectMocks needs a concrete class — not an interface
@InjectMocks
private UserService userService;  // ← must be the IMPLEMENTATION class
                                   //   not the interface IUserService

// ✅ Correct — concrete class, Mockito can instantiate it
@InjectMocks
private UserServiceImpl userService;

🏆 Putting It All Together — Complete Example

This is the template. Copy it, adapt it. Every line has a reason.

Java UserServiceTest — production-quality test class 
package com.example.service;

import static org.junit.jupiter.api.Assertions.*;
import static org.mockito.ArgumentMatchers.*;
import static org.mockito.BDDMockito.*;

@ExtendWith(MockitoExtension.class)  // Wires Mockito into JUnit 5 lifecycle
class UserServiceTest {

    // ── Dependencies (faked) ──────────────────────────────────────
    @Mock
    private UserRepository  userRepository;  // fully mocked — no DB

    @Mock
    private EmailService    emailService;    // fully mocked — no SMTP

    @Mock
    private PasswordEncoder passwordEncoder; // fully mocked

    // ── Class under test ──────────────────────────────────────────
    @InjectMocks
    private UserServiceImpl userService;     // Mockito injects the three mocks above

    // ── Captors ───────────────────────────────────────────────────
    @Captor
    private ArgumentCaptor<User> userCaptor;  // captures what gets saved

    // ═══════════════════════════════════════════════════════════════
    // Happy path: register a valid user
    // ═══════════════════════════════════════════════════════════════
    @Test
    @DisplayName("should register user, hash password, and send welcome email")
    void shouldRegisterUserSuccessfully() {
        // given ─────────────────────────────────────────────────────
        String email    = "alice@example.com";
        String password = "securepass";
        String hash     = "$2a$10$hashed";

        given(userRepository.existsByEmail(email)).willReturn(false);
        given(passwordEncoder.encode(password)).willReturn(hash);
        given(userRepository.save(any(User.class)))
            .willAnswer(inv -> inv.getArgument(0)); // echo back the saved object

        // when ─────────────────────────────────────────────────────
        User result = userService.register(email, password);

        // then — assert return value ───────────────────────────────
        assertAll("registered user",
            () -> assertEquals(email, result.getEmail()),
            () -> assertNotEquals(password, result.getPassword()), // must be hashed
            () -> assertTrue(result.isActive()),
            () -> assertNotNull(result.getCreatedAt())
        );

        // then — verify what was saved matches expectations ──────────
        then(userRepository).should().save(userCaptor.capture());
        User savedUser = userCaptor.getValue();
        assertEquals(hash, savedUser.getPassword()); // confirm hashed password was persisted

        // then — verify side effects ─────────────────────────────────
        then(emailService).should().sendWelcome(email);
        then(emailService).shouldHaveNoMoreInteractions(); // no extra emails
    }

    // ═══════════════════════════════════════════════════════════════
    // Sad path: duplicate email
    // ═══════════════════════════════════════════════════════════════
    @Test
    @DisplayName("should throw when email already exists")
    void shouldThrowWhenEmailAlreadyRegistered() {
        // given
        given(userRepository.existsByEmail("alice@example.com")).willReturn(true);

        // when + then
        DuplicateEmailException ex = assertThrows(
            DuplicateEmailException.class,
            () -> userService.register("alice@example.com", "pass")
        );
        assertEquals("alice@example.com already registered", ex.getMessage());

        // no save, no email
        then(userRepository).should(never()).save(any());
        then(emailService).shouldHaveNoInteractions();
    }

    // ═══════════════════════════════════════════════════════════════
    // Parameterized: invalid inputs
    // ═══════════════════════════════════════════════════════════════
    @ParameterizedTest
    @ValueSource(strings = {"", "notanemail", "@nodomain"})
    @DisplayName("should reject invalid email formats")
    void shouldRejectInvalidEmail(String badEmail) {
        assertThrows(IllegalArgumentException.class,
            () -> userService.register(badEmail, "password"));
        verifyNoInteractions(userRepository, emailService);
    }
}

📋 Quick Reference Tables

Mockito Annotations

Annotation What it creates When to use
@Mock Fully fake object, no real implementation Isolate external dependencies (repos, clients)
@Spy Real object with stubbing capability Mostly real behavior, override a few methods
@InjectMocks The class under test, with mocks injected Always for the class you're testing
@Captor ArgumentCaptor<T> Inspect arguments passed to a mock method
@ExtendWith Wires an extension into JUnit 5 lifecycle Always — at the top of every Mockito test class

Stubbing Methods

Syntax Use case Works on Spy?
when().thenReturn() Return a value from a mock method Caution
when().thenThrow() Throw an exception from a mock method Caution
when().thenAnswer() Dynamic response computed from arguments Safe
when().thenCallRealMethod() Delegate to actual implementation Safe
doReturn().when() Return a value — use this on Spy Preferred for Spy
doNothing().when() Stub a void method to do nothing Safe
doThrow().when() Throw from a void method Safe

Verification Methods

Method What it checks
verify(mock).method() Called exactly once
verify(mock, times(n)).method() Called exactly n times
verify(mock, never()).method() Never called
verify(mock, atLeast(n)).method() Called ≥ n times
verify(mock, atMost(n)).method() Called ≤ n times
verify(mock, only()).method() Only this method, nothing else on mock
verifyNoInteractions(mock) No methods called at all
verifyNoMoreInteractions(mock) Nothing beyond what was already verified

JUnit 5 Architecture at a Glance

Module Artifact Purpose
Platform junit-platform-launcher Universal test execution foundation; discovers engines
Jupiter junit-jupiter-api API you write tests with: @Test, assertions, extensions
Jupiter junit-jupiter-engine Engine that understands and executes Jupiter tests on Platform
Jupiter junit-jupiter-params @ParameterizedTest and all value source annotations
Vintage junit-vintage-engine Runs JUnit 3/4 tests on the Platform (migration only)

🎯 The 5 Things To Remember In Any Interview

  1. JUnit 5 = Platform + Jupiter + Vintage. Platform is the launcher; Jupiter is the API you use.
  2. Mock = fake object. Spy = real object with stubbing. They serve different purposes.
  3. Use doReturn().when() with Spies, not when().thenReturn() — real method runs during setup otherwise.
  4. assertAll() for object-state tests — reports all failures at once, not just the first.
  5. Verify side effects. Assert return values. Don't verify things you already asserted.
