BFS: Level by Level,
Layer by Layer

Breadth-First Search is the algorithm of fairness — every node at distance d is visited before any node at distance d+1. That single guarantee is why it finds shortest paths in unweighted graphs.

Foundation — The Mental Model & Core Template

Why BFS works, how it's structured, and the one rule that prevents every bug.

🌊 The Mental Model — Ripples in a Pond

Drop a stone in still water. The first ripple covers the closest water. The second ripple covers the next ring. Each ring is exactly one step further than the previous — no ring can jump over another.

BFS works identically. Start at a source node. Visit all nodes 1 hop away. Then all nodes 2 hops away. Then 3. You cannot reach a node 3 hops away without having already processed everything at hops 1 and 2.

This is the proof of shortest path: the first time you reach a node, it's guaranteed to be via the fewest possible steps — because you explored all shorter paths first.

💡
Why it only works on unweighted graphs: once you add edge weights, a "longer" path (more hops) can actually be cheaper. That's when you need Dijkstra. BFS treats every edge weight as exactly 1.

The Queue is the Core Mechanism

A queue enforces FIFO order — the first node you enqueue is the first you dequeue. This is exactly why BFS processes nodes level by level: all level-k nodes were enqueued before any level-k+1 node, so they're dequeued first.

DFS uses a stack (or the call stack with recursion). A stack reverses order — go as deep as possible before backtracking. Swap the queue for a stack in BFS code and you have DFS. Same graph, completely different traversal.

⚙️ The Core BFS Template — Fully Annotated

Java Universal BFS Template (Graph / Grid / Tree) 
import java.util.*;

public int bfs(Node start, Node target) {
    // ── Step 1: Guard — handle null / empty input ──────────────
    if (start == null) return -1;

    // ── Step 2: Queue for frontier, Set for visited ─────────────
    Queue<Node> queue   = new LinkedList<>();
    Set<Node>   visited = new HashSet<>();

    // ── Step 3: Seed the queue — mark visited BEFORE enqueue ────
    queue.offer(start);
    visited.add(start);   // ← CRITICAL: mark here, not after dequeue

    int level = 0;         // tracks current distance from source

    // ── Step 4: Main loop — process until queue empty ────────────
    while (!queue.isEmpty()) {

        // ── Step 5: Process entire current level ──────────────────
        int size = queue.size();  // snapshot size BEFORE expanding

        for (int i = 0; i < size; i++) {
            Node curr = queue.poll();

            // ── Step 6: Check if we reached the goal ──────────────
            if (curr == target) return level;

            // ── Step 7: Expand neighbors ──────────────────────────
            for (Node neighbor : curr.neighbors) {
                if (!visited.contains(neighbor)) {
                    visited.add(neighbor);      // mark BEFORE enqueue
                    queue.offer(neighbor);
                }
            }
        }

        level++;  // finished one level, advance distance counter
    }

    return -1;  // target not reachable
}

📌 The 5 BFS Ingredients

  • Queue — the frontier
  • Visited set — prevent cycles
  • Source seeding — add start before loop
  • Level counter — snapshot size each round
  • Goal check — return as soon as found

⚡ Queue Sizing Trick

Capture int size = queue.size() before the inner loop. This freezes the count of nodes at the current level. Nodes added during expansion belong to the next level and must not be included in this round's count.

Four BFS Application Categories

Tree traversal → Grid pathfinding → Graph shortest path → Multi-source flood-fill.

🌳 Category A — Tree Level-Order Traversal

Tree BFS is the simplest form: no visited set needed because trees have no cycles. Parent → child edges only flow downward — you'll never accidentally revisit a node.

The canonical output: a List<List<Integer>> where each inner list is one level of the tree. Mastering this template unlocks a dozen tree-based BFS problems.

Java Binary Tree Level-Order — LC #102 
public List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    if (root == null) return result;

    Queue<TreeNode> queue = new LinkedList<>();
    queue.offer(root);

    while (!queue.isEmpty()) {
        int size = queue.size();
        List<Integer> level = new ArrayList<>();

        for (int i = 0; i < size; i++) {
            TreeNode curr = queue.poll();
            level.add(curr.val);

            if (curr.left  != null) queue.offer(curr.left);
            if (curr.right != null) queue.offer(curr.right);
        }
        result.add(level);
    }
    return result;
}
✅
Tree BFS — no visited set. Trees are directed acyclic graphs. Every child has exactly one parent, so backward traversal is impossible. Adding a visited set is harmless but wasteful. Drop it.

Recognition Signals for Tree Level-Order

  • Problem mentions "level order", "level by level", or "row by row"
  • Asks for "right side view" (last node per level)
  • Asks for "average of levels", "sum per level"
  • Asks for "minimum depth" (BFS finds it faster than DFS)
  • Asks for "zigzag traversal" (alternate direction per level)

Zigzag Variant — One Extra Boolean

Java LC #103 — Binary Tree Zigzag Level Order 
boolean leftToRight = true;

while (!queue.isEmpty()) {
    int size = queue.size();
    LinkedList<Integer> level = new LinkedList<>();

    for (int i = 0; i < size; i++) {
        TreeNode curr = queue.poll();
        if (leftToRight)
            level.addLast(curr.val);   // normal order
        else
            level.addFirst(curr.val);  // reverse: insert at front

        if (curr.left  != null) queue.offer(curr.left);
        if (curr.right != null) queue.offer(curr.right);
    }
    result.add(level);
    leftToRight = !leftToRight;
}

🗺️ Category B — Grid / Matrix BFS

A grid is just a disguised graph. Each cell (r, c) is a node. Edges connect adjacent cells (up/down/left/right, sometimes diagonals). Walls or invalid cells simply mean "no edge exists."

Your visited set is usually the grid itself. Mark a cell as visited by mutating it (grid[r][c] = '#') or by keeping a separate boolean[][] — both work. Mutation is faster; a boolean array is safer when you need the original grid later.

Java Grid BFS — Shortest Path (0 = open, 1 = wall) 
public int shortestPath(int[][] grid, int[] start, int[] end) {
    int rows = grid.length, cols = grid[0].length;
    int[][] dirs = {{-1,0},{1,0},{0,-1},{0,1}};

    Queue<int[]> queue = new LinkedList<>();
    boolean[][] visited = new boolean[rows][cols];

    queue.offer(start);
    visited[start[0]][start[1]] = true; // mark before enqueue!
    int steps = 0;

    while (!queue.isEmpty()) {
        int size = queue.size();

        for (int i = 0; i < size; i++) {
            int[] curr = queue.poll();
            if (curr[0] == end[0] && curr[1] == end[1]) return steps;

            for (int[] d : dirs) {
                int nr = curr[0] + d[0];
                int nc = curr[1] + d[1];
                // bounds + wall + visited check bundled together
                if (nr < 0 || nr >= rows || nc < 0 || nc >= cols) continue;
                if (grid[nr][nc] == 1 || visited[nr][nc]) continue;

                visited[nr][nc] = true; // mark before enqueue
                queue.offer(new int[]{nr, nc});
            }
        }
        steps++;
    }
    return -1;
}
⚠️
Bounds check always comes first. Check nr and nc are inside the grid before accessing grid[nr][nc]. Accessing out-of-bounds throws ArrayIndexOutOfBoundsException in Java — no automatic null, it crashes. Bounds check → wall check → visited check. In that order.

🕸️ Category C — Graph BFS (Unweighted Shortest Path)

Graph BFS is the general case. The graph is usually given as an adjacency list (Map<Node, List<Node>> or int[][] edges). Nodes can be integers, strings, or custom objects.

When the "graph" is implicit (e.g., word ladder — the graph is all words 1-letter apart), you build edges on-the-fly during BFS instead of pre-computing them.

Java Word Ladder — LC #127 (Implicit Graph BFS) 
public int ladderLength(String beginWord, String endWord, List<String> wordList) {
    Set<String> wordSet = new HashSet<>(wordList);
    if (!wordSet.contains(endWord)) return 0;

    Queue<String> queue = new LinkedList<>();
    Set<String>   visited = new HashSet<>();
    queue.offer(beginWord);
    visited.add(beginWord);
    int level = 1;

    while (!queue.isEmpty()) {
        int size = queue.size();
        for (int i = 0; i < size; i++) {
            String curr = queue.poll();
            if (curr.equals(endWord)) return level;

            // Build edges on-the-fly: try every letter at every position
            char[] chars = curr.toCharArray();
            for (int j = 0; j < chars.length; j++) {
                char original = chars[j];
                for (char c = 'a'; c <= 'z'; c++) {
                    chars[j] = c;
                    String next = new String(chars);
                    if (wordSet.contains(next) && !visited.contains(next)) {
                        visited.add(next);
                        queue.offer(next);
                    }
                }
                chars[j] = original; // restore for next position
            }
        }
        level++;
    }
    return 0;
}
💡

Implicit graph = generate neighbors dynamically. Don't build the full adjacency list upfront. Generate only the neighbors of the current node during BFS. This is O(N × L × 26) instead of O(N²) preprocessing.

🔥 Category D — Multi-Source BFS

Normal BFS starts from one source. Multi-source BFS starts from multiple sources simultaneously. Think of it as creating a "virtual super-source" connected to all real sources with zero-cost edges.

This handles problems like: "every rotten orange simultaneously infects neighbors" or "distance from any gate in a room." The trick is just seeding all sources into the queue before the loop begins.

Java Multi-Source BFS Template 
public void multiSourceBFS(int[][] grid) {
    Queue<int[]> queue = new LinkedList<>();

    // ── PHASE 1: Find all sources, add them ALL to the queue ─────
    for (int r = 0; r < grid.length; r++) {
        for (int c = 0; c < grid[0].length; c++) {
            if (isSource(grid[r][c])) {
                queue.offer(new int[]{r, c});
                // mark visited here if needed
            }
        }
    }

    // ── PHASE 2: Standard BFS from here — all sources spread together ─
    int steps = 0;
    while (!queue.isEmpty()) {
        int size = queue.size();
        for (int i = 0; i < size; i++) {
            int[] curr = queue.poll();
            // expand neighbors — same as single-source
        }
        steps++;
    }
}
🎯
Interview Tip: When you see "minimum time/distance for ALL sources to spread" — it's always multi-source BFS. The interviewers love this because candidates who haven't seen it will try nested BFS (O(n² × m²) instead of O(nm)). Seeding all sources upfront is the elegant O(nm) solution.

The Critical BFS Bug & Decision Framework

Mark visited before enqueue — and how to choose BFS over DFS instantly.

🐛 The Visited Set Bug — Mark Before Enqueue, Not After Dequeue

This is the #1 most common BFS bug in interviews. It doesn't always cause a crash — sometimes it just makes your code painfully slow, which is worse because it's harder to catch.

❌ Wrong — Mark After Dequeue

JavaBuggy Version
while (!queue.isEmpty()) {
    Node curr = queue.poll();
    visited.add(curr);  // ← TOO LATE: already added multiple times!

    for (Node neighbor : curr.neighbors) {
        if (!visited.contains(neighbor)) {
            queue.offer(neighbor);  // neighbor can be enqueued MULTIPLE times
        }
    }
}

What goes wrong: If nodes A and B both have neighbor C, and A is processed first: C is enqueued. B is processed — C is still not in visited (not dequeued yet), so C is enqueued again. Now C is in the queue twice. This duplicates work and can cause O(n²) behavior or infinite loops in cyclic graphs.

✅ Correct — Mark Before Enqueue

JavaCorrect Version
while (!queue.isEmpty()) {
    Node curr = queue.poll();

    for (Node neighbor : curr.neighbors) {
        if (!visited.contains(neighbor)) {
            visited.add(neighbor);   // ← mark immediately, before adding to queue
            queue.offer(neighbor);   // guaranteed to appear exactly once
        }
    }
}

Why it works: The moment we decide to enqueue a neighbor, we mark it visited. Any other path trying to enqueue the same node will see it's already visited and skip it. Every node enters the queue exactly once.

🚫
The mental rule: "Visit when you decide to go, not when you arrive." Marking visited = "I've committed to processing this node." The queue is just a waiting room — you're committed the moment you walk in, not when you're called.

⚖️ BFS vs DFS — The Decision Table

Both algorithms visit every node. The difference is order. Choose based on what the problem asks for — never just default to one.

Scenario / Question Type Use BFS Use DFS Why
Shortest path (unweighted) Yes No BFS guarantees minimum hops
Minimum depth of tree Yes Possible BFS stops at first leaf — DFS explores full tree
Maximum depth of tree Possible Yes DFS naturally tracks path length to leaves
All paths from source to target No Yes DFS backtracks naturally; BFS can't reconstruct all paths easily
Detect cycle in graph Possible Yes DFS coloring (white/grey/black) is cleaner
Connected components count Either Either Both work equally; DFS slightly simpler with recursion
Topological sort Kahn's Algo Yes DFS with post-order is the classic approach
Level-by-level processing Yes No BFS naturally produces level groups
Memory concern (wide graph) Bad Better BFS queue can hold an entire level (can be huge); DFS uses O(depth) stack
Memory concern (deep graph) Better Bad DFS stack can hit recursion limit on long paths
🎯

One-line rule: If the problem contains the word "shortest," "minimum steps," or "fewest," reach for BFS first. If it says "all paths," "every combination," "can we reach," reach for DFS first.

Deep Walkthrough — LC #994 Rotting Oranges

Multi-source BFS applied step by step — think aloud, write clean.

🔍 Walkthrough: LC #994 — Rotting Oranges 🟡

Problem Statement

You have a grid where 0 = empty, 1 = fresh orange, 2 = rotten orange. Every minute, a rotten orange infects all 4-directionally adjacent fresh oranges. Return the minimum minutes until no fresh oranges remain, or -1 if impossible.

1
Recognize the Pattern
Multiple rotten oranges spread simultaneously. This is multi-source BFS. All rotten oranges are level-0 sources. Fresh oranges at distance k from the nearest rotten orange get infected at minute k.
2
Scan the Grid — Find Sources & Count Fresh
First pass: add all 2s to the queue. Count all 1s in freshCount. If freshCount == 0, return 0 immediately.
3
BFS — Spread Rot, Decrement freshCount
Standard BFS loop. For each rotten orange dequeued, infect unvisited fresh neighbors. Decrement freshCount for each infected orange. Advance minutes after each full level.
4
Check Remaining Fresh — Return Answer
After BFS ends, if freshCount > 0, return -1 (isolated fresh oranges exist). Otherwise return minutes.

Java LC #994 — Rotting Oranges (Complete Solution) 
public int orangesRotting(int[][] grid) {
    int rows = grid.length, cols = grid[0].length;
    Queue<int[]> queue = new LinkedList<>();
    int freshCount = 0;

    // ── Phase 1: Scan grid ──────────────────────────────────────
    for (int r = 0; r < rows; r++) {
        for (int c = 0; c < cols; c++) {
            if      (grid[r][c] == 2) queue.offer(new int[]{r, c}); // source
            else if (grid[r][c] == 1) freshCount++;               // count fresh
        }
    }

    // ── Edge case: no fresh oranges ─────────────────────────────
    if (freshCount == 0) return 0;

    int[][] dirs = {{-1,0},{1,0},{0,-1},{0,1}};
    int minutes = 0;

    // ── Phase 2: Multi-source BFS ───────────────────────────────
    while (!queue.isEmpty() && freshCount > 0) {
        int size = queue.size();

        for (int i = 0; i < size; i++) {
            int[] curr = queue.poll();

            for (int[] d : dirs) {
                int nr = curr[0] + d[0];
                int nc = curr[1] + d[1];

                if (nr < 0 || nr >= rows || nc < 0 || nc >= cols) continue;
                if (grid[nr][nc] != 1) continue; // only infect fresh oranges

                grid[nr][nc] = 2;   // mutate: marks visited AND converts to rotten
                freshCount--;
                queue.offer(new int[]{nr, nc});
            }
        }
        minutes++;
    }

    // ── Phase 3: Verdict ────────────────────────────────────────
    return freshCount == 0 ? minutes : -1;
}

⏱ Complexity

  • Time: O(rows × cols) — each cell processed at most once
  • Space: O(rows × cols) — queue at worst holds all rotten cells

🧠 Key Insight

We use grid[nr][nc] = 2 as both the visited marker and the state transition. No separate visited array needed. Mutation = "this cell is claimed."

⚠️
Common trap: returning minutes after the loop without checking freshCount == 0. If isolated fresh oranges exist (surrounded by walls or empty cells), BFS will terminate but they'll never rot. Always check freshCount before returning.

🧠 The BFS Cheat Sheet — One Page Summary

5 Recognition Signals

  • "Minimum steps / shortest path" in unweighted graph
  • "Level by level" or "layer by layer"
  • "Spread simultaneously" from multiple points
  • "Minimum depth" of a tree
  • "Distance from nearest ___"

4 Common Traps

  • 🐛 Marking visited after dequeue instead of before enqueue
  • 🐛 Forgetting to snapshot queue.size() before inner loop
  • 🐛 Bounds check after grid access (crashes Java)
  • 🐛 Returning answer without checking residual fresh/unvisited nodes
🏆
The senior engineer's checklist before every BFS problem: (1) Single or multi-source? (2) Tree (no visited set) or Graph (need visited)? (3) Need level count? Snapshot size. (4) Grid? Bounds check first, then wall, then visited. (5) Implicit graph? Generate neighbors dynamically.
