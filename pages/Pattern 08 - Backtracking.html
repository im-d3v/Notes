The Foundation

Mental model, universal template, and the critical "undo" step explained.

🧠 The Mental Model — A GPS That Tries Every Road

Imagine you are navigating a city with GPS. At every intersection, you try one road. If it leads to a dead end or breaks your rules, you turn back to the last intersection and try the next road.

That U-turn — that's backtracking.

You're not random. You're systematic. You explore every possible path in a decision tree but prune branches that can never lead to a valid answer. You never miss a solution. You never explore a dead branch twice.

🧠
Mental Model

Backtracking = DFS on a decision tree + pruning bad branches early. Every node is a partial solution. Every leaf is either valid (collect it) or invalid (ignore it).

The Decision Tree for [1, 2, 3] → All Subsets

[ ] ← start (empty state) / \ [1] [ ] ← include 1 OR skip 1 / \ / \ [1,2] [1] [2] [ ] ← branch on 2 / \ / \ / \ / \ [1,2,3][1,2][1,3][1][2,3][2] [3] [ ] ← all 8 valid leaves (2³)
💡
Why it works

The tree has a finite depth (= input size). At each node you make one choice, recurse, then undo that choice so the next branch starts with a clean state. That undo step is the entire magic of backtracking.

🔧 The Universal Backtracking Template

⚠️
This template solves 95% of backtracking problems.

Master this one structure. Everything else is a specialization of it.

Annotated Template — Every Line Explained

void backtrack(List<Integer> current, int start, int[] nums) {
State: current = partial solution built so far. start = where we can pick from next.
// BASE CASE: valid complete state
if (isComplete(current)) {
result.add(new ArrayList<>(current));
return;
}
Collect: When partial solution is "good enough", save a copy (not reference!) and stop this branch.
// EXPLORE: try each available choice
for (int i = start; i < nums.length; i++) {
Choices: Iterate over remaining valid options. start prevents reusing earlier elements.
// PRUNE: skip bad branches early
if (canPrune(nums[i])) continue;
Prune: If this choice can NEVER lead to a valid solution, skip it immediately. This is the performance win.
// CHOOSE: add to current state
current.add(nums[i]);
Make choice: Commit to this element. Modify state.
// RECURSE: go deeper with updated state
backtrack(current, i + 1, nums);
Recurse: Explore the sub-tree from this choice. i+1 moves forward (no duplicates).
// UNCHOOSE: undo the choice ← THE KEY STEP
current.remove(current.size() - 1);
}
}
★ Undo: Remove the element. Restore state to exactly what it was before this iteration. The NEXT iteration starts clean.
❗
Why undo is non-negotiable

Backtracking shares a single current list across all recursive calls. If you don't undo, when you return from one branch and try the next choice, current still contains the previous choice's elements. You'd be building on dirty state. Undo makes the shared state act like independent branches.

The 3 Problem Types

Subsets, Permutations, and Combinations — with separate templates and clear distinctions.

🟢 Type A — Subsets / Power Set

What it is

No ordering, no repetition. Every element either IN or OUT. For n elements, there are exactly 2ⁿ subsets. Order doesn't matter: {1,2} = {2,1}.

You collect at every node of the tree, not just leaves. That's what makes it different from combinations.

JavaType A — Subsets Template
List<List<Integer>> result = new ArrayList<>();

void subsets(int[] nums) {
    backtrack(new ArrayList<>(), 0, nums);
}

void backtrack(List<Integer> current, int start, int[] nums) {
    // ★ Collect at EVERY node (empty subset is valid too)
    result.add(new ArrayList<>(current));

    for (int i = start; i < nums.length; i++) {
        current.add(nums[i]);           // choose
        backtrack(current, i + 1, nums); // explore (i+1: no reuse)
        current.remove(current.size() - 1); // unchoose
    }
}
✅
Key Signals → Type A (Subsets)

"all subsets", "power set", "all possible selections", "include or exclude each element", no size constraint mentioned

🔴 Type B — Permutations

What it is

Ordering matters. Every element used exactly once. For n elements, there are n! permutations. [1,2,3] ≠ [3,1,2].

You can't use a start index here — you need all elements at every depth. Instead, track what's already been used with a boolean array.

JavaType B — Permutations Template
List<List<Integer>> result = new ArrayList<>();

void permutations(int[] nums) {
    backtrack(new ArrayList<>(), new boolean[nums.length], nums);
}

void backtrack(List<Integer> current, boolean[] used, int[] nums) {
    // ★ Collect only at LEAVES (when all elements are placed)
    if (current.size() == nums.length) {
        result.add(new ArrayList<>(current));
        return;
    }

    for (int i = 0; i < nums.length; i++) {
        if (used[i]) continue; // ★ Skip already used — NO start index!

        used[i] = true;                          // choose
        current.add(nums[i]);
        backtrack(current, used, nums);          // explore (always from 0)
        used[i] = false;                         // unchoose
        current.remove(current.size() - 1);
    }
}
✅
Key Signals → Type B (Permutations)

"all arrangements", "all orderings", "permutations", "every possible order", "rearrange", sequence of all n elements

🔵 Type C — Combinations

What it is

Pick exactly k elements from n, no ordering, no repetition. Like subsets but with a fixed size constraint. Collect only at the leaf when current.size() == k.

Uses a start index (like subsets) but adds the size-check base case.

JavaType C — Combinations Template (pick k from n)
List<List<Integer>> result = new ArrayList<>();

void combinations(int n, int k) {
    backtrack(new ArrayList<>(), 1, n, k);
}

void backtrack(List<Integer> current, int start, int n, int k) {
    // ★ Collect ONLY at target depth k
    if (current.size() == k) {
        result.add(new ArrayList<>(current));
        return;
    }

    // ★ Pruning: not enough elements left to fill k slots
    int remaining = k - current.size();
    for (int i = start; i <= n - remaining + 1; i++) {
        current.add(i);
        backtrack(current, i + 1, n, k);
        current.remove(current.size() - 1);
    }
}
✅
Key Signals → Type C (Combinations)

"choose k from n", "combinations of size k", "groups of size k", "pairs", "triplets", explicit size requirement stated

⚔️ The Critical Difference: start vs used[]

🎯
This is the #1 thing interviewers test about backtracking

Understand exactly WHY each type uses a different mechanism to avoid duplicates.

Mechanism Used In Why it works What it prevents
start index Subsets Combos Only pick elements after the last picked. Enforces sorted selection order. Counting {1,2} and {2,1} as different
used[] boolean array Perms Any element can appear at any position; just can't use the same slot twice. Using element[i] twice in one permutation
No mechanism Never — Generates duplicates and invalid states

✅ Subsets / Combos — start

  • Loop: for i = start → n
  • Recurse with i + 1
  • Ensures {1,2} not re-generated as {2,1}
  • No visited tracking needed
  • Elements have a natural "forward only" order

✅ Permutations — used[]

  • Loop: for i = 0 → n (always full range)
  • Skip if used[i] == true
  • Ensures {1,2} and {2,1} both generated
  • Mark on enter, unmark on exit
  • Every position can hold any unused element

✂️ Pruning Techniques — Cut Branches Early

Pruning is what separates an O(2ⁿ) brute force from a fast backtracking solution. Prune aggressively. The earlier you prune, the more sub-trees you skip.

1. Constraint Pruning

If adding current element already exceeds the target (e.g., sum > target), stop. Works when input is sorted — larger elements will only make it worse.

Java
if (remaining < 0) return; // prune!
// Sort input first for this to work

2. Duplicate Skipping

When input has duplicates and you want unique results — sort first, then skip same-value siblings at the same depth level.

Java
if (i > start && nums[i] == nums[i-1])
    continue; // skip duplicate siblings

3. Size Pruning

In combinations, if not enough elements remain to reach size k, prune. Classic optimization for LC #77.

Java
// loop bound is the pruning
for (int i = start;
     i <= n - (k - cur.size()) + 1; i++)

4. Validity Check

For problems like N-Queens or Sudoku — check if the current choice is valid before recursing. Don't recurse into impossible states.

Java
if (!isValid(board, row, col))
    continue; // prune invalid placement
⚠️
Sort Before Pruning

Constraint pruning and duplicate skipping both require the input to be sorted first. Call Arrays.sort(nums) before you invoke backtrack(). Forgetting this is a common bug.

🪤 The Copy Trap — The Most Common Bug

☠️
The #1 Backtracking Bug in Interviews

You add current to result directly. Your result list fills up with identical empty lists at the end. Entire output is wrong.

❌ WRONG — Reference Bug

Java
result.add(current);
// current is a reference!
// When current changes later,
// every stored "snapshot"
// changes too. All sublists
// end up pointing to the
// SAME final state of current.

✅ CORRECT — Snapshot Copy

Java
result.add(new ArrayList<>(current));
// Creates a NEW list with
// the current contents.
// Future changes to current
// do NOT affect this snapshot.
// This is the correct way.
💡

Rule: Whenever you add a mutable object (List, char[], int[]) to your result, always create a copy first. In Java: new ArrayList<>(current). This one line separates correct from broken code.

🔍 Full Walkthroughs — Template in Action

LC #39 — Combination Sum 🟡 Combo

Problem: Given an array of distinct integers candidates and a target, return all unique combinations that sum to target. The same number may be used unlimited times.

Key twist vs standard combo: You can reuse elements → pass i instead of i+1 when recursing.

JavaLC #39 — Combination Sum
class Solution {
    List<List<Integer>> result = new ArrayList<>();

    public List<List<Integer>> combinationSum(int[] candidates, int target) {
        Arrays.sort(candidates); // sort for pruning
        backtrack(new ArrayList<>(), 0, candidates, target);
        return result;
    }

    void backtrack(List<Integer> current, int start,
                   int[] candidates, int remaining) {

        if (remaining == 0) {                          // ★ exact match
            result.add(new ArrayList<>(current));
            return;
        }

        for (int i = start; i < candidates.length; i++) {
            if (candidates[i] > remaining) break;  // ★ prune (sorted!)

            current.add(candidates[i]);
            backtrack(current, i, candidates,         // ★ i not i+1
                       remaining - candidates[i]);     // allows reuse
            current.remove(current.size() - 1);
        }
    }
}
1
Sort input
Enables the break pruning — once a candidate exceeds remaining, all subsequent (larger) ones will too.
2
Pass i, not i+1
This is the reuse mechanism. Passing i means the same element can be picked again at the next level.
3
Collect when remaining == 0
Not at every node (like subsets). Only when sum is exactly met.

LC #46 — Permutations 🟡 Permutation

JavaLC #46 — Permutations
class Solution {
    List<List<Integer>> result = new ArrayList<>();

    public List<List<Integer>> permute(int[] nums) {
        backtrack(new ArrayList<>(), new boolean[nums.length], nums);
        return result;
    }

    void backtrack(List<Integer> current, boolean[] used, int[] nums) {
        if (current.size() == nums.length) {        // ★ leaf = full perm
            result.add(new ArrayList<>(current));
            return;
        }

        for (int i = 0; i < nums.length; i++) {     // ★ always 0..n, no start!
            if (used[i]) continue;                  // ★ skip used elements

            used[i] = true;
            current.add(nums[i]);
            backtrack(current, used, nums);
            used[i] = false;                        // ★ undo used flag too
            current.remove(current.size() - 1);
        }
    }
}

Spot the Difference — 39 vs 46

LC #39: for i = start → n + recurse with i (no used array). LC #46: for i = 0 → n + used[] check (no start). Two problems, two mechanisms, one template shape.

🎯
The n ≤ 20 Signal

Whenever you see a tiny input constraint like n ≤ 15 or n ≤ 20, the problem almost certainly requires exponential time — which means backtracking, bitmask DP, or brute force enumeration. That small n is a gift — it's telling you O(2ⁿ) is acceptable.

⚖️ Decision Moment — Backtracking vs Alternatives

If the problem asks for…UseNot
Count of valid combinationsDP (top-down)Backtracking (too slow)
All valid combinations (enumerate)BacktrackingDP (can't enumerate)
Exists ANY valid solution (boolean)BFS / DPBacktracking (overkill)
ALL valid paths in a gridBacktracking (DFS)BFS (doesn't enumerate paths)
Shortest valid pathBFSBacktracking (explores all)
Constraint satisfaction (Sudoku, N-Queens)BacktrackingDP (no structure to exploit)
🏆
Interview Decision Rule

If the problem says "return all" → backtracking. If it says "return count" or "return true/false" → think DP or BFS first. Backtracking is the only tool that can enumerate all solutions; DP and BFS are better at optimization and reachability questions.
