Prefix Sum

Pre-compute Everything

The art of paying an upfront O(n) cost so every future range query costs O(1). One of the cleanest space-for-time trades in all of competitive programming.

Range Queries O(1) Query HashMap Combo Subarray Sums 2D Matrix
01

The Foundation

Why prefix sums exist, the mental model, and the 1D + 2D templates.

🧠 The Core Idea

💡

Mental Model: Imagine a highway with toll booths. To know the total toll between exit 3 and exit 8, you don't re-drive that stretch. You pre-record the cumulative toll from the start to every exit — then it's just subtraction.

The Problem Prefix Sum Solves

You're given an array and asked many times for the sum between index i and j. Naïve: loop each query → O(n) per query. With prefix sum: preprocess once in O(n), answer every query in O(1).

arr = [ 3, 1, 4, 1, 5, 9, 2, 6 ] index 0 1 2 3 4 5 6 7 prefix[0] = 0 ← sentinel, always zero prefix[1] = 3 prefix[2] = 4 prefix[3] = 8 prefix[4] = 9 prefix[5] = 14 prefix[6] = 23 prefix[7] = 25 prefix[8] = 31 Sum(i=2, j=5) → prefix[6] - prefix[2] = 23 - 4 = 19 ✓ (verify: 4+1+5+9 = 19)

When to Use It

  • Multiple range sum queries on static array
  • Count subarrays with exact sum k
  • Subarray sum divisible by k
  • 2D rectangle sum in a matrix
  • Problems asking "how many subarrays…"

Recognition Signals 🚨

  • "sum of elements between index i and j"
  • "subarray sum equals k"
  • "number of subarrays where sum…"
  • "range query" + immutable array
  • Contiguous subarray + sum condition

📐 1D Prefix Sum Template

Build + Query

Python 1D Prefix Sum — Universal Template 
def build_prefix(arr):
    n = len(arr)
    # Size n+1, prefix[0] = 0 is the sentinel
    prefix = [0] * (n + 1)
    for i in range(n):
        prefix[i + 1] = prefix[i] + arr[i]
    return prefix

def range_sum(prefix, left, right):
    # Sum of arr[left..right] inclusive (0-indexed)
    return prefix[right + 1] - prefix[left]

# Example:
arr    = [3, 1, 4, 1, 5]
prefix = build_prefix(arr)   # [0, 3, 4, 8, 9, 14]
print(range_sum(prefix, 1, 3))  # 1+4+1 = 6
📌
The Formula

sum(left, right) = prefix[right + 1] − prefix[left]
Think of it as: "everything up to right+1" minus "everything before left". The +1 offset is because prefix is 1-indexed over the array.

🔲 2D Prefix Sum Template

Mental Model First

For a matrix, prefix[r][c] = sum of all elements in the rectangle from (0,0) to (r-1, c-1). To query any sub-rectangle, use inclusion-exclusion: add the big rectangle, subtract two overlapping slabs, add back the doubly-subtracted corner.

Inclusion-Exclusion for 2D: prefix[r2+1][c2+1] ← whole rectangle to bottom-right - prefix[r1][c2+1] ← strip above - prefix[r2+1][c1] ← strip to the left + prefix[r1][c1] ← add back doubly-subtracted corner

Build + Query

Python 2D Prefix Sum — LC #304 
class NumMatrix:
    def __init__(self, matrix):
        m, n = len(matrix), len(matrix[0])
        # (m+1) x (n+1) grid of zeros
        self.p = [[0] * (n + 1) for _ in range(m + 1)]
        for r in range(m):
            for c in range(n):
                self.p[r+1][c+1] = (matrix[r][c]
                    + self.p[r][c+1]      # above
                    + self.p[r+1][c]      # left
                    - self.p[r][c])        # remove double-counted corner

    def sumRegion(self, r1, c1, r2, c2):
        return (self.p[r2+1][c2+1]
              - self.p[r1][c2+1]
              - self.p[r2+1][c1]
              + self.p[r1][c1])
⚠️
Same +1 offset applies in 2D

The prefix grid is always (m+1) × (n+1). The extra row and column of zeros act as sentinels on both axes. Without them, every boundary cell needs an if r > 0 and c > 0 guard — messy and error-prone.

02

The Power Combo

Prefix Sum × HashMap — unlocks subarray counting in O(n).

🎯 The Complement Trick

💡

Core Insight: If the running sum at index j is S, and a sum of k ended at j, then somewhere before j there must be a prefix sum equal to S − k. A HashMap lets you look that up in O(1).

Visualising the Complement

arr = [ 1, 2, 3, 3, -3, 4 ], k = 6 Running sums (prefix): idx: 0 1 2 3 4 5 6 prefix: [0, 1, 3, 6, 9, 6, 10] ↑ At idx 5 prefix = 6. complement = 6 - 6 = 0. 0 exists at idx 0 → subarray arr[1..5] sums to 6 ✓ At idx 3 prefix = 6. complement = 6 - 6 = 0. 0 exists at idx 0 → subarray arr[1..3] sums to 6 ✓ Total subarrays summing to 6: 2

The Core Formula

We want: count of indices where prefix[j] − prefix[i] = k
Rearranged: prefix[i] = prefix[j] − k
So at every step: look in the HashMap for current_sum − k.

🗺️ Prefix Sum + HashMap Pattern

Universal Template

Python Count subarrays with sum = k 
from collections import defaultdict

def subarray_sum(nums, k):
    count = 0
    curr_sum = 0
    seen = defaultdict(int)
    seen[0] = 1   # ← critical: prefix sum 0 exists once before we start

    for num in nums:
        curr_sum += num                  # running prefix sum
        complement = curr_sum - k        # what prefix must exist before us

        if complement in seen:
            count += seen[complement]    # every occurrence is a valid subarray

        seen[curr_sum] += 1             # record this prefix sum for future use

    return count

Step-by-Step Logic

  1. Initialise seen = {0: 1}
  2. Walk array, accumulate curr_sum
  3. Ask: "has curr_sum − k appeared before?"
  4. If yes, add its frequency to count
  5. Record curr_sum in map

Why record AFTER querying?

We add curr_sum to the map after checking the complement. This prevents counting a subarray that starts and ends at the same index — we need a strictly previous prefix sum.

🚫
Never swap the query and the record. Subarray length ≥ 1, so you must look at only past prefixes.

⚖️ Running Sum vs Prefix Array

You don't always need to materialise the full prefix array. Choose based on what you need:

Approach Space Use When Example
Prefix Array
prefix[0..n]		 O(n) Multiple random-access range queries, 2D, or need indices later LC #303, LC #304
Running Sum + HashMap O(n) Counting subarrays, one pass, don't need to revisit old values by index LC #560, LC #974
Running Sum only O(1) Single pass, need current total only (not historical lookup) Running total checks
✅
Decision rule:

If you need to look up "did this prefix sum exist before?" → Running sum + HashMap.
If you need "exact sum between specific indices i and j" → Prefix array.

🔍 Full Walkthrough: LC #560 — Subarray Sum Equals K

Problem

Given an integer array nums and integer k, return the total number of subarrays whose sum equals k.

Text Example 
Input:  nums = [1, 1, 1],  k = 2
Output: 2
Explanation: [1,1] appears at indices (0,1) and (1,2)
1
Initialise
count = 0, curr = 0, seen = {0: 1}. We pre-load the zero prefix to handle subarrays starting at index 0.
2
Process nums[0] = 1
curr = 1. Complement = 1 − 2 = −1. Not in map. Record {0:1, 1:1}.
3
Process nums[1] = 1
curr = 2. Complement = 2 − 2 = 0. In map with count 1 → count = 1. Record {0:1, 1:1, 2:1}.
4
Process nums[2] = 1
curr = 3. Complement = 3 − 2 = 1. In map with count 1 → count = 2. Record {0:1, 1:1, 2:1, 3:1}.
5
Return
count = 2 ✅ — both [1,1] subarrays found in one O(n) pass.

Complexity

MetricValueReason
TimeO(n)Single pass, O(1) HashMap ops
SpaceO(n)HashMap can store up to n distinct prefix sums

⚠️ Common Traps

🔴
Trap #1 — Forgetting seen[0] = 1

This is the most common mistake. Without it, any subarray that starts at index 0 and sums to k is never counted. The zero prefix sum represents the "empty prefix" — the state before touching any element.

WHY is seen[0] = 1 necessary? — Proof by example

Text nums = [3, 1, 2], k = 3 
Without seen[0]=1:
  curr=3, complement=0, not in map → miss!  ← arr[0..0] = [3] counts as 0
  curr=4, complement=1, not in map → miss
  curr=6, complement=3, found!     → count=1
  WRONG ANSWER: 1 (missed [3] itself)

With seen[0]=1:
  curr=3, complement=0, found (count=1) → ✅ [3] counted
  curr=4, complement=1, not in map
  curr=6, complement=3, found (count=2) → ✅ [3,1,2] counted... wait
  Let's recheck: [3,1,2] sum=6 ≠ 3. Only [3] and [1,2] sum to 3.
  curr=4, complement=1, not in map.
  curr=6, complement=3, found (count=2) → ✅ [1,2] counted via prefix diff
  CORRECT ANSWER: 2

The sentinel seen[0] = 1 represents the fact that a prefix sum of zero occurs exactly once: before the array starts. Without it, subarrays that start at index 0 have no way to register their complement.

⚠️
Trap #2 — Off-by-one in 1D prefix array indexing

range_sum(left, right) = prefix[right + 1] − prefix[left] — notice the +1 on right. If you use prefix[right] − prefix[left], you get the wrong range. Trace it on paper once to burn it in.

⚠️
Trap #3 — Prefix Sum with negative numbers

Prefix sum works perfectly with negative numbers. But sliding window does NOT (window contraction logic breaks). If an array has negatives and you need subarray sum = k, reach for prefix sum + HashMap, not sliding window.

🎯
Interview Tip — The "modulo" variant

For "subarray sum divisible by k" (LC #974), store curr_sum % k instead of curr_sum. Two indices with the same modulo mean the subarray between them is divisible by k. Initialise seen[0] = 1 still applies.

✅ Use Prefix Sum + HashMap when

  • Array can have negatives
  • Need exact count of subarrays
  • Sum condition is an equality (= k)
  • Cannot shrink a window meaningfully

✅ Use Sliding Window when

  • Array is all non-negative
  • Need longest/shortest subarray
  • Sum condition is an inequality (≤ k, ≥ k)
  • Window can be contracted greedily
