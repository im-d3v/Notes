Collection Hierarchy
The full class/interface tree you must know

📖 Overview

Java Collections Framework (JCF) provides a unified architecture for storing and manipulating groups of objects. It lives in java.util package. Two root interfaces: Collection and Map (Map does NOT extend Collection).

List — ArrayList & LinkedList
Internals, resizing, when to use each

🔷 ArrayList — Internals Deep Dive

ArrayList is backed by a dynamic array (Object[] internally). Default initial capacity = 10. When full, it grows to 1.5x the current size.

ArrayList Internal Structure
Object[] elementData = new Object[10]; ← backing array int size = 0; ← current element count When capacity is full: newCapacity = oldCapacity + (oldCapacity >> 1) ← roughly 1.5x → new array created, all elements copied (Arrays.copyOf) Example growth: 10 → 15 → 22 → 33 → 49 → ... Memory layout (index-based): Index: [0] [1] [2] [3] [4] ... [9] Data: "A" "B" "C" "D" null ... null size = 4, capacity = 10
JavaArrayList — Core Operations
// Creation with initial capacity (avoid frequent resizing)
List<String> list = new ArrayList<>(100);

// Add operations
list.add("Apple");          // O(1) amortized — append at end
list.add(1, "Banana");      // O(n) — shift elements right
list.addAll(Arrays.asList("C","D"));

// Access — O(1)
String item = list.get(0);

// Remove
list.remove(0);           // O(n) — by index, shifts elements left
list.remove("Apple");     // O(n) — by value, linear search first

// Search
int idx = list.indexOf("Banana"); // O(n)
boolean has = list.contains("C"); // O(n)

// Sort
Collections.sort(list);                        // natural order
list.sort(Comparator.reverseOrder());          // reverse
list.sort(Comparator.comparingInt(String::length)); // by length

// Convert to array
String[] arr = list.toArray(new String[0]);

// Sublist — returns VIEW (backed by original list)
List<String> sub = list.subList(1, 3);

// Unmodifiable list (Java 9+)
List<String> immutable = List.of("A", "B", "C");

🔶 LinkedList — Internals Deep Dive

LinkedList is a doubly linked list. Each element is a Node containing data, a pointer to next, and a pointer to previous. Also implements Deque.

LinkedList Internal Structure
Node { Object item; Node next; Node prev; } LinkedList { Node first; Node last; int size; } Memory layout (doubly linked): null ← [prev|"A"|next] ↔ [prev|"B"|next] ↔ [prev|"C"|next] → null ↑ first ↑ last addFirst("X"): null ← [prev|"X"|next] ↔ [prev|"A"|next] ↔ ... ↑ new first (O(1) — just pointer changes)
JavaLinkedList — As List and Deque
LinkedList<String> ll = new LinkedList<>();

// List operations
ll.add("B");
ll.add(0, "A");   // O(1) at head
ll.get(1);        // O(n) — must traverse from head/tail

// Deque operations (use as Stack or Queue)
ll.addFirst("Head");   // O(1)
ll.addLast("Tail");    // O(1)
ll.peekFirst();        // O(1) — look without remove
ll.pollFirst();        // O(1) — remove and return first
ll.pollLast();         // O(1) — remove and return last

// Use as Stack (LIFO)
ll.push("X");    // addFirst
ll.pop();        // removeFirst

// Use as Queue (FIFO)
ll.offer("Y");   // addLast
ll.poll();       // removeFirst

Common Misconception: People think LinkedList insert is O(1) always. For insert at index i: you still need O(n) traversal to reach position i, then O(1) pointer change. Only insert at head/tail is truly O(1).
🎯 Interview Questions — List
Q1How does ArrayList resize? What is the new capacity?
When full, new capacity = oldCapacity + (oldCapacity >> 1) = roughly 1.5x. A new array is created and elements are copied using Arrays.copyOf(). Default initial = 10. You can set initial capacity to avoid frequent resizing.
Q2What is trimToSize() in ArrayList?
Trims the backing array to the current size (removes unused slots). Useful after bulk add+removal to free memory. E.g., after adding 1000 elements and removing 700, capacity is still ~1500 but size is 300. trimToSize() makes capacity = 300.
Q3When would you prefer LinkedList over ArrayList?
When you need a Queue or Deque (use ArrayDeque actually), frequent insert/delete at the beginning, or implementing a stack. In practice, ArrayList is almost always better due to cache locality. Even for frequent inserts, ArrayList's System.arraycopy is very fast.

Section 2
HashMap — Internal Working
Hashing, buckets, load factor, treeification, Java 8 changes

📖 How HashMap Works — Step by Step

HashMap stores key-value pairs in an array of buckets (Node[]). Each bucket is a linked list (or red-black tree in Java 8+). Internally: transient Node<K,V>[] table.

HashMap Structure — Java 8+
put("name", "Alice") — step by step: Step 1: hashCode() on key "name".hashCode() → some int (e.g., 3373752) Step 2: hash() — spread bits (prevents clustering) hash = key.hashCode() ^ (key.hashCode() >>> 16) Step 3: bucket index = hash & (capacity - 1) index = hash & (16-1) = hash & 15 → e.g., 5 Step 4: Store in table[5] If empty → create new Node and store If collision → add to linked list at that bucket If same key → replace value (equals() used) Bucket layout (default capacity = 16): table[0] → null table[1] → Node("city","NY") → null table[5] → Node("name","Alice") → Node("age","30") → null ← collision table[7] → Node("id","001") → null ... Java 8 TREEIFICATION: If bucket chain length ≥ 8 AND total capacity ≥ 64 → linked list converts to Red-Black Tree → O(log n) search If elements reduced below 6 → converts back to linked list

📐 Load Factor & Rehashing

Default initial capacity = 16
Default load factor = 0.75
Threshold = capacity × load factor = 16 × 0.75 = 12

When the number of entries exceeds the threshold, HashMap rehashes: doubles capacity (16 → 32) and redistributes all entries. Rehashing is expensive — O(n).

JavaHashMap — Important Operations
// Creation — specify initial capacity when size is known
Map<String, Integer> map = new HashMap<>(32, 0.75f);

// Basic operations — O(1) average
map.put("one", 1);
map.get("one");                   // 1
map.getOrDefault("x", 0);         // returns 0 if key not found
map.containsKey("one");           // true
map.containsValue(1);             // O(n) — linear scan
map.putIfAbsent("two", 2);        // only puts if key absent

// Java 8 compute methods
map.computeIfAbsent("three", k -> k.length()); // 5
map.merge("one", 10, Integer::sum);   // 1+10=11 if key exists
map.compute("one", (k, v) -> v == null ? 1 : v + 1); // increment

// Iteration — 3 ways
// Way 1: entrySet (most efficient — one lookup)
for (Map.Entry<String, Integer> e : map.entrySet()) {
    System.out.println(e.getKey() + " = " + e.getValue());
}
// Way 2: keySet + get (extra lookup per key)
for (String key : map.keySet()) {
    System.out.println(key + " = " + map.get(key));
}
// Way 3: forEach (Java 8+)
map.forEach((k, v) -> System.out.println(k + "=" + v));

🔑 hashCode() and equals() — The Contract

  • If equals() returns true → hashCode() MUST be same
  • If hashCode() is same → equals() may or may not be true (collision is ok)
  • Same object → same hashCode during single JVM run
  • Override both or neither — breaking the contract causes bugs in HashMap/HashSet
JavaCustom Key in HashMap — Must Override Both
class StudentKey {
    private int id;
    private String name;

    StudentKey(int id, String name) {
        this.id = id; this.name = name;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof StudentKey)) return false;
        StudentKey other = (StudentKey) o;
        return this.id == other.id && Objects.equals(this.name, other.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id, name); // Java 7+ utility
    }
}

// WITHOUT overriding: two "equal" keys would map to different buckets!
Map<StudentKey, String> m = new HashMap<>();
m.put(new StudentKey(1, "Alice"), "CS");
System.out.println(m.get(new StudentKey(1, "Alice")));
// With override: "CS"  |  Without override: null
🚨
Common Bug: Using a mutable object as a HashMap key and then modifying it. The hashCode changes after modification → you can never retrieve the value again! Always use immutable objects (String, Integer) as HashMap keys.
🎯 Interview Questions — HashMap
Q1What happens when two keys have the same hashCode?
It's a collision. Both are stored in the same bucket as a linked list. Java 8+: if ≥8 nodes in one bucket AND capacity ≥64, the linked list converts to a Red-Black Tree for O(log n) performance. On retrieval, equals() is used to find the exact entry.
Q2Can HashMap have null keys? How many?
Yes — exactly one null key. Null keys are always placed in bucket[0] (no hashCode called). Multiple null values are allowed. Hashtable does NOT allow null keys or values.
Q3HashMap vs Hashtable vs ConcurrentHashMap?
HashMap: Not thread-safe, allows null key/value, fast.
Hashtable: Synchronized (whole table), no null, legacy, slow.
ConcurrentHashMap: Thread-safe via segment/bucket locking (Java 8: CAS + synchronized per node), no null key/value, high concurrency. Prefer over Hashtable always.
Q4Why is String a good HashMap key?
String is immutable (hashCode can't change after creation), String caches its hashCode (computed once, stored in hash field — lazy evaluation, subsequent calls return cached value), and String properly overrides both hashCode() and equals().

No duplicates — how each maintains uniqueness

🔷 HashSet — Internals

HashSet is backed by a HashMap internally! The elements are stored as keys in a HashMap, with a constant dummy value (PRESENT = new Object()). So HashSet inherits all HashMap properties.

JavaHashSet Internals Reveal
// HashSet source code (simplified)
public class HashSet<E> {
    private HashMap<E, Object> map;
    private static final Object PRESENT = new Object(); // dummy value

    public boolean add(E e) {
        return map.put(e, PRESENT) == null; // true if new, false if duplicate
    }

    public boolean contains(Object o) {
        return map.containsKey(o);
    }
}

// Usage
Set<String> set = new HashSet<>();
set.add("Apple");
set.add("Banana");
set.add("Apple");  // ignored — duplicate
System.out.println(set.size()); // 2

// IMPORTANT: HashSet uses hashCode() + equals() to determine duplicates
// Must override both in custom objects!

🔶 LinkedHashSet vs TreeSet

JavaSet Types Comparison
// LinkedHashSet — insertion order maintained (backed by LinkedHashMap)
Set<String> linked = new LinkedHashSet<>();

linked.add("Apple");
linked.add("Cherry");
System.out.println(linked); // [Banana, Apple, Cherry] — insertion order

// TreeSet — natural sorted order (backed by TreeMap = Red-Black Tree)
Set<String> tree = new TreeSet<>();

tree.add("Apple");
tree.add("Cherry");
System.out.println(tree); // [Apple, Banana, Cherry] — alphabetical

// TreeSet NavigableSet operations
TreeSet<Integer> nums = new TreeSet<>(Set.of(1,3,5,7,9));
nums.floor(6);     // 5 — greatest element ≤ 6
nums.ceiling(6);   // 7 — smallest element ≥ 6

nums.tailSet(5);   // [5, 7, 9] — elements ≥ 5
nums.subSet(3,7);  // [3, 5] — 3 inclusive to 7 exclusive
nums.first();      // 1
nums.last();       // 9

Set<String> byLen = new TreeSet<>(Comparator.comparingInt(String::length));
byLen.add("Banana"); byLen.add("Fig"); byLen.add("Apple");
System.out.println(byLen); // [Fig, Apple, Banana] — sorted by length

🔗 LinkedHashMap — Access Order Mode (LRU Cache)

JavaLRU Cache using LinkedHashMap
// accessOrder = true → iterates in order of last access (LRU)
// accessOrder = false → iterates in insertion order (default)

class LRUCache<K, V> extends LinkedHashMap<K, V> {

    private final int capacity;

    LRUCache(int capacity) {
        super(capacity, 0.75f, true); // true = access order!
        this.capacity = capacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        // Remove oldest entry when size exceeds capacity
        return size() > capacity;
    }
}

// Usage
LRUCache<Integer, String> cache = new LRUCache<>(3);
cache.put(1, "A");
cache.put(2, "B");
cache.put(3, "C");
cache.get(1);      // access key 1 → moves to end
cache.put(4, "D"); // capacity exceeded → evicts key 2 (LRU)
System.out.println(cache.keySet()); // [3, 1, 4]

JavaTreeMap NavigableMap Operations
TreeMap<Integer, String> tree = new TreeMap<>();

🔷 Queue Methods — offer vs add, poll vs remove

Operation Throws exception Returns special value
Insert add(e) — throws if full offer(e) — returns false if full
Remove head remove() — throws if empty poll() — returns null if empty
Examine head element() — throws if empty peek() — returns null if empty
✅
Best practice: Use offer(), poll(), peek() — they don't throw exceptions, which is safer for most use cases.

🏔️ PriorityQueue — Min & Max Heap

JavaPriorityQueue Min and Max Heap
// Min Heap (default — natural order)
PriorityQueue<Integer> minHeap = new PriorityQueue<>();
minHeap.offer(5); minHeap.offer(1); minHeap.offer(3);
System.out.println(minHeap.poll()); // 1 — smallest always first

// Max Heap (reverse order)
PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Comparator.reverseOrder());
maxHeap.offer(5); maxHeap.offer(1); maxHeap.offer(3);
System.out.println(maxHeap.poll()); // 5 — largest always first

// PriorityQueue with custom object
PriorityQueue<String> byLength = new PriorityQueue<>(
    Comparator.comparingInt(String::length)
);
byLength.offer("Banana"); byLength.offer("Fig"); byLength.offer("Apple");
System.out.println(byLength.poll()); // "Fig" — shortest first

// NOTE: PriorityQueue does NOT guarantee full sorted iteration!
// Only peek()/poll() return the minimum element.

🔃 ArrayDeque — Preferred over Stack & LinkedList

JavaArrayDeque as Stack and Queue
Deque<String> deque = new ArrayDeque<>();

// As Stack (LIFO)
deque.push("First");   // addFirst
deque.push("Second");
deque.push("Third");
System.out.println(deque.pop()); // Third (LIFO)

// As Queue (FIFO)
Deque<Integer> queue = new ArrayDeque<>();
queue.offer(1); queue.offer(2); queue.offer(3);
System.out.println(queue.poll()); // 1 (FIFO)

// Peek without removing
queue.peekFirst();  // front element
queue.peekLast();   // rear element

💻 Comparable — Natural Ordering

JavaComparable — implement in class itself
class Employee implements Comparable<Employee> {
    private int id;
    private String name;
    private double salary;

    // Constructor, getters...
    Employee(int id, String name, double salary) {
        this.id = id; this.name = name; this.salary = salary;
    }

    @Override
    public int compareTo(Employee other) {
        // Natural order = by id
        // Return: negative if this < other, 0 if equal, positive if this > other
        return Integer.compare(this.id, other.id);
        // DON'T use: return this.id - other.id  ← can overflow!
    }

    @Override
    public String toString() { return name + "(" + id + ")"; }
}

List<Employee> emps = new ArrayList<>();
emps.add(new Employee(3, "Charlie", 70000));
emps.add(new Employee(1, "Alice", 90000));
emps.add(new Employee(2, "Bob", 80000));

Collections.sort(emps);
System.out.println(emps); // [Alice(1), Bob(2), Charlie(3)] — by id

💻 Comparator — Custom & Multiple Ordering

JavaComparator — Lambda + Chaining
List<Employee> emps = /* ... same list ... */;

// Way 1: Anonymous class
Comparator<Employee> bySalary = new Comparator<Employee>() {
    public int compare(Employee a, Employee b) {
        return Double.compare(a.getSalary(), b.getSalary());
    }
};

// Way 2: Lambda (preferred)
Comparator<Employee> bySalaryLambda =
    (a, b) -> Double.compare(a.getSalary(), b.getSalary());

// Way 3: Method reference (cleanest)
Comparator<Employee> bySalaryRef =
    Comparator.comparingDouble(Employee::getSalary);

// Sort by salary descending
emps.sort(Comparator.comparingDouble(Employee::getSalary).reversed());

// CHAINING — sort by salary desc, then by name asc
emps.sort(
    Comparator.comparingDouble(Employee::getSalary)
              .reversed()
              .thenComparing(Employee::getName)
);

// Null-safe comparator
Comparator<Employee> nullSafe =
    Comparator.nullsFirst(Comparator.comparing(Employee::getName));

// Sort string array by length, then alphabetically
String[] words = {"banana", "fig", "apple", "kiwi"};
Arrays.sort(words,
    Comparator.comparingInt(String::length)
              .thenComparing(Comparator.naturalOrder())
);
// Output: [fig, kiwi, apple, banana]
🎯 Interview Questions — Comparable vs Comparator
Q1What does compareTo() return and what do the values mean?
Negative: this < other. Zero: equal. Positive: this > other. Tip: Never use subtraction (this.id - other.id) — it can overflow for large numbers. Use Integer.compare(a, b) instead.
Q2Can you add an object without Comparable to a TreeSet?
No, adding to TreeSet requires comparison. If the object doesn't implement Comparable AND no Comparator is passed to TreeSet constructor, it throws ClassCastException at runtime on the second add() call.
Q3Which sorting algorithm does Collections.sort() use?
Java uses TimSort (hybrid of merge sort + insertion sort). Time complexity: O(n log n) worst/average, O(n) best case. It is stable (maintains relative order of equal elements). Arrays.sort for primitives uses Dual-Pivot Quicksort.

ConcurrentModificationException
Why it happens, how to fix it — 4 ways

❓ What is ConcurrentModificationException?

Thrown when a collection is structurally modified (add/remove) while being iterated using a fail-fast iterator. The iterator has a modCount field — if it changes during iteration, CME is thrown immediately.

🔥 Reproducing the Problem

JavaCME — The Problem
List<String> list = new ArrayList<>(Arrays.asList("A","B","C","D"));

// ❌ WAY 1 — WRONG: using for-each while removing
for (String s : list) {
    if (s.equals("B")) {
        list.remove(s); // ← ConcurrentModificationException!
    }
}

🎯 Interview Questions — CME
Q1What is fail-fast vs fail-safe iterator?
Fail-fast: Throws CME immediately if collection is structurally modified during iteration (ArrayList, HashMap iterators). Uses modCount check.
Fail-safe: Works on a copy of collection — no CME (CopyOnWriteArrayList, ConcurrentHashMap). But might not reflect latest changes during iteration.
Q2Does CME occur in multi-threaded or single-threaded code?
Both! CME is most commonly seen in single-threaded code when you modify a list inside a for-each loop. It's not just a concurrency issue — it's triggered whenever modCount doesn't match expectedModCount, which can happen in the same thread.

🏗️ Custom LinkedHashMap — Bounded + Extra Features

JavaCustom Bounded LinkedHashMap with stats
import java.util.*;

public class BoundedLinkedHashMap<K, V> extends LinkedHashMap<K, V> {

    private final int maxSize;
    private int evictionCount = 0;
    private int hitCount = 0;
    private int missCount = 0;

    public BoundedLinkedHashMap(int maxSize, boolean accessOrder) {
        super(maxSize, 0.75f, accessOrder);
        this.maxSize = maxSize;
    }

    // Called after every put/putAll — remove eldest if over capacity
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        boolean shouldRemove = size() > maxSize;
        if (shouldRemove) {
            evictionCount++;
            System.out.println("Evicting: " + eldest.getKey());
        }
        return shouldRemove;
    }

    // Override get to track hits/misses
    @Override
    public V get(Object key) {
        V val = super.get(key);
        if (val != null) hitCount++;
        else missCount++;
        return val;
    }

    // Extra methods
    public int    getEvictionCount() { return evictionCount; }
    public double getHitRatio()     {
        int total = hitCount + missCount;
        return total == 0 ? 0 : (double) hitCount / total;
    }

    // Peek without updating access order
    public V peek(K key) {
        return super.getOrDefault(key, null);
    }

    public Map.Entry<K,V> getEldestEntry() {
        if (isEmpty()) return null;
        return entrySet().iterator().next(); // first = eldest (insertion order)
    }
}

// Usage
BoundedLinkedHashMap<String, Integer> cache =
    new BoundedLinkedHashMap<>(3, true); // LRU mode

cache.put("a", 1); cache.put("b", 2); cache.put("c", 3);
cache.get("a"); // access "a" → moves to end
cache.put("d", 4); // "b" evicted (LRU)
System.out.println(cache.keySet());       // [c, a, d]
System.out.println(cache.getEvictionCount()); // 1

C5Implement a Custom LinkedList from scratch
JavaCustom Singly LinkedList
public class MyLinkedList<T> {

    // Node inner class
    private static class Node<T> {
        T data;
        Node<T> next;
        Node(T data) { this.data = data; this.next = null; }
    }

    private Node<T> head;
    private int size;

    // Add at end — O(n)
    public void add(T data) {
        Node<T> newNode = new Node<>(data);
        if (head == null) { head = newNode; }
        else {
            Node<T> curr = head;
            while (curr.next != null) curr = curr.next;
            curr.next = newNode;
        }
        size++;
    }

    // Add at head — O(1)
    public void addFirst(T data) {
        Node<T> newNode = new Node<>(data);
        newNode.next = head; head = newNode; size++;
    }

    // Get by index — O(n)
    public T get(int index) {
        if (index < 0 || index >= size)
            throw new IndexOutOfBoundsException("Index: " + index);
        Node<T> curr = head;
        for (int i = 0; i < index; i++) curr = curr.next;
        return curr.data;
    }

    // Remove by value — O(n)
    public boolean remove(T data) {
        if (head == null) return false;
        if (head.data.equals(data)) { head = head.next; size--; return true; }
        Node<T> curr = head;
        while (curr.next != null) {
            if (curr.next.data.equals(data)) {
                curr.next = curr.next.next; size--; return true;
            }
            curr = curr.next;
        }
        return false;
    }

    // Reverse in place — O(n)
    public void reverse() {
        Node<T> prev = null, curr = head, next;
        while (curr != null) {
            next = curr.next;
            curr.next = prev;
            prev = curr;
            curr = next;
        }
        head = prev;
    }

    // Detect cycle — Floyd's algorithm O(n) O(1) space
    public boolean hasCycle() {
        Node<T> slow = head, fast = head;
        while (fast != null && fast.next != null) {
            slow = slow.next;
            fast = fast.next.next;
            if (slow == fast) return true;
        }
        return false;
    }

    public int size() { return size; }

    @Override
    public String toString() {
        StringBuilder sb = new StringBuilder("[");
        Node<T> curr = head;
        while (curr != null) {
            sb.append(curr.data);
            if (curr.next != null) sb.append(" → ");
            curr = curr.next;
        }
        return sb.append("]").toString();
    }
}

📦 General Collections Q&A

Q1What is the difference between Collection and Collections?
Collection: An interface (root of the collection hierarchy — List, Set, Queue extend it). Collections: A utility class in java.util with static helper methods like sort(), shuffle(), min(), max(), unmodifiableList(), synchronizedList().
Q2What is the difference between List.of() and Arrays.asList()?
Arrays.asList(): Fixed-size list (can't add/remove, will throw UnsupportedOperationException), but CAN modify elements (set). Backed by original array.
List.of() (Java 9+): Truly immutable — cannot add, remove, OR set. Doesn't allow null elements. More memory-efficient.
Q3How do you make a Collection thread-safe?
1. Collections.synchronizedList(new ArrayList<>()) — wraps with synchronized methods
2. CopyOnWriteArrayList — thread-safe via copy-on-write
3. Vector — legacy synchronized list
4. ConcurrentHashMap — preferred thread-safe map
5. External sync: synchronized(list) { ... }
Q4What happens if you store null in a TreeSet or TreeMap?
TreeSet/TreeMap: Throws NullPointerException because null cannot be compared using compareTo() (natural ordering). Exception: if a custom Comparator handles null explicitly, it may work. HashSet/HashMap allow one null key.
Q5What is the initial capacity and load factor of HashMap? Why 0.75?
Default capacity = 16, load factor = 0.75 (rehash when 75% full = 12 entries).
Why 0.75? It's a compromise: low load factor = less collision but more memory waste + frequent rehashing. High load factor = more memory efficient but more collisions, slower. 0.75 provides good balance statistically based on Poisson distribution of hash collisions.
Q6How does HashSet ensure uniqueness?
HashSet internally uses HashMap. When you add(e), it calls map.put(e, PRESENT). HashMap uses hashCode() to find bucket, then equals() to check for duplicates in that bucket. If existing key found → put returns old value (not null) → add returns false.
Q7What is CopyOnWriteArrayList? When to use it?
Thread-safe variant of ArrayList. Every write operation (add, remove, set) creates a new copy of the underlying array. Read operations (get, iterate) work on the old array — no synchronization needed for reads. Use when: read-heavy workload, iterations much more frequent than writes. Don't use when: frequent writes (too many copies = slow + memory intensive).
Q8Difference between Iterator and ListIterator?
Iterator: Forward only. Methods: hasNext(), next(), remove(). Works with all Collections.
ListIterator: Forward AND backward. Methods: + hasPrevious(), previous(), add(), set(), nextIndex(), previousIndex(). Only for List implementations.
Q9What is the difference between peek, poll, and remove in Queue?
peek(): Returns head element without removing. Returns null if empty.
poll(): Returns and removes head element. Returns null if empty.
remove(): Returns and removes head element. Throws NoSuchElementException if empty.
Rule: Prefer peek/poll over element/remove — they don't throw exceptions.
Q10How is ConcurrentHashMap different from synchronized HashMap?
synchronizedMap: Entire map locked for every operation — one thread at a time. Low concurrency.
ConcurrentHashMap (Java 7): Divided into 16 segments — locks only the relevant segment. Multiple threads can write to different segments simultaneously.
ConcurrentHashMap (Java 8): Uses CAS (Compare-And-Swap) operations + synchronized on individual bucket node. Even higher concurrency. Also no null keys/values allowed.
