Foundations

Why Spring exists, how configuration evolved, and what Spring Boot actually does under the hood.

🌱 Spring vs Spring Boot

Why Spring Framework Was Introduced

Before Spring (early 2000s), Java enterprise development meant EJB (Enterprise JavaBeans) — a heavyweight, complex, XML-heavy spec that required a full J2EE application server. Problems it caused:

  • Hard to unit-test (EJBs needed a running container)
  • Tightly coupled code — classes managed their own dependencies
  • Enormous XML descriptors just to wire components together
  • Slow startup, slow development feedback loop

Rod Johnson released Spring Framework in 2002 to fix all of this. The core idea: let a container manage your object graph — you write plain Java objects (POJOs), Spring wires them together.

What Spring Framework Solves

Inversion of Control

You don't new your dependencies — the container creates and injects them. Decouples consumers from implementations.

Aspect-Oriented Programming

Cross-cutting concerns (logging, security, transactions) are woven in declaratively — not mixed into business logic.

Data Access Abstraction

JdbcTemplate, ORM support, consistent exception hierarchy. Removes boilerplate JDBC code.

Testability

POJOs are easily unit-tested. Spring Test module provides container-backed integration tests.

The Spring Problem — Why Spring Boot Was Needed

Spring solved EJB complexity but introduced its own: configuration hell. A typical Spring MVC app needed:

  • web.xml — DispatcherServlet registration
  • applicationContext.xml — bean definitions
  • dispatcher-servlet.xml — MVC config
  • Manual dependency version management in pom.xml
  • Manual server deployment (WAR → Tomcat)
⚠️
The 80/20 Problem: 80% of apps use the same configuration. Developers were copy-pasting XML across projects and wasting hours on setup that added zero business value.

Spring Boot's Answer — Convention Over Configuration

✅ Spring Boot

  • Embedded Tomcat — no WAR deployment
  • Auto-configuration — detects classpath and configures beans automatically
  • Starter POMs — curated, compatible dependency sets
  • Opinionated defaults — override only what you need
  • Actuator — production-ready monitoring out of the box
  • No XML required — pure Java config

❌ Plain Spring

  • Manual Tomcat/Jetty setup + WAR packaging
  • Every bean must be explicitly declared
  • Dependency version conflicts — you manage them
  • Pages of XML or @Configuration classes for basic setup
  • No built-in health checks
  • Steep learning curve for new projects

One-Line Mental Model

Spring Framework = The engine (IoC, AOP, Data, MVC).
Spring Boot = A pre-configured car built on that engine — ready to drive, just add your business logic.

⚙️ XML vs Java Configuration

Evolution of Spring Configuration

Spring 1.x — XML Only (2004)
Every bean declared in applicationContext.xml. Verbose but explicit. Separation of config from code.
Spring 2.5 — Annotation-Based (2007)
@Component, @Autowired introduced. Component scanning reduced XML. Still needed XML to bootstrap scanning.
Spring 3.0 — Java Config (2009)
@Configuration + @Bean. 100% XML-free Spring possible. Type-safe, IDE-friendly configuration.
Spring Boot 1.0 — Auto-Config (2014)
Auto-configuration via spring.factories. Write zero config for standard setups. Convention-over-configuration era begins.
Spring Boot 3.x — AOT & Virtual Threads (2022+)
Ahead-of-time compilation for GraalVM native images. Virtual thread support (Java 21). AutoConfiguration.imports replaces spring.factories.

XML Configuration (Legacy)

XMLapplicationContext.xml
<beans xmlns="http://www.springframework.org/schema/beans">

    <!-- Explicit bean declaration -->
    <bean id="userRepository" class="com.app.UserRepositoryImpl"/>

    <!-- Constructor injection -->
    <bean id="userService" class="com.app.UserServiceImpl">
        <constructor-arg ref="userRepository"/>
    </bean>

    <!-- Property (setter) injection -->
    <bean id="emailService" class="com.app.EmailService">
        <property name="smtpHost" value="smtp.gmail.com"/>

</beans>

@Configuration — Java-Based Config

@Configuration marks a class as a bean definition source. It is itself a Spring bean. Spring Boot uses this everywhere.

JavaAppConfig.java
@Configuration                        // Tells Spring: this class defines beans
@ComponentScan("com.app")             // Scan this package for @Component etc.
@PropertySource("classpath:app.properties") // Load properties file
public class AppConfig {

    @Value("${smtp.host}")
    private String smtpHost;

    @Bean                              // Returns a managed bean
    public UserRepository userRepository() {
        return new UserRepositoryImpl();
    }

    @Bean
    public UserService userService(UserRepository userRepository) {
        // Spring auto-injects userRepository from above
        return new UserServiceImpl(userRepository);
    }

    @Bean
    @Conditional(OnProductionCondition.class) // Only in prod
    public EmailService emailService() {
        return new RealEmailService(smtpHost);
    }
}
💡
Internal trick: @Configuration classes are subclassed by CGLIB at runtime. When you call userService() inside the config, CGLIB intercepts the call and returns the existing singleton bean — it does NOT create a new one. This is why @Configuration is different from @Component.

@SpringBootApplication — The Entry Point

JavaWhat @SpringBootApplication expands to
// @SpringBootApplication is a composed annotation equal to:

@SpringBootConfiguration   // ← same as @Configuration for Spring Boot
@EnableAutoConfiguration   // ← load all auto-config classes from the classpath
@ComponentScan             // ← scan current package and sub-packages
public class MyApp {
    public static void main(String[] args) {
        SpringApplication.run(MyApp.class, args);
    }
}
@EnableAutoConfiguration Internals
1
Import AutoConfigurationImportSelector
@EnableAutoConfiguration imports this selector class via @Import.
2
Read AutoConfiguration.imports
The selector reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (SB3) listing hundreds of auto-config classes.
3
Apply @Conditional filters
Each auto-config class has conditions like @ConditionalOnClass, @ConditionalOnMissingBean — only applied when relevant.
4
Register matching beans
Qualifying config classes are loaded into the ApplicationContext, registering their beans.

What Maven Solves

Before Maven: manually download JARs, manage class paths, write build scripts (Ant). Maven standardised the build lifecycle and dependency resolution across all Java projects.

Dependency Management

Declare what you need; Maven fetches it and all transitive dependencies from Central Repository.

Standard Lifecycle

validate → compile → test → package → verify → install → deploy

Spring Boot Parent POM

Inheriting from spring-boot-starter-parent gives you pre-tested dependency versions — no conflicts.

Starters

spring-boot-starter-web pulls Spring MVC + Jackson + Tomcat as one unit. No version juggling.

🔄 IoC & Dependency Injection

Inversion of Control — The Core Principle

Normally, an object controls its own dependencies (calls new). IoC inverts this — a container (the IoC Container) creates objects and wires them. You give up control; the framework takes it.

WITHOUT IoC UserService ──creates──► new UserRepository() └──creates──► new EmailService() → Tight coupling. Hard to test. Impossible to swap implementations. WITH IoC (Spring) Spring Container ──creates──► UserService ──injects──► UserRepository impl ──injects──► EmailService impl → Loose coupling. Easily testable. Swap beans without touching UserService.

🏗️ ApplicationContext vs BeanFactory

BeanFactory — The Base Contract

BeanFactory is the root interface of the Spring IoC container. It provides basic bean instantiation, wiring, and lifecycle management. Beans are created lazily (only when requested).

JavaBeanFactory usage (rare in practice)
Resource resource = new ClassPathResource("beans.xml");
BeanFactory factory = new XmlBeanFactory(resource); // deprecated
MyBean bean = (MyBean) factory.getBean("myBean"); // created NOW (lazy)

ApplicationContext — The Full Container

ApplicationContext extends BeanFactory and adds enterprise features. This is what you always use in Spring Boot.

🌐 Servlet, Tomcat & Containers

What is a Servlet?

A Servlet is a Java class that handles HTTP requests. It implements the javax.servlet.Servlet (or jakarta.servlet.Servlet in Jakarta EE) interface. The lifecycle:

1
init()
Called once when servlet loads. Initialize resources (DB connections, config).
2
service()
Called for every request. Dispatches to doGet(), doPost(), etc.
3
destroy()
Called once when servlet is unloaded. Release resources.

Tomcat — The Servlet Container

Tomcat implements the Servlet specification. It listens on a port, parses HTTP, and invokes the correct Servlet. In Spring Boot, Tomcat is embedded — it's a library inside your JAR, not a separate server you deploy to.

HTTP Request ──► Tomcat (port 8080) │ ├─ Parses HTTP ├─ Creates HttpServletRequest / Response └──► DispatcherServlet ← Spring's front controller │ ├──► HandlerMapping → finds @RequestMapping ├──► HandlerAdapter → invokes @Controller method ├──► ViewResolver → resolves template/JSON └──► HTTP Response

Servlet Container vs Spring Container

🔵 Servlet Container (Tomcat)

  • Manages HTTP lifecycle
  • Creates Servlet instances
  • Thread pool for requests
  • Handles sessions, cookies
  • Scope: web infrastructure

🟢 Spring Container (ApplicationContext)

  • Manages bean lifecycle
  • Handles dependency injection
  • AOP proxy creation
  • Transaction management
  • Scope: application logic
💡
In Spring Boot, both containers start together. SpringApplication.run() starts Tomcat (Servlet container) and inside it registers the Spring ApplicationContext as a Servlet context attribute.

Beans & Annotations

The heart of Spring — how objects are created, configured, scoped, and wired together.

🫘 Bean Lifecycle & Scopes

Complete Bean Lifecycle (in order)

1
Instantiation
Spring calls the constructor (or factory method) to create the raw object.
2
Dependency Injection
Properties and constructor args are injected. @Autowired fields are populated.
3
BeanNameAware / BeanFactoryAware
If bean implements Aware interfaces, Spring calls setBeanName(), setBeanFactory(), etc.
4
BeanPostProcessor — postProcessBeforeInitialization()
All registered BPPs run before init. This is where annotations like @Autowired are processed (AutowiredAnnotationBeanPostProcessor).
5
@PostConstruct / InitializingBean.afterPropertiesSet()
Your custom initialization logic runs here. Use @PostConstruct — it's standard. afterPropertiesSet() couples your class to Spring.
6
BeanPostProcessor — postProcessAfterInitialization()
AOP proxies are created here. The real bean is wrapped in a proxy object that adds cross-cutting concerns.
7
Bean is Ready — In Use
The bean (possibly a proxy) is placed in the singleton cache and served to dependants.
8
@PreDestroy / DisposableBean.destroy()
On application shutdown, cleanup methods run. Close connections, flush caches, etc.
JavaBean lifecycle methods in code
@Component
public class DatabasePool implements InitializingBean, DisposableBean {

    @Autowired
    private DataSourceConfig config; // injected at step 2

    @PostConstruct              // step 5 — preferred
    public void init() {
        System.out.println("Opening connections to: " + config.getUrl());
    }

    @Override                    // also step 5 (after @PostConstruct)
    public void afterPropertiesSet() {
        // avoid — couples to Spring API
    }

    @PreDestroy                 // step 8 — preferred
    public void cleanup() {
        System.out.println("Closing all connections");
    }
}

Prototype Scope — Real-World Use Case
JavaPrototype for stateful PDF report generator
@Component
@Scope("prototype")   // New instance every injection
public class ReportBuilder {
    private List<String> pages = new ArrayList<>();  // mutable state

    public void addPage(String content) { pages.add(content); }
    public Report build() { return new Report(pages); }
}

// ⚠️ Problem: injecting prototype into singleton freezes the instance
@Service
public class ReportService {

    private final ApplicationContext ctx; // Use context to get fresh instance

    public Report generateReport() {
        ReportBuilder builder = ctx.getBean(ReportBuilder.class); // NEW each time
        builder.addPage("Cover");
        builder.addPage("Chapter 1");
        return builder.build();
    }
}
⚠️
Scoped Bean Trap: Injecting a prototype bean via @Autowired into a singleton only creates one prototype instance (at singleton creation). Use ApplicationContext.getBean(), ObjectFactory<T>, or @Lookup to get a fresh prototype every time.

🏷️ Core Annotations Deep Dive

Stereotype Annotations — @Component Family

All four are functionally equivalent to Spring's component scanner — they all register a bean. The difference is semantic and enables Spring to apply layer-specific behaviour.

AnnotationLayerExtra Behaviour Added by Spring
@Component Generic Base annotation. No layer-specific processing.
@Service Business Logic Semantic only. No extra processing currently, but signals intent. Some AOP advice may target this layer.
@Repository Data Access Enables exception translation via PersistenceExceptionTranslationPostProcessor — converts JPA/Hibernate exceptions to Spring's DataAccessException hierarchy.
@Controller Web Layer Marks class as handler for DispatcherServlet. Required for @RequestMapping to work in MVC.
@RestController REST Web Layer @Controller + @ResponseBody. Every method return value is written directly to HTTP response body as JSON/XML.

@Autowired — How It Works Internally

1
AutowiredAnnotationBeanPostProcessor scans
This BPP (registered automatically) finds all @Autowired fields, constructors, and methods during bean initialization.
2
Type matching
Spring looks in the ApplicationContext for beans matching the required type.
3
If multiple matches — name matching
Spring tries to match by field/parameter name. If still ambiguous, throws NoUniqueBeanDefinitionException.
4
Inject via reflection
Uses Field.set() or constructor/method invocation to inject the dependency.
All @Autowired Variants
JavaThree injection styles
@Service
public class OrderService {

    // ────── 1. Constructor Injection (RECOMMENDED) ──────
    private final OrderRepository orderRepo;
    private final PaymentService paymentService;

    // @Autowired is optional since Spring 4.3 for single constructors
    public OrderService(OrderRepository orderRepo, PaymentService paymentService) {
        this.orderRepo = orderRepo;
        this.paymentService = paymentService;
    }

    // ────── 2. Setter Injection ──────
    private NotificationService notifService;

    @Autowired
    public void setNotifService(NotificationService s) {
        this.notifService = s;
    }

    // ────── 3. Field Injection (AVOID in production) ──────
    @Autowired                     // Breaks testability — can't inject mock without Spring
    private AuditService auditService;
}

Controller Layer Annotations — Complete Reference

JavaRestController with all common annotations
@RestController                              // @Controller + @ResponseBody
@RequestMapping("/api/v1/orders")            // base path for all methods
@CrossOrigin(origins = "http://localhost:3000")
@Validated                                   // enable method-level validation
public class OrderController {

    private final OrderService orderService;

    public OrderController(OrderService orderService) {
        this.orderService = orderService;
    }

    @GetMapping                               // GET /api/v1/orders
    public ResponseEntity<List<OrderDTO>> getAll(
            @RequestParam(defaultValue = "0") int page,
            @RequestParam(defaultValue = "20") int size) {
        return ResponseEntity.ok(orderService.findAll(page, size));
    }

    @GetMapping("/{id}")                      // GET /api/v1/orders/42
    public ResponseEntity<OrderDTO> getById(@PathVariable Long id) {
        return ResponseEntity.ok(orderService.findById(id));
    }

    @PostMapping                              // POST /api/v1/orders
    public ResponseEntity<OrderDTO> create(
            @Valid @RequestBody CreateOrderRequest req) {
        OrderDTO created = orderService.create(req);
        URI location = URI.create("/api/v1/orders/" + created.getId());
        return ResponseEntity.created(location).body(created);
    }

    @PutMapping("/{id}")                     // Full update
    public ResponseEntity<OrderDTO> update(
            @PathVariable Long id,
            @Valid @RequestBody UpdateOrderRequest req) {
        return ResponseEntity.ok(orderService.update(id, req));
    }

    @PatchMapping("/{id}/status")           // Partial update
    public ResponseEntity<Void> updateStatus(
            @PathVariable Long id,
            @RequestParam String status) {
        orderService.updateStatus(id, status);
        return ResponseEntity.noContent().build();
    }

    @DeleteMapping("/{id}")                 // DELETE /api/v1/orders/42
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        orderService.delete(id);
        return ResponseEntity.noContent().build();  // 204
    }

    @ExceptionHandler(OrderNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(OrderNotFoundException ex) {
        return ResponseEntity.status(404)
                .body(new ErrorResponse(ex.getMessage()));
    }
}

Service Layer Annotations

JavaService with transaction + validation
@Service
@Transactional(readOnly = true)  // default for all read methods
public class OrderService {

    private final OrderRepository repo;
    private final OrderMapper mapper;

    public OrderService(OrderRepository repo, OrderMapper mapper) {
        this.repo = repo;
        this.mapper = mapper;
    }

    public List<OrderDTO> findAll(int page, int size) {
        return repo.findAll(PageRequest.of(page, size))
                   .stream()
                   .map(mapper::toDTO)
                   .toList();
    }

    @Transactional               // override: write operation needs full tx
    public OrderDTO create(CreateOrderRequest req) {
        Order order = mapper.toEntity(req);
        order.setCreatedAt(Instant.now());
        return mapper.toDTO(repo.save(order));
    }

    public OrderDTO findById(Long id) {
        return repo.findById(id)
                   .map(mapper::toDTO)
                   .orElseThrow(() -> new OrderNotFoundException(id));
    }
}

Circular Dependencies — What They Are & How to Fix

🚫
A ↔ B circular dependency: ServiceA needs ServiceB; ServiceB needs ServiceA. With constructor injection, Spring throws BeanCurrentlyInCreationException at startup — this is actually good, it exposes a design problem.
JavaCircular dependency — detection and fixes
// ❌ CIRCULAR: ServiceA → ServiceB → ServiceA
@Service public class ServiceA {
    public ServiceA(ServiceB b) {} // needs B
}
@Service public class ServiceB {
    public ServiceB(ServiceA a) {} // needs A → BANG! BeanCurrentlyInCreationException
}

// ✅ FIX 1: Break the cycle with an event / mediator
@Service public class ServiceA {
    private final ApplicationEventPublisher events;
    public ServiceA(ApplicationEventPublisher events) { this.events = events; }
}

// ✅ FIX 2: Use @Lazy on one side (defers creation)
@Service public class ServiceA {
    public ServiceA(@Lazy ServiceB b) {} // B created lazily → breaks cycle
}

// ✅ FIX 3: Extract shared logic to a third service
@Service public class SharedService {} // A and B both depend on this

🎯 @Qualifier & @Primary — Resolving Conflicts

When Multiple Beans Match the Same Type

JavaTwo implementations of same interface
public interface NotificationService {
    void send(String message);
}

@Service
@Primary                           // ← preferred candidate when no @Qualifier specified
public class EmailNotificationService implements NotificationService {
    public void send(String msg) { /* send email */ }
}

@Service
@Qualifier("sms")                  // ← named for explicit selection
public class SmsNotificationService implements NotificationService {
    public void send(String msg) { /* send SMS */ }
}

// ── Usage ────────────────────────────────
@Service
public class AlertService {

    private final NotificationService emailNotif;  // gets EmailNotificationService (@Primary)
    private final NotificationService smsNotif;

    public AlertService(
            NotificationService emailNotif,           // @Primary wins
            @Qualifier("sms") NotificationService smsNotif) {
        this.emailNotif = emailNotif;
        this.smsNotif = smsNotif;
    }

    public void alertAll(String msg) {
        emailNotif.send(msg);
        smsNotif.send(msg);
    }
}

Custom Qualifier Annotation

JavaType-safe custom qualifier
// Define the annotation
@Target({ElementType.FIELD, ElementType.PARAMETER})
@Retention(RetentionPolicy.RUNTIME)
@Qualifier
public @interface SmsChannel {}

// Apply to bean
@Service
@SmsChannel
public class SmsNotificationService implements NotificationService { ... }

// Inject with type-safe annotation
@Service
public class AlertService {
    private final NotificationService smsNotif;
    public AlertService(@SmsChannel NotificationService smsNotif) {
        this.smsNotif = smsNotif;
    }
}

⚡ @EnableAutoConfiguration Internals

How Auto-Configuration Works Step-by-Step

App starts │ ▼ @SpringBootApplication │ includes @EnableAutoConfiguration ▼ AutoConfigurationImportSelector.selectImports() │ ├─ reads META-INF/spring/ │ org.springframework.boot.autoconfigure.AutoConfiguration.imports │ (150+ entries like DataSourceAutoConfiguration, WebMvcAutoConfiguration...) │ ▼ Filter by @Conditional annotations │ ├─ @ConditionalOnClass(DataSource.class) → only if JDBC driver on classpath ├─ @ConditionalOnMissingBean(DataSource.class) → only if user hasn't defined one ├─ @ConditionalOnProperty("spring.datasource.url") → only if property set │ ▼ Surviving configs loaded → beans registered

Example: DataSourceAutoConfiguration

JavaSimplified DataSourceAutoConfiguration (Spring Boot source)
@AutoConfiguration                              // marks as auto-config candidate
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })
@ConditionalOnMissingBean(type = "io.r2dbc.spi.ConnectionFactory")
@EnableConfigurationProperties(DataSourceProperties.class) // binds spring.datasource.*
public class DataSourceAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean                    // only if user hasn't defined DataSource
    public DataSource dataSource(DataSourceProperties properties) {
        return properties.initializeDataSourceBuilder().build();
    }
}
✅
Key insight: If you define your own DataSource bean, Spring Boot's auto-configured one is skipped because of @ConditionalOnMissingBean. This is how you override any auto-config — just define the bean yourself.

Aspect-Oriented Programming

Separation of cross-cutting concerns — logging, security, transactions — without polluting business logic.

🔭 What is AOP & Why We Need It

The Problem AOP Solves

Some concerns cut across many classes — they don't belong to any single business domain. Examples:

🔒 Security

Every service method needs authorization checks.

📝 Logging

Log method entry/exit, execution time across all services.

💾 Transaction

Begin/commit/rollback around data operations.

📊 Metrics

Count invocations, measure latency for monitoring.

⚙️ AOP Internals — How It Actually Works

Proxy-Based AOP (Spring's Approach)

Spring uses runtime proxies. When a bean is created that has aspects applied to it, Spring wraps it in a proxy object. Callers interact with the proxy, not the real object.

Advice Types

JavaAll advice types with examples
@Aspect
@Component
public class LoggingAspect {

    // ── @Before ── runs BEFORE the method ──────────────────────
    @Before("execution(* com.app.service.*.*(..))")
    public void logBefore(JoinPoint jp) {
        System.out.println("Calling: " + jp.getSignature().getName());
    }

    // ── @AfterReturning ── runs AFTER successful return ─────────
    @AfterReturning(pointcut = "execution(* com.app.service.*.*(..))",
                    returning = "result")
    public void logAfterReturn(JoinPoint jp, Object result) {
        System.out.println("Returned: " + result);
    }

    // ── @AfterThrowing ── runs if method THROWS ─────────────────
    @AfterThrowing(pointcut = "execution(* com.app.service.*.*(..))",
                   throwing = "ex")
    public void logException(JoinPoint jp, Exception ex) {
        System.out.println("Exception in " + jp.getSignature() + ": " + ex.getMessage());
    }

    // ── @After ── ALWAYS runs (like finally) ────────────────────
    @After("execution(* com.app.service.*.*(..))")
    public void logFinally(JoinPoint jp) {
        System.out.println("Method done: " + jp.getSignature().getName());
    }

    // ── @Around ── WRAPS the method — most powerful ─────────────
    @Around("execution(* com.app.service.*.*(..))")
    public Object measureTime(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        try {
            Object result = pjp.proceed(); // ← actually call the real method
            return result;
        } finally {
            long elapsed = System.currentTimeMillis() - start;
            System.out.println("Took: " + elapsed + "ms");
        }
    }
}

Creating a Custom @Auditable Annotation with AOP

JavaStep 1 — Define the annotation
@Target(ElementType.METHOD)            // apply to methods only
@Retention(RetentionPolicy.RUNTIME)    // visible at runtime (needed for AOP)
public @interface Auditable {
    String action() default "";         // metadata passed to aspect
}
JavaStep 2 — Create the Aspect
@Aspect
@Component
public class AuditAspect {

    private final AuditLogRepository auditRepo;

    // Reusable pointcut definition
    @Pointcut("@annotation(auditable)")
    public void auditableMethods(Auditable auditable) {}

    @Around("auditableMethods(auditable)")
    public Object audit(ProceedingJoinPoint pjp, Auditable auditable)
            throws Throwable {

        String user = SecurityContextHolder.getContext()
                        .getAuthentication().getName();
        String action = auditable.action();
        Instant start = Instant.now();

        try {
            Object result = pjp.proceed(); // run actual method
            auditRepo.save(new AuditLog(user, action, "SUCCESS", start));
            return result;
        } catch (Exception ex) {
            auditRepo.save(new AuditLog(user, action, "FAILED: " + ex.getMessage(), start));
            throw ex;
        }
    }
}
JavaStep 3 — Use the annotation
@Service

Transactions & Concurrency

ACID guarantees, isolation levels, propagation types, and distributed locking strategies.

💾 Transaction Basics

ACID Properties

A — Atomicity

All operations succeed or all fail. No partial updates. If charge succeeds but inventory update fails → rollback the charge.

C — Consistency

DB moves from one valid state to another. Business rules always hold (e.g., account balance never goes negative).

I — Isolation

Concurrent transactions don't see each other's intermediate state. Controlled by isolation level.

D — Durability

Committed transactions survive crashes. Written to disk, not just memory.

@Transactional — Full Syntax

Java@Transactional all attributes
@Transactional(
    propagation  = Propagation.REQUIRED,          // default
    isolation    = Isolation.READ_COMMITTED,        // default for most DBs

    readOnly     = false,                            // hint for optimization
    rollbackFor  = { PaymentException.class },      // roll back for these
    noRollbackFor = { AuditException.class }        // don't roll back for these
)
public void processOrder(Order order) { ... }

// Default rollback behaviour:
// ✅ Rolls back on RuntimeException and Error
// ❌ Does NOT roll back on checked Exception — you must add rollbackFor

🔒 Isolation Levels

Concurrency Problems That Isolation Levels Solve

ProblemDescriptionExample
Dirty Read Read uncommitted data from another transaction T2 reads balance updated by T1, but T1 rolls back. T2 saw wrong data.
Non-Repeatable Read Same row returns different values in same transaction T1 reads balance=100. T2 updates to 200 and commits. T1 reads again → 200.
Phantom Read Same query returns different rows in same transaction T1 counts orders=5. T2 inserts new order. T1 counts again → 6.
Lost Update Two transactions update same row; one overwrites the other T1 and T2 both read balance=100. T1 sets 150. T2 sets 120. T1's update lost.

Isolation Levels — Which Problems They Prevent

LevelDirty ReadNon-RepeatablePhantomPerformance
READ_UNCOMMITTED ❌ Possible❌❌ Fastest
READ_COMMITTED (default PG/Oracle) ✅ Prevented❌❌ Fast
REPEATABLE_READ (default MySQL) ✅✅ Prevented❌ Medium
SERIALIZABLE ✅✅✅ Prevented Slowest

🔀 Propagation Types

What is Propagation?

When a transactional method calls another transactional method, propagation defines what happens to the transaction context — does the inner method join the outer transaction, or start its own?

PropagationExisting Tx?No Existing Tx?Use Case
REQUIRED (default) Join existing Create new Standard service methods
REQUIRES_NEW Suspend existing, create new Create new Audit logs (must commit even if outer rolls back)
NESTED Create savepoint in existing Create new Partial rollback scenarios
SUPPORTS Join existing Run without tx Read methods that work with or without tx
NOT_SUPPORTED Suspend existing, run without tx Run without tx Operations that must not be in a tx
MANDATORY Join existing Throw exception Must always be called within a tx
NEVER Throw exception Run without tx Must never be in a tx

REQUIRES_NEW in Practice — Audit Log Example

JavaAudit must commit even if order fails
@Service
public class OrderService {

    private final AuditService auditService;

    @Transactional                        // Outer transaction TX1
    public void placeOrder(Order order) {
        orderRepo.save(order);            // In TX1
        auditService.log("ORDER_PLACED"); // Starts TX2 (independent)
        throw new RuntimeException("DB error!");
        // TX1 rolls back (order not saved)
        // TX2 already committed (audit log IS saved) ✅
    }
}

@Service
public class AuditService {

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void log(String action) {
        auditRepo.save(new AuditEntry(action));
        // This commits INDEPENDENTLY of the caller's tx
    }
}

JavaProgrammatic with TransactionTemplate
@Service
public class PaymentService {

    private final TransactionTemplate txTemplate;
    private final PlatformTransactionManager txManager;

    // ── Option 1: TransactionTemplate (cleaner) ───────────────
    public void processWithTemplate(Payment payment) {
        txTemplate.execute(status -> {
            paymentRepo.save(payment);
            if (payment.getAmount().compareTo(BigDecimal.ZERO) < 0) {
                status.setRollbackOnly(); // explicit rollback signal
            }
            return null;
        });
    }

    // ── Option 2: Low-level PlatformTransactionManager ────────
    public void processManual(Payment payment) {
        TransactionDefinition def = new DefaultTransactionDefinition();
        TransactionStatus status = txManager.getTransaction(def);
        try {
            paymentRepo.save(payment);
            txManager.commit(status);
        } catch (Exception ex) {
            txManager.rollback(status);
            throw ex;
        }
    }
}

⚡ Optimistic vs Pessimistic Locking

The Problem: Lost Update

Two users read the same inventory count (10). Both try to place an order for 5. Both check "10 ≥ 5 → proceed". Both save inventory=5. Actual inventory should be 0, but it's 5. A lost update.

Optimistic Locking — Version-Based

Assumes conflicts are rare. No DB-level lock. Instead, each row has a version number. On update, check the version still matches — if not, someone else modified it → throw OptimisticLockException.

JavaOptimistic locking with @Version
@Entity
public class Inventory {
    @Id @GeneratedValue
    private Long id;

    private int quantity;

    @Version                    // JPA manages this — increments on every update
    private Long version;
}

// When two threads both try to update:
// Thread 1 reads version=5, updates successfully → version becomes 6
// Thread 2 reads version=5, tries to update with WHERE version=5 → 0 rows updated
// → JPA throws OptimisticLockException → retry or show "please try again"

// Retry strategy using Spring Retry:
@Retryable(value = OptimisticLockException.class, maxAttempts = 3)
@Transactional
public void decreaseInventory(Long productId, int qty) {
    Inventory inv = invRepo.findById(productId).orElseThrow();
    inv.setQuantity(inv.getQuantity() - qty);
    invRepo.save(inv);
}

Pessimistic Locking — DB-Level Locks

Assumes conflicts are frequent. Takes a DB lock before reading. Other transactions trying to read the locked row must wait. More overhead, guaranteed safety.

JavaPessimistic locking in Spring Data JPA
public interface InventoryRepository extends JpaRepository<Inventory, Long> {

    // SELECT ... FOR UPDATE — exclusive lock
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("SELECT i FROM Inventory i WHERE i.id = :id")
    Optional<Inventory> findByIdForUpdate(@Param("id") Long id);

    // SELECT ... FOR SHARE — shared read lock
    @Lock(LockModeType.PESSIMISTIC_READ)
    Optional<Inventory> findById(Long id);
}

// Usage in service
@Transactional
public void bookSeats(Long eventId, int seats) {
    Inventory inv = invRepo.findByIdForUpdate(eventId).orElseThrow();
    // Row is now locked — no other tx can modify until this one commits
    if (inv.getQuantity() < seats) throw new InsufficientSeatsException();
    inv.setQuantity(inv.getQuantity() - seats);
    // Lock released on commit
}

@Async & Thread Management

Non-blocking execution, custom thread pools, and managing transactions in async contexts.

⚡ @Async — Internals

How @Async Works

1
@EnableAsync activates AsyncAnnotationBeanPostProcessor
Add @EnableAsync to a @Configuration class. This BPP scans for @Async methods and creates proxies.
2
CGLIB proxy wraps @Async methods
When an @Async method is called from another bean, the proxy intercepts the call.
3
Task submitted to TaskExecutor
The proxy submits the method body as a Runnable to the configured TaskExecutor (default: SimpleAsyncTaskExecutor — no pooling!).
4
Returns Future/CompletableFuture immediately
The caller gets the Future back and continues. The actual work runs on the pool thread.
Java@Async with all return types
@Configuration
@EnableAsync                             // Step 1: enable async processing
public class AsyncConfig { }

@Service
public class EmailService {

    // ── void: fire and forget ────────────────────────────────
    @Async
    public void sendWelcomeEmail(String email) {
        // runs on separate thread — caller doesn't wait
        smtpClient.send(email, "Welcome!");
    }

    // ── Future<T>: legacy callable style ────────────────────
    @Async
    public Future<String> getReport(Long userId) {
        String report = reportService.generate(userId);
        return new AsyncResult<>(report);
    }

    // ── CompletableFuture<T>: modern, chainable ──────────────
    @Async
    public CompletableFuture<List<Notification>> fetchNotifications(Long userId) {
        List<Notification> notifs = notifRepo.findByUserId(userId);
        return CompletableFuture.completedFuture(notifs);
    }
}

// ── Usage with chaining ──────────────────────────────────────
@Service
public class DashboardService {
    public void loadDashboard(Long userId) {
        CompletableFuture<List<Notification>> notifs = emailService.fetchNotifications(userId);
        CompletableFuture<String> report = emailService.getReport(userId);

        // Wait for both to complete
        CompletableFuture.allOf(notifs, report).join();

        System.out.println(notifs.get());
        System.out.println(report.get());
    }
}
🚫
@Async self-invocation problem: Calling @Async method from the same class doesn't work (proxy is bypassed). Always call from a different bean. Same rule as @Transactional.

🏊 ThreadPoolTaskExecutor — Custom Thread Pools

Why Configure Your Own Pool

Spring Boot's default SimpleAsyncTaskExecutor creates a new thread per task — no pooling! Under load this creates thousands of threads and crashes the JVM. Always define a proper pool.

JavaCustom ThreadPoolTaskExecutor
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();

        executor.setCorePoolSize(5);          // always-alive threads
        executor.setMaxPoolSize(20);           // max threads under load
        executor.setQueueCapacity(100);        // queue before creating new thread
        executor.setKeepAliveSeconds(60);      // idle thread timeout
        executor.setThreadNamePrefix("async-email-");
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy() // run in caller thread if pool full
        );
        executor.initialize();
        return executor;
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) -> {
            System.err.println("Async exception in " + method.getName() + ": " + ex.getMessage());
            // Send to Sentry, log to DB, etc.
        };
    }
}

// Multiple named executors for different tasks
@Bean("emailExecutor")
public Executor emailExecutor() {
    ThreadPoolTaskExecutor e = new ThreadPoolTaskExecutor();
    e.setCorePoolSize(3); e.setMaxPoolSize(10); e.initialize();
    return e;
}

@Bean("reportExecutor")
public Executor reportExecutor() {
    ThreadPoolTaskExecutor e = new ThreadPoolTaskExecutor();
    e.setCorePoolSize(2); e.setMaxPoolSize(5); e.initialize();
    return e;
}

ThreadPoolExecutor Internals — How Tasks Flow

New task submitted │ ▼ Core pool has free threads? ├─ YES ──► assign to free core thread → execute │ ▼ NO Queue has space? (queueCapacity) ├─ YES ──► add to queue → wait for free thread │ ▼ NO (queue full) Thread count < maxPoolSize? ├─ YES ──► create new thread → execute │ ▼ NO (maxPoolSize reached AND queue full) RejectedExecutionHandler ├─ AbortPolicy → throw RejectedExecutionException ├─ CallerRunsPolicy → run in calling thread (backpressure) ├─ DiscardPolicy → silently drop └─ DiscardOldestPolicy → drop oldest queued task, retry

🔀 @Async + @Transactional

Why @Async and @Transactional Don't Mix Directly

Transactions are thread-bound. The transaction started by the calling thread is NOT available on the async thread. If you add @Transactional to an @Async method, each async invocation gets its own independent transaction.

JavaCorrect pattern — transaction inside async
@Service
public class ReportGeneratorService {

    // ✅ @Async + @Transactional on same method = WORKS
    //    Async thread starts its OWN transaction — independent of caller
    @Async("reportExecutor")
    @Transactional
    public CompletableFuture<Report> generateAndSave(Long userId) {
        Report report = buildReport(userId);    // pure computation
        reportRepo.save(report);               // in async thread's own tx
        return CompletableFuture.completedFuture(report);
    }
}

// ── Exception Handling in @Async ─────────────────────────────
@Async
public CompletableFuture<String> riskyOperation() {
    try {
        // do work
        return CompletableFuture.completedFuture("done");
    } catch (Exception ex) {
        // For CompletableFuture, caller sees exception via .get()
        // or .exceptionally() chain
        return CompletableFuture.failedFuture(ex);
    }
}

// Caller handles exception
riskyOperation()
    .exceptionally(ex -> { log.error("Failed: ", ex); return null; })
    .thenAccept(result -> System.out.println(result));

🏷️ JPA Entity Annotations

Rules for Defining a JPA Entity

📋
7 Rules every JPA entity must follow:
1. Annotated with @Entity
2. Has a no-arg constructor (can be protected)
3. Has a primary key annotated with @Id
4. Class must not be final (Hibernate proxies it)
5. Fields/properties must not be final
6. Should implement Serializable (if used in distributed cache)
7. Override equals() and hashCode() based on business key (not DB id)
JavaFully annotated Entity
@Entity
@Table(name = "orders",
       indexes = { @Index(name = "idx_customer", columnList = "customer_id") },
       uniqueConstraints = { @UniqueConstraint(columnNames = "order_number") })
@EntityListeners(AuditingEntityListener.class)  // for @CreatedDate etc.
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY) // DB auto-increment
    private Long id;

    @Column(name = "order_number", nullable = false, length = 50)
    private String orderNumber;

    @Enumerated(EnumType.STRING)          // store as "PENDING" not 0
    private OrderStatus status;

    @Column(precision = 10, scale = 2)
    private BigDecimal total;

    @CreatedDate                           // auto-set by Spring Data Auditing
    @Column(updatable = false)
    private Instant createdAt;

    @LastModifiedDate
    private Instant updatedAt;

    @Version
    private Long version;               // optimistic locking

    @ManyToOne(fetch = FetchType.LAZY)   // ALWAYS lazy for @ManyToOne
    @JoinColumn(name = "customer_id")
    private Customer customer;

    @OneToMany(mappedBy = "order",
               cascade = CascadeType.ALL,
               orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();

    // ── Always use business key for equals/hashCode ──
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Order other)) return false;
        return orderNumber != null && orderNumber.equals(other.orderNumber);
    }

    @Override
    public int hashCode() {
        return Objects.hashCode(orderNumber);
    }
}

N+1 Problem & How to Solve It

🔥

// ✅ Solution 2: @EntityGraph (declarative)
@EntityGraph(attributePaths = { "customer", "items", "items.product" })

// Generates single SQL with LEFT JOIN for customer, items, and product

Composite Primary Key

JavaComposite key using @EmbeddedId
// 1. Define the key class
@Embeddable
public class OrderItemId implements Serializable {
    private Long orderId;
    private Long productId;
    // equals + hashCode required!
}

// 2. Use it in entity
@Entity
public class OrderItem {
    @EmbeddedId
    private OrderItemId id;

    @ManyToOne
    @MapsId("orderId")         // maps orderId field of key to order FK
    private Order order;

    @ManyToOne
    @MapsId("productId")
    private Product product;

    private int quantity;
}
