01

The Core Intuition

Why two runners moving at different speeds always collide in a loop.

🧠 The Track Runner Mental Model

The Analogy That Makes It Click

Imagine two runners on a circular track. One runs at 1 lap/hour, the other at 2 laps/hour. No matter where they start — they will meet. The faster runner laps the slower one. Guaranteed.

This is the entire pattern. A slow pointer moves 1 step at a time. A fast pointer moves 2 steps. If there's a cycle — anywhere — they collide. If there's no cycle, the fast pointer falls off the end (hits null).

💡
Mental Model Lock-In

Fast = hare. Slow = tortoise. If the track loops, the hare comes back around and bumps into the tortoise. If the track ends, the hare falls off a cliff. The tortoise never gets to the cliff edge — it's still walking when the hare is already gone.

📐 Why They Always Meet (Intuitive Math)

You don't need to memorise a proof. You need to feel why this works. Here's the intuition in plain English:

STEP-BY-STEP INTUITION Let F = length of tail before cycle starts
Let C = length of the cycle
Let k = distance from cycle entry to meeting point

When slow enters the cycle, fast is already F steps inside it.
Fast is chasing slow. The gap closes by 1 step each iteration
(fast moves 2, slow moves 1 → gap shrinks by 1 each step).

∴ They MUST meet. The gap can never grow — only shrink or stay zero.
🎯
The Key Insight for Phase 2 (Finding Cycle Start)

When they meet: the distance from meeting point → cycle start equals the distance from head → cycle start. This is the magic that makes Floyd's Phase 2 work. Reset one pointer to head, keep one at meeting point, advance both at speed 1 — they meet exactly at the cycle start.

02

The Three Core Templates

Cycle detection · Finding cycle start · Finding the middle.

🔁 Template 1 — Cycle Detection

The Question It Answers

"Does this linked list have a cycle?"

If fast ever catches slow, yes. If fast hits null, no. Simple as that. This is LC #141.

Java LC #141 — Linked List Cycle (Detect) 
public boolean hasCycle(ListNode head) {
    // Edge case: empty or single node can't have a cycle
    if (head == null || head.next == null) return false;

    ListNode slow = head;
    ListNode fast = head;

    while (fast != null && fast.next != null) {
        slow = slow.next;          // 1 step
        fast = fast.next.next;     // 2 steps

        if (slow == fast) return true; // Pointers collided → cycle!
    }

    return false; // fast hit null → no cycle
}
⚠️
Common Trap: Check Both fast AND fast.next

Your while condition must be fast != null && fast.next != null. Checking only fast != null will throw a NullPointerException when you try fast.next.next on the last node.

📍 Template 2 — Finding the Cycle Start (Floyd's Phase 2)

The Two-Phase Algorithm

Phase 1 Run fast & slow until they meet. This confirms a cycle exists.

Phase 2 Reset one pointer to head. Move both at speed 1. Where they meet again = the cycle entry node.

Java LC #142 — Linked List Cycle II (Find Start) 
public ListNode detectCycle(ListNode head) {
    if (head == null || head.next == null) return null;

    ListNode slow = head, fast = head;

    // ── Phase 1: Find the meeting point ──
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) break; // meeting point found
    }

    // No cycle detected
    if (fast == null || fast.next == null) return null;

    // ── Phase 2: Find cycle start ──
    // Reset slow to head, keep fast at meeting point
    slow = head;
    while (slow != fast) {
        slow = slow.next; // both move 1 step now
        fast = fast.next;
    }

    return slow; // They meet at the cycle start node
}
💡

Why Phase 2 works: When fast & slow meet, the meeting point is exactly F steps from the cycle start (where F = tail length). Head is also F steps from cycle start. Move both at speed 1 from their respective positions — they cover equal distance and land on the start. Magic.

⚖️ Template 3 — Finding the Middle of a Linked List

When fast reaches the end, slow is at the middle. Fast travels 2x the distance, so when fast has covered the whole list, slow has covered half. It's like measuring half a rubber band — stretch it and fold.

Java LC #876 — Middle of the Linked List 
public ListNode middleNode(ListNode head) {
    ListNode slow = head, fast = head;

    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }

    return slow; // For even-length list: returns SECOND middle
}

// ── If you need the FIRST middle (for palindrome check) ──
public ListNode firstMiddle(ListNode head) {
    ListNode slow = head, fast = head.next; // ← fast starts 1 ahead

    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }

    return slow; // Returns FIRST middle for even-length list
}
⚠️
The Off-By-One Trap — Even-Length Lists

For list 1→2→3→4: fast=head → slow returns node 3 (second middle). fast=head.next → slow returns node 2 (first middle). Always clarify with your interviewer which middle they want. State this out loud.

🌀 Beyond Linked Lists — Cycle Detection in Value Space

The Key Generalisation

Fast & slow pointers work on any structure where you can define a "next" function. It doesn't have to be a physical linked list. If you can write f(x) and keep applying it, you can use Floyd's.

The Happy Number problem is the perfect example. You follow a number through a transformation sequence. If it loops → not happy. If it reaches 1 → happy.

Java LC #202 — Happy Number (Cycle in Value Space) 
public boolean isHappy(int n) {
    int slow = n;
    int fast = getNext(n);

    // fast reaching 1 = happy number found (exit before meeting)
    while (fast != 1 && slow != fast) {
        slow = getNext(slow);           // 1 transformation
        fast = getNext(getNext(fast));  // 2 transformations
    }

    return fast == 1; // true = happy, false = cycle detected
}

private int getNext(int n) {
    int sum = 0;
    while (n > 0) {
        int digit = n % 10;
        sum += digit * digit;
        n /= 10;
    }
    return sum;
}
✅
The Mental Shift

Think of getNext() as a .next pointer. The number sequence 19 → 82 → 68 → 100 → 1 is just a linked list in disguise. Floyd's algorithm doesn't care whether nodes are objects or integers.

Deep Dive & Distinctions

Full walkthrough of LC #142 + why this isn't just "Two Pointers."

🔬 Full Walkthrough: LC #142 — Linked List Cycle II

Problem Statement

Given a linked list, return the node where the cycle begins. If no cycle, return null.

Input: 3 → 2 → 0 → -4, where -4 points back to node 2 (index 1).

Output: Node with value 2.

1
Recognise the Pattern
Problem says "linked list" + asks about cycles → Fast & Slow. The ask is the start node, not just existence → need Phase 2.
2
Phase 1 — Run Until Meeting
slow=3, fast=3 → slow=2, fast=0 → slow=0, fast=2 → slow=-4, fast=-4. They meet at -4.
3
Phase 2 — Reset slow to Head
slow=3, fast=-4. Both move 1 step: slow=2, fast=2. They meet at node 2. ✅ That's our cycle start.
4
Verify the Math
Tail length F=1 (just node 3). From meeting point -4 to cycle start 2 = 1 step. From head 3 to cycle start 2 = 1 step. Equal. ✅
🗣️
What to Say in the Interview

"I'll use Floyd's cycle detection. Phase 1 finds a meeting point — O(n). Phase 2 exploits the mathematical property that the tail length equals the distance from meeting point to cycle start. Overall O(n) time, O(1) space. No hash set needed."

⚖️ Fast & Slow vs Same-Direction Two Pointers

Both patterns use two pointers moving in the same direction. Junior devs often confuse them. The distinction is sharp once you know it.

Dimension Fast & Slow (Floyd's) Same-Direction Two Pointers
Speed Ratio Always 2:1 (fast = 2× slow) Variable — often 1:1 with a gap (window)
Primary Goal Detect cycles, find middle Remove Nth from end, find pairs
Structure Required Any "follow the next" structure Array or linked list with indexing
Key Question "Do they collide?" "Does a k-step gap give me info?"
Stops When Pointers meet OR fast hits null One pointer exits the structure
Classic Problems LC #141, #142, #202, #287 LC #19 (Remove Nth), #26 (Remove Dups)

✅ Use Fast & Slow When

  • Problem involves cycles or repetition
  • You need the middle of a linked list
  • Structure could be infinite (no guaranteed end)
  • You need O(1) space and can't afford a HashSet
  • "Follow next pointer" is the core operation

✅ Use Same-Dir Two Pointers When

  • You need an element k steps before the end
  • Structure is finite and acyclic
  • You're maintaining a sliding window
  • You need two views of a sequence simultaneously
  • Think: "gap" not "speed ratio"

Recognition & Practice

Signals that scream "Floyd's" + 8 problems to build muscle memory.

🎯 Recognition Signals

🔴 Strong Signals

  • Problem says "detect cycle" explicitly
  • Asks for cycle entry node
  • Linked list problem, O(1) space required
  • Asks to find the middle of a linked list
  • Number sequence that might loop (Happy Number)

🟡 Softer Signals

  • Palindrome linked list (need middle first)
  • "Find duplicate without extra space" in array where values are indices
  • Circular array — does traversal loop?
  • Reorder list — split at middle, reverse second half
  • Any problem with a functional "next state" that repeats

Decision Moment — "Should I Use Floyd's or a HashSet?"

HashSet approach is always simpler to code (O(n) space). Use Floyd's when: the problem explicitly requires O(1) space, or when the "visited" set would be too large (infinite domain). In interviews, mentioning both and then choosing Floyd's shows depth.

🌟 Bonus Deep Dive: LC #287 — Find the Duplicate

This problem looks like a math or sort problem. The insight: treat the array as a linked list where index i points to nums[i]. Since values are in range [1, n] and there's a duplicate, there must be a cycle. Floyd's finds it.

Java LC #287 — Find the Duplicate (Array as Implicit LinkedList) 
public int findDuplicate(int[] nums) {
    // Treat array as: index i → nums[i] → nums[nums[i]] → ...
    // Duplicate value creates two indices that point to same "next"
    // That's your cycle entry point!

    int slow = nums[0];
    int fast = nums[0];

    // Phase 1: Find meeting point
    do {
        slow = nums[slow];              // follow once
        fast = nums[nums[fast]];        // follow twice
    } while (slow != fast);

    // Phase 2: Find cycle start (= duplicate number)
    slow = nums[0];                   // reset to "head"
    while (slow != fast) {
        slow = nums[slow];
        fast = nums[fast];
    }

    return slow; // The duplicate value
}
🎯
Interview Tip: The "No Extra Space" Trigger

Whenever a problem says "find duplicate in array of size n+1 where values are 1 to n — without modifying the array and using O(1) space" — that's Floyd's in disguise. This is a famous problem at top companies. Know it cold.
