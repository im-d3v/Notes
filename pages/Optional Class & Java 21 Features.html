1. Optional Class

Java 8 — Null safety without NullPointerException

1 What is Optional? — Why was it introduced?

Before Java 8, developers frequently encountered NullPointerException (NPE) — famously called "the billion-dollar mistake" by Tony Hoare. Optional is a container object that may or may not contain a non-null value.

❌ Before Optional (NPE Prone)
✅ With Optional (Safe)
Forces developers to remember null checks
Makes null possibility explicit in return type
Runtime NPE without warning
Compile-time awareness of absent values
Nested if-null checks everywhere
Elegant chaining with map, flatMap, filter
⚠️
Important: Optional is NOT meant to replace every null check. It's designed for method return types to signal "this might not return a value". Don't use Optional for fields, constructor params, or method parameters.
2 Creating Optional — 3 Factory Methods
MethodUse WhenThrows if null?
Optional.of(value) Value is guaranteed non-null Yes — NullPointerException
Optional.ofNullable(value) Value may or may not be null No — returns empty Optional
Optional.empty() Explicitly represent "no value" N/A — creates empty Optional
Java — Creating Optional
// Optional.of() — use when value is NEVER null
Optional<String> opt1 = Optional.of("Hello");           // ✅ OK
Optional<String> opt2 = Optional.of(null);            // ❌ NullPointerException!

// Optional.ofNullable() — use when value CAN be null
String name = getUserNameFromDB();                     // might return null
Optional<String> opt3 = Optional.ofNullable(name);     // ✅ Safe always

// Optional.empty() — represent absence of value
Optional<String> opt4 = Optional.empty();                // empty container

// Check if value present
boolean hasValue = opt1.isPresent();                    // true
boolean isEmpty  = opt4.isEmpty();                      // true (Java 11+)
3 All Optional Methods — Complete Reference
isPresent()
Returns true if value is present. Use sparingly — prefer functional style.
isEmpty() Java 11
Returns true if no value present. Opposite of isPresent().
get()
Returns value if present, else throws NoSuchElementException. Avoid if possible!
orElse(default)
Returns value if present, else returns the given default. Default is ALWAYS evaluated.
orElseGet(Supplier)
Returns value if present, else invokes Supplier. Lazy — Supplier only runs if empty.
orElseThrow()
Returns value or throws NoSuchElementException. Can pass custom exception Supplier.
map(Function)
Transforms value if present. Returns Optional of result or empty Optional.
flatMap(Function)
Like map but for functions that return Optional (avoids Optional<Optional>).
filter(Predicate)
If value present and matches predicate, returns Optional with value, else empty.
ifPresent(Consumer)
Executes Consumer if value is present. Does nothing if empty.
ifPresentOrElse() Java 9
Execute Consumer if present, else run empty-action Runnable.
or(Supplier) Java 9
If empty, use Supplier to provide another Optional (chain fallbacks).
stream() Java 9
Returns Stream of one element if present, else empty Stream.
Java — All Methods in Action
Optional<String> opt = Optional.ofNullable(getName());

// ─── orElse vs orElseGet ───────────────────────────────────────
String r1 = opt.orElse("Guest");               // "Guest" always created
String r2 = opt.orElseGet(() -> generateDefault()); // lazy: called only if empty

// ─── orElseThrow ───────────────────────────────────────────────
String r3 = opt.orElseThrow(() -> new UserNotFoundException("User not found"));

// ─── map + orElse chain ────────────────────────────────────────
String upper = opt
    .map(String::toUpperCase)
    .orElse("UNKNOWN");

// ─── filter ────────────────────────────────────────────────────
Optional<String> adult = Optional.ofNullable(age)
    .filter(a -> a >= 18)
    .map(a -> "Age: " + a);

// ─── ifPresentOrElse (Java 9+) ─────────────────────────────────
opt.ifPresentOrElse(
    v -> System.out.println("Found: " + v),
    ()  -> System.out.println("Not found")
);

// ─── or (Java 9+) — chain fallback Optionals ──────────────────
Optional<String> result = opt
    .or(() -> getFromCache())
    .or(() -> getFromDefault());

// ─── stream (Java 9+) — integrate with Stream API ─────────────
List<Optional<String>> opts = List.of(opt1, opt2, opt3);
List<String> values = opts.stream()
    .flatMap(Optional::stream)   // only present values
    .collect(Collectors.toList());
4 🔥 orElse vs orElseGet — Most Common Interview Trap!
🚨
Key Difference: orElse() always evaluates the default — even if Optional has a value! orElseGet() only evaluates the Supplier if Optional is empty (lazy evaluation).
Java — The Trap Explained
public class OrElseTrap {

    static String expensiveDefault() {
        System.out.println("⚡ Expensive call executed!");
        return "DEFAULT";
    }

    public static void main(String[] args) {
        Optional<String> opt = Optional.of("REAL_VALUE"); // has value!

        // orElse — ALWAYS evaluates expensiveDefault()
        String r1 = opt.orElse(expensiveDefault());
        // Prints: ⚡ Expensive call executed!  ← even though opt has value!
        // r1 = "REAL_VALUE" but the method still ran — waste!

        // orElseGet — LAZY: only calls if empty
        String r2 = opt.orElseGet(OrElseTrap::expensiveDefault);
        // Prints: nothing — method NOT called because opt has value
        // r2 = "REAL_VALUE" — efficient!
    }
}
Output of orElse block:
⚡ Expensive call executed!
r1 = REAL_VALUE

Output of orElseGet block:
(nothing printed)
r2 = REAL_VALUE
🎯 Interview Tip "Use orElseGet() when the default value is expensive to compute (DB call, API call, object creation). Use orElse() only for cheap, already-computed constants like orElse("N/A")."

    return Optional.ofNullable(System.getenv("API_KEY"))

public void patchUser(Long id, UserPatchDTO patch) {
    User user = userRepository.findById(id).orElseThrow();
    Optional.ofNullable(patch.getEmail()).ifPresent(user::setEmail);
    Optional.ofNullable(patch.getPhone()).ifPresent(user::setPhone);
    userRepository.save(user);
}

6 flatMap — Avoid Optional Nesting
Java — map vs flatMap with Optional
public class Order {
    private Optional<Customer> customer; // field itself is Optional
    public Optional<Customer> getCustomer() { return customer; }
}

public class Customer {
    private Optional<String> email;
    public Optional<String> getEmail() { return email; }
}

// ❌ BAD — map creates Optional<Optional<Customer>>
Optional<Optional<Customer>> bad = Optional.of(order).map(Order::getCustomer);

// ✅ GOOD — flatMap flattens Optional<Optional<T>> → Optional<T>
Optional<String> email = Optional.of(order)
    .flatMap(Order::getCustomer)        // Optional<Customer>
    .flatMap(Customer::getEmail)         // Optional<String>
    .filter(e -> e.contains("@"));      // Optional<String>

2. Serialization & Deserialization

Object → Bytes → Object, exceptions, best practices

1 What is Serialization? — Basic to Deep
📤
Serialization
Converting a Java object into a byte stream so it can be:
  • Saved to a file
  • Sent over a network
  • Stored in database/cache
  • Shared between JVMs
📥
Deserialization
Converting the byte stream back to a Java object. The reverse process.
  • Read from file
  • Received from network
  • Reconstructed from stored bytes
Java — Basic Serialization
import java.io.*;

// Step 1: Class MUST implement Serializable
public class Employee implements Serializable {

    // ALWAYS define serialVersionUID — explained below
    private static final long serialVersionUID = 1L;

    private String name;
    private int age;
    private transient String password;  // transient = NOT serialized

    public Employee(String name, int age, String password) {
        this.name = name;
        this.age = age;
        this.password = password;
    }
    // getters, toString...
}

// Step 2: SERIALIZE — write object to file
try (ObjectOutputStream oos = new ObjectOutputStream(
        new FileOutputStream("employee.ser"))) {
    oos.writeObject(new Employee("Alice", 30, "secret123"));
    System.out.println("✅ Serialized!");
} catch (IOException e) {
    e.printStackTrace();
}

// Step 3: DESERIALIZE — read object from file
try (ObjectInputStream ois = new ObjectInputStream(
        new FileInputStream("employee.ser"))) {
    Employee emp = (Employee) ois.readObject();
    System.out.println(emp.getName());      // "Alice"
    System.out.println(emp.getAge());       // 30
    System.out.println(emp.getPassword());  // null — transient not saved!
} catch (IOException | ClassNotFoundException e) {
    e.printStackTrace();
}
2 serialVersionUID — Why It Matters

serialVersionUID is a unique identifier for each Serializable class. It's used during deserialization to verify that the sender and receiver are compatible.

1
Class serialized with serialVersionUID = 1L
Object written to file successfully
2
Developer adds a new field to class
If no serialVersionUID declared, JVM auto-computes a new one
3
Deserialization tries to read old file
UIDs don't match → InvalidClassException thrown!
Java — serialVersionUID Exception
// ❌ WITHOUT serialVersionUID — DANGEROUS
public class User implements Serializable {
    // JVM auto-generates UID from class structure
    // Adding a field changes the auto-generated UID
    // → InvalidClassException during deserialization!
    private String name;
    // later added: private String email; ← breaks old serialized files!
}

// ✅ WITH serialVersionUID — SAFE
public class User implements Serializable {
    private static final long serialVersionUID = 1L;
    private String name;
    private String email; // Added later — UID same → compatible!
    // Old serialized files still load. New field gets default value (null)
}
💡
Exceptions in Serialization:
  • NotSerializableException — class doesn't implement Serializable
  • InvalidClassException — serialVersionUID mismatch
  • ClassNotFoundException — during deserialization, class not found in classpath
  • StreamCorruptedException — corrupted byte stream
3 transient Keyword + Custom Serialization
Java — transient + writeObject/readObject
public class SecureEmployee implements Serializable {
    private static final long serialVersionUID = 1L;

    private String name;
    private transient String password;     // NOT serialized
    private transient Connection dbConn;    // NOT serialized (not serializable!)

    // Custom serialization logic
    private void writeObject(ObjectOutputStream oos) throws IOException {
        oos.defaultWriteObject(); // write non-transient fields
        // Optionally encrypt and write password
        oos.writeObject(encrypt(password));
    }

    private void readObject(ObjectInputStream ois)
            throws IOException, ClassNotFoundException {
        ois.defaultReadObject(); // read non-transient fields
        // Decrypt password when reading back
        this.password = decrypt((String) ois.readObject());
    }
}

// transient summary:
// ✅ Use for sensitive data (password, tokens)
// ✅ Use for non-serializable objects (threads, connections, streams)
// ✅ Use for derived/calculated fields (no need to store)
// ⚠️  Transient fields get default values on deserialization (null, 0, false)

3. Records Java 16 — Stable

Concise, immutable data carriers — Java's answer to Lombok @Value

1 What is a Record? — The Problem It Solves

Before Records, creating a simple data class required enormous boilerplate: private fields, constructor, getters, equals, hashCode, toString. Records eliminate ALL of this.

❌ Traditional Class (~40 lines)
✅ Record (1 line!)
public class Point {
  private final int x;
  private final int y;
  public Point(int x, int y) {
    this.x = x;
    this.y = y;
  }
  public int x() { return x; }
  public int y() { return y; }
  @Override public boolean equals(Object o){...}
  @Override public int hashCode(){...}
  @Override public String toString(){...}
}
public record Point(int x, int y) {}

// Auto-generated by compiler:
// ✅ Private final fields
// ✅ Canonical constructor
// ✅ Accessor methods x(), y()
// ✅ equals() — value-based
// ✅ hashCode() — value-based
// ✅ toString() — Point[x=1, y=2]
2 Records — Complete Feature Set
Java — Records All Features
// ── Basic Record Declaration ───────────────────────────────────
public record Person(String name, int age) {

    // ① Compact Canonical Constructor (validation)
    // No need to declare parameters — automatically assigned
    public Person {
        if (age < 0) throw new IllegalArgumentException("Age cannot be negative");
        if (name == null) throw new NullPointerException("Name required");
        name = name.trim();  // can normalize here
    }

    // ② Explicit Canonical Constructor (full control)
    // Use only when you need different logic:
    // public Person(String name, int age) { ... }

    // ③ Static fields allowed
    static final String SPECIES = "Human";

    // ④ Static methods allowed
    public static Person guest() {
        return new Person("Guest", 0);
    }

    // ⑤ Instance methods allowed
    public boolean isAdult() {
        return age >= 18;
    }

    // ⑥ Can override accessor
    @Override
    public String name() {
        return "Mr/Ms. " + name;  // customize accessor
    }

    // ⑦ Can override toString
    @Override
    public String toString() {
        return "Person{" + name + ", age=" + age + "}";
    }

    // ❌ NOT ALLOWED in records:
    // - instance fields (beyond components)
    // - extending another class (implicitly extends Record)
    // - non-final components
}

// ── Usage ─────────────────────────────────────────────────────
Person p1 = new Person("Alice", 30);
Person p2 = new Person("Alice", 30);

// Accessors — note: getName() does NOT exist, use name()
p1.name();           // "Alice"
p1.age();            // 30

// Value-based equality
p1.equals(p2);       // true — compares field VALUES, not reference

p1.toString();       // "Person{Alice, age=30}" (overridden)

// Records implement Serializable? NO by default
// Add explicitly: public record Person(...) implements Serializable { }

// ── Records with Generics ──────────────────────────────────────
public record Pair<A, B>(A first, B second) {}
Pair<String, Integer> pair = new Pair<>("hello", 42);

// ── Records can implement interfaces ──────────────────────────
public record Product(String name, double price) implements Comparable<Product> {
    @Override
    public int compareTo(Product other) {
        return Double.compare(this.price, other.price);
    }
}

// ── Local Records (Java 16+) ────────────────────────────────────
public void processOrders() {
    record OrderSummary(String id, double total) {}  // local to method!
    List<OrderSummary> summaries = orders.stream()
        .map(o -> new OrderSummary(o.getId(), o.getTotal()))
        .collect(Collectors.toList());
}
3 Records — Rules, Restrictions & When to Use
✅
Records CAN:
  • • Implement interfaces
  • • Have static fields & methods
  • • Have instance methods
  • • Be generic
  • • Have multiple constructors
  • • Be declared locally in a method
  • • Implement Serializable
  • • Be annotated
❌
Records CANNOT:
  • • Extend other classes (implicit extends Record)
  • • Declare additional instance fields
  • • Be abstract
  • • Have mutable components (fields are final)
  • • Be used as JPA @Entity (Hibernate needs mutable)
🎯
When to use Records (Real Scenarios):
  • DTOs (Data Transfer Objects) — record UserDTO(String name, String email) {}
  • API Response/Request bodies
  • Value objects in domain model
  • Temporary intermediate data in streams
  • Map keys (need equals/hashCode)
  • Configuration holders
🎯 Interview Tip "Records are NOT replacements for all classes. They're for pure data carriers. Don't use records for: JPA Entities (need mutable + no-arg constructor), classes with complex behavior, or Spring components (@Service, @Component)."

4. Sealed Classes Java 17 — Stable

Restricted class hierarchies — you control who can extend

1 What are Sealed Classes? — The Problem They Solve

Before sealed classes, anyone could extend your class from anywhere in the codebase. Sealed classes give you explicit control over who can extend or implement your class/interface.

❌
Without sealed:
Anyone can extend Shape — including malicious or unintended subclasses. Can't exhaustively handle all types safely.
✅
With sealed:
Only permitted subclasses can extend. Compiler knows ALL possible subtypes. Works beautifully with Pattern Matching switch.
Java — Sealed Classes Complete Example
// ── SEALED CLASS — controls who can extend ────────────────────
public sealed class Shape
    permits Circle, Rectangle, Triangle {  // ONLY these 3 can extend
    public abstract double area();
}

// ── Permitted subclasses have 3 options ───────────────────────

// Option 1: final — no further extension allowed
public final class Circle extends Shape {
    private final double radius;
    public Circle(double radius) { this.radius = radius; }
    @Override public double area() { return Math.PI * radius * radius; }
}

// Option 2: sealed — further restricted extension
public sealed class Rectangle extends Shape
    permits Square {   // only Square can extend Rectangle
    protected final double width, height;
    public Rectangle(double w, double h) { width = w; height = h; }
    @Override public double area() { return width * height; }
}

public final class Square extends Rectangle {
    public Square(double side) { super(side, side); }
}

// Option 3: non-sealed — opens back up for extension
public non-sealed class Triangle extends Shape {
    // Anyone can extend Triangle now
    private final double base, height;
    public Triangle(double b, double h) { base = b; height = h; }
    @Override public double area() { return 0.5 * base * height; }
}

// ❌ This will NOT compile — GreenShape not in permits list!
// public class GreenShape extends Shape {}
2 Sealed Classes + Pattern Matching — The Power Combo

Sealed classes pair perfectly with Pattern Matching in switch (Java 21). The compiler knows ALL permitted types and can warn you about missing cases.

Java — Pattern Matching with Sealed
// ── Sealed Interface for Result type ──────────────────────────
public sealed interface ApiResult<T>
    permits ApiResult.Success, ApiResult.Failure, ApiResult.Loading {

    record Success<T>(T data) implements ApiResult<T> {}
    record Failure<T>(String error, int code) implements ApiResult<T> {}
    record Loading<T>() implements ApiResult<T> {}
}

// ── Usage with switch pattern matching (Java 21) ───────────────
public void handleResult(ApiResult<User> result) {
    switch (result) {
        case ApiResult.Success<User>(var user)  ->
            System.out.println("User: " + user.name());
        case ApiResult.Failure<User>(var err, var code) ->
            System.out.println("Error " + code + ": " + err);
        case ApiResult.Loading<User> l ->
            System.out.println("Loading...");
        // No default needed! Compiler knows all cases covered.
    }
}

// ── Real World: Domain Events ──────────────────────────────────
public sealed interface PaymentEvent
    permits PaymentInitiated, PaymentCompleted, PaymentFailed, PaymentRefunded {}

public record PaymentInitiated(String txnId, double amount)
    implements PaymentEvent {}

public record PaymentCompleted(String txnId, String receiptNo)
    implements PaymentEvent {}

public record PaymentFailed(String txnId, String reason)
    implements PaymentEvent {}

public record PaymentRefunded(String txnId, double amount)
    implements PaymentEvent {}

// Handler: compiler ensures all events handled!
public String describeEvent(PaymentEvent event) {
    return switch (event) {
        case PaymentInitiated e  -> "Initiated: "  + e.txnId();
        case PaymentCompleted e  -> "Completed: "  + e.receiptNo();
        case PaymentFailed e     -> "Failed: "     + e.reason();
        case PaymentRefunded e   -> "Refunded: "   + e.amount();
    };
}
🎯 Interview Tip "Sealed classes are designed to work with Pattern Matching in switch expressions (Java 21). They're great for sum types / discriminated unions — modeling a fixed set of possible states. Use case: API result types (Success/Failure), domain events, state machines."

// ── Switch Pattern Matching ────────────────────────────────────
public String format(Object obj) {
    return switch (obj) {
        case Integer i   -> "int: "    + i;
        case Double d    -> "double: " + d;
        case String s    -> "String: " + s.toUpperCase();
        case null        -> "null";      // null case (new!)
        default          -> "other: "   + obj;
    };
}

// ── Guarded Patterns ──────────────────────────────────────────
public String classify(Object o) {
    return switch (o) {
        case Integer i when i > 0  -> "Positive int";
        case Integer i when i < 0  -> "Negative int";
        case Integer i             -> "Zero";
        case String s when s.isEmpty() -> "Empty string";
        case String s               -> "String: " + s;
        default                     -> "Other";
    };
}

6. Sequenced Collections Java 21

New interfaces for ordered collections — first and last access

1 What Problem Does It Solve?

Before Java 21, accessing the first or last element of collections was inconsistent and awkward:

Java — The Problem (Before Java 21)
// ── Getting FIRST element (inconsistent across collections!) ──
list.get(0);                          // ArrayList — index 0
linkedList.getFirst();               // LinkedList — special method
deque.peekFirst();                   // Deque — another name
sortedSet.first();                   // SortedSet — yet another!
// 😵 No unified API!

// ── Getting LAST element ──────────────────────────────────────
list.get(list.size() - 1);           // ArrayList — awkward
linkedList.getLast();               // LinkedList
sortedSet.last();                    // SortedSet
// 😵 Still inconsistent!

// ── Getting REVERSED view ─────────────────────────────────────
Collections.reverse(list);          // mutates original — WRONG!
// No clean way to get reversed view!
2 New Interfaces — SequencedCollection, SequencedSet, SequencedMap
Java 21 — Sequenced Collection Interfaces
// ── 3 New Interfaces added to Collection hierarchy ────────────

// 1. SequencedCollection — for List, Deque
interface SequencedCollection<E> extends Collection<E> {
    SequencedCollection<E> reversed();  // NEW — returns reversed VIEW
    // Also inherits first/last via addFirst/addLast/getFirst/getLast/removeFirst/removeLast
}

// 2. SequencedSet — for LinkedHashSet, SortedSet
interface SequencedSet<E> extends SequencedCollection<E>, Set<E> {
    SequencedSet<E> reversed();
}

// 3. SequencedMap — for LinkedHashMap, SortedMap
interface SequencedMap<K,V> extends Map<K,V> {
    Map.Entry<K,V> firstEntry();    // NEW
    Map.Entry<K,V> lastEntry();     // NEW
    void putFirst(K k, V v);        // NEW
    void putLast(K k, V v);         // NEW
    SequencedMap<K,V> reversed();   // NEW — reversed view
    // Also: sequencedKeySet(), sequencedValues(), sequencedEntrySet()
}

// ── Using Sequenced Collections (Java 21) ─────────────────────
List<String> list = new ArrayList<>(List.of("a", "b", "c"));

// ✅ Unified API now!
list.getFirst();                   // "a"  — NEW uniform method

7. Virtual Threads Java 21 — Stable

Project Loom — Revolutionary concurrency for high-throughput apps

1 The Problem with Platform Threads (Before Virtual Threads)
❌
Platform Threads (Old)
  • 1 Java thread = 1 OS thread
  • Each thread ~1-2MB stack memory
  • OS can manage ~thousands of threads
  • Blocking I/O wastes OS thread resources
  • Thread creation is expensive (slow)
  • Cannot handle millions of concurrent requests
✅
Virtual Threads (New)
  • Managed by JVM, not OS
  • Very small memory footprint (~few KB)
  • JVM can handle millions of virtual threads
  • Blocking unmounts from carrier thread
  • Creation is nearly free (very fast)
  • True high-throughput I/O concurrency
💡
Key Mental Model: Virtual threads are scheduled onto a small pool of carrier threads (platform threads). When a virtual thread blocks on I/O, it unmounts from its carrier thread, freeing that carrier to run other virtual threads. When I/O completes, it remounts on any available carrier thread.

⚠️
Virtual Thread Pitfalls:
  • Pinning: Using synchronized blocks inside virtual threads "pins" them to carrier thread — defeats the purpose. Use ReentrantLock instead.
  • ThreadLocal abuse: Millions of virtual threads with heavy ThreadLocals = memory explosion. Use ScopedValue (Java 21 preview) instead.
  • CPU-intensive tasks: Virtual threads don't help for CPU-bound work — platform threads are better.
4 Virtual Threads — Real World Use Case (Interview Answer)
Java — Virtual Thread Real-World Example
// ── Scenario: Handle 10,000 concurrent HTTP API calls ─────────
// Before Virtual Threads — thread pool limited to ~200 threads
public class OldApproach {
    private final ExecutorService pool = Executors.newFixedThreadPool(200);

    public void fetchUserData(List<Long> userIds) {
        // 200 concurrent requests MAX — rest queue up!
        userIds.forEach(id -> pool.submit(() -> callExternalApi(id)));
    }
}

// After Virtual Threads — handles all 10,000 concurrently!
public class NewApproach {
    public void fetchUserData(List<Long> userIds) {
        try (ExecutorService ex = Executors.newVirtualThreadPerTaskExecutor()) {
            // 10,000 virtual threads — no problem!
            userIds.forEach(id -> ex.submit(() -> callExternalApi(id)));
        }
        // When blocked waiting for HTTP response,
        // carrier thread freed to run OTHER virtual threads!
    }
}

// ── Spring Boot 3.2+ — One line to enable ────────────────────
// application.properties
// spring.threads.virtual.enabled=true
// → Tomcat, async, @Async all use virtual threads automatically!

// ── Structured Concurrency (Java 21 Preview) ──────────────────
// Run multiple tasks, wait for all, cancel on failure
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Future<User>     user  = scope.fork(() -> fetchUser(userId));
    Future<Order>    order = scope.fork(() -> fetchOrder(orderId));
    Future<Product>  prod  = scope.fork(() -> fetchProduct(productId));

    scope.join();         // wait for all
    scope.throwIfFailed(); // propagate exception if any failed

    // All 3 ran concurrently as virtual threads!
    return new Dashboard(user.get(), order.get(), prod.get());
}
🎯 Interview Answer for "What are Virtual Threads?" "Virtual threads are JVM-managed lightweight threads introduced in Java 21 as part of Project Loom. Unlike platform threads (which map 1:1 to OS threads), virtual threads are multiplexed onto a small pool of carrier threads. When a virtual thread performs blocking I/O, it unmounts from the carrier thread, freeing it to run other virtual threads. This enables writing simple, readable synchronous-style code that scales to millions of concurrent requests — perfect for I/O-bound applications like REST APIs."
