The Problem Space

Why static URLs break in a dynamic microservices world — and the two fundamental solutions.

§1 — The Service Discovery Problem

The Monolith World vs The Microservices World

In a monolith, all code runs in one process. orderService.placeOrder() is a direct method call — no network, no address, no configuration. You always know where everything is.

In microservices on Docker/Kubernetes, every service is an independent process running on some machine somewhere. And here's the problem:

💥
IP addresses in Kubernetes are ephemeral

Every time a pod restarts, scales up, crashes, or gets rescheduled — it gets a new IP address. If you hardcode http://192.168.1.42:8080, you're already wrong by the time your code ships.

The Mental Model — A Self-Updating Phone Directory

Imagine a hotel with 100 rooms. Guests (services) check in and out constantly. You don't know which room your friend is in — you call the front desk (service registry), they tell you the current room number.

If your friend moves to a different room, the front desk updates their record. You never need to memorize room numbers — you always ask fresh.

The Two Actors

Service Registry

A database that maps service names to IP:port. Services register themselves on startup and deregister on shutdown. Examples: Eureka Consul Zookeeper

Service Discovery

The act of looking up a service's address before calling it. The key question: who does the lookup — the client or an intermediary? That's what the next section is about.

§2 — Client-Side vs Server-Side Discovery

The Core Question

When Service A needs to call Service B, who is responsible for finding Service B's current address? There are exactly two answers, and they lead to fundamentally different architectures.

Client-Side Discovery — "You do your own homework"

1
Service A asks the Registry
Service A calls Eureka: "Give me all instances of SERVICE-B". Gets back a list of IPs and ports.
2
Service A picks an instance
Using its own load balancing logic (Ribbon / Spring Cloud LB), it selects one instance — e.g., round robin.
3
Service A calls Service B directly
No intermediary. Direct network call from A → B. Fast. One network hop.

Server-Side Discovery — "Delegate to the router"

1
Service A calls the Load Balancer
Service A doesn't know anything about Service B's location. It just calls http://load-balancer/service-b/...
2
LB consults the Registry
The Load Balancer queries the service registry, gets the instance list, picks one.
3
LB routes to Service B
Service A is completely unaware. Extra hop but zero client complexity.
DimensionClient-Side DiscoveryServer-Side Discovery
Who finds the instance? The calling service (client) The load balancer / router
Network hops 1 hop (direct) 2 hops (via LB)
Client complexity High — needs discovery lib Low — just HTTP call
Language agnostic ❌ Each language needs a registry client ✅ Any HTTP client works
LB single point of failure ✅ No central LB bottleneck ⚠️ LB must be HA

Example tools Eureka + Ribbon, Spring Cloud LB AWS ALB, K8s Service + kube-proxy
💡
Kubernetes is Server-Side by default

A Kubernetes Service resource with kube-proxy IS server-side discovery. You call http://order-service:8080, Kubernetes figures out which pod to send it to. Most K8s users never write a line of service discovery code.

The Netflix Stack

§3 — Eureka: Netflix Service Registry Deep Dive

What Eureka Actually Is

A REST-based service registry developed by Netflix. Services register themselves, send heartbeats to prove they're alive, and Eureka keeps an up-to-date map of every service → its IP:port(s).

Architecture: one or more Eureka Server nodes (clustered for HA) + Eureka Client embedded in every application service.

Heartbeat every 30 seconds → no heartbeat for 90 seconds → service deregistered

Eureka Client Lifecycle — Step by Step

1
Register on Startup
POST /eureka/apps/{appName} — sends hostname, IP, port, health URL, metadata. Eureka stores it.
2
Send Heartbeat Every 30s
PUT /eureka/apps/{appName}/{instanceId} — "I'm still alive." Resets the 90s eviction timer.
3
Cache the Registry Locally
The Eureka client fetches the full registry and caches it locally (refresh every 30s). This is critical — your service doesn't call Eureka for every outgoing request. It uses the local cache. Eureka failure doesn't instantly break your calls.
4
Deregister on Graceful Shutdown
DELETE /eureka/apps/{appName}/{instanceId} — tells Eureka "I'm going down." Eureka removes it immediately, not after 90s timeout.

⚠️
Self-Preservation Mode — The Most Misunderstood Eureka Behavior

If Eureka stops receiving heartbeats from a large percentage of services simultaneously, it assumes there's a network partition (not mass service failure). It enters self-preservation mode: it stops evicting service registrations, even stale ones.

This is an AP choice (Availability over Consistency from CAP theorem). The logic: it's better to return a possibly-stale address than to return nothing at all. But it means callers may occasionally get a dead instance — your retry logic must handle this.

🔍
When to pick Consul over Eureka

1. You need strong consistency (CP) — Consul uses Raft consensus. 2. You need active health checks (Eureka relies on passive heartbeats). 3. You have polyglot services (Go, Python, Node) — Consul has native clients for everything. 4. You need secret management alongside service discovery (Consul KV + Vault integration).

§4 — Ribbon: Client-Side Load Balancing

Eureka gives you a list. Ribbon picks one.

Eureka tells you: "SERVICE-B is running at [192.168.1.10:8080, 192.168.1.11:8080, 192.168.1.12:8080]". Now what? You need a rule to decide which one to call. That's Ribbon.

Ribbon is a client-side IPC (inter-process communication) library. It sits inside your service. For every outgoing request to SERVICE-B, it applies a load balancing rule to select one instance from the list Eureka provided.

🔄
Round Robin Rule
Default. Cycles through instances in order. Request 1 → instance A, Request 2 → instance B, etc. Simple and fair.
🚦
Availability Filtering Rule
Skips instances where the circuit breaker is open, or that have too many concurrent connections (configurable threshold).
⚡
Weighted Response Time
Slower instances get fewer requests. Calculates a weight inversely proportional to average response time. Best for heterogeneous hardware.
🌍
Zone Affinity Rule
Prefers instances in the same availability zone. Reduces cross-AZ latency and data transfer costs in cloud environments.
JavaRestTemplate with Ribbon (Classic Pattern)
@Configuration
public class AppConfig {

    @Bean
    @LoadBalanced  // ← This single annotation enables Ribbon!
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}

@Service
public class OrderService {

    @Autowired
    private RestTemplate restTemplate;

    public User getUserById(Long id) {
        // Use SERVICE NAME, not IP:port — Ribbon resolves it
        String url = "http://USER-SERVICE/users/" + id;
        return restTemplate.getForObject(url, User.class);
        // Ribbon intercepts this, asks Eureka for USER-SERVICE instances,
        // picks one via round-robin, replaces URL, makes the actual call
    }
}
⚠️
Ribbon is in maintenance mode — Use Spring Cloud LoadBalancer instead

Spring Cloud deprecated Ribbon in 2020. The modern replacement is spring-cloud-starter-loadbalancer. The @LoadBalanced annotation still works the same way, but it now uses Spring Cloud LoadBalancer under the hood. For new projects, exclude Ribbon explicitly and use the new module.

JavaSpring Cloud LoadBalancer — Custom Rule
// Custom Load Balancer configuration (modern approach)
@Configuration
public class LoadBalancerConfig {

    @Bean
    ReactorLoadBalancer<ServiceInstance> randomLoadBalancer(
            Environment environment,
            LoadBalancerClientFactory loadBalancerClientFactory) {
        String name = environment.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        // RandomLoadBalancer instead of default RoundRobinLoadBalancer
        return new RandomLoadBalancer(
            loadBalancerClientFactory.getLazyProvider(name, ServiceInstanceListSupplier.class),
            name
        );
    }
}

§5 — API Gateway Deep Dive

The Mental Model — Airport Security

Every passenger (request) entering the airport (system) goes through one checkpoint. Security checks ID (authentication), customs limits (rate limiting), and routes you to the right terminal (service routing). No one sneaks through a side door.

An API Gateway is the single entry point for all external client traffic. Clients never know or call individual services directly.

What an API Gateway Does — Full Feature List

🗺️
Request Routing
/users → User Service, /orders → Order Service. Based on path, host, headers, method.
🔐
Authentication
Validate JWT/OAuth tokens before forwarding. Services trust the gateway.
🚦
Rate Limiting
100 req/min per API key. Protect downstream services from abuse.
🔒
SSL Termination
Decrypt HTTPS at the gateway. Internal traffic is plain HTTP — simpler cert management.
✂️
Request Transform
Rewrite URLs, modify headers, add/remove request params before forwarding.
🧩
Aggregation
Fan out to multiple services, merge responses. One client call → multiple service calls.
⚡
Circuit Breaking
Fail fast at the edge if downstream is unhealthy. Return fallback response.
💾
Caching
Cache GET responses. Reduce load on downstream services for repeated reads.
📊
Logging & Metrics
Centralized access logs, latency tracking, error rate dashboards.
🌐
CORS Handling
Manage cross-origin policies in one place, not in every service.
⚖️
Load Balancing
Distribute across multiple instances of the same service.
📦
Compression
GZIP response bodies. Reduce bandwidth for mobile/slow clients.

Spring Cloud Gateway — Route Configuration Explained Line by Line

JavaSpring Cloud Gateway — Routes as Java Config
@Configuration
public class GatewayConfig {

    @Bean
    public RouteLocator customRouteLocator(RouteLocatorBuilder builder) {
        return builder.routes()

            .route("user-service", r -> r
                // PREDICATE: only match requests where path starts with /api/users/
                .path("/api/users/**")
                .filters(f -> f
                    // StripPrefix=1 removes the first path segment BEFORE forwarding
                    // /api/users/1 → /users/1 when it reaches USER-SERVICE
                    .stripPrefix(1)
                    // Add a custom header so services know traffic came via gateway
                    .addRequestHeader("X-Request-Source", "gateway")
                    // Circuit breaker: if USER-SERVICE is down, forward to /fallback endpoint
                    .circuitBreaker(c -> c
                        .setName("userServiceCB")
                        .setFallbackUri("forward:/fallback/users"))
                )
                // URI: lb:// means "use load balancer + service discovery"
                // USER-SERVICE is the name registered in Eureka
                .uri("lb://USER-SERVICE")
            )

            .route("order-service", r -> r
                .path("/api/orders/**")
                // Weight-based routing: send 90% to v2, 10% to v1 (canary deploy)
                .weight("order-group", 90)
                .filters(f -> f
                    .stripPrefix(1)
                    // Rate limiter: 10 requests/sec per user
                    .requestRateLimiter(c -> c
                        .setRateLimiter(redisRateLimiter())
                        .setKeyResolver(userKeyResolver()))
                )
                .uri("lb://ORDER-SERVICE")
            )

            .build();
    }
}

Predicates — The "When to Match" Rules

Built-in Predicates

  • Path=/api/users/** — match by URL path
  • Method=GET,POST — match HTTP method
  • Header=X-Token, \d+ — header regex match
  • Query=page — query param exists
  • Host=*.example.com — match by hostname
  • Cookie=session, abc — match cookie value
  • After=2024-01-01T00:00:00Z — time-based routing
  • Weight=group, 80 — weighted routing (canary)

Built-in Filters (key ones)

  • StripPrefix=1 — remove N path segments
  • AddRequestHeader=K, V — inject header
  • AddResponseHeader=K, V — add to response
  • RewritePath=regex, replace — URL rewrite
  • CircuitBreaker — Resilience4j CB
  • RequestRateLimiter — Redis-backed rate limit
  • Retry — auto-retry on failure
  • RequestSize — reject oversized bodies
🎯
Interview Tip: Gateway vs BFF

An API Gateway is general-purpose routing + cross-cutting concerns. A Backend for Frontend (BFF) is a specialized gateway variant: one gateway instance per client type (mobile BFF, web BFF). The mobile BFF returns trimmed payloads optimized for mobile, the web BFF returns richer data. Mention this when discussing API gateway design — it shows depth.

§6 — Feign Client: Declarative HTTP Calls

❌ Without Feign — RestTemplate Boilerplate

  • Build URL string manually every time
  • Handle serialization/deserialization yourself
  • Write error handling per call
  • Add headers, timeouts, retries manually
  • 10–20 lines of code per service method

✅ With Feign — Just an Interface

  • Define an interface — Feign generates the impl
  • Annotations map to HTTP methods
  • Ribbon LB + Eureka discovery automatic
  • Hystrix circuit breaking per method
  • Looks and feels like a local method call
JavaFeign Client — Complete Example
// Step 1: Declare the interface. That's it. Feign creates the implementation.
@FeignClient(
    name = "USER-SERVICE",         // ← Service name in Eureka registry
    fallback = UserClientFallback.class,  // ← Hystrix fallback class
    configuration = UserFeignConfig.class  // ← Custom timeouts, interceptors
)
public interface UserServiceClient {

    @GetMapping("/users/{id}")
    UserDto getUserById(@PathVariable("id") Long id);

    @PostMapping("/users")
    UserDto createUser(@RequestBody CreateUserRequest request);

    @GetMapping("/users")
    List<UserDto> getUsersByIds(@RequestParam("ids") List<Long> ids);
}

// Step 2: Inject and use it like a local service
@Service
public class OrderService {

    @Autowired
    private UserServiceClient userClient; // Feign-generated proxy injected here

    public OrderDto getOrderWithUser(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        // Looks like a local call. Under the hood: Eureka lookup → Ribbon LB → HTTP call
        UserDto user = userClient.getUserById(order.getUserId());
        return new OrderDto(order, user);
    }
}

// Step 3: Fallback for Hystrix circuit breaking
@Component
public class UserClientFallback implements UserServiceClient {

    @Override
    public UserDto getUserById(Long id) {
        // Circuit is open — return a safe default or throw a custom exception
        return new UserDto(id, "Unknown User", "N/A");
    }

    @Override
    public UserDto createUser(CreateUserRequest request) {
        throw new ServiceUnavailableException("User Service is down");
    }
}

JavaCustom ErrorDecoder — Map HTTP Errors to Exceptions
// FeignException is the default: wraps all HTTP error responses generically.
// ErrorDecoder lets you map specific HTTP status codes to meaningful exceptions.

@Component
public class UserServiceErrorDecoder implements ErrorDecoder {

    private final ErrorDecoder defaultDecoder = new Default();

    @Override
    public Exception decode(String methodKey, Response response) {
        switch (response.status()) {
            case 404:
                // HTTP 404 from User Service → UserNotFoundException in Order Service
                return new UserNotFoundException("User not found");
            case 401:
                return new UnauthorizedException("Invalid token passed to User Service");
            case 503:
                // Retryable exception — Feign will retry this if retry is configured
                return new RetryableException(
                    response.status(), "User service temporarily unavailable",
                    Request.HttpMethod.GET, (Long) null, response.request()
                );
            default:
                return defaultDecoder.decode(methodKey, response);
        }
    }
}
JavaFeign Timeout Configuration
// application.yml
// feign:
//   client:
//     config:
//       USER-SERVICE:         ← per-client config (or 'default' for all)
//         connectTimeout: 1000  ← milliseconds to establish TCP connection
//         readTimeout: 3000     ← milliseconds to wait for response AFTER connected
//         loggerLevel: FULL     ← NONE, BASIC, HEADERS, FULL

// Java config equivalent:
@Configuration
public class UserFeignConfig {

    @Bean
    public Request.Options feignOptions() {
        return new Request.Options(
            1000, TimeUnit.MILLISECONDS,  // connectTimeout
            3000, TimeUnit.MILLISECONDS,  // readTimeout
            true                           // followRedirects
        );
    }

    @Bean
    public Retryer feignRetryer() {
        // Default Retryer: only retries on RetryableException (connection errors)
        // NOT on HTTP 5xx — those go to ErrorDecoder first
        return new Retryer.Default(
            100,    // initial retry interval (ms)
            1000,   // max retry interval (ms)
            3       // max attempts
        );
    }
}
⚠️
Critical: Feign does NOT retry on HTTP 5xx by default

Feign's default Retryer only triggers on RetryableException — which is thrown for connection-level failures (socket timeout, connection refused). A 500 Internal Server Error from the downstream service goes to ErrorDecoder, not to retry. To retry on 5xx, you must throw RetryableException from your ErrorDecoder explicitly.

§7 — Sidecar Pattern & Service Mesh

The Code Duplication Problem

You've implemented circuit breaking, distributed tracing, retries, mTLS in Service A. Now you need all of it in Service B, C, D... In Java. Then someone joins with a Go microservice. Do it again in Go.

After 50 services, you're maintaining infrastructure code in 50 places. And upgrading a tracing library means touching all 50. This is the problem service mesh solves.

💡
The Sidecar Pattern — "Attach a motorbike sidecar"

Deploy a proxy container alongside every service container in the same pod. This proxy (Envoy) intercepts ALL inbound and outbound network traffic for that service. The service talks to localhost; the sidecar handles everything else. The service doesn't know the sidecar exists.

What the Sidecar Proxy Handles (so your app doesn't have to)

Traffic Management

  • Load balancing (L7 aware)
  • Retries with backoff
  • Circuit breaking
  • Timeout enforcement
  • Traffic splitting (canary)

Security

  • mTLS between all services
  • Certificate rotation (automatic)
  • Authorization policies (RBAC per service)
  • JWT validation

Observability

  • Distributed tracing (auto-inject headers)
  • Metrics: latency, error rate, throughput
  • Access logs for every request

Resilience (language-agnostic)

  • No Resilience4j needed in Go/Python
  • Consistent policy across all languages
  • Centrally configurable via YAML

⚠️
Don't recommend a service mesh for a 5-service system

Istio adds 300+ Kubernetes objects, a control plane with 4+ pods, and significant CPU/memory overhead per sidecar. For most startups and teams, Resilience4j + Spring Cloud Gateway is the right answer. Mention service mesh when the interviewer asks about "very large scale" or "multi-language environments."

§8 — Interview Playbook

The Golden Rule

In interviews, always anchor your answer in a trade-off. Never say "X is better than Y." Say "X is better than Y when [context], because [trade-off]."

Q: "How do services discover each other in your architecture?"
30-second answer

"In our Spring Boot microservices, we used client-side discovery with Eureka and Spring Cloud LoadBalancer. Each service registers with Eureka on startup and sends heartbeats every 30 seconds. When Service A needs to call Service B, it resolves the service name via the local Eureka cache and Spring Cloud LB picks an instance using round-robin. We moved away from Ribbon as it's in maintenance mode. The advantage is no extra network hop — but if we were on Kubernetes, I'd use K8s Service DNS instead since that's server-side discovery built in."

Q: "Client-side vs server-side service discovery — when would you choose each?"
Decision framework

Client-side: Spring Cloud + Eureka ecosystem, single language (Java), need routing intelligence in the client, no existing LB infrastructure.

Server-side: Already on Kubernetes (use K8s Service), polyglot services where you can't embed a registry client in every language, or AWS where ALB handles routing. K8s is server-side by default — most modern systems use this without even knowing it.

Q: "What does an API Gateway do that a load balancer doesn't?"
Common confusion area

A load balancer operates at L4/L7 and distributes traffic — it routes based on IP/port and maybe URL. It does one thing well.

An API Gateway is application-aware. It does routing PLUS: authentication (JWT validation), rate limiting per client, request/response transformation, circuit breaking, aggregation (fan-out across multiple services), SSL termination, CORS, caching. It's the business logic layer of your network edge.

Q: "Why use Feign over RestTemplate?"
Practical answer

RestTemplate is imperative — you write boilerplate for each call: build URL, handle exceptions, deserialize. With Feign, you write an interface. The proxy handles the rest.

More importantly: Feign integrates with Ribbon, Eureka, Hystrix, and Sleuth automatically. With RestTemplate, you wire each integration manually. The interface-based model also means the caller reads like a contract — which endpoint, what params, what return type — which is cleaner for reviews and onboarding.

Q: "Explain Eureka Self-Preservation Mode"
Shows depth

If Eureka suddenly stops receiving heartbeats from many services at once, it could mean either: all those services died, OR there's a network partition and Eureka can't hear them.

Eureka bets on the second scenario (network partition is more common in distributed systems). It enters self-preservation mode — it stops evicting stale registrations. This means callers might get stale addresses and need to retry, but it's far better than Eureka wiping out valid registrations during a temporary network blip.

This is the A (Availability) over C (Consistency) choice from CAP theorem. Eureka is an AP system by design.

Q: "When would you use a service mesh vs circuit breakers in code?"
Architecture maturity signal

For small to medium scale (<30 services, Java-only): Resilience4j in code. Simple, less operational overhead, easy to debug.

For large scale (50+ services, polyglot): Service mesh (Istio + Envoy sidecar). You don't want to implement retries, circuit breaking, and mTLS in Go, Python, Rust, and Java separately. Centralize it in the mesh, configure via Kubernetes CRDs. But be honest about the tradeoff: service mesh adds significant operational complexity. Don't recommend it for startups or simple systems.

The answer interviewers love: "It depends on our scale and team maturity. For a Spring Boot system <20 services, Eureka + Feign + Spring Cloud Gateway gives 90% of what you need with minimal ops overhead. For Kubernetes, I'd lean on built-in service DNS and use Spring Cloud Gateway as the ingress. Service mesh only enters the picture when infrastructure concerns are duplicated across 50+ services in multiple languages."
