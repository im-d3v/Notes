Who coined SOLID? Robert C. Martin (Uncle Bob) introduced these principles in early 2000s. The acronym SOLID was later popularized by Michael Feathers. These are design principles — not design patterns. They guide how to structure your code.
S

Single Responsibility Principle

The "S" in SOLID
"A class should have only one reason to change — meaning it should have only one job or responsibility."
🧠
Simple way to remember: If you can describe what a class does using the word "AND", it probably violates SRP. "This class reads data AND validates it AND saves it to DB" → violates SRP.
❌ Violation
Bad — Multiple Responsibilities 
// ❌ This class has 3 responsibilities:
// 1. Business logic (generate report)
// 2. Formatting (format report)
// 3. Persistence (save to file)
public class Report {

    private String title;
    private String content;

    // Responsibility 1: Business logic
    public String generateReport() {
        return "Report: " + title + "\n" + content;
    }

    // Responsibility 2: Formatting
    public String formatAsHTML() {
        return "<h1>" + title + "</h1>"
              + "<p>" + content + "</p>";
    }

    // Responsibility 3: Saving to file
    public void saveToFile(String fileName) {
        // file writing logic...
    }
}
⚠️
If formatting changes, this class changes. If file saving changes, this class changes. If business logic changes, this class changes. 3 reasons to change!
✅ Fixed — SRP Applied
Good — Each Class has 1 Job 
// ✅ Class 1: Only holds report data
public class Report {
    private String title;
    private String content;

    public String generateReport() {
        return "Report: " + title + content;
    }
    // getters, setters...
}

// ✅ Class 2: Only responsible for formatting
public class ReportFormatter {
    public String formatAsHTML(Report report) {
        return "<h1>" + report.getTitle()
              + "</h1><p>"
              + report.getContent() + "</p>";
    }
}

// ✅ Class 3: Only responsible for persistence
public class ReportSaver {
    public void saveToFile(
            Report report, String fileName) {
        // file writing logic...
    }
}
✅
Now each class has exactly one reason to change. Easy to test, easy to maintain.
Real-World Spring Boot Example 🏆 Interview Favourite
SRP in Spring Boot — Layered Architecture 
// ✅ Controller — only handles HTTP requests/responses
@RestController
@RequestMapping("/users")
public class UserController {
    private final UserService userService;
    public UserController(UserService userService) {
        this.userService = userService;
    }
    @PostMapping
    public ResponseEntity<User> createUser(@RequestBody UserDTO dto) {
        return ResponseEntity.ok(userService.createUser(dto));
    }
}

// ✅ Service — only handles business logic
@Service
public class UserService {
    private final UserRepository repo;
    private final EmailService    emailService;

    public User createUser(UserDTO dto) {
        User user = mapToUser(dto);
        User saved = repo.save(user);
        emailService.sendWelcomeEmail(saved);
        return saved;
    }
}

// ✅ Repository — only handles data access
public interface UserRepository extends JpaRepository<User, Long> { }

// ✅ EmailService — only handles email sending
@Service
public class EmailService {
    public void sendWelcomeEmail(User user) { /* send email */ }
}
🎯
Interview Answer: "A class should have only one reason to change. In Spring Boot, we follow this via layered architecture — Controller handles HTTP, Service handles business logic, Repository handles DB operations. If I put DB logic inside Controller, it violates SRP because the Controller would now change for both HTTP-related AND DB-related reasons."
O

Open / Closed Principle

The "O" in SOLID
"Software entities should be open for extension but closed for modification."
🧠
Simple way to remember: You should be able to add new features by adding new code, NOT by changing existing code. Once a class is tested and working, you shouldn't need to modify it to add new behaviour — just extend it.
❌ Violation
Every new shape requires modifying the class 
// ❌ Adding new shape requires changing this class
public class AreaCalculator {

    public double calculateArea(Object shape) {
        if (shape instanceof Circle c) {
            return Math.PI * c.radius * c.radius;
        }
        else if (shape instanceof Rectangle r) {
            return r.width * r.height;
        }
        // ❌ Adding Triangle → must MODIFY this method!
        else if (shape instanceof Triangle t) {
            return 0.5 * t.base * t.height;
        }
        throw new IllegalArgumentException("Unknown shape");
    }
}
✅ Fixed — OCP Applied
New shapes added without modifying existing code 
// ✅ Define abstraction — closed for modification
public interface Shape {
    double calculateArea();
}

// ✅ Extend by adding new classes — open for extension
public class Circle implements Shape {
    private double radius;
    @Override
    public double calculateArea() {
        return Math.PI * radius * radius;
    }
}

public class Rectangle implements Shape {
    private double width, height;
    @Override
    public double calculateArea() {
        return width * height;
    }
}

// ✅ AreaCalculator never changes!
public class AreaCalculator {
    public double calculate(Shape shape) {
        return shape.calculateArea(); // polymorphism
    }
}

// ✅ Adding Triangle: ZERO changes to existing code!
public class Triangle implements Shape {
    private double base, height;
    @Override
    public double calculateArea() {
        return 0.5 * base * height;
    }
}
OCP in Spring Boot — Payment Strategy Example 🏆 Asked in Interview
Strategy Pattern + OCP — Add new payment without changing existing code 
// ✅ Define abstraction (closed for modification)
public interface PaymentStrategy {
    void pay(double amount);
}

// ✅ Concrete implementations (open for extension)
@Component("creditCard")
public class CreditCardPayment implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " via Credit Card");
    }
}

@Component("upi")
public class UPIPayment implements PaymentStrategy {
    @Override
    public void pay(double amount) {
        System.out.println("Paid $" + amount + " via UPI");
    }
}

// ✅ Context class — NEVER changes when new payment added
@Service
public class PaymentService {
    private final Map<String, PaymentStrategy> strategies;

    public PaymentService(Map<String, PaymentStrategy> strategies) {
        this.strategies = strategies;
    }

    public void processPayment(String type, double amount) {
        strategies.getOrDefault(type,
            a -> System.out.println("Unknown payment"))
            .pay(amount);
    }
    // ✅ Adding "PayPal" payment = add new @Component class only!
}
🎯
Interview Answer: "OCP means your classes should be open to extension but closed to modification. In my STEP project, I used the Strategy Pattern for payment methods — instead of if-else chains, each payment type implements a PaymentStrategy interface. Adding a new payment type requires zero changes to existing code."
L

Liskov Substitution Principle

The "L" in SOLID — introduced by Barbara Liskov (1987)
"Objects of a subclass should be replaceable with objects of the superclass without breaking the program. If S is a subtype of T, then objects of T may be replaced with objects of S."
🧠
Simple way to remember: A child class should NOT surprise the caller. If you use a parent class reference pointing to a child object, the program should work exactly as expected — no exceptions thrown, no broken contracts.
❌ Classic LSP Violation
Penguin can't fly — breaks the contract 
// ❌ Violation — classic Bird/Penguin problem
class Bird {
    public void fly() {
        System.out.println("Bird is flying...");
    }
}

class Sparrow extends Bird {
    @Override
    public void fly() {
        System.out.println("Sparrow flying..."); // ✅ OK
    }
}

class Penguin extends Bird {
    @Override
    public void fly() {
        // ❌ Penguins CAN'T fly! Breaking the contract
        throw new UnsupportedOperationException(
            "Penguins cannot fly!");
    }
}

// Usage — program BREAKS at runtime!
public void makeBirdFly(Bird bird) {
    bird.fly(); // ❌ Throws exception for Penguin!
}

// Child cannot substitute parent safely
✅ Fixed — LSP Applied
Proper hierarchy — separate interfaces 
// ✅ Base class — only what ALL birds can do
abstract class Bird {
    public abstract void eat();
    public abstract void makeSound();
}

// ✅ Separate interface for flying birds only
public interface Flyable {
    void fly();
}

// ✅ Sparrow CAN fly
class Sparrow extends Bird
        implements Flyable {
    @Override public void eat() { /* ... */ }
    @Override public void makeSound() { /* chirp */ }
    @Override
    public void fly() {
        System.out.println("Sparrow flying!");
    }
}

// ✅ Penguin CANNOT fly — no Flyable interface
class Penguin extends Bird {
    @Override public void eat() { /* ... */ }
    @Override public void makeSound() { /* honk */ }
    public void swim() {
        System.out.println("Penguin swimming!"); // ✅
    }
}

// ✅ makeBirdFly only accepts Flyable
public void makeBirdFly(Flyable bird) {
    bird.fly(); // Safe — no surprise exceptions!
}
LSP in Real Application — Account Example 💼 Real World
Bank Account — proper substitution 
abstract class BankAccount {
    protected double balance;

    public abstract void deposit(double amount);
    public abstract void withdraw(double amount);

    // LSP contract: deposit must increase balance
    // LSP contract: withdraw must decrease balance or throw InsufficientFundsException
}

class SavingsAccount extends BankAccount {
    @Override
    public void deposit(double amount) {
        balance += amount; // ✅ Respects contract
    }
    @Override
    public void withdraw(double amount) {
        if (amount > balance) throw new IllegalArgumentException("Insufficient funds");
        balance -= amount; // ✅ Respects contract
    }
}

class FixedDepositAccount extends BankAccount {
    @Override
    public void deposit(double amount) {
        balance += amount; // ✅ Respects contract
    }
    @Override
    public void withdraw(double amount) {
        // ❌ VIOLATION if we just throw exception silently
        // ✅ CORRECT: Document the restriction clearly
        throw new UnsupportedOperationException(
            "FD accounts cannot withdraw before maturity");
        // ✅ Better: separate interface for withdrawable accounts
    }
}

// Service works with any BankAccount
public void processDeposit(BankAccount account, double amount) {
    account.deposit(amount); // ✅ Safe substitution
}
🎯
Interview Answer: "LSP says a subclass must be usable wherever the parent class is used, without breaking the program. The classic violation is Penguin extends Bird with a fly() that throws an exception. The fix is to use interfaces — Flyable — only implemented by birds that can actually fly. In real apps, I ensure child classes honour the contract of the parent without surprising the caller."
LSP Rules to Remember
Rule Description Example
Precondition Subclass cannot strengthen preconditions If parent accepts any int, child cannot restrict to positive only
Postcondition Subclass cannot weaken postconditions If parent guarantees non-null return, child must too
Exception Subclass cannot throw new checked exceptions If parent's method throws IOException, child can't add SQLException
Invariant Subclass must preserve invariants of parent If parent guarantees balance >= 0, child must too
I

Interface Segregation Principle

The "I" in SOLID
"A client should never be forced to implement an interface that it doesn't use. Many client-specific interfaces are better than one general-purpose interface."
🧠
Simple way to remember: Don't create "fat interfaces". If a class is forced to implement methods it doesn't need and just writes throw new UnsupportedOperationException() — that's an ISP violation. Break the big interface into smaller, focused ones.
❌ Fat Interface Violation
Bloated Animal interface forces all to implement everything 
// ❌ ONE fat interface — forces all animals to implement EVERYTHING
public interface Animal {
    void eat();
    void fly();    // ❌ Not all animals fly
    void swim();   // ❌ Not all animals swim
    void run();    // ❌ Not all animals run
    void bark();   // ❌ Not all animals bark
}

// ❌ Dog forced to implement fly() and swim()
class Dog implements Animal {
    @Override public void eat()  { /* eats */ }
    @Override public void run()  { /* runs */ }
    @Override public void bark() { /* barks */ }
    @Override
    public void fly() {
        throw new UnsupportedOperationException(
            "Dogs can't fly!"); // ❌ Forced!
    }
    @Override
    public void swim() {
        throw new UnsupportedOperationException(
            "This dog can't swim!"); // ❌ Forced!
    }
}
✅ Fixed — ISP Applied
Segregated interfaces — implement only what you need 
// ✅ Small, focused interfaces (ISP)
public interface Eatable  { void eat();  }
public interface Flyable  { void fly();  }
public interface Swimmable{ void swim(); }
public interface Runnable { void run();  }
public interface Barkable { void bark(); }

// ✅ Dog only implements what it needs
class Dog implements
        Eatable, Runnable, Barkable {
    @Override public void eat()  { /* eats */ }
    @Override public void run()  { /* runs */ }
    @Override public void bark() { /* barks */ }
    // ✅ No forced methods!
}

// ✅ Eagle only implements what it needs
class Eagle implements
        Eatable, Flyable {
    @Override public void eat() { /* eats */ }
    @Override public void fly() { /* flies */ }
}

// ✅ Duck implements multiple
class Duck implements
        Eatable, Flyable, Swimmable {
    @Override public void eat()  { /* eats */ }
    @Override public void fly()  { /* flies */ }
    @Override public void swim() { /* swims */ }
}
ISP in Spring Boot — Repository Interfaces 💼 Real World
Spring Data JPA follows ISP naturally 
// ✅ Spring Data JPA is a great ISP example!
// Repository — basic contract
public interface Repository<T, ID> { }

// CrudRepository — adds CRUD operations
public interface CrudRepository<T, ID>
        extends Repository<T, ID> {
    T save(T entity);
    Optional<T> findById(ID id);
    void deleteById(ID id);
}

// PagingAndSortingRepository — adds pagination
public interface PagingAndSortingRepository<T, ID>
        extends CrudRepository<T, ID> {
    Page<T> findAll(Pageable pageable);
}

// JpaRepository — adds JPA-specific methods
public interface JpaRepository<T, ID>
        extends PagingAndSortingRepository<T, ID> {
    List<T> findAll();
    void flush();
}

// ✅ Use only what you need in your app!
public interface UserRepository
        extends CrudRepository<User, Long> {
    // If you only need basic CRUD, don't extend JpaRepository!
}

// ✅ Custom ISP example in services
public interface Readable<T> {
    List<T> findAll();
    Optional<T> findById(Long id);
}

public interface Writable<T> {
    T save(T entity);
    void delete(Long id);
}

// Read-only service only gets read capability
public class ReportService implements Readable<Report> {
    // Only read operations — no accidental writes!
}
🎯
Interview Answer: "ISP says don't force clients to depend on methods they don't use. A classic example: a fat Animal interface with fly(), swim(), bark() forces Dog to implement fly(). The fix is to split into Flyable, Swimmable, Barkable. Spring Data JPA actually demonstrates ISP beautifully — Repository, CrudRepository, JpaRepository are separate interfaces with increasing capabilities."
D

Dependency Inversion Principle

The "D" in SOLID — Most important in Spring!
"High-level modules should not depend on low-level modules. Both should depend on abstractions. Abstractions should not depend on details. Details should depend on abstractions."
🧠
Simple way to remember: Depend on interfaces, not on concrete classes. Your high-level business logic (OrderService) should depend on an abstraction (PaymentGateway interface), not on a specific implementation (StripePaymentGateway).
❌ Tight Coupling Violation
High-level depends on low-level concrete class 
// ❌ Tight coupling — high depends on low
class PDFFormatter {
    public String format(String content) {
        return "PDF: " + content;
    }
}

class ReportGenerator {
    // ❌ Directly depends on PDFFormatter (concrete class)
    private PDFFormatter formatter = new PDFFormatter();

    public String generate(String data) {
        return formatter.format(data);
    }
    // ❌ To switch to HTML output: MUST MODIFY this class
    // ❌ Cannot unit test without real PDFFormatter
    // ❌ Cannot mock in tests
}
✅ Fixed — DIP Applied
Both depend on abstraction (interface) 
// ✅ Abstraction (interface)
public interface Formatter {
    String format(String content);
}

// ✅ Low-level depends on abstraction
class PDFFormatter implements Formatter {
    @Override
    public String format(String content) {
        return "PDF: " + content;
    }
}

class HTMLFormatter implements Formatter {
    @Override
    public String format(String content) {
        return "<p>" + content + "</p>";
    }
}

// ✅ High-level depends on abstraction (not concrete)
class ReportGenerator {
    private final Formatter formatter; // interface!

    // ✅ Injected from outside (Constructor DI)
    public ReportGenerator(Formatter formatter) {
        this.formatter = formatter;
    }

    public String generate(String data) {
        return formatter.format(data);
    }
    // ✅ Swap PDF → HTML: zero code change here!
    // ✅ Mock Formatter easily in unit tests
}
DIP in Spring Boot — @Autowired on interface 🌱 Spring Core
Spring IoC perfectly implements DIP 
// ✅ Abstraction layer
public interface NotificationService {
    void send(String message, String recipient);
}

// ✅ Concrete implementation 1
@Service
@Primary
public class EmailNotificationService
        implements NotificationService {
    @Override
    public void send(String msg, String recipient) {
        System.out.println("Email sent to: " + recipient);
    }
}

// ✅ Concrete implementation 2
@Service
@Qualifier("sms")
public class SMSNotificationService
        implements NotificationService {
    @Override
    public void send(String msg, String recipient) {
        System.out.println("SMS sent to: " + recipient);
    }
}

// ✅ High-level module depends on interface — DIP!
@Service
public class OrderService {

    private final NotificationService notificationService;

    // ✅ Spring injects the concrete impl at runtime
    // ✅ OrderService has ZERO knowledge of Email vs SMS
    public OrderService(NotificationService notificationService) {
        this.notificationService = notificationService;
    }

    public void placeOrder(Order order) {
        // business logic...
        notificationService.send(
            "Order placed: #" + order.getId(),
            order.getUserEmail()
        );
    }
}

// ✅ Unit Test — mock the interface easily!
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    @Mock
    NotificationService notificationService; // mocked interface
    @InjectMocks
    OrderService orderService;

    @Test
    void placeOrder_shouldSendNotification() {
        orderService.placeOrder(new Order(1L, "test@test.com"));
        verify(notificationService).send(any(), any());
    }
}
🎯
Interview Answer: "DIP says high-level modules should not depend on low-level modules — both should depend on abstractions. In Spring Boot, this is the foundation — we inject interfaces (NotificationService), not concrete classes. Spring IoC container manages which concrete bean gets injected at runtime. This makes testing easy (we can mock interfaces), and we can swap implementations (Email → SMS) with zero code changes in the business layer."

⚡ DIP vs DI vs IoC — The Confusing Trio

Most interviewers ask this. Know the exact difference!

⚠️
Common Mistake: Many candidates confuse DIP (design principle), Dependency Injection (design pattern), and IoC (architectural principle). They are related but different concepts.
DIP
Design Principle (SOLID)
"Depend on abstractions, not concretions."

It's a guideline / principle — a rule that tells you what you should aim for in your design. It does NOT say HOW to achieve it.
Dependency Injection
Design Pattern
"Provide dependencies from outside, don't create them inside."

It's a pattern / technique — a concrete way to implement DIP. Constructor DI, Setter DI, Field DI are all forms of this.
IoC
Architectural Principle
"The control of object creation is inverted — given to a container."

It's a broader architectural concept. Spring IoC Container creates, manages, and injects beans — YOU don't call new.
How They Relate Together
DIP → achieved via → DI → powered by → IoC Container 
// ──────────────────────────────────────────────
// 1. DIP (principle) says: depend on abstractions
// ──────────────────────────────────────────────
public interface MessageSender {   // ← abstraction
    void send(String msg);
}

// ──────────────────────────────────────────────
// 2. DI (pattern) says: inject from outside
// ──────────────────────────────────────────────
@Service
public class AlertService {

    private final MessageSender sender; // DIP: depends on interface

    // Constructor DI — injected from outside (not created here)
    public AlertService(MessageSender sender) { // DI pattern
        this.sender = sender;
    }

    public void sendAlert(String msg) {
        sender.send(msg);
    }
}

// ──────────────────────────────────────────────
// 3. IoC (Spring container) wires everything
// ──────────────────────────────────────────────
@Component
public class EmailSender implements MessageSender {
    @Override
    public void send(String msg) {
        System.out.println("Email: " + msg);
    }
}
// Spring IoC container auto-creates EmailSender bean
// and injects it into AlertService — YOU don't call new!
// Control of object creation is INVERTED to the container
Concept Type What it says Spring Example
DIP Design Principle Depend on abstractions Inject interface, not concrete class
DI Design Pattern Provide dependencies externally @Autowired, constructor injection
IoC Architectural Principle Invert control of object creation Spring ApplicationContext, @Bean
🎯
Perfect Interview Answer: "DIP is a design principle — it says depend on abstractions, not concretions. Dependency Injection is a design pattern that implements DIP by providing dependencies from outside. IoC is an architectural principle where the control of object creation is given to a container (Spring). So: DIP tells WHAT, DI tells HOW, IoC tells WHO does it."

🧹 Clean Code Principles

Related to SOLID — often asked together in interviews

DRY — Don't Repeat Yourself
❌ Magic Numbers / Repeated Constants 
// ❌ Magic numbers — meaning unclear, hard to maintain
if (user.getAge() >= 18) { /* allow */ }
if (password.length() >= 8) { /* allow */ }
if (session.getDuration() > 1800) { /* expire */ }

// ❌ Duplicated connection string in multiple classes
String url1 = "jdbc:mysql://localhost:3306/mydb";
String url2 = "jdbc:mysql://localhost:3306/mydb";
✅ Constants class / interface — DRY principle 
// ✅ Option 1: Constants class (recommended)
public final class AppConstants {
    private AppConstants() {} // prevent instantiation

    public static final int MINIMUM_AGE       = 18;
    public static final int MIN_PASSWORD_LEN  = 8;
    public static final int SESSION_TIMEOUT   = 1800;
    public static final String DB_URL = "jdbc:mysql://...";
}

// ✅ Option 2: Enum for related constants
public enum UserStatus {
    ACTIVE, INACTIVE, SUSPENDED, DELETED
}

// ✅ Option 3: @Value from application.properties (Spring)
@Value("${app.minimum.age}")
private int minimumAge;

// ✅ Usage is now self-documenting
if (user.getAge() >= AppConstants.MINIMUM_AGE) { }
Clean Naming & Guard Clauses
❌ Bad naming & deep nesting 
// ❌ Meaningless names
int d; String s; void proc() {}

// ❌ Comment explaining what code DOES (code should be self-explanatory)
// Check if age > 18
if (user.getAge() > 18) { }

// ❌ Method name doing 3 things
public void processDataAndSaveToDbIfValid() {}

// ❌ Deep nesting (arrow code)
if (user != null) {
    if (user.isActive()) {
        if (user.hasPermission("ADMIN")) {
            if (request.isValid()) {
                // actual logic here... 😱
            }
        }
    }
}
✅ Guard Clauses — early returns to reduce nesting 
// ✅ Self-documenting code
boolean isAdult = user.getAge() >= MINIMUM_AGE;

// ✅ Meaningful method names (each does ONE thing)
public void processPayment(PaymentRequest request) {}
public void validateRequest(PaymentRequest request) {}
public void saveTransaction(Transaction txn) {}

// ✅ Guard Clauses — fail fast, reduce nesting
public void processRequest(User user, Request request) {
    if (user == null)             throw new IllegalArgumentException("User is null");
    if (!user.isActive())         throw new UserInactiveException("User inactive");
    if (!user.hasPermission("ADMIN")) throw new AccessDeniedException("No permission");
    if (!request.isValid())       throw new InvalidRequestException("Invalid");

    // ✅ Happy path — no nesting!
    executeRequest(request);
}
Quick Clean Code Rules — Interview Cheat Sheet
Rule Bad Example Good Example
Naming int d; int daysUntilExpiry;
Functions saveAndNotify() save() + notify() — one job each
Magic Numbers if (age > 18) if (age >= LEGAL_AGE)
Comments // loop through items Code should be self-documenting; comment WHY, not WHAT
Nesting 3+ levels of if-else Guard clauses — early return
DRY Same logic in 3 places Extract to shared utility/constant
