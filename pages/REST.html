🌐 Chapter 1 — Introduction to REST & Spring REST

What is REST? (The Real Answer)

REST is not a protocol. It is not a standard. It is an architectural style — a set of constraints that, when followed, produce a scalable, uniform, and stateless distributed system. Roy Fielding defined it in his 2000 PhD dissertation at UC Irvine.

Most developers think REST just means "JSON over HTTP." That's wrong. A system can use JSON over HTTP and still violate every REST constraint. REST is about how components interact, not what format they use.

📜 The Roy Fielding Story (Why This Matters for Interviews)

Roy Fielding was one of the principal authors of the HTTP/1.1 specification. He didn't invent REST out of thin air — he looked at what made the World Wide Web scale to millions of nodes and reverse-engineered the constraints that made it work.

His dissertation, "Architectural Styles and the Design of Network-based Software Architectures", describes REST as a way to design systems that are as scalable and fault-tolerant as the web itself. Chapter 5 defines REST specifically.

🎯
Interview Tip

When an interviewer asks "what is REST?", saying "it's an architectural style derived from observing what made the web scale" immediately separates you from candidates who say "it uses HTTP verbs."

🔒 The 6 REST Constraints — Deep Dive

These are not suggestions. Violate one and your system is no longer RESTful.

1. Client–Server

What: Separate concerns. Client handles UI, server handles data.

Why it matters: Allows them to evolve independently. You can rewrite your React frontend without changing your API. You can migrate from Postgres to MongoDB without touching the client.

Violation: Server returning HTML with embedded business logic that only makes sense for one specific client.

2. Statelessness

What: Each request must contain ALL information needed to process it. Server holds zero session state between requests.

Why it matters: Any server in a cluster can handle any request. Horizontal scaling becomes trivial. Load balancer doesn't need sticky sessions.

Violation: Server-side sessions (HttpSession). The server "remembering" who you are between requests.

3. Cacheability

What: Responses must label themselves cacheable or non-cacheable via HTTP headers.

Why it matters: Clients, proxies, and CDNs can cache responses, reducing load and latency dramatically.

Violation: GET /users always returning fresh data but not setting Cache-Control, ETag, or Last-Modified headers.

4. Uniform Interface

What: This is REST's most defining constraint. Four sub-constraints:

  • Resource identification in requests (URIs)
  • Resource manipulation through representations
  • Self-descriptive messages (Content-Type)
  • HATEOAS (hypermedia as the engine of state)

Why it matters: Decouples client from server implementation. Clients navigate by following links, not hardcoding paths.

5. Layered System

What: Client cannot know if it's talking directly to the origin server or an intermediary (proxy, load balancer, CDN).

Why it matters: Enables security layers, caching layers, and load balancing without client awareness. Think: Cloudflare in front of your API.

6. Code on Demand (Optional)

What: Server can extend client functionality by sending executable code (JavaScript in browsers).

Why it's optional: Not applicable to most APIs. The only optional constraint — your system is still RESTful without it.

Example: A server sending a JavaScript validation snippet that a rich client executes.

📡 HTTP Methods — The Real Theory

These are not just CRUD wrappers. Each verb has precise semantic contracts.

GET
Safe Idempotent Cacheable

Safe: Does not modify state. Calling 1000 times = same result.

Use: Retrieve a resource or collection. Never use GET for mutations — search crawlers will trigger it.

Body: Technically allowed by spec but universally ignored by servers.

POST
Not Safe Not Idempotent

Use: Create a new resource. Process a command. Submit a form.

Critical: Not idempotent. Submitting the same POST twice creates two resources. Networks retry — design for this with idempotency keys.

Response: 201 Created + Location header pointing to new resource.

PUT
Not Safe Idempotent

Use: Full replacement of a resource. Client sends the complete representation. Fields not in body get nulled out.

Idempotent: PUT /users/1 with same body 10 times = same state.

If not exists: May create (upsert) or return 404 — document clearly.

PATCH
Not Safe Conditionally Idempotent

Use: Partial update. Only fields in request body change. Missing fields are untouched.

PUT vs PATCH: If client only knows the fields it wants to update, use PATCH. If client sends the whole resource, use PUT.

Trap: PATCH is not guaranteed idempotent — an increment-by-one operation via PATCH is not idempotent.

DELETE
Not Safe Idempotent

Use: Remove a resource.

Idempotent: DELETE /users/1 twice = user is gone either way. Second call returns 404 but the state (no user) is the same.

Soft vs Hard delete: 204 for hard delete. Consider 200 with body for soft delete with audit trail.

HEAD / OPTIONS
Safe Idempotent

HEAD: Same as GET but returns headers only, no body. Used to check if a resource exists or if cached version is stale.

OPTIONS: Returns allowed methods for a URL. CORS preflight requests use this. Spring handles automatically.

⚙️ Spring Internal Request Chain — What Actually Happens

Every HTTP request that hits a Spring REST application passes through this chain. Understanding this is essential for debugging and performance.

1. OS Kernel
TCP Socket Accept — OS receives TCP packet, Tomcat's acceptor thread picks up the connection from the accept queue.
2. Embedded Tomcat
Connector → Engine → Host → Context — Tomcat parses raw HTTP bytes into an HttpServletRequest object. Thread from NIO thread pool assigned.
3. FilterChain
javax.servlet.Filter chain executes. Spring Security filters live here. CORS filters, request logging, compression — all Filters. Each Filter can abort the chain or modify request/response.
4. DispatcherServlet
The Spring Front Controller. Every request flows through exactly one DispatcherServlet. It is the entry point into the Spring MVC machinery. It delegates to HandlerMapping → HandlerAdapter → MessageConverter.
5. HandlerMapping
RequestMappingHandlerMapping inspects the request URL and HTTP method, matches against @RequestMapping annotations registered at startup. Returns a HandlerExecutionChain containing the matched method and any applicable HandlerInterceptors.
6. HandlerInterceptors
preHandle() runs before the controller. postHandle() runs after. afterCompletion() always runs (like finally). Used for logging, auth checks, metrics. Different from Filters — Interceptors are Spring-aware and can access the handler method.
7. ArgumentResolvers
HandlerMethodArgumentResolver chain inspects each controller method parameter and resolves its value. @PathVariable, @RequestParam, @RequestBody each have a dedicated resolver. @RequestBody triggers Jackson deserialization here.
8. Controller Method
Your code executes. Returns a Java object or ResponseEntity. Spring detects return type — if @ResponseBody (included in @RestController), triggers message conversion.
9. MessageConverter
HttpMessageConverter (Jackson's MappingJackson2HttpMessageConverter) serializes the Java object to JSON. Content-Type negotiation happens here. Writes to HttpServletResponse output stream.
10. Response Flush
Tomcat flushes response buffer to OS TCP send buffer. OS kernel sends TCP segments to client. Connection may be kept alive (HTTP keep-alive) for reuse.
💡
Filter vs Interceptor vs AOP

Filter: Servlet-level, runs before DispatcherServlet, can intercept any request including static resources.
Interceptor: Spring MVC-level, only for controller-mapped requests, has access to handler method info.
AOP Aspect: Method-level within Spring beans, most granular, can apply to any Spring bean not just controllers.

⚙️ Chapter 2 — Setting Up Spring REST

📜 Pre-Boot History — Why Spring Boot Exists

To appreciate Spring Boot's auto-configuration, you must understand the pain it replaced.

Spring MVC Era (2004–2013) — XML Hell
Every Spring REST application required: web.xml configuring DispatcherServlet, servlet-context.xml with component scan, view resolvers, message converters — all manually wired. Deploying meant packaging a WAR and deploying to an external Tomcat. Startup was slow, config was fragile, and every new developer had to understand 500 lines of XML before writing a single controller.
Spring Boot 1.0 (2014) — Convention Over Configuration
Introduced @SpringBootApplication, embedded servers, starter POMs, and auto-configuration. The question changed from "how do I configure this?" to "what is the default and when do I need to override it?"
Spring Boot 2.x (2018) — Reactive + Actuator
Added Spring WebFlux (reactive stack), Micrometer metrics, improved actuator endpoints, and R2DBC for reactive database access.
Spring Boot 3.x (2022) — Jakarta EE + Native
Migrated from javax.* to jakarta.* namespace (Jakarta EE 9+). Added GraalVM Native Image support (AOT compilation). Requires Java 17 minimum. This is what production systems are migrating to now.

🔮 Auto-Configuration Theory — How @SpringBootApplication Works

@SpringBootApplication is a meta-annotation composing three annotations. Understanding each is a favourite interview question.

Java @SpringBootApplication decomposed 
// @SpringBootApplication is shorthand for all three:

@SpringBootConfiguration   // ← Marks as configuration class (@Configuration)
@EnableAutoConfiguration   // ← THE magic. Triggers auto-config loading.
@ComponentScan             // ← Scans current package + sub-packages
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
1
@EnableAutoConfiguration triggers the mechanism
Spring Boot reads all META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports files from every JAR on the classpath. Each entry is a potential @Configuration class.
2
Conditional evaluation
Each auto-configuration class has @Conditional annotations: @ConditionalOnClass (applies only if a class is present), @ConditionalOnMissingBean (applies only if you haven't already configured the bean yourself), @ConditionalOnProperty (applies based on property value).
3
Ordered evaluation chain
Auto-configurations respect @AutoConfigureBefore and @AutoConfigureAfter to handle dependencies between configurations.
4
Your beans take precedence
@ConditionalOnMissingBean means if you define your own DataSource bean, Spring Boot's auto-configured DataSource does not register. This is the override mechanism.

📊 Chapter 3 — Richardson Maturity Model

The Four Levels of REST Maturity

Leonard Richardson created a model (popularised by Martin Fowler) to measure how "RESTful" an API is. Most production APIs reach Level 2. Reaching Level 3 is rare but impressive.

L0

The Swamp of POX

Single URI, single HTTP method (usually POST). Just tunnelling RPC over HTTP.

Example: POST /api?action=getUser&id=5

SOAP services are here. XML-RPC is here. This is not REST at all.

L1

Resources

Multiple URIs for different resources. Still using only POST.

Example:
POST /users → get user
POST /orders → get order

Better structure but no HTTP verb semantics.

L2

HTTP Verbs

Resources + correct HTTP verbs + correct status codes.

Example:
GET /users/5 → 200
DELETE /users/5 → 204
POST /users → 201 + Location

This is what 95% of APIs call "REST." Industry standard.

L3

HATEOAS

Resources + Verbs + Hypermedia Controls. Responses include links to related actions and resources.

Client discovers what it can do next by reading the response, not reading documentation.

True REST per Fielding. Rarely implemented fully in practice.

🔗 HATEOAS — Explained In-Depth

HATEOAS stands for Hypermedia As The Engine Of Application State. The mental model: think of a web browser. You go to amazon.com and follow links. You never hardcode "/checkout/step-2" — you click the "Checkout" link the page gives you. HATEOAS is this same principle for APIs.

Java HATEOAS with Spring HATEOAS library 
// Maven: spring-boot-starter-hateoas

@RestController
@RequestMapping("/users")
public class UserController {

    @GetMapping("/{id}")
    public EntityModel<UserDTO> getUser(@PathVariable Long id) {
        UserDTO user = userService.findById(id);

        // Wrap user with hypermedia links
        return EntityModel.of(user,
            // Link to self
            WebMvcLinkBuilder.linkTo(
                WebMvcLinkBuilder.methodOn(UserController.class).getUser(id)
            ).withSelfRel(),
            // Link to user's orders
            WebMvcLinkBuilder.linkTo(
                WebMvcLinkBuilder.methodOn(OrderController.class).getOrdersByUser(id)
            ).withRel("orders"),
            // Link to delete action
            WebMvcLinkBuilder.linkTo(
                WebMvcLinkBuilder.methodOn(UserController.class).deleteUser(id)
            ).withRel("delete")
        );
    }
}

/* JSON Response:
{
  "id": 1,
  "name": "Alice",
  "_links": {
    "self":   { "href": "http://localhost:8080/users/1" },
    "orders": { "href": "http://localhost:8080/users/1/orders" },
    "delete": { "href": "http://localhost:8080/users/1" }
  }
}
*/

When Should You Use HATEOAS? (Senior Answer)

Use it when clients are long-lived and you want to evolve server URLs without breaking them. Don't use it for internal microservices where both sides are versioned together. Don't use it if you don't have the budget to maintain link generation logic. Most startups don't need it. Most platform APIs (payment gateways, data APIs) benefit from it.

REST is one of many protocol choices. Know when to use REST, GraphQL, gRPC, WebSocket, SSE, or SOAP — and how to define a contract that survives team boundaries.

Every API protocol is a trade-off. The axis that matters most: latency, payload size, coupling, browser support, and streaming needs.

When asked "how would you design X?", state your protocol choice first with a one-sentence justification: "I'd use gRPC for the Order→Inventory call because it's internal, high-frequency, and we need strong typing — and REST for the public-facing consumer API because browsers need it." This signals that you understand trade-offs, not just syntax.

🔵 REST — Representational State Transfer

REST is an architectural style, not a protocol. It runs over HTTP and has no official schema — the contract is defined separately via OpenAPI.

✅ Use REST when…

  • Consumers are browsers or mobile apps
  • You need a public / partner-facing API
  • Resources are CRUD-shaped (users, orders, products)
  • Cacheability matters (CDN, HTTP cache headers)
  • Team is polyglot — any language speaks JSON/HTTP

❌ Avoid REST when…

  • You have high-frequency internal service calls (gRPC wins)
  • Client needs only 3 of 40 fields (over-fetching → GraphQL)
  • You need real-time push (WebSocket / SSE wins)
  • Strict transactional ACID guarantees needed (consider SOAP or messaging)

Classic REST Over-fetching / Under-fetching Problem

This is the #1 reason GraphQL was invented. Understand this deeply.

Text Over-fetch — GET /users/42 returns everything 
{
  "id": 42,
  "name": "Alice",
  "email": "alice@example.com",
  "address": { ... },       // mobile app doesn't need this
  "paymentMethods": [ ... ], // or this
  "orderHistory": [ ... ],   // or this — 40 KB wasted on 3G
  "preferences": { ... }
}
Text Under-fetch — dashboard needs 4 extra round trips 
GET /users/42          → user data
GET /users/42/orders   → order list
GET /orders/9/items    → line items
GET /products/7        → product details
// 4 sequential HTTP calls = 4× latency on mobile
💡
Interview answer: REST's solution is to add query params (?fields=id,name,email) or use a BFF (Backend-for-Frontend) layer. GraphQL's solution is to make this the default. Neither is universally better — it depends on who controls the client.

REST Maturity Model (Richardson)

0
Plain Old XML / JSON over HTTP
One endpoint, all verbs do the same thing. POST /api for everything. SOAP lives here.
1
Resources
Separate URLs per resource: /users, /orders. But still only POST everywhere.
2
HTTP Verbs + Status Codes
GET/POST/PUT/DELETE used correctly. 404 for missing, 201 for created. Most real-world "REST" APIs stop here.
3
HATEOAS
Responses include hypermedia links to next valid actions. True REST per Fielding. Rarely implemented in practice — Spring HATEOAS supports it.
⚠️
Interview trap: Most companies claim "we use REST" but are at Level 2. Level 3 (HATEOAS) is theoretically correct but operationally painful. Know the levels, but don't advocate for HATEOAS unless specifically asked.

🟣 GraphQL — Query Language for APIs

GraphQL is a query language AND a runtime. The client specifies exactly what data it needs. Developed by Facebook (Meta) in 2012, open-sourced 2015. Used at Meta, GitHub, Shopify, Twitter.

How GraphQL Works

[ Client ] ──POST /graphql──► [ GraphQL Server ] │ ┌─────────────┼─────────────┐ ▼ ▼ ▼ [ UserDB ] [ OrderDB ] [ ProductAPI ] One HTTP endpoint. One request. Data assembled server-side.

Schema Definition (SDL)

Text GraphQL Schema Definition Language 
type User {
  id:     ID!
  name:   String!
  email:  String!
  orders: [Order!]!     # nested — fetched only if requested
}

type Order {
  id:       ID!
  total:    Float!
  status:   OrderStatus!
}

enum OrderStatus { PENDING SHIPPED DELIVERED }

type Query {
  user(id: ID!): User
  users(limit: Int): [User!]!
}

type Mutation {
  createOrder(userId: ID!, items: [ItemInput!]!): Order!
}

type Subscription {
  orderStatusChanged(orderId: ID!): Order!   # real-time via WS
}

Query vs Mutation vs Subscription

Text Client requests only what it needs 
# Query — read only exact fields needed
query {
  user(id: "42") {
    name
    orders {
      id
      status
    }
  }
}

# Mutation — write operation (like POST/PUT/DELETE)
mutation {
  createOrder(userId: "42", items: [{productId: "7", qty: 2}]) {
    id
    total
    status
  }
}

# Subscription — real-time push (uses WebSocket under the hood)
subscription {
  orderStatusChanged(orderId: "99") {
    status
    updatedAt
  }
}

✅ Use GraphQL when…

  • Multiple clients (web, iOS, Android) need different shapes of the same data
  • Building a BFF (Backend for Frontend) aggregation layer
  • Rapid product iteration — frontend team can fetch new fields without backend deploy
  • Complex nested object graphs (social graphs, e-commerce catalogs)
  • GitHub, Shopify, Meta scale patterns

❌ Avoid GraphQL when…

  • Simple CRUD API with predictable shapes (REST is simpler)
  • File upload is primary use case
  • HTTP caching is critical (GraphQL uses POST → no GET caching)
  • Team unfamiliar — N+1 query problem will destroy your DB
  • Public third-party API that needs versioning (REST versioning is simpler)
🔥
N+1 Problem — GraphQL's Biggest Pitfall

Requesting 100 users with their orders fires 1 query for users + 100 queries for orders = 101 DB hits. Solution: DataLoader (batching) — collect all order IDs, fire 1 query with WHERE order_id IN (...). This is a guaranteed Meta/Google interview follow-up if you mention GraphQL.

🟢 gRPC — Google Remote Procedure Call

gRPC uses HTTP/2 as transport and Protocol Buffers as the serialization format. It generates client and server stubs from a .proto schema. Used extensively inside Google, Netflix, Uber, Dropbox.

Why gRPC is Fast

HTTP/2 Multiplexing

Multiple requests share one TCP connection. No head-of-line blocking. Persistent connection = no TCP handshake overhead per call.

Protobuf Binary Encoding

JSON: "status":"ACTIVE" = 18 bytes. Protobuf: field tag + varint = 2 bytes. 5–10× smaller payload. No parsing ambiguity.

Proto File — The Contract

Text order_service.proto 
syntax = "proto3";
package com.example.order;

service OrderService {
  // Unary — one request, one response (like REST)
  rpc GetOrder (GetOrderRequest) returns (Order);

  // Server streaming — one request, stream of responses
  rpc ListOrders (ListOrdersRequest) returns (stream Order);

  // Client streaming — stream of requests, one response
  rpc BulkCreateOrders (stream CreateOrderRequest) returns (BulkResult);

  // Bidirectional streaming — both sides stream
  rpc TrackOrders (stream TrackRequest) returns (stream OrderUpdate);
}

message GetOrderRequest {
  string order_id = 1;
}

message Order {
  string  id      = 1;
  string  user_id = 2;
  float   total   = 3;
  Status  status  = 4;
}

enum Status {
  PENDING   = 0;
  SHIPPED   = 1;
  DELIVERED = 2;
}

From this .proto file, protoc generates client stubs in Java, Go, Python, C++, etc. Both sides compile from the same file → contract-first development.

Four gRPC Communication Patterns

Pattern Request Response Use Case
Unary Single Single GetUser, PlaceOrder — same as REST
Server Streaming Single Stream Download large dataset, live price feed
Client Streaming Stream Single Upload file chunks, batch sensor data
Bidirectional Stream Stream Chat, collaborative editing, game state sync

✅ Use gRPC when…

  • Internal microservice-to-microservice calls
  • High throughput, low latency required
  • Multiple language services (polyglot)
  • Streaming data (IoT, financial ticks)
  • Strong schema enforcement needed

❌ Avoid gRPC when…

  • Browser clients (needs gRPC-Web + Envoy proxy)
  • Public / partner API (tooling not universal)
  • Simple CRUD where REST is sufficient
  • Human-readable debugging needed (binary not curl-friendly)
💡
Protobuf Field Numbers are the Contract

Never change a field number once deployed — it's the binary wire identity. You can add new fields (consumers ignore unknown fields). To remove a field, mark it reserved. This is how Google ships proto schema changes across hundreds of services without downtime.

🔵 WebSocket — Full-Duplex Persistent Connection

WebSocket starts as an HTTP/1.1 connection and upgrades to a TCP socket. Both client and server can send messages at any time. No polling, no overhead per message.

WebSocket Handshake

HTTP Upgrade from HTTP → WebSocket 
# Client sends HTTP Upgrade request
GET /chat HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13

# Server accepts — connection switches protocol
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

# After this: raw TCP frames, no HTTP overhead

✅ Use WebSocket when…

  • Real-time chat or collaboration (Google Docs, Slack)
  • Live dashboards (trading, monitoring)
  • Multiplayer gaming
  • Both client and server need to push messages
  • Latency matters more than simplicity

❌ Avoid WebSocket when…

  • Only server needs to push (use SSE — simpler)
  • Stateless horizontal scaling needed (WebSocket is sticky)
  • CDN caching required (WebSocket is uncacheable)
  • Firewall/proxy environments that block upgrades
⚠️
WebSocket Scaling Problem — Know This for System Design

WebSocket connections are stateful and sticky. If you have 1M concurrent connections across 10 servers, a user on Server 3 can't receive a message published by a user on Server 7 without a message broker. Standard solution: Redis Pub/Sub or Kafka as a backplane. Every server subscribes to relevant topics and fans out to local connections. This is exactly how Slack, Discord, and WhatsApp scale WebSockets.

📡 SSE & Long Polling — Simpler Push Alternatives

When you only need server-to-client push, SSE is simpler than WebSocket. Long polling is the fallback when SSE isn't available.

Why This Matters in Debugging

If you accidentally annotate a @RestController method to return a String that looks like a path (e.g., "redirect:/login"), Spring does NOT redirect — it serializes the string "redirect:/login" as JSON. Jackson is Spring's default JSON library. Understanding its pipeline prevents production bugs around null handling, date formats, and unknown fields.

Request: JSON → Java (Deserialization)

📦 ResponseEntity Patterns

ResponseEntity<T> gives you full control over HTTP status, headers, and body. Use it when the response status or headers need to vary based on logic.

Java ResponseEntity patterns — every common use case 
@RestController
@RequestMapping("/users")
public class UserController {

    // Pattern 1: 201 Created with Location header
    @PostMapping
    public ResponseEntity<UserDTO> create(@RequestBody CreateUserRequest req) {
        UserDTO created = userService.create(req);
        URI location = URI.create("/users/" + created.getId());
        return ResponseEntity.created(location).body(created);
        // HTTP 201 + Location: /users/123 + body
    }

    // Pattern 2: 404 if not found, 200 if found
    @GetMapping("/{id}")
    public ResponseEntity<UserDTO> getById(@PathVariable Long id) {
        return userService.findById(id)
            .map(ResponseEntity::ok)                   // 200 + body
            .orElse(ResponseEntity.notFound().build()); // 404, no body
    }

    // Pattern 3: 204 No Content for delete
    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        userService.delete(id);
        return ResponseEntity.noContent().build(); // 204
    }

    // Pattern 4: Custom headers (e.g., rate limiting info)
    @GetMapping
    public ResponseEntity<List<UserDTO>> list() {
        List<UserDTO> users = userService.findAll();
        return ResponseEntity.ok()
            .header("X-Total-Count", String.valueOf(users.size()))
            .header("Cache-Control", "max-age=60")
            .body(users);
    }

    // Pattern 5: Conditional update with ETag
    @PutMapping("/{id}")
    public ResponseEntity<UserDTO> update(
            @PathVariable Long id,
            @RequestBody UpdateUserRequest req,
            @RequestHeader(value = "If-Match", required = false) String ifMatch) {
        if (ifMatch != null && !userService.versionMatches(id, ifMatch)) {
            return ResponseEntity.status(412).build(); // 412 Precondition Failed
        }
        UserDTO updated = userService.update(id, req);
        return ResponseEntity.ok(updated);
    }
}

Why Global Exception Handling Matters

Without global handling: every controller method has try-catch blocks, error format is inconsistent, stack traces leak to clients, and Spring's default Whitelabel error page returns HTML to JSON clients.

With @ControllerAdvice: one place handles all exceptions, consistent error response format, no stack traces in production, and complete control.

Java Global exception handler — production pattern 
@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    // ── 1. Business not-found exception ──────────────────
    @ExceptionHandler(ResourceNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(
            ResourceNotFoundException ex,
            HttpServletRequest request) {
        return buildError(404, "NOT_FOUND", ex.getMessage(), request);
    }

    // ── 2. Validation failure (@Valid) ────────────────────
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(
            MethodArgumentNotValidException ex,
            HttpServletRequest request) {

        List<FieldError> errors = ex.getBindingResult().getFieldErrors()
            .stream()
            .map(fe -> new FieldError(fe.getField(), fe.getDefaultMessage()))
            .toList();

        ErrorResponse resp = buildError(400, "VALIDATION_FAILED",
            "Request validation failed", request);
        resp.setFieldErrors(errors);
        return resp;
    }

    // ── 3. Business conflict (duplicate email, etc.) ──────
    @ExceptionHandler(DuplicateResourceException.class)
    @ResponseStatus(HttpStatus.CONFLICT)
    public ErrorResponse handleConflict(
            DuplicateResourceException ex,
            HttpServletRequest request) {
        return buildError(409, "CONFLICT", ex.getMessage(), request);
    }

    // ── 4. Catch-all — NEVER expose internals ─────────────
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleAll(
            Exception ex,
            HttpServletRequest request) {
        log.error("Unhandled exception for {} {}",
            request.getMethod(), request.getRequestURI(), ex);
        // NEVER put ex.getMessage() here — may contain internal details
        return buildError(500, "INTERNAL_ERROR",
            "An unexpected error occurred. Please try again later.", request);
    }

    private ErrorResponse buildError(int status, String error,
                                         String message, HttpServletRequest request) {
        return ErrorResponse.builder()
            .status(status).error(error).message(message)
            .path(request.getRequestURI())
            .timestamp(Instant.now())
            .build();
    }
}

DTO (Data Transfer Object) is the object that crosses the HTTP boundary. It is NOT your JPA entity. The distinction is crucial.

✅ DTO (what crosses HTTP)

  • Immutable or minimal setters (records are perfect)
  • Only fields the client needs to see
  • Validation annotations on request DTOs
  • Jackson annotations for JSON shaping
  • No JPA annotations
  • Can be generated from OpenAPI spec

❌ Entity (stays inside service layer)

  • JPA annotations (@Entity, @Table, @Column)
  • Relationships (@OneToMany, @ManyToOne)
  • Database-specific fields (version, audit fields)
  • Lazy-loaded associations
  • Password hashes, internal state
  • Never expose directly via @ResponseBody

🔄 @Transactional Deep Dive

@Transactional is one of Spring's most powerful and most misunderstood annotations. It works via AOP proxying.

Java @Transactional — key properties and common mistakes 
@Service
public class UserService {

    // Default: REQUIRED propagation, READ/WRITE, rollback on RuntimeException
    @Transactional
    public UserResponse createUser(CreateUserRequest req) {
        User user = mapToEntity(req);
        User saved = userRepository.save(user);
        emailService.sendWelcomeEmail(saved); // TRAP: if this throws, user save is rolled back
        return toResponse(saved);
    }

    // Read-only transactions — enables Hibernate optimizations
    @Transactional(readOnly = true)
    public List<UserResponse> findAll() {
        return userRepository.findAll().stream().map(this::toResponse).toList();
    }

    // REQUIRES_NEW — nested independent transaction
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void saveAuditLog(String event) {
        // Runs in its OWN transaction. Commits even if outer TX rolls back.
        auditRepo.save(new AuditLog(event));
    }

    // Custom rollback rules
    @Transactional(rollbackFor = Exception.class)  // Roll back on checked exceptions too
    public void riskyOperation() throws IOException { ... }

    // ⚠️ SELF-INVOCATION TRAP — @Transactional is IGNORED here:
    public void outerMethod() {
        this.innerTransactionalMethod(); // Bypasses proxy! No transaction!
    }

    @Transactional
    public void innerTransactionalMethod() { ... }
}
🚨
The Self-Invocation Trap (Most Common @Transactional Bug)

@Transactional works via Spring AOP proxies. When a method calls another method on the same object (this.method()), it bypasses the proxy and @Transactional is completely ignored. Solution: inject the service into itself (@Autowired UserService self;), use ApplicationContext.getBean(), or restructure into separate service classes.

⚡ The N+1 Problem — The Most Common Performance Bug

If you fetch 100 users, then for each user you fetch their orders, you make 1 + 100 = 101 database queries. This is the N+1 problem. It silently kills production performance.

Java N+1 problem — cause and fix 
// ❌ N+1: UserDTO contains orderCount → fetched lazily per user
public List<UserDTO> findAllBad() {
    return userRepository.findAll()   // 1 query: SELECT * FROM users
        .stream()
        .map(user -> {
            int orderCount = user.getOrders().size(); // N queries: SELECT * FROM orders WHERE user_id=?
            return new UserDTO(user.getId(), user.getName(), orderCount);
        }).toList();
}

// ✅ Fix 1: JOIN FETCH in JPQL
@Query("SELECT u FROM User u LEFT JOIN FETCH u.orders")

// ✅ Fix 2: @EntityGraph (preferred for Spring Data)
@EntityGraph(attributePaths = {"orders"})
@Query("SELECT u FROM User u")
List<User> findAllWithOrders();

// ✅ Fix 3: Projection with @Query to count in SQL

📊 Page vs Slice — Choose the Right Type

Type SQL Generated Contains Total Count Use When
Page<T> SELECT + COUNT(*) Yes Pagination UI (show "Page 3 of 47")
Slice<T> SELECT only (fetches size+1 to detect hasNext) No Infinite scroll, mobile feeds — no total needed
List<T> SELECT only (no auto LIMIT) No Never for large tables — use with explicit @Query LIMIT
⚠️
Page<T> COUNT query on large tables is expensive

On a table with 50 million rows, SELECT COUNT(*) can take seconds. Consider using Slice<T> with cursor-based pagination for performance-critical endpoints. Alternatively, maintain a counter in Redis that you increment/decrement on insert/delete.

🔑 OFFSET vs Keyset (Cursor-Based) Pagination

OFFSET Pagination

SQL
SELECT * FROM users
ORDER BY id
LIMIT 20 OFFSET 10000;

Problem: Database must scan and discard 10,000 rows before returning 20. Gets slower as you go deeper. Page 500 scans 10,000 rows to skip them.

Also: Inconsistent results if rows are inserted/deleted between page requests.

Use for admin panels, small datasets, when total count is needed

Keyset (Cursor) Pagination

SQL
SELECT * FROM users
WHERE id > :lastSeenId
ORDER BY id
LIMIT 20;

Why fast: Uses the indexed primary key. B-tree index lookup is O(log n), not O(n) like OFFSET. Performance is constant regardless of page depth.

Also: Consistent results — inserting a row won't cause items to be skipped or duplicated.

Use for feeds, infinite scroll, large datasets, APIs

🔢 API Versioning Strategies

URL Path Versioning

/api/v1/users

Pros: Most visible, easy to route at infrastructure level, cacheable, bookmark-friendly.

Cons: URIs should identify resources, not versions. Breaks REST purity.

Use when: Public APIs, mobile backends. Most common in practice.

Header Versioning

X-API-Version: 2

Pros: Clean URIs, REST-compliant, same URL can serve multiple versions.

Cons: Not visible in URL, harder to test in browser, caching more complex.

Use when: Internal APIs, when URI cleanliness is priority.

Accept Header Versioning

Accept: application/vnd.company.v2+json

Pros: Technically most REST-correct. Version is part of the media type.

Cons: Complex to implement and consume. Hard to test manually.

Use when: Public APIs where REST purity matters (GitHub API uses this).

Query Parameter Versioning

/users?version=2

Pros: Simple to add, optional parameter.

Cons: Parameters pollute the resource URL, cache-unfriendly.

Avoid for anything significant. Acceptable for minor variations.

🔐 Security Checklist

✓
Use HTTPS everywhere, no exceptions
HTTP transmits tokens and data in plaintext. Use TLS 1.2+ minimum, TLS 1.3 preferred. Set HSTS header: Strict-Transport-Security: max-age=31536000; includeSubDomains
✓
Never expose stack traces or internal details
Set server.error.include-stacktrace=never and server.error.include-message=never in production application.properties.
✓
Validate ALL input, including path variables
Don't trust any client data. PathVariables, query params, headers, cookies — validate them all. Reject oversized payloads with spring.servlet.multipart.max-file-size.
✓
Implement rate limiting at API gateway or application level
Without rate limiting, a single client can exhaust your thread pool. Use Bucket4j, Spring Cloud Gateway, or nginx rate limiting. Return 429 with Retry-After header.
✓
Implement CORS correctly
Configure allowed origins explicitly — never use * for authenticated APIs. Spring: @CrossOrigin(origins = "https://yourdomain.com") or global config via CorsConfigurationSource.
✓
Use idempotency keys for non-idempotent operations
Network retries will replay POST requests. If a payment POST is replayed, the user is charged twice. Accept an Idempotency-Key header, store it with the result, return the cached result for duplicate keys.
✓
Return 403 (not 404) for authorization failures on existing resources
Returning 404 to "hide" resources leaks information through timing attacks. Be consistent: if authenticated, return 403. If not authenticated, return 401. 404 only when the resource truly doesn't exist.

🔁 Idempotency — Deep Dive

An operation is idempotent if calling it once has the same effect as calling it N times. This is essential for reliable distributed systems where retries are common.

Java Idempotency key pattern for POST requests 
@PostMapping("/payments")
public ResponseEntity<PaymentResponse> createPayment(
        @RequestHeader("Idempotency-Key") String idempotencyKey,
        @RequestBody @Valid PaymentRequest request) {

    // Check if we've seen this key before
    Optional<PaymentResponse> cached = idempotencyStore.get(idempotencyKey);
    if (cached.isPresent()) {
        // Return the original response — don't process again
        return ResponseEntity.ok(cached.get());
    }

    // Process the payment
    PaymentResponse response = paymentService.process(request);

    // Store response keyed by idempotency key (with TTL)
    idempotencyStore.store(idempotencyKey, response, Duration.ofHours(24));

    return ResponseEntity.status(HttpStatus.CREATED).body(response);
}

// Redis-backed idempotency store
@Service
public class RedisIdempotencyStore {
    private final RedisTemplate<String, PaymentResponse> redis;

    public Optional<PaymentResponse> get(String key) {
        return Optional.ofNullable(redis.opsForValue().get("idempotency:" + key));
    }

    public void store(String key, PaymentResponse response, Duration ttl) {
        redis.opsForValue().set("idempotency:" + key, response, ttl);
    }
}

Rate Limiting with Bucket4j

Bucket4j implements the Token Bucket algorithm. Each client gets a "bucket" with tokens. Each request consumes one token. Tokens refill at a configured rate. When bucket is empty → 429. Store buckets in Redis for distributed rate limiting across multiple instances.

🌐 Q&A — REST Fundamentals

Q1. What is the difference between REST and HTTP?
HTTP is a protocol — a set of rules for data transfer over a network. REST is an architectural style — a set of constraints for designing distributed systems. REST can theoretically run over any protocol, though in practice it uses HTTP. HTTP can be used for non-RESTful communication (e.g., RPC-style POX services are HTTP but not REST). You need HTTP to build REST APIs in practice, but using HTTP does not make your API RESTful.
Q2. What does "stateless" mean in REST? Give a concrete production implication.
Stateless means the server holds zero session state between requests. Each request must carry all context needed to process it (authentication token, user ID, etc.).

Production implication: You can run 10 instances of your API behind a load balancer, and any instance can handle any request. No sticky sessions needed. Horizontal scaling is trivial. Compare to stateful: if the server stores session in memory, all requests from a user must route to the same instance (sticky sessions), which complicates scaling and causes problems when that instance crashes.

Common interview trap: JWT tokens make an API stateless if all needed information is in the token and the server doesn't validate against a session store. If you store JWT in a Redis blacklist for revocation, you've reintroduced server-side state.

Q3. What is the Uniform Interface constraint and why is it REST's most important constraint?
The Uniform Interface constraint says all REST components must interact through a single, generalized interface. It has four sub-constraints:
  • Resource identification: Resources are identified by URIs
  • Manipulation through representations: Clients interact with representations, not the resource directly
  • Self-descriptive messages: Each message carries enough information to describe how to process it (Content-Type header)
  • HATEOAS: Clients navigate by following hypermedia links, not hardcoded paths

It's the most important because it decouples client from server. Any client that understands JSON and HTTP can interact with any server that exposes a REST API, without knowing the server's implementation details. This is what made the web work at global scale.

Q4. What is idempotency? Which HTTP methods are idempotent and why does this matter?
An operation is idempotent if calling it N times has the same effect as calling it once.
Method Idempotent Why
GET Yes Read-only, no side effects
PUT Yes Full replacement — same input → same state
DELETE Yes Resource gone after first call, subsequent calls find nothing
POST No Creates new resource each call
PATCH Conditional Depends on operation — set operations yes, increment no

Why it matters: Networks are unreliable. When a request times out, clients retry. If a safe/idempotent method is retried, no harm done. If POST is retried, a user gets charged twice. This is why payment APIs require idempotency keys — to make POST effectively idempotent from the business perspective.

Q5. When would you use PUT vs PATCH? What's wrong with "just always use PUT"?

PUT = full replacement. Client sends the complete resource representation. Server replaces the entire stored resource. Fields not in the body are nulled out.

PATCH = partial update. Client sends only the fields to change. Unchanged fields retain their values.

Why "always use PUT" is wrong:

  • Client doesn't always have the full object — a mobile client editing just the bio field should not need to load and resend all 40 profile fields
  • Race condition risk — Client A reads resource (name=Alice, email=alice@x.com), Client B updates email to b@x.com. Client A then PUTs with old email, wiping Client B's change. PATCH would only update the specific field, avoiding the race
  • Bandwidth waste — sending 40 fields to update 1 is wasteful

Practical guidance: Use PUT for resource "upsert" scenarios. Use PATCH for real-world form-based partial updates.

Q6. Why should you return 201 instead of 200 for a successful POST? What else should the response include?

201 Created semantically communicates that a new resource was created, distinguishing it from a simple retrieval (200). This matters for clients that need to differentiate creation from update/read in audit logs or metrics.

A complete 201 response should include:

  • Location header: Location: /api/v1/users/123 — tells the client where the new resource lives. Client can then GET it without knowing the ID convention.
  • Body (optional but recommended): The created resource representation, so the client doesn't need a second GET request to see the assigned ID and server-generated fields.

    .created(URI.create("/users/" + created.getId()))
    .body(created);
Q7. Explain HATEOAS. In your experience, when is it actually worth implementing?

HATEOAS means responses include hypermedia links that describe what the client can do next. A GET /users/1 response includes links to update, delete, get orders — the client doesn't need to know these URLs in advance.

When it's worth it:

  • External/public APIs where third-party clients exist that you can't update
  • Long-lived APIs where URL structure might evolve
  • Complex workflows where available actions depend on resource state (e.g., an order can only be cancelled if status=PENDING — include cancel link only then)

When it's not worth it:

  • Internal microservice APIs where client and server are deployed together
  • Simple CRUD APIs with stable URLs
  • Teams without the discipline to maintain link generation

Most production systems implement Level 2 (HTTP verbs) and call it REST. That's pragmatically acceptable. True HATEOAS is aspirational for most teams.

Q8. What is the difference between 401 Unauthorized and 403 Forbidden? Getting this wrong in production has security implications.

401 Unauthorized = Authentication failure. The request has no or invalid credentials. The server is saying: "I don't know who you are. Please authenticate." The response should include a WWW-Authenticate header telling the client how to authenticate.

403 Forbidden = Authorization failure. The server knows who you are (authentication succeeded) but you don't have permission for this resource. The server is saying: "I know who you are, and you're not allowed here."

Security implication: Some teams return 404 for protected resources instead of 403, to "hide" that the resource exists. This is called "security through obscurity" and is generally discouraged — an attacker can probe via timing differences. However, for highly sensitive resources (admin endpoints, other users' private data), returning 404 regardless is a valid security pattern to not reveal resource existence to unauthorized actors.

⚙️ Q&A — Spring Internals

Q9. What is the DispatcherServlet and what is its role in the Spring MVC request lifecycle?

DispatcherServlet is the Front Controller in Spring MVC. Every HTTP request that reaches your application goes through exactly one DispatcherServlet. It is the entry point into the Spring MVC machinery.

Its responsibilities:

  • Delegates to HandlerMapping to find which controller method should handle the request
  • Delegates to HandlerAdapter to invoke the controller method
  • Delegates to HandlerExceptionResolver if an exception occurs
  • Delegates to ViewResolver (for MVC) or HttpMessageConverter (for REST)

Why one DispatcherServlet? The Front Controller pattern centralizes cross-cutting concerns (security, logging, content negotiation) in one place, making the rest of the application code clean. Spring Boot configures exactly one DispatcherServlet mapped to "/" by default.

Q10. How does Spring Boot auto-configuration work? Explain the mechanism step by step.
  1. @SpringBootApplication includes @EnableAutoConfiguration
  2. Spring Boot reads META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports from all JARs on the classpath — this lists hundreds of auto-configuration classes
  3. Each auto-config class has @Conditional annotations that guard its activation: @ConditionalOnClass (only if library is present), @ConditionalOnMissingBean (only if you haven't defined it yourself)
  4. Qualifying auto-configs create beans — e.g., if H2 is on classpath and no DataSource bean exists, DataSourceAutoConfiguration creates an in-memory H2 DataSource
  5. Your own @Bean definitions take precedence because they're processed before auto-configs

How to debug: Run with --debug flag or add debug=true to application.properties. Spring Boot prints a Conditions Evaluation Report showing which auto-configs applied and which were skipped and why.

Q12. How does @RequestBody work? Walk me through what happens from receiving the HTTP request to the Java object being populated.
  1. Tomcat receives the HTTP request, parses headers, makes body available as an InputStream
  2. DispatcherServlet finds the matching controller method
  3. HandlerAdapter runs HandlerMethodArgumentResolver chain to resolve each parameter
  4. RequestResponseBodyMethodProcessor handles @RequestBody parameters
  5. It reads the Content-Type header (must be application/json for JSON)
  6. It finds a compatible HttpMessageConverter — for JSON, MappingJackson2HttpMessageConverter
  7. Jackson ObjectMapper reads the InputStream, creates a Java object via reflection (requires default no-arg constructor unless @JsonCreator is used)
  8. Maps JSON field names to Java field names (case-sensitive, or as configured)
  9. If @Valid is on the parameter, Hibernate Validator validates the populated object
  10. If validation fails → MethodArgumentNotValidException. If passes → method executes with the object
Q13. What is content negotiation in Spring REST and how does it work?

Content negotiation is the process where client and server agree on the representation format for a response. Spring's ContentNegotiationStrategy determines the format to use.

Mechanism (in priority order):

  1. URL path extension (/users.json) — deprecated, disabled in Spring Boot 3
  2. Request parameter (?format=json) — disabled by default
  3. Accept header — default. Server inspects registered HttpMessageConverters to find one that can produce the requested media type

What if no match? Server returns 406 Not Acceptable.

Adding XML support: Add jackson-dataformat-xml dependency. Spring Boot auto-registers MappingJackson2XmlHttpMessageConverter. Now the same endpoint serves JSON or XML based on the Accept header — zero code changes.

Controlling per-endpoint: Use produces attribute: @GetMapping(produces = "application/json") to restrict to JSON only.

Q14. What is Jackson's ObjectMapper and why should it be a singleton?

ObjectMapper is Jackson's core class for JSON serialization/deserialization. It is expensive to create — it builds internal introspection caches, registers modules, and configures type factories.

Why singleton: Creating a new ObjectMapper per request causes: (1) significant CPU overhead from re-initializing caches, (2) memory pressure from generating and then GC-ing large configuration objects.

Thread safety: ObjectMapper is thread-safe after configuration (once you stop modifying it). Spring Boot creates exactly one ObjectMapper bean and reuses it for all serialization/deserialization. Injecting it into your services for manual JSON processing is correct and recommended.

How to customize it in Spring Boot:

Q15. Explain how Spring resolves ambiguous @RequestMapping methods. What happens at startup vs at runtime?

Spring resolves mapping ambiguity at startup time, not at runtime. During context initialization, RequestMappingHandlerMapping scans all @Controller beans, registers each @RequestMapping method in a lookup map keyed by method + URL + conditions.

Resolution priority (most specific wins):

  1. Exact URL match beats wildcard match
  2. More specific wildcard beats less specific (/users/{id}/orders beats /users/**)
  3. More specific media type (produces/consumes) wins
  4. More specific headers/params win

If two methods are equally specific and match the same request, Spring throws AmbiguousRequestMappingException at startup — the application fails to start. This is intentional: fail early, not on a production request.

At runtime: Only an O(1) lookup in the pre-built map. No resolution logic runs per request.

Q17. Why does @Transactional fail silently when called from within the same class?

Spring's @Transactional works via AOP proxies. When you inject a @Service bean into another class, Spring injects a proxy around your class. When an external caller invokes a @Transactional method, the proxy intercepts the call and wraps it in a transaction.

When you call your own method (this.method()), you're calling it directly on the real object, bypassing the proxy. The proxy doesn't intercept it. No transaction is started.

Solutions:

  1. Inject the service into itself via @Autowired @Lazy UserService self; and call self.method()
  2. Use ApplicationContext.getBean(UserService.class).method()
  3. Extract the inner method into a separate service class
  4. Use AspectJ mode (@EnableTransactionManagement(mode = AdviceMode.ASPECTJ)) for compile-time weaving — avoids proxy limitation but requires AspectJ setup
Q18. What is the N+1 problem? How do you detect it in production?

When you fetch N entities and then for each entity you issue one more query to fetch a related collection, you have N+1 total queries instead of 1.

Detection:

  • Development: Enable logging.level.org.hibernate.SQL=DEBUG and logging.level.org.hibernate.type.descriptor.sql=TRACE. Watch for repetitive queries with different ID values.
  • Hibernate statistics: Enable spring.jpa.properties.hibernate.generate_statistics=true — Hibernate logs query counts per session.
  • Production: Use APM tools (Datadog, New Relic, Elastic APM) that track DB queries per HTTP request. Alerts on requests exceeding N queries.
  • P6Spy: Add the p6spy JDBC driver wrapper to log and timestamp every SQL statement with its caller's stack trace.
Q19. What is @Valid vs @Validated? When does each one apply?

@Valid is from Jakarta Bean Validation (JSR-380). It triggers standard Bean Validation on the annotated parameter. It does NOT support validation groups.

@Validated is Spring's extension. It wraps @Valid and adds validation group support. It can also be used at the class level to enable method-level validation (for @Service methods, not just controller parameters).

Java
// @Valid — standard, no groups
@PostMapping
public UserDTO create(@Valid @RequestBody CreateUserRequest req) {}

// @Validated — with groups (different validation on create vs update)
@PostMapping
public UserDTO create(@Validated(Create.class) @RequestBody UserRequest req) {}

@PutMapping("/{id}")
public UserDTO update(@Validated(Update.class) @RequestBody UserRequest req) {}

// In the request: @NotNull on id only for Update group
@NotNull(groups = Update.class)
private Long id;

@NotBlank(groups = {Create.class, Update.class})
private String name;
Q20. Design an error response format for a production REST API. What fields must it have?

The industry standard is RFC 7807 "Problem Details for HTTP APIs." A well-designed error response must have:

JSON
{
  "status": 422,              // HTTP status code (machine-readable)
  "error": "VALIDATION_FAILED", // Stable error code (machine-readable, never changes)
  "message": "Email is already registered", // Human-readable (can change)
  "path": "/api/v1/users",   // Which endpoint
  "timestamp": "2024-01-15T10:30:00Z", // When (for log correlation)
  "traceId": "abc123",        // Distributed tracing ID (for log search)
  "fieldErrors": [            // For validation failures
    { "field": "email", "message": "must not be blank" },
    { "field": "age",   "message": "must be 18 or older" }
  ]
}

Rules:

  • Use a stable machine-readable error code (VALIDATION_FAILED) — clients can switch on this. Don't use the human message as a code.
  • Include traceId for distributed tracing correlation (from Spring Cloud Sleuth/Micrometer)
  • Never include stack traces or internal class names
  • Always set the correct HTTP status code — body code should match header code
Q21. What happens when a @Service method throws a checked exception — does @Transactional roll back?

By default, @Transactional only rolls back for unchecked exceptions (RuntimeException and its subclasses). It commits on checked exceptions, even if the exception propagates up.

This is a notorious footgun. You throw an IOException, the method exits "exceptionally," but the database changes are committed anyway.

Fix: use rollbackFor

Java
// Roll back for ALL exceptions (checked and unchecked)
@Transactional(rollbackFor = Exception.class)
public void riskyMethod() throws IOException { ... }

// Roll back for specific checked exception
@Transactional(rollbackFor = BusinessException.class)
public void businessMethod() throws BusinessException { ... }

// Prevent rollback for a specific exception (e.g., when you want to log but commit)
@Transactional(noRollbackFor = NotFoundException.class)
public void method() { ... }

🗄️ Q&A — CRUD, Pagination & Data Layer

Q22. Why should you never expose JPA entities directly from REST controllers?

Three serious problems occur when you serialize entities directly:

  1. Security: You expose internal fields (password hashes, audit fields, internal IDs, database-specific metadata) to the client. You'd need @JsonIgnore scattered everywhere and one missed field is a security leak.
  2. N+1 queries during serialization: Jackson traverses all getters to serialize. A lazy-loaded @OneToMany collection gets triggered during serialization — outside any transaction (transaction closed after repository call). You get LazyInitializationException or, if open-session-in-view is enabled, an N+1 query storm.
  3. Coupling: Your API contract becomes your database schema. Any database refactoring (renaming a column, adding a field) immediately breaks the API contract.

The correct approach: map entities to DTOs in the service layer before returning. Use record types or MapStruct for efficient, type-safe mapping.

Q23. What is the difference between Page and Slice in Spring Data? Which one should you use for an infinite scroll feed?

Page<T> runs two SQL queries: the data query (with LIMIT/OFFSET) AND a COUNT(*) query to get total elements. This lets you show "Page 3 of 47".

Slice<T> runs only the data query but fetches size + 1 rows. If it gets size+1 rows back, it knows there's a next page (hasNext = true) and strips the extra row from the result. No COUNT query.

For infinite scroll → use Slice. Reasons:

  • COUNT(*) on a large table (millions of rows) is expensive and gets slower over time
  • Infinite scroll doesn't show "page X of Y" — it only needs to know "is there more?"
  • Slice is more performant at scale

For paginated tables → use Page. The total count is needed to render page numbers and "showing 41-60 of 234 results."

Q24. Explain cursor-based pagination. Why is it better than OFFSET at scale?

OFFSET problem: LIMIT 20 OFFSET 10000 forces the database to scan and discard 10,000 rows before returning 20. With an index, each "row skip" still has a cost. At OFFSET 1,000,000, the query takes seconds even with an index.

Cursor-based approach:

SQL
-- First page
SELECT * FROM posts ORDER BY created_at DESC, id DESC LIMIT 20;

-- Next page (cursor = {created_at: '2024-01-15T10:00:00Z', id: 500})
SELECT * FROM posts
WHERE (created_at, id) < ('2024-01-15T10:00:00Z', 500)
ORDER BY created_at DESC, id DESC
LIMIT 20;

The compound index on (created_at, id) makes this an O(log n) B-tree seek. Performance is constant regardless of how deep you paginate.

Trade-offs: Can't jump to page 50 directly. Can't show total count. Cursor must be encoded (base64 or JWT) to hide implementation details. Perfect for feeds (Twitter, Instagram), not for admin tables that need page jumping.

Q25. What is @Transactional(readOnly = true) and what does it actually do?

Setting readOnly = true tells Spring (and Hibernate) that this transaction will not modify data. This enables several optimizations:

  • Hibernate's dirty checking is skipped: Normally, Hibernate takes a snapshot of all loaded entities and compares on flush to detect changes. For read-only transactions, this snapshot is never taken — saving memory and CPU.
  • Hibernate flushes nothing: The flush mode is set to MANUAL. Even if you accidentally modify a loaded entity, Hibernate won't flush it.
  • JDBC driver hint: Some drivers optimize read-only connections (e.g., PostgreSQL driver can send to read replicas in a read-replica setup).
  • Spring DataSource routing: Combined with AbstractRoutingDataSource, readOnly = true can automatically route to a read replica.

Rule: Mark ALL read methods with @Transactional(readOnly = true). Only mark write methods with plain @Transactional.

Q26. What is Open Session in View (OSIV) and why should it be disabled in production?

OSIV is a pattern where the Hibernate session (and database transaction) stays open for the entire HTTP request, including the controller and view rendering phases. Spring Boot enables it by default (spring.jpa.open-in-view=true).

Why it seems helpful: Prevents LazyInitializationException when serializing entities in the controller — the session is still open when Jackson triggers lazy loading.

Why it's harmful in production:

  • Database connections are held for the entire request (including slow I/O, rendering, etc.), exhausting the connection pool under load
  • Lazy loading silently fires in the controller/view layer — invisible N+1 queries that don't show up in service-layer profiling
  • Creates a false sense of security — makes bad practices (serializing entities directly) work without errors in development but cause performance collapse at scale

Fix: spring.jpa.open-in-view=false. Use DTOs. Load everything you need within the service transaction boundary.

📄 Q&A — Documentation, Versioning & OpenAPI

Q27. What is OpenAPI 3.0 and how does SpringDoc generate it automatically?

OpenAPI 3.0 (formerly Swagger) is a language-agnostic specification format (YAML/JSON) that describes REST APIs: endpoints, HTTP methods, parameters, request/response schemas, authentication mechanisms.

SpringDoc generation process:

  1. At application startup, SpringDoc scans all @RestController beans
  2. It reads @RequestMapping, @GetMapping etc. for path/method
  3. It reads method parameter types and annotations for request schema
  4. It reads return types for response schema
  5. It builds an in-memory OpenAPI 3.0 model
  6. This model is served at /v3/api-docs as JSON
  7. Swagger UI at /swagger-ui.html fetches this JSON and renders interactive docs

SpringDoc respects @Operation, @ApiResponse, @Schema annotations for documentation enrichment. Without annotations, it still generates a functional spec from types alone.

🏆 Q&A — Best Practices, Design & Advanced Topics

Q30. How would you implement rate limiting for a REST API? What algorithm would you use?

The most common production-grade algorithms:

Token Bucket (recommended): Each client has a "bucket" with a max token capacity. Tokens refill at a constant rate. Each request consumes one token. If bucket is empty → 429. Allows burst traffic (uses saved-up tokens) while enforcing average rate. This is what Stripe and GitHub use.

Fixed Window Counter: Count requests in a fixed time window (e.g., 100 req/min). Simple but has a "boundary burst" problem — 100 requests at 0:59 + 100 at 1:01 = 200 requests in 2 seconds.

Sliding Window Log: Track timestamp of each request. Count requests in the rolling window. Most accurate but memory-intensive.

Java
// Bucket4j + Redis implementation
@Component
public class RateLimitFilter implements Filter {

    private final ProxyManager<String> proxyManager;

    @Override
    public void doFilter(ServletRequest req, ServletResponse resp,
                          FilterChain chain) throws IOException, ServletException {
        String clientIp = ((HttpServletRequest) req).getRemoteAddr();

        BucketConfiguration config = BucketConfiguration.builder()
            .addLimit(Bandwidth.classic(100, Refill.greedy(100, Duration.ofMinutes(1))))
            .build();

        Bucket bucket = proxyManager.builder().build(clientIp, config);

        if (bucket.tryConsume(1)) {
            chain.doFilter(req, resp);
        } else {
            HttpServletResponse httpResp = (HttpServletResponse) resp;
            httpResp.setStatus(429);
            httpResp.setHeader("Retry-After", "60");
            httpResp.getWriter().write("{\"error\": \"Rate limit exceeded\"}");
        }
    }
}
Q31. What is CORS and how does Spring handle it? What's the security implication of misconfiguring it?

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that restricts JavaScript code from making requests to a different origin (protocol + domain + port) than the page it's loaded from. Without CORS headers, the browser blocks the response.

How it works: For "complex" requests (non-GET, or with custom headers), browsers send a preflight OPTIONS request first. The server responds with CORS headers specifying what's allowed. Browser then decides to proceed or block.

Spring configuration:

Java
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://myapp.com")       // ← NEVER use * for auth'd APIs
            .allowedMethods("GET", "POST", "PUT", "DELETE", "PATCH")
            .allowedHeaders("Authorization", "Content-Type")
            .allowCredentials(true)  // ← Required for cookies/auth headers
            .maxAge(3600);             // ← Cache preflight for 1 hour
    }
}

Security risk of allowedOrigins("*"): Allows any website to make authenticated requests to your API using the user's browser cookies. If your API uses cookie-based auth, this enables CSRF attacks from any malicious site.

Q32. When should you use 422 vs 400 for request failures?

400 Bad Request = The request itself is malformed. The server cannot parse it. Examples: invalid JSON syntax, missing required Content-Type header, URL path variable is not a valid number.

422 Unprocessable Entity = The request is syntactically valid (server can parse it) but semantically invalid — the data violates business rules. Examples: email field is properly formatted but already registered, age is a valid integer but below the minimum, start date is after end date.

Practical guidance used by major APIs:

  • Stripe uses 400 for both, with a detailed error code in the body
  • GitHub uses 422 for semantic validation failures
  • The distinction matters for clients that want to differentiate "retry with correct syntax" (400) from "this data is logically invalid" (422)

For Spring Boot: @Valid failures throw MethodArgumentNotValidException, which maps to 400 by default. You can change to 422 in your @ExceptionHandler.

Q33. How do you handle partial failures in a REST API that orchestrates multiple services?

This is a distributed systems design question. When a single API call triggers multiple downstream operations, partial failure is possible.

Option 1: Saga Pattern (Event-driven) — Each step publishes an event. If step 3 fails, compensating transactions roll back steps 1 and 2. Eventual consistency. Complex to implement but most resilient.

Option 2: Outbox Pattern for side effects — Save the main entity AND the outbox event in the same DB transaction. A separate worker processes the outbox and calls downstream services. The main response is immediate; side effects are guaranteed to eventually happen.

Option 3: 202 Accepted + polling — Return 202 immediately with a job ID. The API processes asynchronously. Client polls GET /jobs/{id} for status (PENDING/COMPLETED/FAILED). Appropriate when operations take seconds or minutes.

What NOT to do: Return 200 from the main operation and silently ignore failures in downstream calls. Clients will think everything succeeded when data is inconsistent.

Q34. What is ETag and how does it enable efficient HTTP caching for REST APIs?

ETag (Entity Tag) is a hash/version string representing a specific version of a resource. The server includes it in GET responses. The client caches both the response body and ETag.

Conditional GET flow:

  1. Client: GET /users/1 → Server: 200 OK, ETag: "v1abc123"
  2. Client caches response + ETag
  3. Client: GET /users/1, If-None-Match: "v1abc123"
  4. Server checks: has user 1 changed? If ETag still matches → 304 Not Modified (no body!) → client uses its cache. If changed → 200 OK with new ETag and body.

Optimistic locking with ETag:

Java
// PUT with If-Match prevents "lost update" race condition
// Client A reads user (ETag: "v1")
// Client B updates user (ETag becomes "v2")
// Client A tries PUT with If-Match: "v1" → server returns 412 Precondition Failed
// Client A must re-fetch, see the conflict, and re-apply changes

@PutMapping("/{id}")
public ResponseEntity<UserDTO> update(
        @PathVariable Long id,
        @RequestBody UpdateUserRequest req,
        @RequestHeader(value = "If-Match", required = false) String ifMatch) {

    UserDTO current = userService.findById(id);
    if (ifMatch != null && !ifMatch.equals(current.getEtag())) {
        return ResponseEntity.status(HttpStatus.PRECONDITION_FAILED).build();
    }
    return ResponseEntity.ok(userService.update(id, req));
}

Q36. What is the correct way to handle PATCH requests in Spring REST? What is the JSON Merge Patch vs JSON Patch standard?

There are two RFC standards for PATCH request bodies:

JSON Merge Patch (RFC 7396) — content type: application/merge-patch+json Simple: send only the fields you want to change. Set a field to null to delete it. Cannot do array operations.

Java
// JSON Merge Patch — simplest approach
// PATCH /users/1
// Content-Type: application/merge-patch+json
// Body: {"email": "new@email.com"}  ← only email changes, everything else untouched

@PatchMapping(value = "/{id}", consumes = "application/merge-patch+json")
public ResponseEntity<UserDTO> patch(
        @PathVariable Long id,
        @RequestBody JsonMergePatch patch) {

    UserDTO current = userService.findById(id);
    JsonNode currentNode = objectMapper.valueToTree(current);
    JsonNode patched = patch.apply(currentNode);     // Apply merge
    UserDTO updated = objectMapper.treeToValue(patched, UserDTO.class);
    return ResponseEntity.ok(userService.save(updated));
}

JSON Patch (RFC 6902) — content type: application/json-patch+json Operations: add, remove, replace, move, copy, test. More powerful but complex. Use for array manipulation or atomic multi-field operations.

Simple approach for most teams: Use a standard request body with Optional fields — null means "don't change this." Not RFC-standard but simpler to implement and common in production.

Q37. How would you implement API key authentication in Spring REST?
Java
// Step 1: Custom authentication filter
public class ApiKeyAuthFilter extends OncePerRequestFilter {

    private static final String API_KEY_HEADER = "X-API-Key";
    private final ApiKeyService apiKeyService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                       HttpServletResponse response,
                                       FilterChain chain) throws IOException, ServletException {
        String apiKey = request.getHeader(API_KEY_HEADER);

        if (apiKey == null || apiKey.isBlank()) {
            sendError(response, 401, "Missing API key");
            return;
        }

        var details = apiKeyService.validateKey(apiKey);
        if (details.isEmpty()) {
            sendError(response, 401, "Invalid API key");
            return;
        }

        // Store in SecurityContext for downstream use
        UsernamePasswordAuthenticationToken auth =
            new UsernamePasswordAuthenticationToken(
                details.get(), null, details.get().getAuthorities());
        SecurityContextHolder.getContext().setAuthentication(auth);

        chain.doFilter(request, response);
    }
}

// Step 2: Store API keys hashed in database (NEVER plaintext)
public Optional<ApiKeyDetails> validateKey(String rawKey) {
    // Hash the incoming key and compare to stored hash
    // (similar to password verification — use SHA-256 for API keys, BCrypt for passwords)
    String keyHash = DigestUtils.sha256Hex(rawKey);
    return apiKeyRepository.findByKeyHashAndActiveTrue(keyHash);
}

Key security points:

  • Never store API keys in plaintext — store a SHA-256 hash
  • API key lookups must be constant-time to prevent timing attacks
  • Log the first few characters (prefix) for debugging, never the full key
  • Implement key scopes (read-only vs read-write keys)
  • API keys over HTTPS only — never HTTP
Q38. What is the Richardson Maturity Model? At what level is a "real" REST API?

The RMM is a 4-level model measuring how RESTful an API is:

  • Level 0: HTTP as tunnelling. Single URI (e.g., /api), only POST. SOAP-style RPC. Not REST at all.
  • Level 1: Resources. Multiple URIs (one per resource type). Still no proper HTTP verb usage.
  • Level 2: HTTP Verbs. Multiple resources + correct GET/POST/PUT/DELETE usage + correct status codes. This is what 95% of "REST" APIs do.
  • Level 3: HATEOAS. Responses include hypermedia controls — links to what the client can do next. This is "true REST" per Roy Fielding.

What Fielding actually says: Fielding famously stated that Level 2 without HATEOAS is NOT REST — he was frustrated that people called HTTP + JSON APIs "REST" when they didn't implement hypermedia. In practice, the industry accepted Level 2 as the pragmatic standard. Most engineers (and job descriptions) mean "Level 2" when they say "REST API."

Interview answer: Acknowledge the distinction, explain that Level 3 is theoretically correct REST, but Level 2 is the production standard for practical reasons.

Q39. How does Spring Boot's embedded Tomcat differ from deploying to an external Tomcat? What are the deployment implications?

Embedded Tomcat: Tomcat is a library inside your JAR. Your main class starts Tomcat programmatically. You ship a self-contained fat JAR and run it with java -jar app.jar.

External Tomcat (WAR deployment): Tomcat runs independently. You package a WAR file and deploy it to the Tomcat webapps directory. Tomcat manages the lifecycle.

Aspect Embedded JAR External WAR
Deployment unit Executable JAR WAR file
Startup java -jar app.jar Deploy to /webapps/
Containerization Docker-native, simple Complex (Tomcat image)
Multiple apps per server No (one app = one JVM) Yes
Config application.properties Tomcat's server.xml
Modern preference Yes (cloud-native) Legacy enterprise

For cloud-native microservices, embedded is strictly better — simpler Dockerfiles, no shared Tomcat to manage, independent scaling and versioning of each service.

Q40. How would you approach debugging a REST endpoint that returns the wrong status code sporadically?

This is a debugging methodology question. Structure your answer systematically:

1. Reproduce reliably: Find the exact request that triggers it. Check logs for patterns — is it only under load? Only for specific user IDs? Only with specific headers?

2. Add correlation IDs: If not already present, add request tracing. Every request gets a UUID in MDC. All log statements for that request carry it. Now you can trace the exact path of a failing request.

3. Check exception handler ordering: Multiple @ControllerAdvice classes may conflict. Check @Order annotations — higher order = lower priority in Spring. One @ControllerAdvice may be "stealing" exceptions from another.

4. Check @ResponseStatus on exception classes: If the exception class has @ResponseStatus, that takes effect even if @ControllerAdvice handles it — unless @ControllerAdvice explicitly sets a status.

5. Spring Boot Actuator: Check /actuator/conditions to verify auto-configurations. Check /actuator/mappings to verify all URL mappings.

6. Filter vs Interceptor interference: A filter may be modifying the response status after your controller runs. Check all registered filters.

57 What is the difference between @NotNull and @NotBlank? @NotNull: value != null. @NotEmpty: not null AND not empty (empty string/collection). @NotBlank: not null AND not whitespace-only (for strings). Most form inputs need @NotBlank.
58 How does Spring Security integrate with Spring REST? Spring Security operates as a FilterChain that runs BEFORE DispatcherServlet. SecurityFilterChain processes authentication/authorization. If rejected → 401/403 response directly, controller is never reached.
59 What causes LazyInitializationException in REST controllers? Accessing a lazy-loaded JPA relationship (e.g., user.getOrders()) after the transaction/session has closed. Occurs when entity is returned from service (session closes) and Jackson tries to serialize the lazy collection in the controller. Fix: use DTOs, fetch eagerly, or use @Transactional on controller (anti-pattern).
60 What is the best practice for returning empty collections? Return an empty array [] not null. Never return 404 for an empty collection — 404 means the resource (the collection endpoint) doesn't exist. An empty collection at GET /users is 200 OK with [].
61 What is @RestController's base annotation and what does it mean? @Controller marks it for component scanning and as a handler. @ResponseBody tells DispatcherServlet: don't look for a view, write the return value directly to HTTP response body using HttpMessageConverter.
62 How do you test a REST controller without starting the full server? @WebMvcTest(UserController.class) starts only the web layer (controllers, filters, etc.) in a mock servlet environment. MockMvc performs HTTP requests. @MockBean mocks service dependencies. Tests run in milliseconds.
63 How does @Transactional(propagation = REQUIRES_NEW) behave? Suspends the current transaction, starts a new independent transaction, executes, commits/rolls back independently, then resumes the original transaction. Used for audit logs that must commit even if outer transaction rolls back.
64 What is Spring Boot Actuator's /health endpoint used for in REST APIs? Health checks for load balancers and container orchestration (Kubernetes liveness/readiness probes). Returns 200 if UP, 503 if DOWN. Custom HealthIndicators can check DB, message brokers, external dependencies.
65 How do you handle file uploads in Spring REST? @PostMapping with MultipartFile parameter. Set Content-Type: multipart/form-data. Configure size limits in application.properties. Never store files locally in production — use object storage (S3, GCS).
66 What is the difference between 200 OK and 202 Accepted? 200 = request processed, result in body. 202 = request accepted for processing but not done yet. Use 202 for asynchronous operations (email sending, batch processing, payment charging). Include a job/status URL in response body.
67 How do you prevent SQL injection in Spring REST? Use parameterized queries (JPA/Spring Data do this by default with @Query). NEVER concatenate user input into query strings. Use @Param annotations to bind values safely. Never use native SQL with string concatenation.
68 What does spring.jpa.open-in-view=false do? Disables Open Session In View pattern. Hibernate session closes when service transaction ends, not when HTTP response is sent. Forces proper DTO usage. Prevents silent N+1 queries in the web layer. Should be false in production.
69 How does Spring handle request parameter type conversion? ConversionService converts String (all HTTP params are strings) to the target type. Supports primitives, enums, LocalDate, UUID, etc. You can register custom Converter<String, MyType> beans for custom types.
70 What is a DTO vs a VO vs an Entity? Entity = JPA-managed, maps to database row, has lifecycle. DTO (Data Transfer Object) = crosses boundary (HTTP/service), no JPA annotations, transport-focused. VO (Value Object) = immutable, defined by its attributes, business concept. Records are ideal for VOs and DTOs in Java.
71 What HTTP method is correct for a "search" operation with complex criteria? Controversial: GET with query params for simple criteria. POST to a /search sub-resource for complex criteria (when the filter doesn't fit in a URL). GET is semantically correct but POST avoids URL length limits and allows JSON body for complex filters. GitHub, Elasticsearch use POST for search.
72 What is the Content-Type vs Accept header distinction? Content-Type: format of the request body being sent. Accept: format the client wants in the response. Example: Content-Type: application/json (I'm sending JSON), Accept: application/xml (please respond in XML).
73 How does Spring Boot choose which HttpMessageConverter to use? For serialization: DispatcherServlet calls converter.canWrite(returnType, contentType). First converter that says yes wins. Order matters — register custom converters before or after built-ins using configureMessageConverters() or extendMessageConverters().
74 What is the difference between @PathVariable and @MatrixVariable? @PathVariable extracts the segment (/users/42). @MatrixVariable extracts name-value pairs from URL segments (/users/42;role=admin;status=active). MatrixVariable is rarely used in practice and requires enabling in Spring config.
75 How do you implement request-response logging for all REST endpoints? Implement OncePerRequestFilter that wraps request in ContentCachingRequestWrapper and response in ContentCachingResponseWrapper. Log after chain.doFilter(). This allows reading both bodies. Be careful: log body size limits, skip binary content types, mask sensitive fields.
76 What is the Outbox Pattern and why is it important for REST APIs that emit events? Instead of publishing to a message broker in a separate transaction (which can fail silently), write the event to an "outbox" table in the same DB transaction as your main data change. A separate worker reads the outbox and publishes events. Guarantees at-least-once delivery. Prevents inconsistency between DB and broker.
77 What is @Async and how does it interact with @Transactional? @Async runs the method in a different thread. Since @Transactional is thread-bound (stored in ThreadLocal), the async method runs in a NEW transaction, not the caller's transaction. Data committed by the caller is visible. Data not yet committed is NOT visible in the async thread.
78 How do you version a field in the response DTO without breaking existing clients? Additive changes (new fields) are backward compatible — existing clients ignore unknown fields (if @JsonIgnoreProperties(ignoreUnknown=true) or default Jackson config). Breaking changes (removing/renaming) require new API version. Use @JsonAlias to accept old and new field names during transition period.
79 What is the correct HTTP status for "this feature is not yet implemented"? 501 Not Implemented. Not 404 (resource exists, endpoint exists, feature doesn't work). Not 500 (would suggest an error). 501 explicitly communicates "the server does not support the functionality required to fulfill the request."
80 How would you design a REST API to support both synchronous and asynchronous clients? For synchronous: respond within timeout with the result. For async: support Prefer: respond-async header. If received, return 202 + Location: /jobs/{id}. Client polls GET /jobs/{id} → PENDING/COMPLETED/FAILED. When COMPLETED, GET /jobs/{id}/result. This is the REST async result pattern.
