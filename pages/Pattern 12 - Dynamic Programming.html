The Foundation

Build the mental model that makes every DP problem feel familiar.

🧠 §1 — What is DP, Really?

💡

Mental Model: DP is just recursion with a memory upgrade. You've already solved every DP problem recursively — DP just refuses to re-solve the same sub-problem twice.

Without DP (Naive Recursion)

Imagine calculating Fibonacci. fib(5) calls fib(4) and fib(3). fib(4) calls fib(3) again. By fib(50) you're making billions of calls — 90% are repeats.

Exponential time. Completely useless.

With DP (Memoized / Tabulated)

The first time you solve fib(3), you write the answer down. Every future call just looks it up in O(1). You solve each unique sub-problem exactly once.

Polynomial time. Interview-ready.

The 2 Conditions for DP

Every DP problem satisfies both of these — if either is missing, DP doesn't apply:

1. Optimal Substructure

The optimal answer to the whole problem can be built from optimal answers to sub-problems.

✅
Example: Shortest path from A→C through B = shortest(A→B) + shortest(B→C)

2. Overlapping Subproblems

The recursive breakdown hits the same sub-problem multiple times. Without this, plain recursion or divide-and-conquer already works fine.

⚠️
Merge Sort does NOT have overlapping subproblems — each sub-array is unique. DP doesn't help there.
🎯
Interview Tip

When an interviewer asks "can DP solve this?" — your first move is to check these 2 conditions out loud. It shows you understand the theory, not just the technique.

🪜 §1.2 — The 5-Step DP Framework

Use this framework on every single DP problem. It's not optional. Engineers fail DP interviews because they jump to code without defining the state properly.

1
Define the State
What does dp[i] (or dp[i][j]) represent in English? Write this down explicitly. A wrong state definition dooms every step after it.

Ask: "What do I need to know at position i to answer the problem?"
2
Find the Recurrence
Express dp[i] in terms of smaller states. This is the mathematical heart of DP. Usually looks like: dp[i] = f(dp[i-1], dp[i-2], ...)

Ask: "If I already know all answers up to i-1, how do I compute i?"
3
Identify Base Cases
These are the smallest inputs you can answer directly without the recurrence. Every DP table needs at least one base case — it's where the chain starts.

Common trap: Off-by-one in base cases. Should it be dp[0]=0 or dp[1]=1? Draw it out.
4
Determine Traversal Order
Fill states in the order they are needed. If dp[i] needs dp[i-1], you must fill left-to-right. For 2D DP the direction matters critically.

Rule: The state you need must already be computed before you compute the current state.
5
Extract the Answer
The answer is not always dp[n]. Sometimes it's max(dp[0..n]), sometimes dp[m][n], sometimes a running max across all cells.

Common trap: Assuming dp[n] is always the answer and not verifying.
📐
Worked Example — Climbing Stairs (LC #70)

1. State: dp[i] = number of distinct ways to reach step i
2. Recurrence: dp[i] = dp[i-1] + dp[i-2] (came from 1 step back OR 2 steps back)
3. Base cases: dp[1]=1, dp[2]=2
4. Order: left to right, i from 3 to n
5. Answer: dp[n]

⚖️ §1.3 — Top-Down vs Bottom-Up

🔽 Top-Down (Memoization)

  • Write the recursive solution first
  • Add a HashMap or array to cache results
  • Only computes states that are actually needed
  • Easier to write, closer to intuition
  • Stack overflow risk on very deep recursion
JavaTop-Down Template
Map<Integer, Long> memo = new HashMap<>();

private long solve(int i) {
    if (i <= 1) return i; // base case
    if (memo.containsKey(i))
        return memo.get(i);
    long result = solve(i-1) + solve(i-2);
    memo.put(i, result);
    return result;
}

🔼 Bottom-Up (Tabulation)

  • Build a table starting from base cases
  • Iteratively fill states using the recurrence
  • Computes all states in range (may do extra work)
  • No recursion stack — safer for large inputs
  • Often faster in practice (cache-friendly)
JavaBottom-Up Template
long[] dp = new long[n + 1];
dp[0] = 0;
dp[1] = 1; // base cases

for (int i = 2; i <= n; i++) {
    dp[i] = dp[i-1] + dp[i-2]; // recurrence
}

return dp[n]; // extract answer

🧭 Practical Decision Rule

Use Top-Down when: The state space is sparse (not all states will be visited) or the recurrence has complex conditional branching that's clearer as recursion.
Use Bottom-Up when: You need the absolute best performance, the state space is dense, or you want to later optimize space (rolling array trick).
In interviews: Top-down is faster to write correctly — start there, mention you can optimize to bottom-up if asked.

Core Template

JavaLinear DP Template
// dp[i] = answer at index i
int[] dp = new int[n + 1];
dp[0] = /* base case 0 */;
dp[1] = /* base case 1 */;

for (int i = 2; i <= n; i++) {
    dp[i] = /* recurrence using dp[i-1], dp[i-2], ... */;
}

// ── Space Optimization (if dp[i] only needs last 2 states) ──
int prev2 = dp[0], prev1 = dp[1];
for (int i = 2; i <= n; i++) {
    int curr = prev1 + prev2; // example recurrence
    prev2 = prev1;
    prev1 = curr;
}
return prev1;

Problem Walkthroughs

House Robber Pattern

State: dp[i] = max money robbing houses 0..i

dp[i] = max(dp[i-1], dp[i-2] + nums[i])// skip or rob house i

Key insight: You can't rob adjacent houses, so at house i you either skip it (keep dp[i-1]) or rob it (dp[i-2] + value).

Max Subarray (Kadane's)

State: dp[i] = max subarray ending at index i

dp[i] = max(nums[i], dp[i-1] + nums[i])// fresh start or extend

Key insight: If the previous subarray total is negative, restart from current element — extending it would only hurt.

Longest Increasing Subsequence

State: dp[i] = LIS length ending at index i

dp[i] = max(dp[j]+1) for all j<i where nums[j]<nums[i]

Answer: max(dp[0..n]) — not dp[n]! The longest sequence doesn't have to end at the last element.

Word Break

State: dp[i] = can we segment s[0..i-1] into words?

dp[i] = true if ∃ j<i: dp[j] && s[j..i] ∈ dict

Key insight: A boolean DP — checking if any valid "cut point" j exists where everything before j works and the word from j..i is in the dictionary.

Pattern 2 §4 — Knapsack DP

🎒

Recognition Signal: You have items, each with a weight/cost and value. You have a capacity budget. You want to maximize/minimize value within the budget. Or: "can you reach exactly sum X?"

The Critical Distinction

0/1 Knapsack

Each item can be used at most once. Take it or leave it.

Loop order: inner loop runs BACKWARDS (right to left)

Why? Forward iteration would let you "use the same item again" within the same item pass, since you'd read updated values from the same row.

Java
for (int i = 0; i < n; i++) {
  for (int w = W; w >= weight[i]; w--) {
    dp[w] = Math.max(dp[w],
        dp[w - weight[i]] + value[i]);
  }
}

Unbounded Knapsack

Each item can be used unlimited times. Take as many as you want.

Loop order: inner loop runs FORWARDS (left to right)

Why? Forward iteration allows reading an already-updated dp[w - weight[i]], meaning we CAN use the same item multiple times in one pass.

Java
for (int i = 0; i < n; i++) {
  for (int w = weight[i]; w <= W; w++) {
    dp[w] = Math.max(dp[w],
        dp[w - weight[i]] + value[i]);
  }
}
🚨
Loop Order is THE Trap

Swapping the loop direction (forward vs backward) for 0/1 vs Unbounded is one of the top interview mistakes. If you can't remember which is which, remember this: Backwards = Once, Forwards = Many times. The inner loop direction controls whether an item gets counted multiple times.

Subset Sum / Partition Equal Subset

JavaSubset Sum — Can we reach exactly target T?
// State: dp[w] = can we form sum w using elements so far?
boolean[] dp = new boolean[T + 1];
dp[0] = true; // empty subset sums to 0

for (int num : nums) {
    for (int w = T; w >= num; w--) { // backward = 0/1 knapsack
        dp[w] = dp[w] || dp[w - num];
    }
}
return dp[T];

// Partition Equal Subset: just check if sum/2 is reachable
// (if total sum is odd → immediately false)

Pattern 3 §5 — Two-Sequence (LCS) DP

🔑

Recognition Signal: You have two sequences (strings, arrays). The answer involves aligning, comparing, or transforming one into the other. The state needs to track position in BOTH sequences.

Animated Matrix Fill — LCS of "ABCDE" and "ACE"

Green = characters match (diagonal + 1). Blue = filled from best of up/left.

The Match-or-Skip Recurrence — Explained

if s1[i] == s2[j]: dp[i][j] = dp[i-1][j-1] + 1 // characters match → extend diagonal else: dp[i][j] = max(dp[i-1][j], dp[i][j-1]) // skip one char from either string
🧩
Visual Intuition

Think of the 2D table as a map. You start at (0,0) and want to reach (m,n). When characters match, you move diagonally (progress in both). When they don't, you choose to skip in s1 (move down) or skip in s2 (move right), taking whichever direction gave the better score.

Three Classic Two-Sequence Problems

LCS (Longest Common Subsequence)
match: dp[i-1][j-1] + 1 skip: max(dp[i-1][j], dp[i][j-1]) answer: dp[m][n]
Edit Distance (Levenshtein)
match: dp[i-1][j-1] (no cost) replace: dp[i-1][j-1] + 1 delete: dp[i-1][j] + 1 insert: dp[i][j-1] + 1 dp[i][j] = min of above
Longest Common Substring
match: dp[i-1][j-1] + 1 no match: dp[i][j] = 0 (reset!) answer: max value seen in table

⚠️ Unlike LCS, reset to 0 on mismatch. Answer is the global max, NOT dp[m][n].

JavaEdit Distance — Full Template
public int minDistance(String word1, String word2) {
    int m = word1.length(), n = word2.length();
    int[][] dp = new int[m+1][n+1];

    // Base cases: transform to/from empty string
    for (int i = 0; i <= m; i++) dp[i][0] = i;
    for (int j = 0; j <= n; j++) dp[0][j] = j;

    for (int i = 1; i <= m; i++) {
        for (int j = 1; j <= n; j++) {
            if (word1.charAt(i-1) == word2.charAt(j-1)) {
                dp[i][j] = dp[i-1][j-1]; // no cost on match
            } else {
                dp[i][j] = 1 + Math.min(dp[i-1][j-1],  // replace
                               Math.min(dp[i-1][j],    // delete
                                        dp[i][j-1]));  // insert
            }
        }
    }
    return dp[m][n];
}

Pattern 4 §6 — Interval DP

🔑

Recognition Signal: The problem asks about a subarray or substring from index i to j. You need answers for all intervals, and a larger interval depends on smaller intervals inside it.

🚨
The #1 Interval DP Trap — Wrong Iteration Order

You CANNOT iterate by i from 0 to n. When computing dp[i][j], you need dp[i][k] and dp[k+1][j] — both are smaller intervals. You must iterate by interval LENGTH, from small to large. Start with length 1, then 2, then 3...

JavaInterval DP Template — Iterate by Length
int[][] dp = new int[n][n];

// Base case: length-1 intervals
for (int i = 0; i < n; i++) {
    dp[i][i] = /* base value */;
}

// Fill by increasing interval length
for (int len = 2; len <= n; len++) {
    for (int i = 0; i <= n - len; i++) {
        int j = i + len - 1; // right boundary
        dp[i][j] = Integer.MAX_VALUE; // or MIN for max problems
        for (int k = i; k < j; k++) { // try all split points
            dp[i][j] = Math.min(dp[i][j],
                dp[i][k] + dp[k+1][j] + /* cost of merge */);
        }
    }
}
return dp[0][n-1];

Palindromic Substrings — The Interval DP Classic

State: dp[i][j] = true if s[i..j] is a palindrome

Recurrence: dp[i][j] = (s[i] == s[j]) && dp[i+1][j-1]

Base: dp[i][i] = true (single char), dp[i][i+1] = (s[i]==s[i+1])

Answer: Count all true entries, OR for Longest Palindromic Substring, track the max length seen.

⚠️
Direction matters! For palindromes, the outer iteration must go from bottom (large i) to top (small i), because dp[i][j] needs dp[i+1][j-1].

Pattern 5 §7 — Grid / Matrix DP

🔑

Recognition Signal: You're moving through a 2D grid. Movement is typically right and down (or all 4 directions). Answer at each cell depends on neighbors already computed.

JavaGrid DP Template — Min/Max Path
int[][] dp = new int[m][n];
dp[0][0] = grid[0][0]; // base case: top-left

// Fill first row (can only come from the left)
for (int j = 1; j < n; j++)
    dp[0][j] = dp[0][j-1] + grid[0][j];

// Fill first column (can only come from above)
for (int i = 1; i < m; i++)
    dp[i][0] = dp[i-1][0] + grid[i][0];

// Fill rest — came from top OR left, take minimum
for (int i = 1; i < m; i++) {
    for (int j = 1; j < n; j++) {
        dp[i][j] = grid[i][j] +
            Math.min(dp[i-1][j], dp[i][j-1]);
    }
}
return dp[m-1][n-1];

Grid DP Variants

Unique Paths

Count ways to reach bottom-right. Grid has no obstacles.

dp[i][j] = dp[i-1][j] + dp[i][j-1] base: dp[0][j] = 1, dp[i][0] = 1
Unique Paths II (with obstacles)

Same but cells with obstacle = 0 ways, block propagation.

if obstacle[i][j]: dp[i][j] = 0 else: dp[i][j] = dp[i-1][j] + dp[i][j-1]
Dungeon Game (right-to-left)

Minimum HP needed to reach bottom-right. Fill backwards — start from destination.

dp[i][j] = max(1, min(dp[i+1][j], dp[i][j+1]) - dungeon[i][j])

⚠️ Must fill right-to-left, bottom-to-top.

Maximal Square

Largest square of 1s in a binary matrix.

if matrix[i][j]=='1': dp[i][j] = min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) + 1 answer: max(dp)²

💀 §9 — The DP Trap Hall of Fame

These 8 mistakes cause engineers to fail DP problems they actually understand. Read once. Internalize forever.

1
Wrong State Definition

Defining dp[i] vaguely like "best answer" without specifying exactly what constraint is applied. Example: for LIS, saying dp[i] = "LIS length in array" instead of "LIS length ending at index i". One is correct. One makes the recurrence impossible.

2
Assuming dp[n] is Always the Answer

For LIS, the answer is max(dp[0..n]). For Max Subarray, same. The longest/best sequence doesn't have to end at the last element. Always ask: "could the answer involve a middle state?"

3
Wrong Loop Order in Knapsack (0/1 vs Unbounded)

Using forward iteration for 0/1 knapsack allows items to be reused multiple times, silently giving wrong answers. The bug won't crash — it'll just compute wrong values. Always verify loop direction with a small manual example.

4
Iterating by i in Interval DP Instead of Length

If you iterate i from 0 to n in interval DP, when you compute dp[i][j], the smaller interval dp[i+1][j] may not be filled yet. Always iterate by interval length from small to large.

5
Off-by-One in Base Cases

Initializing dp arrays at index 0 when the problem is 1-indexed (or vice versa). The rule: if dp[i] represents "using the first i elements," then dp[0] = empty set = no elements. Draw the first 3 states manually before coding.

6
Forgetting to Handle Negative Values

Initializing dp with 0 when values can be negative (e.g., Max Subarray, Dungeon Game). Use Integer.MIN_VALUE for max problems with negatives, and always clamp with Math.max(1, ...) for "minimum health" style problems.

7
Not Resetting dp[i][j]=0 for Longest Common Substring

LCS (subsequence) extends across mismatches. Substring (contiguous) must reset to 0 on mismatch. Mixing them up gives wrong length. Substring answer = global max table value. Subsequence answer = dp[m][n].

8
Stack Overflow with Top-Down on Large Inputs

Java's default stack is ~500 recursive calls deep. For n=10000, pure recursion even with memoization will overflow. Always mention this trade-off and switch to bottom-up for large n (LCS with 1000-char strings, etc.).

🎯
Interview Protocol

Before writing any code: state your dp definition in one sentence, write the recurrence, write 2 base cases, say the iteration order, and say where the answer is. This takes 90 seconds and prevents 7 of the 8 traps above.

🏁 Master Summary — The DP Compact

Always Do First

  1. Define state in English
  2. Write the recurrence
  3. State 2 base cases
  4. Say iteration order
  5. Say where answer is

Pattern Triggers

  • Grid → Grid DP
  • Two strings → LCS DP
  • Items+budget → Knapsack
  • Subarray range → Interval DP
  • 1D sequence → Linear DP

Never Forget

  • 0/1 Knapsack → backwards loop
  • Unbounded → forward loop
  • Interval DP → iterate by length
  • LIS answer → max(dp) not dp[n]
  • Substring → reset to 0 on miss

Space Optimization

  • Linear DP: 2 variables
  • Knapsack: 1D array
  • LCS: 2 rows (current + prev)
  • Grid: 1 row + prev value
  • Interval/Tree: usually no shortcut
