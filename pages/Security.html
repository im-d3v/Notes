Why Web Security Exists

The threat model, attack surface, and why every layer matters

🌐 The Internet Was Never Designed to Be Secure

The original internet (ARPANET, 1969) was designed for academic and military communication between trusted parties. The assumption was: everyone on the network is trustworthy. That assumption died when the internet went public.

Every piece of data traveling over a network passes through multiple machines owned by different people. Without security, anyone between you and your destination can read, modify, or replay your data.

🎯 The Core Problem

Without security: your username, password, credit card, and private messages travel as plain text through dozens of servers you've never heard of. Anyone who controls one of those servers owns your data.

The CIA Triad — Three Goals of Security

🔒 Confidentiality

Only authorized parties can read the data. Encryption achieves this. Without it, your data is readable by anyone.

✅ Integrity

Data is not modified in transit. Hashing/HMAC/Digital Signatures detect tampering. Without it, attackers change your bank transfer amount.

⚡ Availability

Systems remain accessible to legitimate users. Rate limiting, DDoS protection achieve this. Without it, attackers take your service down.

Additional Security Properties

  • Authentication — Who are you? Prove your identity.
  • Authorization — What are you allowed to do? Access control.
  • Non-repudiation — You cannot deny performing an action. Digital signatures provide this.
  • Auditability — Complete trail of who did what and when.
💡
Mental Model for Spring Security

Spring Security answers these questions: Who is this request from? (Authentication) What are they allowed to do? (Authorization) Everything else — filters, tokens, sessions — is implementation detail supporting these two questions.

🔐 HTTPS, TLS, and SSL — The Foundation

HTTPS = HTTP + TLS (Transport Layer Security). It wraps HTTP inside an encrypted tunnel, providing confidentiality, integrity, and authentication of the server (and optionally the client).

📚
SSL vs TLS — History

SSL (Secure Sockets Layer) was developed by Netscape in 1995. SSL 1.0, 2.0, and 3.0 were released and all were found insecure. TLS (Transport Layer Security) 1.0 was released in 1999 as SSL 3.1. Engineers still say "SSL" colloquially but always mean TLS. TLS 1.3 (2018) is the current standard. SSL 2.0, 3.0, TLS 1.0, TLS 1.1 are deprecated and should be disabled.

What TLS Provides

🔒 Encryption (Confidentiality)

All data encrypted with AES symmetric encryption using a session key established during the handshake. No one in the middle can read it.

✅ Data Integrity

Every TLS record includes a MAC (Message Authentication Code). Any modification to data in transit is detected and the connection is terminated.

🪪 Server Authentication

Server presents a digital certificate signed by a trusted Certificate Authority. Client verifies the server is who it claims to be — prevents impersonation.

🔑 Symmetric Encryption

Same key encrypts and decrypts. Fast. The challenge: how do you share the key securely? (The "key distribution problem" — solved by asymmetric cryptography).

AES — Advanced Encryption Standard

AES is the gold standard symmetric cipher. The US government uses it for TOP SECRET data. It's the cipher used inside TLS for all HTTPS traffic after the handshake.

JavaAES Encryption in Java
import javax.crypto.*;
import javax.crypto.spec.*;
import java.security.*;
import java.util.Base64;

public class AESExample {

    // AES-256 with GCM mode (authenticated encryption)
    public static String encrypt(String plainText, SecretKey key) throws Exception {
        byte[] iv = new byte[12]; // 96-bit IV for GCM
        new SecureRandom().nextBytes(iv);

        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.ENCRYPT_MODE, key, new GCMParameterSpec(128, iv));

        byte[] encrypted = cipher.doFinal(plainText.getBytes("UTF-8"));

        // Prepend IV to ciphertext (IV is not secret, needed for decryption)
        byte[] combined = new byte[iv.length + encrypted.length];
        System.arraycopy(iv, 0, combined, 0, iv.length);
        System.arraycopy(encrypted, 0, combined, iv.length, encrypted.length);

        return Base64.getEncoder().encodeToString(combined);
    }

    // Generate a 256-bit AES key
    public static SecretKey generateKey() throws Exception {
        KeyGenerator keyGen = KeyGenerator.getInstance("AES");
        keyGen.init(256, new SecureRandom());
        return keyGen.generateKey();
    }
}

ECB encrypts identical blocks to identical ciphertext. You can literally see patterns in the "encrypted" data. The famous ECB-encrypted Linux penguin image shows the penguin outline clearly after "encryption." Always use AES-GCM.

🗝️ Asymmetric Encryption (Public Key Cryptography)

Asymmetric encryption uses two mathematically linked keys: a public key (share with everyone) and a private key (never share). What one key encrypts, only the other can decrypt.

RSA — The Classic Algorithm

RSA security is based on the difficulty of factoring the product of two large prime numbers. A 2048-bit RSA key is currently considered secure. 4096-bit for long-term secrets.

JavaRSA Key Pair Generation + Sign/Verify
import java.security.*;
import java.util.Base64;

public class RSAExample {

    public static void main(String[] args) throws Exception {

        // ── 1. Generate RSA key pair ──────────────────────────────
        KeyPairGenerator keyGen = KeyPairGenerator.getInstance("RSA");
        keyGen.initialize(2048, new SecureRandom());
        KeyPair keyPair = keyGen.generateKeyPair();

        PublicKey publicKey = keyPair.getPublic();   // Share with world
        PrivateKey privateKey = keyPair.getPrivate(); // NEVER share

        String message = "Transfer \$1000 to Account #456";

        // ── 2. Sign with PRIVATE key ──────────────────────────────
        Signature signer = Signature.getInstance("SHA256withRSA");
        signer.initSign(privateKey);
        signer.update(message.getBytes());
        byte[] signature = signer.sign();

        // ── 3. Verify with PUBLIC key ─────────────────────────────
        Signature verifier = Signature.getInstance("SHA256withRSA");
        verifier.initVerify(publicKey);
        verifier.update(message.getBytes());
        boolean valid = verifier.verify(signature);

        System.out.println("Signature valid: " + valid); // true

        // Tamper with message:
        String tampered = "Transfer \$9999 to Account #456";
        verifier.initVerify(publicKey);
        verifier.update(tampered.getBytes());
        boolean validAfterTamper = verifier.verify(signature);
        System.out.println("After tamper: " + validAfterTamper); // false
    }
}

🔍 Cryptographic Hashing

A cryptographic hash function transforms any amount of data into a fixed-size output. The key properties make it fundamentally different from encryption.

Required Properties of a Cryptographic Hash

  • Deterministic — same input always produces same output.
  • One-way (Pre-image resistant) — cannot compute input from output. Given hash H, finding any input M such that hash(M) = H is computationally infeasible.
  • Collision resistant — computationally infeasible to find two different inputs with same hash.
  • Avalanche effect — changing one bit of input changes ~50% of output bits.
  • Fixed output size — SHA-256 always outputs 256 bits regardless of input size.

Why We Hash Passwords

Databases get breached. It's not if, it's when. If you store passwords in plaintext and your database is breached, every user's account on every site where they reuse that password is compromised — forever.

❌ Storing Plain Text

  • Database breach → all passwords exposed
  • DB admins can see all passwords
  • Anyone with DB access can log in as any user
  • LinkedIn 2012: 6.5M plaintext passwords stolen

✅ Hashing Passwords

  • Hash cannot be reversed to password
  • On login: hash input, compare to stored hash
  • DB breach exposes only hashes, not passwords
  • Strong hash (BCrypt) makes brute-force impractical

The Rainbow Table Attack — Why Salting Exists

If you hash password123 with SHA-256, it always produces the same hash. Attackers precompute a huge table: (plaintext → hash) for millions of common passwords. This is called a rainbow table. Lookup is instant.

🧂 Salt — The Solution to Rainbow Tables

A salt is a random value unique per user, prepended to the password before hashing. hash(salt + password). The salt is stored alongside the hash (it doesn't need to be secret). Rainbow tables are precomputed for common passwords — adding a unique salt means the attacker would need to build a separate rainbow table for each user's salt. Computationally infeasible.

JavaSHA-256 with Manual Salt — The Old Way
// Manual salting — this is what BCrypt automates
import java.security.MessageDigest;
import java.security.SecureRandom;

public class ManualSaltExample {

    public static String hashPassword(String password) throws Exception {
        // Generate random 16-byte salt
        byte[] salt = new byte[16];
        new SecureRandom().nextBytes(salt);

        MessageDigest md = MessageDigest.getInstance("SHA-256");
        md.update(salt);                          // add salt first
        byte[] hash = md.digest(password.getBytes());

        // Store: salt + hash together
        return Base64.encode(salt) + ":" + Base64.encode(hash);
    }
}

// PROBLEM: SHA-256 is FAST. Fast hashing is BAD for passwords.
// Modern GPU: ~10 billion SHA-256 hashes per second.
// That means brute-forcing "password123" takes microseconds.
// Solution: Use an intentionally SLOW hash → BCrypt, Argon2, PBKDF2

🔐 BCrypt — Complete Deep Dive

🎯 BCrypt is the Most Important Password Security Algorithm You Need to Know

Designed by Niels Provos and David Mazières in 1999. Built on the Blowfish cipher. Has three properties that make it ideal for passwords: it's slow by design, it has a work factor that scales with hardware, and it includes salt automatically.

BCrypt Hash Format — Anatomy

\$12  = Cost factor (work factor) — 2^12 = 4096 iterations

You never need to store salt separately.

Why Slow is Good for Password Hashing

If BCrypt takes 300ms on one CPU core: an attacker with a GPU farm trying to brute-force a BCrypt hash can try ~100 passwords/second. An attacker trying the same with MD5 can try 10 billion/second. That's a difference of 100 million times. BCrypt's intentional slowness is its most important security feature.

BCrypt in Spring Security

JavaBCrypt — Complete Spring Security Implementation
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;
import org.springframework.security.crypto.password.PasswordEncoder;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class SecurityConfig {

    // ── BCryptPasswordEncoder Bean ────────────────────────────────
    @Bean
    public PasswordEncoder passwordEncoder() {
        // Default cost = 10. Production: use 12
        return new BCryptPasswordEncoder(12);
    }
}

// ── Usage in Service ─────────────────────────────────────────────
@Service
public class UserService {

    private final PasswordEncoder passwordEncoder;
    private final UserRepository userRepository;

    // ── REGISTERING A USER ────────────────────────────────────────
    public User register(String username, String rawPassword) {
        // NEVER store raw password
        String hashedPassword = passwordEncoder.encode(rawPassword);
        // Each call generates NEW random salt → different hash every time!
        // passwordEncoder.encode("same") → different hash each call

        User user = new User(username, hashedPassword);
        return userRepository.save(user);
    }

    // ── VERIFYING A PASSWORD ──────────────────────────────────────
    public boolean verifyPassword(String rawPassword, String storedHash) {
        // matches() extracts salt from storedHash, recomputes, compares
        // Time-safe comparison (prevents timing attacks)
        return passwordEncoder.matches(rawPassword, storedHash);
    }

    // ── UPGRADING COST FACTOR ─────────────────────────────────────
    // As hardware gets faster, upgrade cost factor on next login
    public void upgradePasswordIfNeeded(User user, String rawPassword) {
        if (passwordEncoder instanceof BCryptPasswordEncoder b) {
            if (b.upgradeEncoding(user.getPassword())) {
                user.setPassword(passwordEncoder.encode(rawPassword));
                userRepository.save(user);
            }
        }
    }
}

DelegatingPasswordEncoder — Future-Proof Password Storage

What happens when BCrypt becomes insecure or you want to upgrade to Argon2? DelegatingPasswordEncoder solves this by storing the algorithm identifier in the hash.

// Hash format: {algorithm}encodedPassword
// Examples:
// {bcrypt}$$2a\$12$$N9qo8uLOickgx2ZMRZoMye...
// {argon2}$$argon2id$$v=19$$m=16384,t=2,p=1$$...
// {pbkdf2}5d923b44a6d129f3ddf3e3c8d29412723dcbde72445e8ef6bf3b508fbf17fa4f

@Bean
public PasswordEncoder passwordEncoder() {
    // Use PasswordEncoderFactories for auto-configured DelegatingPasswordEncoder
    return PasswordEncoderFactories.createDelegatingPasswordEncoder();
    // Default: bcrypt with cost=10
    // But can verify older hashes encoded with {noop}, {sha256}, etc.
}

// Custom DelegatingPasswordEncoder:
@Bean
public PasswordEncoder customPasswordEncoder() {
    String idForEncode = "bcrypt";
    Map<String, PasswordEncoder> encoders = new HashMap<>();
    encoders.put("bcrypt", new BCryptPasswordEncoder(12));
    encoders.put("argon2", new Argon2PasswordEncoder(16, 32, 1, 65536, 10));
    encoders.put("pbkdf2", new Pbkdf2PasswordEncoder("", 8, 185000, 256));
    encoders.put("noop", NoOpPasswordEncoder.getInstance()); // legacy only

    return new DelegatingPasswordEncoder(idForEncode, encoders);
    // New passwords: encoded with bcrypt (idForEncode)
    // Old passwords: verified with whichever algo prefix says
    // Migration: on next login, re-encode with current algorithm
}

🔏 HMAC — Hash-Based Message Authentication Code

HMAC combines a cryptographic hash with a shared secret key. It proves both integrity (message wasn't modified) and authenticity (message came from someone who knows the key). This is how JWT HS256 tokens are signed.

Why Plain Hashing Isn't Enough

If you just hash a message, anyone can modify the message and recompute the hash. The hash proves integrity only if the recipient can verify the hash independently. But if the attacker intercepts and modifies the message, they can compute a valid new hash. HMAC solves this by requiring the secret key.

HMAC — HOW IT WORKS ───────────────────────────────────────────────────────── Input: message, secret_key Step 1: Prepare two keys from secret_key: ipad = key XOR 0x36363636... (inner padding) opad = key XOR 0x5C5C5C5C... (outer padding) Step 2: Compute HMAC HMAC = Hash(opad || Hash(ipad || message)) Simplified: HMAC(key, message) = SHA256(key ⊕ opad || SHA256(key ⊕ ipad || message)) SENDER: message = "Transfer \$100 to account 456" key = "supersecretkey" hmac = HMAC-SHA256(key, message) = "7a3f91c8d2b4e6f1..." Send: message + hmac RECEIVER: Recompute HMAC with known key Compare with received HMAC If match → message is authentic AND unmodified If no match → message tampered OR wrong key ATTACKER (doesn't know key): Can modify message CANNOT compute valid HMAC → Forgery impossible without key

HMAC in Java

JavaHMAC-SHA256 Implementation
import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;
import java.util.Base64;

public class HMACExample {

    public static String computeHMAC(String message, String secretKey)
            throws Exception {

        Mac mac = Mac.getInstance("HmacSHA256");
        SecretKeySpec keySpec = new SecretKeySpec(
            secretKey.getBytes("UTF-8"), "HmacSHA256"
        );
        mac.init(keySpec);
        byte[] hmacBytes = mac.doFinal(message.getBytes("UTF-8"));
        return Base64.getUrlEncoder().encodeToString(hmacBytes);
    }

    public static boolean verifyHMAC(String message, String secretKey,
                                    String receivedHmac) throws Exception {
        String computedHmac = computeHMAC(message, secretKey);
        // MessageDigest.isEqual() — CONSTANT TIME comparison
        // Prevents timing attacks (string.equals() exits early on first diff)
        return MessageDigest.isEqual(
            computedHmac.getBytes(),
            receivedHmac.getBytes()
        );
    }

    public static void main(String[] args) throws Exception {
        String message = "userId=123&amount=1000¤cy=INR";
        String key = "payment-webhook-secret-2024";

        String hmac = computeHMAC(message, key);
        System.out.println("HMAC: " + hmac);

        // Verify webhook signature (Stripe, GitHub do this)
        boolean valid = verifyHMAC(message, key, hmac);
        System.out.println("Valid: " + valid); // true

        // Tampered message:
        boolean tamperedValid = verifyHMAC("userId=123&amount=9999¤cy=INR", key, hmac);
        System.out.println("Tampered valid: " + tamperedValid); // false
    }

⚠️
Timing Attacks — Always Use Constant-Time Comparison

Regular string comparison (hmac1.equals(hmac2)) returns false as soon as it finds the first different character. An attacker can measure response time to determine how many characters of the HMAC they got right — potentially recovering the HMAC byte by byte. Always use MessageDigest.isEqual() which compares ALL bytes regardless of where the difference is.

🚦 Authorization — Role-Based and Permission-Based

Authorization = what you're allowed to do. Occurs AFTER authentication. Spring Security has two levels: URL-based (filter chain) and method-based (@PreAuthorize).

RBAC vs ABAC

🏷️ RBAC — Role-Based Access Control

  • Users assigned to roles (ADMIN, USER, MANAGER)
  • Permissions granted to roles
  • Simple, widely understood
  • Spring Security's default model
  • Good for simple permission structures

📋 ABAC — Attribute-Based Access Control

  • Decisions based on attributes: user, resource, environment
  • Example: "Manager can edit reports if in same department"
  • More flexible but complex
  • Spring Security ACL or SpEL expressions
  • Good for fine-grained access control

Method Security — The Right Way

Java@PreAuthorize, @PostAuthorize, @Secured
@Service
public class PaymentService {

    // ── @PreAuthorize — evaluated BEFORE method executes ─────────

    // Simple role check
    @PreAuthorize("hasRole('ADMIN')")
    public List<Payment> getAllPayments() { ... }

    // Multiple roles (any)
    @PreAuthorize("hasAnyRole('ADMIN', 'FINANCE_MANAGER')")
    public Report getReport() { ... }

    // Permission-based (not role-based)
    @PreAuthorize("hasAuthority('payment:write')")
    public Payment createPayment(CreatePaymentRequest req) { ... }

    // Use method parameter in expression (SpEL)
    @PreAuthorize("#userId == authentication.principal.id or hasRole('ADMIN')")
    public UserProfile getProfile(@Param("userId") Long userId) { ... }
    // Users can only view their own profile, unless ADMIN

    // ── @PostAuthorize — evaluated AFTER method executes ─────────
    // Good when you need to check properties of the RETURNED object
    @PostAuthorize("returnObject.ownerId == authentication.principal.id or hasRole('ADMIN')")
    public Payment getPayment(Long paymentId) {
        return paymentRepository.findById(paymentId).orElseThrow();
        // Will throw AccessDeniedException after return if condition fails
    }

    // ── @PreFilter / @PostFilter — filter collections ────────────
    @PostFilter("filterObject.ownerId == authentication.principal.id")
    public List<Payment> getMyPayments() {
        return paymentRepository.findAll(); // returns ALL, Spring filters the list
    }

    // ── Complex SpEL with custom beans ───────────────────────────
    @PreAuthorize("@permissionEvaluator.canAccessPayment(authentication, #paymentId)")
    public Payment processRefund(Long paymentId) { ... }
}

@Component("permissionEvaluator")
public class PaymentPermissionEvaluator {

    private final PaymentRepository paymentRepository;

    public boolean canAccessPayment(Authentication auth, Long paymentId) {
        UserPrincipal user = (UserPrincipal) auth.getPrincipal();
        Payment payment = paymentRepository.findById(paymentId).orElse(null);

        if (payment == null) return false;

        // Admin can access any payment
        if (user.getAuthorities().stream()
               .anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"))) {

        // User can only access their own payments
        return payment.getCustomerId().equals(user.getUser().getId());

@Component
public class LoginFailureHandler implements AuthenticationFailureHandler {

    private final UserRepository userRepository;

    @Override
    public void onAuthenticationFailure(HttpServletRequest request,
                                         HttpServletResponse response,
                                         AuthenticationException exception) throws IOException {

        String username = request.getParameter("email");

        // Increment failed attempts, lock after 5 failures
        userRepository.findByUsernameIgnoreCase(username).ifPresent(user -> {
            user.setFailedLoginAttempts(user.getFailedLoginAttempts() + 1);
            if (user.getFailedLoginAttempts() >= 5) {
                user.setLockedUntil(LocalDateTime.now().plusMinutes(30));
                user.setAccountNonLocked(false);
            }
            userRepository.save(user);
        });

        response.setStatus(HttpStatus.UNAUTHORIZED.value());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        response.getWriter().write("""
            {"error": "Invalid credentials", "code": "AUTH_FAILED"}
            """);
    }

🪙 JWT — JSON Web Token Complete Deep Dive

🎯 JWT is Signed, NOT Encrypted

By default, JWT claims are Base64-encoded (readable by anyone) and signed (tamper-evident). Anyone who intercepts a JWT can decode and read its contents. Never put sensitive data in JWT claims. If you need confidentiality, use JWE (JSON Web Encryption).

JavaJwtService — Production Implementation
@Service
@Slf4j
public class JwtService {

    @Value("${jwt.secret}")
    private String jwtSecret;

    @Value("${jwt.access-token-expiry:3600}")   // 1 hour default
    private long accessTokenExpiry;

    @Value("${jwt.refresh-token-expiry:604800}") // 7 days default
    private long refreshTokenExpiry;

    private SecretKey signingKey;

    @PostConstruct
    public void init() {
        // Decode base64-encoded secret from config
        byte[] keyBytes = Decoders.BASE64.decode(jwtSecret);
        this.signingKey = Keys.hmacShaKeyFor(keyBytes);
        // Key must be at least 256 bits for HS256
    }

    // ── Generate Access Token ────────────────────────────────────
    public String generateAccessToken(UserDetails userDetails) {
        Map<String, Object> claims = new HashMap<>();

        // Add roles as claim
        claims.put("roles", userDetails.getAuthorities().stream()
            .map(GrantedAuthority::getAuthority)
            .collect(Collectors.toList()));

        // Add user type
        claims.put("token_type", "access");

        return buildToken(claims, userDetails, accessTokenExpiry);
    }

    // ── Generate Refresh Token ───────────────────────────────────
    public String generateRefreshToken(UserDetails userDetails) {
        Map<String, Object> claims = Map.of("token_type", "refresh");
        return buildToken(claims, userDetails, refreshTokenExpiry);
    }

    private String buildToken(Map<String, Object> extraClaims,
                               UserDetails user, long expiry) {
        return Jwts.builder()
            .claims(extraClaims)
            .subject(user.getUsername())
            .issuer("payments-api")
            .audience().add("payments-client").and()
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expiry * 1000))
            .id(UUID.randomUUID().toString()) // jti — unique token ID
            .signWith(signingKey)
            .compact();
    }

    // ── Validate Token ───────────────────────────────────────────
    public boolean isTokenValid(String token, UserDetails userDetails) {
        try {
            Claims claims = extractAllClaims(token);

            String username = claims.getSubject();
            Date expiration = claims.getExpiration();

            return username.equals(userDetails.getUsername()) &&
                   expiration.after(new Date()) &&
                   !isTokenRevoked(claims.getId()); // jti blocklist check

        } catch (ExpiredJwtException e) {
            log.warn("Token expired for user");
            return false;
        } catch (JwtException e) {
            log.warn("Invalid token: {}", e.getMessage());
            return false;
        }
    }

    // ── Extract Claims ───────────────────────────────────────────
    public Claims extractAllClaims(String token) {
        return Jwts.parser()
            .verifyWith(signingKey)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }

    public String extractUsername(String token) {
        return extractAllClaims(token).getSubject();
    }

    // ── Token Revocation (using Redis blocklist) ─────────────────
    private boolean isTokenRevoked(String jti) {
        return redisTemplate.hasKey("revoked:jwt:" + jti);
    }

    public void revokeToken(String token) {
        Claims claims = extractAllClaims(token);
        long ttl = claims.getExpiration().getTime() - System.currentTimeMillis();
        if (ttl > 0) {
            redisTemplate.opsForValue().set(
                "revoked:jwt:" + claims.getId(), "revoked",
                ttl, TimeUnit.MILLISECONDS
            );
        }
    }
}
JavaJWT Authentication Filter
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    // OncePerRequestFilter: guaranteed to execute ONCE per request
    // (some filters can execute multiple times in forward scenarios)

    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                     HttpServletResponse response,
                                     FilterChain filterChain)
            throws ServletException, IOException {

        // ── 1. Extract JWT from Authorization header ─────────────
        final String authHeader = request.getHeader("Authorization");

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response); // no JWT → pass through
            return;
        }

        final String jwt = authHeader.substring(7); // remove "Bearer "

        try {
            // ── 2. Extract username from token ───────────────────────
            final String username = jwtService.extractUsername(jwt);

            // ── 3. Check if already authenticated ───────────────────
            if (username != null &&
                SecurityContextHolder.getContext().getAuthentication() == null) {

                // ── 4. Load user from database ───────────────────────
                UserDetails userDetails = userDetailsService.loadUserByUsername(username);

                // ── 5. Validate token ────────────────────────────────
                if (jwtService.isTokenValid(jwt, userDetails)) {

                    // ── 6. Create authentication token ──────────────────
                    UsernamePasswordAuthenticationToken authToken =
                        new UsernamePasswordAuthenticationToken(
                            userDetails,
                            null,                        // no credentials needed
                            userDetails.getAuthorities()  // roles from UserDetails
                        );

                    // Add request details (IP, session) to auth token
                    authToken.setDetails(
                        new WebAuthenticationDetailsSource().buildDetails(request)
                    );

                    // ── 7. Set in SecurityContext ────────────────────────
                    SecurityContextHolder.getContext().setAuthentication(authToken);
                    // All downstream filters + controller see this as authenticated!
                }
            }
        } catch (ExpiredJwtException e) {
            sendError(response, HttpStatus.UNAUTHORIZED, "Token expired");
            return;
        } catch (JwtException e) {
            sendError(response, HttpStatus.UNAUTHORIZED, "Invalid token");
            return;
        }

        // ── 8. Continue filter chain ─────────────────────────────
        filterChain.doFilter(request, response);
    }

    private void sendError(HttpServletResponse response,
                           HttpStatus status, String message) throws IOException {
        response.setStatus(status.value());
        response.setContentType(MediaType.APPLICATION_JSON_VALUE);
        response.getWriter().write("""
            {"error": "%s", "status": %d}
            """.formatted(message, status.value()));
    }
}

Refresh token rotation: Issue new refresh token on each use (prevents refresh token reuse after theft)
JavaAuth Controller — Login + Refresh Token Endpoint
@RestController
@RequestMapping("/api/auth")
@RequiredArgsConstructor
public class AuthController {

    private final AuthenticationManager authManager;
    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;
    private final RefreshTokenRepository refreshTokenRepo;

    @PostMapping("/login")
    public ResponseEntity<AuthResponse> login(
            @Valid @RequestBody LoginRequest request) {

        // Authenticate using Spring Security's mechanism
        Authentication authentication = authManager.authenticate(
            new UsernamePasswordAuthenticationToken(
                request.getUsername(),
                request.getPassword()
            )
        ); // throws AuthenticationException if invalid

        UserDetails userDetails = (UserDetails) authentication.getPrincipal();

        String accessToken = jwtService.generateAccessToken(userDetails);
        String refreshToken = jwtService.generateRefreshToken(userDetails);

        // Store refresh token hash in DB for validation/revocation
        refreshTokenRepo.save(RefreshToken.builder()
            .tokenHash(sha256(refreshToken))
            .username(userDetails.getUsername())
            .expiresAt(LocalDateTime.now().plusDays(7))
            .build());

        return ResponseEntity.ok(AuthResponse.builder()
            .accessToken(accessToken)
            .refreshToken(refreshToken)
            .tokenType("Bearer")
            .expiresIn(3600L)
            .build());
    }

    @PostMapping("/refresh")
    public ResponseEntity<AuthResponse> refresh(@RequestBody RefreshRequest request) {

        String refreshToken = request.getRefreshToken();
        String tokenHash = sha256(refreshToken);

        // Validate refresh token exists and not expired
        RefreshToken stored = refreshTokenRepo.findByTokenHash(tokenHash)
            .orElseThrow(() -> new InvalidTokenException("Invalid refresh token"));

        if (stored.getExpiresAt().isBefore(LocalDateTime.now())) {
            refreshTokenRepo.delete(stored);
            throw new TokenExpiredException("Refresh token expired, login again");
        }

        // Refresh token rotation — invalidate old, issue new
        refreshTokenRepo.delete(stored);

        UserDetails userDetails = userDetailsService
            .loadUserByUsername(stored.getUsername());

        String newAccessToken = jwtService.generateAccessToken(userDetails);
        String newRefreshToken = jwtService.generateRefreshToken(userDetails);

        refreshTokenRepo.save(RefreshToken.builder()
            .tokenHash(sha256(newRefreshToken))
            .username(userDetails.getUsername())
            .expiresAt(LocalDateTime.now().plusDays(7))
            .build());

        return ResponseEntity.ok(AuthResponse.builder()
            .accessToken(newAccessToken)
            .refreshToken(newRefreshToken)
            .tokenType("Bearer")
            .expiresIn(3600L)
            .build());
    }

    @PostMapping("/logout")
    public ResponseEntity<Void> logout(
            @RequestHeader("Authorization") String bearerToken,
            @RequestBody LogoutRequest request) {

        String accessToken = bearerToken.substring(7);
        jwtService.revokeToken(accessToken); // add to Redis blocklist

        // Delete refresh token from DB
        refreshTokenRepo.deleteByTokenHash(sha256(request.getRefreshToken()));

        return ResponseEntity.noContent().build();
    }
}

🔑 OAuth 2.0 — The Authorization Framework

📚
OAuth 2.0 is NOT an Authentication Protocol

OAuth 2.0 is an authorization framework. It answers "what data can this app access on behalf of this user?" NOT "who is this user?" Authentication was added later via OpenID Connect (OIDC) built on top of OAuth 2.0.

Resource Server Server that hosts protected resources Google Calendar API
Client Application requesting access on behalf of user Your calendar app
Authorization Server Issues access tokens after user consents Google's OAuth server (accounts.google.com)

    public OidcUser loadUser(OidcUserRequest userRequest) {

        String name = oidcUser.getFullName();
        String providerId = oidcUser.getSubject();

                .username(email)
                .name(name)
                .provider(provider)
                .providerId(providerId)
                .password("") // no password for OAuth users
                .role("USER")
                .enabled(true)
                .build());

        return new DefaultOidcUser(
            List.of(new SimpleGrantedAuthority("ROLE_USER")),
            oidcUser.getIdToken(),
            oidcUser.getUserInfo()
        );

🔐 OpenID Connect (OIDC) — Authentication Layer

OIDC is an identity layer built on top of OAuth 2.0. It adds standardized user authentication to OAuth's authorization. It introduces the ID Token — a JWT containing user identity information.

OAUTH 2.0 vs OIDC OAuth 2.0 only ┌─────────┐ access_token ┌──────────────────┐ │ Client │ ───────────────► │ Resource Server │ │ │ │ /api/user │ │ │ ◄─────────────── │ → user data │ └─────────┘ └──────────────────┘ Problem: access_token alone doesn’t say WHO logged in. 🛡️ CSRF — Cross-Site Request Forgery

CSRF tricks a user's browser into making an unwanted authenticated request to your server. Works because browsers automatically attach cookies to any request matching the cookie's domain.

CSRF ATTACK — HOW IT WORKS 1. Alice logs into bank.com Browser stores: Cookie: session=abc123 2. Alice visits evil.com (while still logged into bank.com) 3. evil.com contains: <form action="https://bank.com/transfer" method="POST"> <input name="to" value="attacker_account"/> <input name="amount" value="10000"/> </form> <script>document.forms[0].submit()</script> 4. Browser sends: POST https://bank.com/transfer Cookie: session=abc123 ← auto-attached by browser! {to: "attacker", amount: 10000} 5. Bank sees valid session cookie → processes transfer! Alice's money transferred without her knowledge. DEFENSE — CSRF Token: Server generates random CSRF token per session. Token embedded in every form (hidden field) or in header. Browser cannot read it from another origin (SOP). Server validates token on every state-changing request. Attacker on evil.com cannot read or forge the CSRF token.
JavaCSRF Configuration in Spring Security
// ── For Web Applications (with sessions + forms) ─────────────────
http.csrf(csrf -> csrf
    .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
    // Stores CSRF token in a cookie (XSRF-TOKEN)
    // Client JS reads cookie, sends in X-XSRF-TOKEN header
    // withHttpOnlyFalse = JS can read the cookie (needed for SPAs)
);

// Angular/React automatically read XSRF-TOKEN cookie and
// send it as X-XSRF-TOKEN header on every state-changing request

// ── For Stateless REST APIs (JWT-based) ───────────────────────────
http.csrf(AbstractHttpConfigurer::disable);
// WHY safe to disable for REST APIs:
// CSRF exploits browser's automatic cookie sending.
// JWT is sent in Authorization header, NOT a cookie.
// Browser never auto-attaches Authorization headers to cross-site requests.
// Therefore: no session cookie = no CSRF risk for REST/JWT APIs.

// ── Partial CSRF protection (protect some, not all) ───────────────
http.csrf(csrf -> csrf
    .ignoringRequestMatchers("/api/webhooks/**")  // webhooks use HMAC sig
    .ignoringRequestMatchers("/api/mobile/**")    // mobile uses JWT
);

// ── SameSite Cookie — Modern Defense ─────────────────────────────
// SameSite=Strict: cookie never sent on cross-site requests
// SameSite=Lax: sent on top-level navigations only (GET)
// SameSite=None: sent on all cross-site (must set Secure)

@Bean
public CookieSerializer cookieSerializer() {
    DefaultCookieSerializer serializer = new DefaultCookieSerializer();
    serializer.setSameSite("Strict");
    serializer.setUseSecureCookie(true);
    serializer.setUseHttpOnlyCookie(true);
    return serializer;
}

🌐 CORS — Cross-Origin Resource Sharing

CORS is a browser security mechanism. Browsers block JavaScript from making requests to a different origin (domain/protocol/port) unless the server explicitly permits it. This is enforced by the browser — servers don't enforce it, they advertise policies.

CORS — SAME ORIGIN POLICY + PREFLIGHT Same Origin Policy (built into browsers): https://app.payments.com/dashboard ↓ JS tries to fetch: https://api.payments.com/data ← DIFFERENT origin (different subdomain) → Browser blocks request (CORS error)! CORS Preflight (for non-simple requests): 1. Browser sends OPTIONS request first: OPTIONS /api/payments Origin: https://app.payments.com Access-Control-Request-Method: POST Access-Control-Request-Headers: Authorization, Content-Type 2. Server responds: Access-Control-Allow-Origin: https://app.payments.com Access-Control-Allow-Methods: GET, POST, PUT Access-Control-Allow-Headers: Authorization, Content-Type Access-Control-Max-Age: 86400 (cache preflight for 24h) 3. Browser proceeds with actual request if server allowed it. 4. If server doesn't respond with CORS headers → browser blocks. Note: CORS is a BROWSER mechanism. curl, Postman, server-to-server = NOT affected by CORS.

        source.registerCorsConfiguration("/**", config);
        return source;
    }
}

XSS allows attackers to inject malicious JavaScript into your web pages that runs in other users' browsers. Once JS runs in a victim's browser, it has full access to their session, cookies, DOM, and can make requests on their behalf.

@Component
public class LoginRateLimitFilter extends OncePerRequestFilter {

    // Bucket4j or Resilience4j RateLimiter
    private final Map<String, Bucket> buckets = new ConcurrentHashMap<>();

    private Bucket getBucketForIp(String ip) {
        return buckets.computeIfAbsent(ip, k ->
            Bucket.builder()
                .addLimit(Bandwidth.classic(5, Refill.greedy(5, Duration.ofMinutes(1))))
                // 5 attempts per minute per IP
                .build()
        );
    }

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
                                     FilterChain chain) throws IOException, ServletException {

        if (req.getRequestURI().equals("/api/auth/login") &&
            req.getMethod().equals("POST")) {

            String clientIp = getClientIp(req);
            Bucket bucket = getBucketForIp(clientIp);

            if (!bucket.tryConsume(1)) {
                res.setStatus(429); // Too Many Requests
                res.setHeader("Retry-After", "60");
                res.getWriter().write("""
                    {"error": "Too many login attempts. Try again in 60 seconds."}""");
                return;
            }
        }

        chain.doFilter(req, res);
    }

    private String getClientIp(HttpServletRequest request) {
        // Check X-Forwarded-For for clients behind proxy/load balancer
        String forwarded = request.getHeader("X-Forwarded-For");
        if (forwarded != null && !forwarded.isEmpty()) {
            return forwarded.split(",")[0].trim(); // first IP is client
        }
        return request.getRemoteAddr();
    }
}

🏢 Multi-Tenancy Security

JavaJWT-Based Multi-Tenant Authentication
// Tenant resolved from subdomain, header, or JWT claim
// Each tenant may have different auth configs

@Component
public class MultiTenantJwtAuthFilter extends OncePerRequestFilter {

    private final Map<String, JwtDecoder> tenantDecoders = new ConcurrentHashMap<>();

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
                                     FilterChain chain) throws IOException, ServletException {

        String token = extractToken(req);
        if (token == null) { chain.doFilter(req, res); return; }

        // Extract tenant from subdomain: tenant1.app.com
        String tenantId = extractTenantFromHost(req.getServerName());
        // Or from JWT claim (decode header.payload without verifying first)
        // Or from request header: X-Tenant-ID

        JwtDecoder decoder = tenantDecoders.computeIfAbsent(tenantId,
            tid -> buildDecoderForTenant(tid));

        try {
            Jwt jwt = decoder.decode(token);

            // Store tenant in request scope for downstream use
            req.setAttribute("tenantId", tenantId);

            UsernamePasswordAuthenticationToken auth =
                new UsernamePasswordAuthenticationToken(
                    jwt.getSubject(), null,
                    extractAuthorities(jwt)
                );
            SecurityContextHolder.getContext().setAuthentication(auth);
        } catch (JwtException e) {
            res.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            res.getWriter().write("{\"error\": \"Invalid or expired token\"}");
            return;
        }

        chain.doFilter(req, res);
    }

    private String extractToken(HttpServletRequest req) {
        String header = req.getHeader("Authorization");
        if (header != null && header.startsWith("Bearer ")) {
            return header.substring(7);
        }
        return null;
    }

    private String extractTenantFromHost(String host) {
        // tenant1.app.com → "tenant1"
        String[] parts = host.split("\\.");
        return parts.length >= 3 ? parts[0] : "default";
    }

    private JwtDecoder buildDecoderForTenant(String tenantId) {
        // Look up per-tenant JWKS URI from config/DB
        String jwksUri = "https://auth.company.com/" + tenantId + "/.well-known/jwks.json";
        return NimbusJwtDecoder.withJwkSetUri(jwksUri).build();
    }

    private Collection<GrantedAuthority> extractAuthorities(Jwt jwt) {
        List<String> roles = jwt.getClaimAsStringList("roles");
        if (roles == null) return List.of();
        return roles.stream()
            .map(r -> new SimpleGrantedAuthority("ROLE_" + r))
            .collect(Collectors.toList());
    }
}

How do you implement JWT authentication in a Reactive (WebFlux) application?
In WebFlux, security filters are WebFilter implementations that return Mono<Void>. The JWT is extracted, validated, and the authentication is stored in the reactive security context.

Java
// JWT Authentication Web Filter
@Component
public class JwtAuthenticationWebFilter implements WebFilter {

    private final JwtService jwtService;
    private final ReactiveUserDetailsService userDetailsService;

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                              WebFilterChain chain) {
        return Mono.justOrEmpty(
                exchange.getRequest().getHeaders().getFirst("Authorization"))
            .filter(header -> header.startsWith("Bearer "))
            .map(header -> header.substring(7))
            .flatMap(token -> validateToken(token))
            .flatMap(auth ->
                chain.filter(exchange)
                    .contextWrite(ReactiveSecurityContextHolder
                        .withAuthentication(auth))
            )
            .switchIfEmpty(chain.filter(exchange)); // no token – continue
    }

    private Mono<Authentication> validateToken(String token) {
        return Mono.fromCallable(() -> jwtService.extractUsername(token))
            .flatMap(userDetailsService::findByUsername)
            .filter(user -> jwtService.isTokenValid(token, user))
            .map(user -> new UsernamePasswordAuthenticationToken(
                user, null, user.getAuthorities()))
            .onErrorResume(e -> Mono.empty()); // invalid token = empty
    }
}

        }

            acl.insertAce(acl.getEntries().size(), perm, sid, true);
        }
        aclService.updateAcl(acl);
    }

@Service

    private final Map<String, AtomicInteger> attempts = new ConcurrentHashMap<>();
    private static final int MAX_ATTEMPTS = 5;
    private static final int LOCK_DURATION_MINUTES = 15;

    public void recordFailedAttempt(String username, String ip) {
        attempts.computeIfAbsent(username, k -> new AtomicInteger())
                .incrementAndGet();
        if (attempts.get(username).get() >= MAX_ATTEMPTS) {
            userService.lockAccount(username, LOCK_DURATION_MINUTES);
            log.warn("ACCOUNT_LOCKED user={} after {} failed attempts",
                      username, MAX_ATTEMPTS);
        }
    }

        attempts.remove(username);
    }

// AES-256-GCM Encryption Service
@Service
public class AesEncryptionService {
    @Value("${encryption.key}")  // 32-byte key from Vault/Secrets Manager
    private String encryptionKey;

    public String encrypt(String plainText) throws Exception {
        byte[] iv = new byte[12];
        new SecureRandom().nextBytes(iv);

        SecretKeySpec key = new SecretKeySpec(
            Base64.getDecoder().decode(encryptionKey), "AES");
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.ENCRYPT_MODE, key,
            new GCMParameterSpec(128, iv));

        byte[] encrypted = cipher.doFinal(plainText.getBytes(StandardCharsets.UTF_8));
        byte[] combined  = new byte[iv.length + encrypted.length];
        System.arraycopy(iv,       0, combined, 0,         iv.length);
        System.arraycopy(encrypted,0, combined, iv.length, encrypted.length);
        return Base64.getEncoder().encodeToString(combined);
    }

    public String decrypt(String cipherText) throws Exception {
        byte[] combined = Base64.getDecoder().decode(cipherText);
        byte[] iv       = Arrays.copyOfRange(combined, 0, 12);
        byte[] encrypted= Arrays.copyOfRange(combined, 12, combined.length);

        SecretKeySpec key = new SecretKeySpec(
            Base64.getDecoder().decode(encryptionKey), "AES");
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.DECRYPT_MODE, key,
            new GCMParameterSpec(128, iv));
        return new String(cipher.doFinal(encrypted), StandardCharsets.UTF_8);
    }
}

🛡️ Zero Trust Security

What is Zero Trust Architecture?
Zero Trust is a security model based on the principle: "Never trust, always verify." No user, device, or service is trusted by default — even inside the corporate network.

Traditional SecurityZero Trust Security
"Trust but verify""Never trust, always verify"
Perimeter-based (firewall)Identity-based (every request authenticated)
Internal traffic trustedInternal traffic verified same as external
Implicit trust on networkExplicit verification always required
Broad access once insideLeast privilege access at all times

Zero Trust Pillars:
  1. Verify explicitly – always authenticate and authorise every request.
  2. Least privilege access – grant minimum permissions needed.
  3. Assume breach – design as if attacker is already inside.

How do you implement token rotation and refresh token patterns?
Token rotation limits the window of opportunity if a token is stolen. Short-lived access tokens + longer-lived refresh tokens is the standard pattern.

Java
// Token strategy
// Access token  → short TTL (15 minutes) – sent with every API request
// Refresh token → longer TTL (7 days) – stored securely, used to renew

@Entity
@Table(name = "refresh_tokens")
public class RefreshToken {
    @Id @GeneratedValue private Long id;
    private String token;           // UUID or random bytes
    private String username;
    private LocalDateTime expiresAt;
    private boolean revoked;
    private String deviceInfo;      // track which device
    private String ipAddress;
}

@Service
public class AuthService {

    public AuthResponse login(LoginRequest req) {
        authenticate(req.getUsername(), req.getPassword());

        UserDetails user = userDetailsService.loadUserByUsername(req.getUsername());
        String accessToken  = jwtService.generateAccessToken(user);  // 15 min
        String refreshToken = generateRefreshToken(user.getUsername()); // 7 days

        return new AuthResponse(accessToken, refreshToken);
    }

    public AuthResponse refreshTokens(String refreshToken) {
        RefreshToken stored = refreshTokenRepo
            .findByToken(refreshToken)
            .orElseThrow(() -> new InvalidTokenException("Refresh token not found"));

        if (stored.isRevoked() ||
            stored.getExpiresAt().isBefore(LocalDateTime.now())) {
            // Possible token theft – revoke all tokens for this user
            revokeAllUserTokens(stored.getUsername());
            throw new InvalidTokenException("Refresh token expired or revoked");
        }

        // Rotate – invalidate old refresh token, issue new one
        stored.setRevoked(true);
        refreshTokenRepo.save(stored);

        UserDetails user =
            userDetailsService.loadUserByUsername(stored.getUsername());
        String newAccessToken  = jwtService.generateAccessToken(user);
        String newRefreshToken = generateRefreshToken(user.getUsername());

        return new AuthResponse(newAccessToken, newRefreshToken);
    }

    public void logout(String refreshToken, String username) {
        // Revoke specific refresh token
        refreshTokenRepo.findByToken(refreshToken)
            .ifPresent(t -> {
                t.setRevoked(true);
                refreshTokenRepo.save(t);
            });
        // Optionally: add access token to blacklist until expiry
    }

    private String generateRefreshToken(String username) {
        RefreshToken token = new RefreshToken();
        token.setToken(UUID.randomUUID().toString());
        token.setUsername(username);
        token.setExpiresAt(LocalDateTime.now().plusDays(7));
        token.setRevoked(false);
        refreshTokenRepo.save(token);
        return token.getToken();
    }
}

public String generateToken(UserDetails user, HttpServletRequest req) {

public boolean validateToken(String token, HttpServletRequest req) {
    Claims claims  = extractAllClaims(token);
    String stored  = claims.get("fingerprint", String.class);
    String current = DigestUtils.sha256Hex(generateFingerprint(req));
    return stored.equals(current); // reject if different fingerprint
}

What is PKCE and why is it needed in OAuth2?
PKCE (Proof Key for Code Exchange) prevents authorisation code interception attacks in public clients (mobile apps, SPAs) that cannot securely store a client secret.

// API Key Authentication Filter
@Component
public class ApiKeyAuthFilter extends OncePerRequestFilter {

    private static final String API_KEY_HEADER = "X-API-Key";

    @Override
    protected void doFilterInternal(HttpServletRequest req,
                                    HttpServletResponse res,
                                    FilterChain chain)
            throws ServletException, IOException {

        String apiKey = req.getHeader(API_KEY_HEADER);
        if (apiKey == null) {
            chain.doFilter(req, res); // no API key – try other auth methods
            return;
        }

        String keyHash = DigestUtils.sha256Hex(apiKey);
        Optional<ApiKey> stored = apiKeyRepo.findByKeyHash(keyHash);

        if (stored.isEmpty() || !stored.get().isActive() ||
            stored.get().getExpiresAt().isBefore(LocalDateTime.now())) {
            res.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Invalid API Key");
            return;
        }

        ApiKey key = stored.get();
        List<GrantedAuthority> authorities = key.getScopes().stream()
            .map(SimpleGrantedAuthority::new)
            .collect(Collectors.toList());

        ApiKeyAuthentication auth =
            new ApiKeyAuthentication(key.getClientName(), authorities);
        SecurityContextHolder.getContext().setAuthentication(auth);

        // Audit API key usage
        apiKeyAuditService.logUsage(key.getId(), req.getRequestURI());
        chain.doFilter(req, res);
    }
}

// Security config – support both JWT and API Key
@Bean

            UsernamePasswordAuthenticationFilter.class)    // API Key first

            .requestMatchers("/api/public/**").permitAll()
