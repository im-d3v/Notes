Creational Pattern · Most Asked
Singleton Pattern
Ensure a class has only ONE instance throughout the application
❓ What Problem Does It Solve?
  • Prevent multiple instances of resource-heavy objects
  • Provide a single global access point
  • Control shared state (e.g., config, DB connection)
  • Avoid inconsistent state across instances
🌍 Real-World Use Cases
  • Database connection pool
  • Logger (single log instance)
  • Spring IoC Container itself
  • Configuration manager
  • Thread pool manager

Type 1 Eager Initialization

📌
Instance is created when the class is loaded — even if it's never used. Simple but wastes memory if instance is heavy.
EagerSingleton.java Java 
public class EagerSingleton {

    // Instance created at class loading time
    private static final EagerSingleton INSTANCE
            = new EagerSingleton();

    // Private constructor — prevents new EagerSingleton()
    private EagerSingleton() {
        // initialization code
    }

    public static EagerSingleton getInstance() {
        return INSTANCE;
    }

    public void doSomething() {
        System.out.println("Eager Singleton working!");
    }
}

// Usage
EagerSingleton s1 = EagerSingleton.getInstance();
EagerSingleton s2 = EagerSingleton.getInstance();
System.out.println(s1 == s2); // true — same instance
✅ Pros
  • Thread-safe by default (JVM class loading is thread-safe)
  • Simple to implement
❌ Cons
  • Instance created even if never used
  • Wastes memory for heavy objects

Type 2 Lazy Initialization

⚠️
Instance created only when first needed. But NOT thread-safe! Two threads can both pass the null check simultaneously and create two instances.
LazySingleton.java Java 
public class LazySingleton {

    private static LazySingleton instance; // null by default

    private LazySingleton() {}

    public static LazySingleton getInstance() {
        if (instance == null) {             // ⚠️ NOT thread-safe
            instance = new LazySingleton();
        }
        return instance;
    }
}

Type 3 Synchronized Method (Thread-Safe but Slow)

SynchronizedSingleton.java Java 
public class SynchronizedSingleton {

    private static SynchronizedSingleton instance;

    private SynchronizedSingleton() {}

    // Thread-safe but synchronized on every call = SLOW
    public static synchronized SynchronizedSingleton getInstance() {
        if (instance == null) {
            instance = new SynchronizedSingleton();
        }
        return instance;
    }
}

Type 4 ⭐ Double-Checked Locking (DCL) — Best Practice

🌟
Most asked in interviews! DCL checks null twice — once without lock (fast path) and once inside the synchronized block (safe path). The volatile keyword is CRITICAL here.
DCLSingleton.java Java 
public class DCLSingleton {

    // volatile prevents instruction reordering
    private static volatile DCLSingleton instance;

    private DCLSingleton() {
        // Prevent reflection attack
        if (instance != null) {
            throw new RuntimeException("Use getInstance()");
        }
    }

    public static DCLSingleton getInstance() {
        // ✅ Check 1: No lock needed if already created (performance)
        if (instance == null) {
            // Only one thread enters this block at a time
            synchronized (DCLSingleton.class) {
                // ✅ Check 2: Another thread may have created it
                if (instance == null) {
                    instance = new DCLSingleton();
                }
            }
        }
        return instance;
    }
}

🤔 Why is volatile needed in DCL?

1
Object Creation has 3 steps in bytecode
new DCLSingleton() internally: (a) Allocate memory, (b) Initialize object, (c) Assign reference to variable
2
JVM/CPU can reorder instructions
Without volatile, steps can be reordered: (a) → (c) → (b). So another thread sees a non-null but partially initialized object!
3
volatile prevents reordering
volatile creates a memory barrier — guarantees assignment happens AFTER full initialization.

Type 5 ⭐ Enum Singleton (Joshua Bloch's Recommendation)

💎
Best way to implement Singleton as per Effective Java. It handles serialization, reflection, and thread safety automatically!
EnumSingleton.java Java 
public enum EnumSingleton {
    INSTANCE; // One and only instance

    private int counter = 0;

    public void doSomething() {
        counter++;
        System.out.println("Counter: " + counter);
    }
}

// Usage
EnumSingleton.INSTANCE.doSomething();
EnumSingleton.INSTANCE.doSomething();
// Output: Counter: 1, Counter: 2 (same instance)

⚠️ 3 Ways to Break Singleton (and How to Fix)

🪞 1. Reflection Attack
  • Access private constructor via reflection
  • Fix: Throw exception if instance already exists in constructor
  • Or use Enum (Enum prevents reflection)
📦 2. Serialization Attack
  • Deserializing creates a new object
  • Fix: Implement readResolve() method that returns existing instance
  • Or use Enum (handles serialization natively)
🧬 3. Cloning Attack
  • Calling clone() creates a new instance
  • Fix: Override clone() and throw CloneNotSupportedException
BreakingSingleton.java — How to break + fix Java 
// ─── REFLECTION ATTACK ───────────────────────────────
Constructor<DCLSingleton> constructor
        = DCLSingleton.class.getDeclaredConstructor();
constructor.setAccessible(true);
DCLSingleton brokenInstance = constructor.newInstance();
// Creates a NEW instance — breaks singleton!

// FIX: In constructor check if instance already exists
private DCLSingleton() {
    if (instance != null) {
        throw new IllegalStateException("Already instantiated!");
    }
}

// ─── SERIALIZATION FIX ───────────────────────────────
protected Object readResolve() {
    return getInstance(); // Return existing instead of deserialized
}

// ─── CLONING FIX ─────────────────────────────────────
@Override
protected Object clone() throws CloneNotSupportedException {
    throw new CloneNotSupportedException("Cloning not allowed!");
}
🎤 Frequently Asked Interview Questions
Q: What happens if I remove the first null check in DCL and keep only the synchronized block?
A: It would still work correctly, but every single call to getInstance() would acquire a lock — even after the object is created. This causes unnecessary synchronization overhead and kills performance in multi-threaded applications. The outer null check is the performance optimization.
Q: Why is Enum the best way to implement Singleton?
A: Because JVM guarantees Enum instances are unique and handles (1) thread safety, (2) serialization, (3) protection from reflection attacks — all automatically. You get a bullet-proof Singleton in 3 lines of code.
Q: Singleton is Eager — what problem does it solve?
A: It resolves the problem of multiple inconsistent instances of resource-heavy or stateful objects (like DB connections, logger, config). Before Singleton pattern, developers unknowingly created multiple instances causing resource waste and inconsistent state.

Creational Pattern
Factory Pattern
Create objects without exposing creation logic — let subclass/factory decide which class to instantiate
❓ What Problem Does It Solve?
  • Avoid tight coupling between client and concrete classes
  • Open/Closed principle — add new types without modifying existing code
  • Centralize object creation logic
🌍 Real Use Cases
  • Payment gateway (Credit Card, PayPal, UPI)
  • Notification service (SMS, Email, Push)
  • Logger factory (FileLogger, DBLogger)
  • Spring's BeanFactory itself
PaymentFactory.java — Complete Implementation Java 
// ─── STEP 1: Define the common interface ──────────────
public interface Payment {
    void processPayment(double amount);
    String getPaymentType();
}

// ─── STEP 2: Concrete implementations ─────────────────
public class CreditCardPayment implements Payment {
    @Override
    public void processPayment(double amount) {
        System.out.println("Processing ₹" + amount + " via Credit Card");
    }
    @Override
    public String getPaymentType() { return "CREDIT_CARD"; }
}

public class UPIPayment implements Payment {
    @Override
    public void processPayment(double amount) {
        System.out.println("Processing ₹" + amount + " via UPI");
    }
    @Override
    public String getPaymentType() { return "UPI"; }
}

public class PayPalPayment implements Payment {
    @Override
    public void processPayment(double amount) {
        System.out.println("Processing ₹" + amount + " via PayPal");
    }
    @Override
    public String getPaymentType() { return "PAYPAL"; }
}

// ─── STEP 3: Enum for type-safe selection ─────────────
public enum PaymentType {
    CREDIT_CARD, UPI, PAYPAL, NET_BANKING
}

// ─── STEP 4: The Factory ───────────────────────────────
public class PaymentFactory {

    // Static factory method
    public static Payment createPayment(PaymentType type) {
        return switch (type) {
            case CREDIT_CARD  -> new CreditCardPayment();
            case UPI          -> new UPIPayment();
            case PAYPAL       -> new PayPalPayment();
            default           -> throw new IllegalArgumentException(
                                      "Unsupported payment: " + type);
        };
    }
}

// ─── STEP 5: Client usage ──────────────────────────────
public class CheckoutService {

    public void processOrder(PaymentType type, double amount) {
        Payment payment = PaymentFactory.createPayment(type);
        payment.processPayment(amount);
        System.out.println("Type: " + payment.getPaymentType());
    }

    public static void main(String[] args) {
        CheckoutService service = new CheckoutService();
        service.processOrder(PaymentType.UPI, 500.0);
        service.processOrder(PaymentType.CREDIT_CARD, 1200.0);
        service.processOrder(PaymentType.PAYPAL, 300.0);
    }
}
📤 Output
Processing ₹500.0 via UPI Type: UPI Processing ₹1200.0 via Credit Card Type: CREDIT_CARD Processing ₹300.0 via PayPal Type: PAYPAL
💡
Why use Enum instead of String? Enum provides compile-time type safety. If you pass a wrong string like "CREDIT CARD" (with space), it fails at runtime. With Enum, it fails at compile time — caught by IDE immediately.
🎤 Interview Questions
Q: What is the difference between Factory and Abstract Factory?
A: Factory creates objects of one product type (e.g., different payments). Abstract Factory creates families of related objects (e.g., a UIFactory that creates Button + TextBox + Dialog all in consistent theme — Dark/Light).
Q: How does Factory relate to SOLID?
A: Factory supports OCP (Open-Closed Principle) — you can add new payment types by adding a new class + enum constant, without modifying existing code. It also satisfies DIP — client depends on abstract Payment, not concrete CreditCardPayment.

Behavioral Pattern
Strategy Pattern
Define a family of algorithms, encapsulate each one, and make them interchangeable at runtime
❓ What Problem Does It Solve?
  • Eliminate multiple if-else / switch for algorithm selection
  • Change behavior at runtime without changing the class
  • Follows OCP — add new strategies without modifying existing
🌍 Real Use Cases
  • Sorting strategy (Bubble, Quick, Merge) — switch at runtime
  • Discount strategy (Festival, Coupon, Clearance)
  • Tax calculation strategy by region
  • Compression (ZIP, RAR, GZIP)
DiscountStrategy.java — E-commerce Discount System Java 
// ─── STEP 1: Strategy Interface ────────────────────────
public interface DiscountStrategy {
    double applyDiscount(double originalPrice);
    String getStrategyName();
}

// ─── STEP 2: Concrete Strategies ──────────────────────
public class NoDiscount implements DiscountStrategy {
    @Override
    public double applyDiscount(double price) { return price; }
    @Override
    public String getStrategyName() { return "No Discount"; }
}

public class FestivalDiscount implements DiscountStrategy {
    private final double discountPercent;

    public FestivalDiscount(double percent) {
        this.discountPercent = percent;
    }

    @Override
    public double applyDiscount(double price) {
        return price - (price * discountPercent / 100);
    }
    @Override
    public String getStrategyName() {
        return "Festival Discount (" + discountPercent + "%)";
    }
}

public class CouponDiscount implements DiscountStrategy {
    private final double flatDiscount;

    public CouponDiscount(double flat) {
        this.flatDiscount = flat;
    }

    @Override
    public double applyDiscount(double price) {
        return Math.max(0, price - flatDiscount);
    }
    @Override
    public String getStrategyName() {
        return "Coupon Flat ₹" + flatDiscount;
    }
}

// ─── STEP 3: Context class ─────────────────────────────
public class ShoppingCart {

    private DiscountStrategy discountStrategy; // injected
    private double totalAmount;

    public ShoppingCart(double amount) {
        this.totalAmount = amount;
        this.discountStrategy = new NoDiscount(); // default
    }

    // ✅ Strategy can be CHANGED at runtime!
    public void setDiscountStrategy(DiscountStrategy strategy) {
        this.discountStrategy = strategy;
    }

    public void checkout() {
        double finalPrice = discountStrategy.applyDiscount(totalAmount);
        System.out.printf("Strategy: %s | Original: ₹%.2f | Final: ₹%.2f%n",
                discountStrategy.getStrategyName(), totalAmount, finalPrice);
    }
}

// ─── STEP 4: Usage ─────────────────────────────────────
public static void main(String[] args) {
    ShoppingCart cart = new ShoppingCart(1000.0);

    cart.checkout();                                       // No discount

    cart.setDiscountStrategy(new FestivalDiscount(20));   // 20% off
    cart.checkout();

    cart.setDiscountStrategy(new CouponDiscount(150));    // Flat ₹150 off
    cart.checkout();
}
📤 Output
Strategy: No Discount | Original: ₹1000.00 | Final: ₹1000.00 Strategy: Festival Discount (20.0%) | Original: ₹1000.00 | Final: ₹800.00 Strategy: Coupon Flat ₹150.0 | Original: ₹1000.00 | Final: ₹850.00

🚀 Strategy with Lambda (Modern Java)

💡
Since DiscountStrategy is a functional interface (one method), you can pass lambdas directly without creating separate classes!
LambdaStrategy.java Java 
ShoppingCart cart = new ShoppingCart(1000.0);

// Lambda as a Strategy — no class needed!
cart.setDiscountStrategy(price -> price * 0.85); // 15% off inline
cart.checkout();

// Store and reuse lambdas
DiscountStrategy newYearDeal = price -> price - 200;
DiscountStrategy flashSale   = price -> price * 0.5; // 50% off

cart.setDiscountStrategy(flashSale);
cart.checkout();
🎤 Interview Questions
Q: Where have you used Strategy pattern in your STEP project?
A: Example answer: "In my STEP project, I used Strategy for discount calculation. Different product categories had different discount rules — electronics had percentage discount, clothing had flat discount, and clearance items had minimum price guarantee. Instead of if-else in checkout, each rule was a DiscountStrategy implementation injected at runtime based on product category."
Q: What is the difference between Strategy and Factory?
A: Factory is about object creation — which class to instantiate. Strategy is about behavior selection — which algorithm to execute. They often work together — Factory creates the right Strategy based on input.

Creational Pattern
Builder Pattern
Construct complex objects step by step — separate construction from representation
❓ What Problem Does It Solve?
  • Avoid "telescoping constructor" anti-pattern
  • Handle optional fields without many overloaded constructors
  • Construct objects in multiple steps
  • Create immutable objects with many fields
🌍 Real Use Cases
  • Building HTTP requests (OkHttp, HttpClient)
  • Creating entity objects in tests
  • StringBuilder itself is a Builder!
  • Lombok's @Builder annotation

❌ The Problem — Telescoping Constructor

BAD_Example.java — Telescoping Constructor Anti-Pattern Java 
public class User {
    public User(String name) { ... }
    public User(String name, String email) { ... }
    public User(String name, String email, String phone) { ... }
    public User(String name, String email, String phone, String address) { ... }
    public User(String name, String email, String phone, String address, int age) { ... }
    // Ugly! Hard to read, hard to maintain!
}

// Client code is a mystery:
User u = new User("John", "john@mail.com", null, null, 25);
// What does null, null mean? Which position is what?

✅ The Solution — Builder Pattern

User.java — Builder Pattern (Thread-Safe & Immutable) Java 
public final class User {

    // ✅ Required fields — final for immutability
    private final Long   id;
    private final String name;
    private final String email;

    // ✅ Optional fields
    private final String phone;
    private final String address;
    private final int    age;
    private final String role;

    // Private constructor — only Builder can call
    private User(Builder builder) {
        this.id      = builder.id;
        this.name    = builder.name;
        this.email   = builder.email;
        this.phone   = builder.phone;
        this.address = builder.address;
        this.age     = builder.age;
        this.role    = builder.role;
    }

    // Getters only — no setters (immutable!)
    public Long   getId()      { return id; }
    public String getName()    { return name; }
    public String getEmail()   { return email; }
    public String getPhone()   { return phone; }
    public String getAddress() { return address; }
    public int    getAge()     { return age; }

    @Override
    public String toString() {
        return "User{id="+id+", name='"+name+"', email='"+email+"', role='"+role+"'}";
    }

    // ══════════════════════════════════════════
       STATIC INNER BUILDER CLASS
    ══════════════════════════════════════════ 
    public static class Builder {

        // Required fields — set via constructor
        private final Long   id;
        private final String name;
        private final String email;

        // Optional fields — set via fluent methods
        private String phone   = null;
        private String address = null;
        private int    age     = 0;
        private String role    = "USER"; // default

        // Required fields via constructor
        public Builder(Long id, String name, String email) {
            if (name == null || name.isBlank())
                throw new IllegalArgumentException("Name required");
            if (email == null || !email.contains("@"))
                throw new IllegalArgumentException("Valid email required");
            this.id    = id;
            this.name  = name;
            this.email = email;
        }

        // Fluent setter methods — each returns 'this' for chaining
        public Builder phone(String phone) {
            this.phone = phone;
            return this;
        }

        public Builder address(String address) {
            this.address = address;
            return this;
        }

        public Builder age(int age) {
            if (age < 0 || age > 150)
                throw new IllegalArgumentException("Invalid age");
            this.age = age;
            return this;
        }

        public Builder role(String role) {
            this.role = role;
            return this;
        }

        // Terminal build() — creates immutable User
        public User build() {
            return new User(this);
        }
    }
}

// ─── Usage — Clean, readable, immutable ───────────────
User admin = new User.Builder(1L, "Alice", "alice@example.com")
        .phone("+91-9999999999")
        .address("Delhi, India")
        .age(28)
        .role("ADMIN")
        .build();

// Minimal — only required fields
User guest = new User.Builder(2L, "Bob", "bob@example.com").build();

System.out.println(admin);
System.out.println(guest);
📤 Output
User{id=1, name='Alice', email='alice@example.com', role='ADMIN'} User{id=2, name='Bob', email='bob@example.com', role='USER'}

🎤 Interview Questions
Q: How is Builder thread-safe?
A: The Builder accumulates state in a local object (not shared between threads). The final User object is immutable (all final fields, no setters) — once built, it can be safely shared between threads without synchronization.
Q: Builder vs Constructor — when to prefer Builder?
A: Use Builder when (1) class has 4+ parameters, (2) many are optional, (3) you need validation at build time, (4) you want readable code with named parameters. Use Constructor when object has 1-3 required fields only.

Structural Pattern
Decorator Pattern
Add behavior to objects dynamically without changing their class — wrapping, not subclassing
📌
"Prefer composition over inheritance" — Decorator is the textbook implementation of this principle. Instead of creating subclasses for every combination, wrap objects with decorators.
❓ What Problem Does It Solve?
  • Avoid class explosion from inheritance combinations
  • Add/remove features at runtime, not compile-time
  • Follow Open-Closed Principle (add features without modifying)
🌍 Real Use Cases (Not Pizza/Coffee!)
  • Java I/O Streams (BufferedInputStream wraps FileInputStream)
  • HTTP middleware / filter chains
  • Logging decorators in Spring AOP
  • UI components (scroll, border, shadow wrapping)
  • Data encryption layer on top of storage
NotificationDecorator.java — Multi-channel Notification System Java 
// ─── STEP 1: Component Interface ──────────────────────
public interface NotificationService {
    void send(String message, String recipient);
}

// ─── STEP 2: Concrete Component (base) ────────────────
public class EmailNotification implements NotificationService {
    @Override
    public void send(String msg, String recipient) {
        System.out.println("📧 EMAIL → [" + recipient + "] : " + msg);
    }
}

// ─── STEP 3: Abstract Decorator ───────────────────────
// Implements same interface AND holds a reference to wrapped object
public abstract class NotificationDecorator implements NotificationService {

    protected final NotificationService wrapped; // delegation

    public NotificationDecorator(NotificationService service) {
        this.wrapped = service;
    }

    @Override
    public void send(String msg, String recipient) {
        wrapped.send(msg, recipient); // delegate to wrapped
    }
}

// ─── STEP 4: Concrete Decorators ──────────────────────
public class SMSDecorator extends NotificationDecorator {

    public SMSDecorator(NotificationService service) {
        super(service);
    }

    @Override
    public void send(String msg, String recipient) {
        super.send(msg, recipient);            // original email first
        System.out.println("📱 SMS  → [" + recipient + "] : " + msg);
    }
}

public class SlackDecorator extends NotificationDecorator {

    public SlackDecorator(NotificationService service) {
        super(service);
    }

    @Override
    public void send(String msg, String recipient) {
        super.send(msg, recipient);            // all previous channels
        System.out.println("💬 SLACK→ [" + recipient + "] : " + msg);
    }
}

public class LoggingDecorator extends NotificationDecorator {

    public LoggingDecorator(NotificationService service) {
        super(service);
    }

    @Override
    public void send(String msg, String recipient) {
        System.out.println("📋 LOG  → Sending notification to: " + recipient);
        super.send(msg, recipient);
        System.out.println("📋 LOG  → Notification sent successfully!");
    }
}

// ─── STEP 5: Dynamic composition ──────────────────────
public static void main(String[] args) {

    // Basic: Email only
    NotificationService emailOnly = new EmailNotification();

    // Email + SMS
    NotificationService emailSms
            = new SMSDecorator(new EmailNotification());

    // Email + SMS + Slack + Logging — all stacked!
    NotificationService allChannels
            = new LoggingDecorator(
                new SlackDecorator(
                    new SMSDecorator(
                        new EmailNotification()
                    )
                )
              );

    System.out.println("=== Email Only ===");
    emailOnly.send("Server down!", "admin@company.com");

    System.out.println("\n=== All Channels ===");
    allChannels.send("CRITICAL: DB unreachable", "ops@company.com");
}
📤 Output
=== Email Only === 📧 EMAIL → [admin@company.com] : Server down! === All Channels === 📋 LOG → Sending notification to: ops@company.com 📧 EMAIL → [ops@company.com] : CRITICAL: DB unreachable 📱 SMS → [ops@company.com] : CRITICAL: DB unreachable 💬 SLACK→ [ops@company.com] : CRITICAL: DB unreachable 📋 LOG → Notification sent successfully!

InputStream raw       = new FileInputStream("file.txt");

🎤 Interview Questions
Q: Give a real-world example of Decorator (not pizza or coffee)
A: Java I/O Streams — FileInputStream is the base. You wrap it with BufferedInputStream for buffering, then with GZIPInputStream for compression, then with CipherInputStream for encryption. Each decorator adds a feature without modifying the original. Another example: Spring Security filter chain — each security filter wraps the request adding authentication, authorization, CSRF checks as decorators.
Q: Decorator vs Inheritance — what's the difference?
A: Inheritance is static (decided at compile time). Decorator is dynamic (decided at runtime). With inheritance, you need a new class for every combination (EmailSMSNotification, EmailSlackNotification, etc.) — this causes class explosion. Decorator lets you compose features on-the-fly by stacking wrappers.

System Design + Inner Classes
Parking Lot Design
Classic system design question using inner classes, enums, and OOP principles
📌
This question tests your ability to model real-world systems using OOP. Key focus: proper use of inner classes, enums, and encapsulation.

📋 Requirements

  • ✅ Multiple floors in parking lot
  • ✅ Different vehicle types (Car, Bike, Truck)
  • ✅ Different spot sizes (Small, Medium, Large)
  • ✅ Park a vehicle — find available spot
  • ✅ Unpark a vehicle — free the spot
  • ✅ Generate ticket on parking
  • ✅ Calculate fee on exit
  • ✅ Check available spots
ParkingLot.java — Complete Design with Inner Classes Java 
import java.time.*;
import java.util.*;

public class ParkingLot {

    // ══ INNER ENUMS ═══════════════════════════════

    public enum VehicleType { BIKE, CAR, TRUCK }

    public enum SpotSize    { SMALL, MEDIUM, LARGE }

    // ══ INNER CLASS: Vehicle ═══════════════════════

    public static class Vehicle {
        private final String      licensePlate;
        private final VehicleType type;

        public Vehicle(String plate, VehicleType type) {
            this.licensePlate = plate;
            this.type         = type;
        }

        public String      getLicensePlate() { return licensePlate; }
        public VehicleType getType()         { return type; }

        @Override
        public String toString() {
            return type + "[" + licensePlate + "]";
        }
    }

    // ══ INNER CLASS: ParkingSpot ════════════════════

    public static class ParkingSpot {
        private final String    spotId;
        private final SpotSize  size;
        private       boolean   occupied = false;
        private       Vehicle   currentVehicle;

        public ParkingSpot(String id, SpotSize size) {
            this.spotId = id;
            this.size   = size;
        }

        public boolean isAvailable() { return !occupied; }

        public boolean canFit(Vehicle v) {
            return switch (v.getType()) {
                case BIKE  -> size == SpotSize.SMALL
                             || size == SpotSize.MEDIUM;
                case CAR   -> size == SpotSize.MEDIUM
                             || size == SpotSize.LARGE;
                case TRUCK -> size == SpotSize.LARGE;
            };
        }

        public void occupy(Vehicle v) {
            this.occupied       = true;
            this.currentVehicle = v;
        }

        public void vacate() {
            this.occupied       = false;
            this.currentVehicle = null;
        }

        public String   getSpotId()        { return spotId; }
        public SpotSize getSize()          { return size; }
        public Vehicle  getCurrentVehicle() { return currentVehicle; }
    }

    // ══ INNER CLASS: Ticket ═════════════════════════

    public static class Ticket {
        private static int ticketCounter = 1000;

        private final String        ticketId;
        private final Vehicle       vehicle;
        private final ParkingSpot   spot;
        private final LocalDateTime entryTime;

        public Ticket(Vehicle v, ParkingSpot spot) {
            this.ticketId  = "TKT-" + (++ticketCounter);
            this.vehicle   = v;
            this.spot      = spot;
            this.entryTime = LocalDateTime.now();
        }

        public double calculateFee() {
            long hours = Duration
                    .between(entryTime, LocalDateTime.now())
                    .toHours() + 1; // minimum 1 hour
            double ratePerHour = switch (vehicle.getType()) {
                case BIKE  -> 20.0;
                case CAR   -> 50.0;
                case TRUCK -> 100.0;
            };
            return hours * ratePerHour;
        }

        @Override
        public String toString() {
            return String.format(
                "Ticket[%s] Vehicle:%s Spot:%s Entry:%s",
                ticketId, vehicle, spot.getSpotId(),
                entryTime.toLocalTime()
            );
        }

        public ParkingSpot getSpot()    { return spot; }
        public String      getTicketId() { return ticketId; }
    }

    // ══ INNER CLASS: ParkingFloor ══════════════════

    public static class ParkingFloor {
        private final int                   floorNumber;
        private final List<ParkingSpot>    spots;

        public ParkingFloor(int num, int small, int medium, int large) {
            this.floorNumber = num;
            this.spots       = new ArrayList<>();
            for (int i = 0; i < small;  i++) spots.add(new ParkingSpot("F"+num+"-S"+i,  SpotSize.SMALL));
            for (int i = 0; i < medium; i++) spots.add(new ParkingSpot("F"+num+"-M"+i, SpotSize.MEDIUM));
            for (int i = 0; i < large;  i++) spots.add(new ParkingSpot("F"+num+"-L"+i,  SpotSize.LARGE));
        }

        // Find first available spot that fits the vehicle
        public Optional<ParkingSpot> findAvailableSpot(Vehicle v) {
            return spots.stream()
                    .filter(s -> s.isAvailable() && s.canFit(v))
                    .findFirst();
        }

        public long getAvailableCount() {
            return spots.stream().filter(ParkingSpot::isAvailable).count();
        }
    }

    // ══ MAIN ParkingLot FIELDS ═════════════════════

    private final String                   name;
    private final List<ParkingFloor>      floors;
    private final Map<String, Ticket>    activeTickets; // plate → ticket

    public ParkingLot(String name, int numFloors) {
        this.name          = name;
        this.floors        = new ArrayList<>();
        this.activeTickets = new HashMap<>();

        // Each floor: 10 small, 10 medium, 5 large spots
        for (int i = 1; i <= numFloors; i++) {
            floors.add(new ParkingFloor(i, 10, 10, 5));
        }
    }

    // Park a vehicle
    public Ticket park(Vehicle vehicle) {
        if (activeTickets.containsKey(vehicle.getLicensePlate())) {
            throw new IllegalStateException("Vehicle already parked!");
        }

        for (ParkingFloor floor : floors) {
            Optional<ParkingSpot> spot = floor.findAvailableSpot(vehicle);
            if (spot.isPresent()) {
                spot.get().occupy(vehicle);
                Ticket ticket = new Ticket(vehicle, spot.get());
                activeTickets.put(vehicle.getLicensePlate(), ticket);
                System.out.println("✅ Parked: " + ticket);
                return ticket;
            }
        }
        throw new IllegalStateException("No available spot for " + vehicle);
    }

    // Unpark a vehicle and get fee
    public double unpark(String ticketId) {
        Ticket ticket = activeTickets.values().stream()
                .filter(t -> t.getTicketId().equals(ticketId))
                .findFirst()
                .orElseThrow(() -> new IllegalArgumentException("Ticket not found"));

        double fee = ticket.calculateFee();
        ticket.getSpot().vacate();
        activeTickets.remove(ticket.getSpot().getCurrentVehicle() != null
                ? ticket.getSpot().getCurrentVehicle().getLicensePlate()
                : "");
        System.out.printf("🚗 Unparked. Fee: ₹%.2f%n", fee);
        return fee;
    }

    public void displayStatus() {
        System.out.println("\n📊 Parking Lot: " + name);
        floors.forEach(f -> System.out.println(
            "  Floor " + f.floorNumber + ": " + f.getAvailableCount() + " spots free"
        ));
    }

    // ─── DEMO ──────────────────────────────────────
    public static void main(String[] args) {
        ParkingLot lot = new ParkingLot("City Center Parking", 3);

        Vehicle car1  = new Vehicle("DL01AB1234", VehicleType.CAR);
        Vehicle bike1 = new Vehicle("DL02CD5678", VehicleType.BIKE);
        Vehicle truck = new Vehicle("HR05EF9999", VehicleType.TRUCK);

        Ticket t1 = lot.park(car1);
        Ticket t2 = lot.park(bike1);
        Ticket t3 = lot.park(truck);

        lot.displayStatus();
    }
}

🎤 Interview Follow-up Questions
Q: How would you make this thread-safe for concurrent parking requests?
A: Use synchronized on the park() and unpark() methods, or use ReentrantLock per floor. activeTickets should be a ConcurrentHashMap. For better performance, lock per floor rather than the entire lot.
Q: Why use inner classes instead of separate files?
A: Vehicle, ParkingSpot, Ticket, and ParkingFloor are domain objects tightly coupled to ParkingLot. Making them inner classes (static) provides logical grouping, better encapsulation, and prevents these classes from being misused in other contexts.

Relationships.java — Hands-On Code Java 
// ═══════════════════════════════════════════════
   1. ASSOCIATION — "uses-a" relationship
   Objects are aware of each other, no ownership
═══════════════════════════════════════════════ 
class Teacher {
    private String name;

    public Teacher(String name) { this.name = name; }

    // Teacher uses Student (parameter) — association
    public void teach(Student student) {
        System.out.println(name + " is teaching " + student.getName());
    }
}

class Student {
    private String name;
    public Student(String n) { name = n; }
    public String getName() { return name; }
}

// ═══════════════════════════════════════════════
   2. AGGREGATION — "has-a" weak ownership
   Child can exist without parent
═══════════════════════════════════════════════ 
class Professor {
    private String name;
    public Professor(String n) { name = n; }
    public String getName() { return name; }
}

class Department {
    private String           deptName;
    private List<Professor> professors; // passed in, not created here

    // Professors are PASSED IN — they exist independently
    public Department(String name, List<Professor> profs) {
        this.deptName   = name;
        this.professors = profs; // Aggregation: shared reference
    }
    // If Department is deleted, Professors still exist elsewhere!
}

// ═══════════════════════════════════════════════
   3. COMPOSITION — "has-a" STRONG ownership
   Child CANNOT exist without parent
═══════════════════════════════════════════════ 
class Room {
    private String type;
    public Room(String type) { this.type = type; }
    public String getType() { return type; }
}

class House {
    private final Room bedroom;   // Room CREATED inside House
    private final Room kitchen;
    private final Room bathroom;

    public House() {
        // Rooms are CREATED here — they exist ONLY within this House
        this.bedroom  = new Room("Bedroom");
        this.kitchen  = new Room("Kitchen");
        this.bathroom = new Room("Bathroom");
        // House demolished → Rooms are gone too!
    }

    public void describe() {
        System.out.println("House has: "
            + bedroom.getType() + ", "
            + kitchen.getType() + ", "
            + bathroom.getType());
    }
}

// ─── Usage Demo ────────────────────────────────
public class Main {
    public static void main(String[] args) {

        // Association — temporary use
        Teacher t = new Teacher("Mr. Sharma");
        Student s = new Student("Aditya");
        t.teach(s);  // Both exist independently

        // Aggregation — shared ownership
        Professor p1 = new Professor("Dr. Mehta");
        Professor p2 = new Professor("Dr. Khan");
        Department dept = new Department("CS", List.of(p1, p2));
        // p1, p2 still accessible even without dept

        // Composition — strong ownership
        House house = new House();
        house.describe();
        // Rooms are not accessible from outside — fully owned by House
    }
}
🎤 Interview Questions
Q: What is the key technical difference between Aggregation and Composition in code?
A: In Aggregation, the child object is passed as a parameter (constructed outside) — so it can exist independently. In Composition, the child is created inside the parent's constructor (new Room() inside House) — so it exists only within the parent's scope and is garbage collected together.
Q: IS-A vs HAS-A?
A: IS-A = Inheritance — Dog IS-A Animal (class Dog extends Animal). HAS-A = Composition/Aggregation — Car HAS-A Engine (class Car { Engine e; }). Rule: "Prefer HAS-A over IS-A" because inheritance creates tight coupling, while composition is more flexible.

Java Design
Immutable Class
Create objects whose state cannot change after construction — thread-safe by design

📜 5 Rules to Create Immutable Class

1
Declare class as final
Prevents subclassing which could override methods and break immutability
2
Make all fields private final
private prevents external access, final prevents reassignment after construction
3
No setters — only getters
Setters would allow mutation after creation. Only provide read-only access
4
Initialize all fields in constructor
All values must be set at construction time — no lazy initialization
5
Deep copy mutable objects (e.g., List, Date)
If a field is a mutable object (List, Date), return a copy — not the original reference. Otherwise the object can be mutated externally!
ImmutableEmployee.java — Production-Grade Immutable Class Java 
import java.util.*;

// Rule 1: final class — prevents subclassing
public final class ImmutableEmployee {

    // Rule 2: private final fields
    private final int          id;
    private final String       name;
    private final double       salary;
    private final List<String> skills;    // mutable! — needs deep copy
    private final Date         joiningDate; // mutable! — needs deep copy

    // Rule 4: All fields initialized in constructor
    public ImmutableEmployee(int id, String name, double salary,
                              List<String> skills, Date joiningDate) {
        this.id     = id;
        this.name   = name;
        this.salary = salary;

        // Rule 5: Deep copy of mutable objects
        this.skills      = new ArrayList<>(skills);   // defensive copy
        this.joiningDate = new Date(joiningDate.getTime()); // defensive copy
    }

    // Rule 3: Only getters — NO setters
    public int    getId()     { return id; }
    public String getName()  { return name; }
    public double getSalary() { return salary; }

    // Rule 5: Return COPY of mutable fields
    public List<String> getSkills() {
        return Collections.unmodifiableList(skills); // read-only view
    }

    public Date getJoiningDate() {
        return new Date(joiningDate.getTime()); // copy, not original
    }

    @Override
    public String toString() {
        return "Employee{id="+id+", name='"+name+"', salary="+salary+"}";
    }
}

// ─── Proof of immutability ─────────────────────
public class Demo {
    public static void main(String[] args) {
        List<String> skills = new ArrayList<>(List.of("Java", "Spring"));
        Date date = new Date();

        ImmutableEmployee emp = new ImmutableEmployee(1, "Alice", 60000, skills, date);
        System.out.println(emp.getSkills()); // [Java, Spring]

        // Try to modify original list — emp should NOT be affected
        skills.add("Python");
        System.out.println(emp.getSkills()); // [Java, Spring] — unchanged! ✅

        // Try to modify returned date
        emp.getJoiningDate().setTime(0); // modifying returned copy
        System.out.println(emp.getJoiningDate()); // original date unchanged! ✅

        // emp is completely immutable — thread-safe!
    }
}

☕ Why is String Immutable in Java?

  • Security — Class loading uses String for class names. Mutability would be a security risk
  • String Pool — Multiple references can point to same object safely
  • Thread Safety — Immutable objects can be shared without synchronization
  • Hashcode Caching — hashCode is computed once and cached (used in HashMap)
🎤 Interview Questions
Q: What is the main risk if you forget Rule 5 (deep copy)?
A: If you store a direct reference to a mutable List or Date, the caller can mutate your internal state after creation. For example: List<String> skills = new ArrayList<>(), if you store it directly — skills.add("hacked") outside would change the employee's skills! This is called "mutable reference leakage".
Q: Are Java Records immutable?
A: Java Records (Java 16+) are shallowly immutable — fields are final and only getters are generated. But if a record holds a mutable object (like List), that object can still be mutated. For deep immutability in Records, you still need defensive copies.
Q: Why is immutability important in microservices?
A: In Spring Boot, DTO objects passed between services should be immutable — this prevents accidental mutation during processing, simplifies debugging, and is naturally thread-safe in async processing pipelines.
